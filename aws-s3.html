AWS S3 CLI — from ls to put-bucket-policy

Two CLI families, one storage service. aws s3 is the friendly, high-level wrapper for everyday work — copy, sync, presign. aws s3api is the low-level mirror of the REST API, for fine-grained operations like versioning, lifecycles, multipart uploads, and bucket policies. Type your way through both.

aws s3 — high-level
File-system-feeling commands: ls, cp, mv, sync, rm, mb, rb, presign. It does smart things automatically — multi-threaded uploads, multipart for big files, retries — and is what you reach for 90% of the time.
aws s3api — low-level
One subcommand per S3 API verb: list-objects-v2, head-object, put-bucket-versioning, create-multipart-upload, … Use this when you need a feature aws s3 doesn't expose, or when you want raw JSON output to feed into --query / jq.

The S3 URI you'll type a thousand times

# scheme   bucket name      key (object path inside the bucket)
s3://my-bucket/photos/2026/cat.jpg

# a "folder" is just a key prefix — it doesn't really exist as an object
s3://my-bucket/photos/2026/
Gotcha: S3 keys are not a real filesystem. The "directory" photos/2026/ only exists because there are objects whose keys start with that prefix. Empty folders are zero-byte objects ending in /.

List & Inspect — ls, head-object, list-objects-v2, --query

First commandment of cloud storage: know what's in the bucket before you touch it. Both CLI families list objects; s3api gives you JSON you can slice with JMESPath via --query.

aws s3 ls — the friendly listing

# All buckets in the account
$ aws s3 ls
2026-04-01 10:22:13 my-bucket
2026-04-12 09:01:55 logs-bucket

# Objects at the root of one bucket
$ aws s3 ls s3://my-bucket
                           PRE photos/
2026-05-01 14:11:02     1024 readme.txt

# Recursive, human-readable, with a summary line
$ aws s3 ls s3://my-bucket/photos/ --recursive --human-readable --summarize
2026-05-01 14:11:02   2.1 MiB photos/2026/cat.jpg
2026-05-01 14:11:03   1.7 MiB photos/2026/dog.jpg

Total Objects: 2
   Total Size: 3.8 MiB
1 Recursively list everything under the photos/ prefix in my-bucket.
$ aws s3 ls s3://my-bucket/photos/ --
2 Add the flag that prints sizes in MiB / GiB instead of bytes.
$ aws s3 ls s3://my-bucket --recursive --

aws s3api list-objects-v2 — programmatic listing

# List up to 10 keys under prefix p/, project just key + size
$ aws s3api list-objects-v2 \

    --prefix p/ \
    --max-items 10 \
    --query 'Contents[].{Key:Key,Size:Size}'

# Pagination via continuation token
$ aws s3api list-objects-v2 --bucket my-bucket --max-keys 1000 \
    --starting-token "$NEXT_TOKEN"
3 Project just the Key field from every object listed.
$ aws s3api list-objects-v2 --bucket my-bucket \
    --query 'Contents[].'

JMESPath crash course — the --query flag

Every aws call returns JSON. --query 'EXPR' runs JMESPath on it client-side, before --output formats it. 

# Just the bucket names
$ aws s3api list-buckets --query 'Buckets[].Name'

# Objects bigger than 1 MiB (note the backticks for numbers)
$ aws s3api list-objects-v2 --bucket B \
    --query 'Contents[?Size > `1048576`].Key'

# Newest 5 objects
$ aws s3api list-objects-v2 --bucket B \
    --query 'reverse(sort_by(Contents, &LastModified))[:5].[Key,LastModified]' \
    --output table

# Total size of a prefix (sum)
$ aws s3api list-objects-v2 --bucket B --prefix logs/ \
    --query 'sum(Contents[].Size)'
5 Return only bucket names from list-buckets.
$ aws s3api list-buckets --query 'Buckets[].'
6 JMESPath: keep only objects strictly larger than 1 MiB. Fill in the keyword.
--query 'Contents[Size > `1048576`].Key'

Upload, Download, Sync — cp, mv, sync, exclude/include

These four verbs (cp, mv, sync, rm) handle 95% of bytes moving in or out of S3. Master --exclude / --include ordering and --dryrun before you ever type --delete.

cp — single file or recursive

# Upload
$ aws s3 cp ./report.pdf s3://my-bucket/reports/

# Download
$ aws s3 cp s3://my-bucket/reports/report.pdf ./

# Recursive — entire directory
$ aws s3 cp ./build/ s3://my-bucket/site/ --recursive

# With storage class, encryption, content-type, and metadata
$ aws s3 cp big.bin s3://my-bucket/data/big.bin \
    --storage-class STANDARD_IA \
    --sse aws:kms --sse-kms-key-id alias/data \
    --content-type application/octet-stream \
    --metadata ingest=2026-05-20,team=ml \
    --acl private
1 Upload an entire local directory to a prefix in S3.
$ aws s3 cp ./build/ s3://my-bucket/site/ --
2 Tag the upload as Infrequent-Access. Fill in the storage class.
$ aws s3 cp file.txt s3://b/k --storage-class

mv — copy + delete source

# Promote a build artefact from staging to release prefix
$ aws s3 mv s3://b/staging/v1.tar.gz s3://b/release/v1.tar.gz

# Move local → S3 (deletes the local file on success)
$ aws s3 mv ./done.zip s3://b/archive/

sync — the directory-level superpower

# Mirror local → S3, *adding* and *updating* but never deleting
$ aws s3 sync ./public/ s3://my-site/

# Mirror with deletion: S3 ends up exactly matching local
$ aws s3 sync ./public/ s3://my-site/ --delete

# Skip junk files, only ship JS and CSS
$ aws s3 sync ./dist/ s3://my-site/ \
    --exclude "*" \
    --include "*.js" \
    --include "*.css"

# See what would change without doing it
$ aws s3 sync ./public/ s3://my-site/ --delete --dryrun
Order matters: --exclude / --include are evaluated left-to-right and the last match wins. Pattern --exclude "*" first, then re-include the bits you want.
3 Mirror a local directory to S3 so deleted local files are also deleted in S3, while skipping .tmp files.
$ aws s3  local/ s3://bucket/prefix/ -- --exclude "*.tmp"
4 Preview what a destructive sync would do without performing it.
$ aws s3 sync ./public/ s3://site/ --delete --

      rm — careful, especially with --recursive

      # A single object
$ aws s3 rm s3://b/old.log

# Whole prefix — confirms nothing!
$ aws s3 rm s3://b/tmp/ --recursive

# Selective: nuke logs/ but keep .json files
$ aws s3 rm s3://b/logs/ --recursive --exclude "*.json"
      Versioning warning: aws s3 rm --recursive on a versioned bucket only adds delete markers — old versions are still there. To truly free space, you have to delete versions explicitly via s3api.

      s3api Power Tools — one verb per S3 API action

      When the friendly aws s3 commands run out, drop down to s3api. The output is always JSON — pipe it through --query or jq.

      Object-level operations

      # Upload from local file with explicit content-type and metadata
$ aws s3api put-object \
    --bucket my-bucket \
    --key data/blob.bin \
    --body ./blob.bin \
    --content-type application/octet-stream \
    --metadata sha256=abcd...,team=ml

# Download to a file
$ aws s3api get-object --bucket my-bucket --key data/blob.bin out.bin

# Server-side copy (no bytes through your laptop)
$ aws s3api copy-object \
    --bucket dest-bucket \
    --key archive/blob.bin \
    --copy-source my-bucket/data/blob.bin

# Delete one specific version
$ aws s3api delete-object \
    --bucket my-bucket --key data/blob.bin \
    --version-id 3HL4kqtJlcpXroDTDmJ+rmSpXd3dIbrHY
      1 Server-side copy a key from srcB/srcK to dstB/dstK. Fill the verb and the source flag.
      $ aws s3api  \
    --bucket dstB --key dstK \
    -- srcB/srcK

      Object tagging

      $ aws s3api put-object-tagging \

$ aws s3api get-object-tagging --bucket my-bucket --key data/x.bin

      Buckets & Policies — mb, rb, policy, CORS, lifecycle, public-access block

      Bucket-level configuration is where security and cost most often live or die. These commands set up versioning, public-access blocks, CORS, lifecycle rules, and bucket policies.

      Make & remove a bucket

      # Create. Region matters for the LocationConstraint (us-east-1 is implicit / "none")
$ aws s3 mb s3://my-new-bucket --region us-east-1

# Delete. --force also empties it (high-level only).
$ aws s3 rb s3://my-new-bucket --force

# s3api equivalents (more knobs)
$ aws s3api create-bucket --bucket my-new-bucket \
    --create-bucket-configuration LocationConstraint=eu-west-1

$ aws s3api delete-bucket --bucket my-new-bucket
      1 Make a new bucket called cat-photos in us-east-1.
      $ aws s3  s3://cat-photos --region us-east-1
      2 Delete a non-empty bucket in one shot. Pick the flag.
      $ aws s3 rb s3://cat-photos --

      Block all public access (do this first!)

      $ aws s3api put-public-access-block \
    --bucket my-bucket \
    --public-access-block-configuration \
        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

$ aws s3api get-public-access-block --bucket my-bucket
      Account-level shield: the same flags exist as aws s3control put-public-access-block --account-id … — set there too so a future bucket policy can't accidentally open a hole.

      Object Lock — write-once-read-many

      # Enabled at bucket creation only — can't be turned on later
$ aws s3api create-bucket --bucket vault \
    --object-lock-enabled-for-bucket

# Default retention: 7 years in COMPLIANCE mode (immutable for everyone)
$ aws s3api put-object-lock-configuration \
    --bucket vault \
    --object-lock-configuration \
    '{"ObjectLockEnabled":"Enabled","Rule":{"DefaultRetention":{"Mode":"COMPLIANCE","Years":7}}}'
      GOVERNANCE vs COMPLIANCE: GOVERNANCE lets a privileged IAM user override retention with s3:BypassGovernanceRetention; COMPLIANCE blocks everyone, including the root user, until the retention period ends.

      Multipart & Pre-signed URLs — for big files and shared links

      The high-level aws s3 cp uses multipart automatically for files bigger than ~8 MB. When you drive uploads programmatically, here are the s3api primitives.

      Multipart upload — the four-step dance

      ## 1. Start — get an UploadId
$ UID=$(aws s3api create-multipart-upload \
    --bucket B --key big.bin \
    --query 'UploadId' --output text)

## 2. Upload each part (5 MB minimum, except the last)
$ ETAG1=$(aws s3api upload-part \
    --bucket B --key big.bin \
    --part-number 1 --upload-id $UID \
    --body part1.bin \
    --query 'ETag' --output text)
#  …repeat for parts 2..N…

## 3. Complete — give S3 the part list and it stitches them server-side
$ aws s3api complete-multipart-upload \
    --bucket B --key big.bin --upload-id $UID \
    --multipart-upload '{"Parts":[{"ETag":"...","PartNumber":1},...]}'

## 4 (alt). Cancel — frees the storage you'd otherwise be billed for
$ aws s3api abort-multipart-upload \
    --bucket B --key big.bin --upload-id $UID

## See orphan uploads (these cost money!)
$ aws s3api list-multipart-uploads --bucket B
      Add a lifecycle rule with AbortIncompleteMultipartUpload: { DaysAfterInitiation: 7 } so failed uploads don't silently inflate your bill.
      1 Stop a stuck multipart upload to free its storage.
      $ aws s3api  \
    --bucket B --key K --upload-id $UID

      Boss Drill — End-to-End S3 Pipeline

      Pipeline: cold-storage research dataset

      Compliance says: KMS-encrypted, no public access, versioned, 365-day expiry, tiered to Glacier after 90 days. You're operating from the research profile in eu-west-1. 

      # 1. Pick the right profile + region for this whole shell session
$ export AWS_=research
$ export AWS_REGION=eu-west-1

# 2. Confirm we're really logged in as the research IAM user
$ aws sts 

# 3. Make the bucket
$ aws s3  s3://lab-genomics-2026 --region eu-west-1

        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

# 5. Turn on versioning so accidental deletes are recoverable
$ aws s3api put-bucket-versioning --bucket lab-genomics-2026 \
    --versioning-configuration Status=

# 6. Default-encrypt with KMS so any future upload that "forgets" --sse still gets encrypted
$ aws s3api put-bucket-encryption --bucket lab-genomics-2026 \
    --server-side-encryption-configuration \
    '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"","KMSMasterKeyID":"alias/lab-data"}}]}'

# 7. Upload the first batch with explicit encryption + IA storage class
$ aws s3 cp ./batch-001/ s3://lab-genomics-2026/raw/2026-05/ \
    --recursive \
    --sse aws:kms --sse-kms-key-id alias/lab-data \
    --storage-class 

# 8. Verify — list with sizes in MiB
$ aws s3 ls s3://lab-genomics-2026/raw/ --recursive -- --summarize

# 9. Generate a 24h share link for the PI to download one file
$ aws s3  s3://lab-genomics-2026/raw/2026-05/cohort-A.tsv --expires-in 86400

# 10. Install lifecycle: tier to GLACIER at 90d, expire at 365d, kill old versions at 30d
$ aws s3api put-bucket- \
    --bucket lab-genomics-2026 \

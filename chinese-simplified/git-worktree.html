01 worktree 解决的问题

你正埋头写一个特性，生产环境炸了，必须立刻修 main。没有工作树时，怎么选都很疼。

痛点 1 上下文切换的代价

你要 stash、切到 main、修复、推送、切回来、pop stash，再祈祷不冲突。在 stash pop 时遇到正在改的代码冲突，是一种独特的折磨。

痛点 2 并行工作

你的构建要 6 分钟。CI 在分支 A 跑的同时，你想在分支 B 写代码而不重建。一次检出 = 同时只能在一个分支上。

痛点 3 评审 PR

你想 git checkout pr/42 在本地跑一下 —— 但又不想毁掉当前分支上正在改的修改。

worktree 的答案

加一个钉在另一个分支上的工作目录。两者共享同一份 .git 对象库 —— fetch、ref、钩子、配置全部共享 —— 但 磁盘上的文件 各自独立。靠 cd 切，不靠 checkout。

# 之前：stash 大挪移
git stash push -m wip
git checkout main
# …修复、提交、推送…
git checkout feature/long-thing
git stash pop   # 🤞

# 之后：开一个并行的工作目录
git worktree add ../hotfix main
cd ../hotfix
# …修复、提交、推送，然后：
cd ../cro-economy-service
git worktree remove ../hotfix

02 剖面：共享的 .git、独立工作目录

工作树是一份位于主仓库目录之外、但仍由主仓库的 .git 跟踪的检出。

共享的部分

对象库（.git/objects）：commit、tree、blob。

引用：分支、标签、远程 —— 唯一真相来源。

配置与钩子：在主仓库里只定义一次。

独立的部分

工作目录：磁盘上实际检出的文件。

暂存区（index）：每个工作树有自己独立的暂存区。

HEAD：每个工作树指向自己的 commit / 分支。

元数据放在哪里

主仓库会把每个关联工作树记录在 .git/worktrees/<名字>/ 下。每个关联工作树自己有一个 .git 文件（不是目录），通过 gitdir: 指回那份元数据。

04 常见模式

三种立刻就能让你回本的工作流。

Hotfix 抛下一切

生产挂了，你正在某个特性上提交了 14 次。

git fetch origin
git worktree add -b hotfix/login \
  ../hotfix origin/main
cd ../hotfix
# 修复、提交、推送
cd -
git worktree remove ../hotfix

Review 在本地跑 PR

评审 PR #42，又不想毁掉手头还没收尾的工作。

git fetch origin pull/42/head:pr/42
git worktree add ../review pr/42
cd ../review
# 跑测试、四处看看
cd -
git worktree remove ../review

长测试 让 CI 在旁边跑

在 release/v2 上跑集成测试套件，自己继续写代码。

git worktree add ../e2e release/v2
cd ../e2e
make integration-test &
cd -
# 主仓库继续干活

05 值得记住的坑

你大概率会踩到的工作树 bug，几乎都属于下面这五种。

! add 时进入游离 HEAD

如果传的是某个 commit / tag 而不是分支，新工作树就会处于游离 HEAD。要新建分支，加 -b。

# 游离：
git worktree add ../tmp v1.4.0
# 新建分支：
git worktree add -b fix-1.4 ../tmp v1.4.0

! 分支已被检出

git 不允许一个分支同时被两个工作树检出（否则两个 HEAD 会踩同一个 ref）。要么换个分支，要么真知道自己在干嘛时用 --force。

fatal: 'feature-x' is already checked out at '/path/A'
# 强制覆盖（基本不是好主意）：
git worktree add --force ../B feature-x

! 手动 rm 留下残余元数据

如果直接 rm -rf 一个工作树目录，.git/worktrees/<名字> 还在。git worktree list 仍把它列为 可清理。跑一下 git worktree prune。

git worktree list
# 显示：../old  (prunable: gitdir is missing)
git worktree prune -v

! 子模块与按工作树的配置

子模块不会自动跨工作树初始化 —— 在每个工作树都跑一遍 git submodule update --init。一些配置（稀疏检出、钩子）只有在你用 --worktree 显式声明时才按工作树生效。

! 删错对象

如果工作树里还有未提交修改，remove 会拒绝。不要条件反射地按 --force —— 先 cd 进去看看。

· 用 lock 来保护

如果工作树位于可移动磁盘或慢吞吞的 NFS 上，用 git worktree lock 告诉 git：路径暂时不可达不代表要清掉它。

07 裸仓库 + worktree：单仓库（monorepo）小技巧

有些团队干脆不要"主检出"，把仓库当作一个装满工作树的文件夹。

初始化方式

git clone --bare git@github.com:org/repo.git repo.git
cd repo.git
git worktree add ../repo/main main
git worktree add ../repo/feature-x feature-x
git worktree add ../repo/release release/v2

目录结构：

repo.git/        # 共享对象库
repo/
  main/          # main 分支的工作树
  feature-x/     # feature-x 分支的工作树
  release/       # release/v2 的工作树

它的好处

没有所谓"主"工作树会让人迷糊 —— 每个分支都是平级目录。工具链（IDE、devcontainer、按分支的构建缓存）正好对齐到一个个文件夹。

代价：在裸仓库里跑某些 git 命令略显怪。多数人会给 cd repo/main 起个别名，干脆忘掉裸仓库。

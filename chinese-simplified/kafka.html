Kafka 速查表
⌕
v3.7+

概念

brokers 组成集群。topic 划分为多个 partition。生产者追加，消费者按 offset 读取。整盘游戏就这样。

集群剖面

生产者向某个 topic 写入消息。partitioner 决定写到哪个 partition。每个 partition 都有副本。同一消费组内的消费者瓜分 partition。

词汇表

术语含义
brokerKafka 服务器。集群 = 多个 broker。每个 broker 由一个数字 broker.id 标识。
topic命名的消息日志。创建时指定分区数和副本因子。
partition有序、仅追加的提交日志。同一 partition 内严格有序；跨 partition 不保证顺序。
offsetpartition 内单调递增的 64 位位置。消费者跟踪自己读到了哪个 offset。
leader / follower对每个 partition，一个 broker 是 leader（处理读写），其余 broker 复制。leader 失效时，follower 被提升。
ISRIn-Sync Replicas：在 replica.lag.time.max.ms 内追上的 follower。只有 ISR 成员可以成为 leader。
consumer group共享同一 group.id 的一组消费者。Kafka 任一时刻把每个 partition 分给组里某一个消费者。
rebalance当组成员变化（消费者加入/离开/挂掉）时，重新分配 partition。
__consumer_offsets内置压缩 topic，存放已提交 offset（默认 50 个 partition）。

顺序规则

仅 partition 内有序
同一 partition 严格有序；跨 partition 无顺序。同 key → 默认走同一 partition → 该 key 内有序。
# 顺序坑
# key=user-42 → 进入 partition hash(user-42) % N
# 所有 user-42 事件保持顺序。user-99 可能交错出现。
acks 与持久性
acks=all + min.insync.replicas≥2 = 单 broker 故障时不丢数据。
acks=0    # 一发即走
acks=1    # 仅等 leader
acks=all  # 等待所有 ISR
投递语义
默认至少一次。精确一次需要幂等生产者 + 事务 + read_committed 消费者。
# 精确一次配方
enable.idempotence=true
transactional.id=order-app-1
# 消费者：
isolation.level=read_committed
陷阱：事后增加 partition 数不会迁移已有数据。已有 key 留在原 partition；新 key 可能落到别处 —— 该 key 的顺序在两个 partition 上分裂一段时间。提前规划 partition 数，或使用按 key 的重分区作业。

生产者

生产者向 topic 追加消息。选择 key、ack 策略和 partitioner 占据 80% 的设计。

快速上手（kafka-python）

基础生产者
from kafka import KafkaProducer
import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",
    enable_idempotence=True,
    linger_ms=5,
    compression_type="zstd",
)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()

自定义 partitioner
默认在有 key 时使用 murmur2(key) % N；无 key 时使用 sticky partitioner。
def tenant_partitioner(key, all_parts, available):
    # 把 tenant=premium 路由到专属 partition 0..3
    if key and key.startswith(b"premium:"):
        return hash(key) % 4
    return hash(key) % len(all_parts)

KafkaProducer(partitioner=tenant_partitioner, …)

def on_delivery(err, msg):

acks vs 持久性 vs 延迟

acks持久性延迟适用场景
0无 — 一发即走最低指标，可完全替代的遥测数据。
1leader 不挂就还在低偶尔丢失可接受的热路径。

压缩与批处理

参数作用
linger.ms即使批未满也至多等这么久再发送。5–20 ms 能带来显著的吞吐提升。
batch.size每个 partition 批次的字节数（默认 16 KB）。高吞吐生产者宜 64 KB+。
compression.typenone · gzip · snappy · lz4 · zstd。zstd 通常在压缩比 + CPU 上表现最佳。
buffer.memory客户端可用于缓冲未发送消息的总字节（默认 32 MB）。

消息流

生产者 → leader → follower（复制）→ 消费者。

消费者代码

基础 poll 循环
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",
    auto_offset_reset="earliest",   # 或 "latest"
    enable_auto_commit=False,
    isolation_level="read_committed",
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # 同步提交
手动 poll + offset
仅在处理成功后提交 → 至少一次。
while True:
    batch = c.poll(timeout_ms=1000)
    for tp, msgs in batch.items():
        for m in msgs:
            handle(m)
        c.commit({tp: OffsetAndMetadata(msgs[-1].offset+1, None)})
Rebalance 监听器
在 partition 被收回前清空在途任务。
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()
        consumer.commit()
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)

c.subscribe(["orders"], listener=L())
按 offset / 时间 seek
# 按 offset
c.seek(TopicPartition("orders", 0), 12345)

# 按时间戳（毫秒，自 epoch 起）
ts = {TopicPartition("orders", 0): 1714000000000}
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():
    c.seek(tp, off.offset)
静态成员
稳定的 group.instance.id 可避免滚动重启时触发 rebalance。
KafkaConsumer(
  group_id="billing",
  group_instance_id="billing-pod-7",
  session_timeout_ms=30_000,
)
协作式 rebalance
增量 rebalance —— 只有易主的 partition 暂停，其它继续处理。
partition_assignment_strategy=[
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor"
]

提交语义

模式保证风险
enable_auto_commit=true（默认 5 秒）近似至多一次提交后、处理前崩溃 → 消息丢失。
处理后手动 commit()至少一次处理后、提交前崩溃 → 重启时重处理，需让 handler 幂等。
事务 send_offsets_to_transaction()精确一次（Kafka 内的 read-process-write）当副作用在 Kafka 外（如非幂等的 HTTP POST）时无效。
陷阱：消息处理函数耗时超过 max.poll.interval.ms（默认 5 分钟）会被踢出组 → 触发 rebalance 风暴。要么调大限制，要么减小 max.poll.records，要么把工作交给 worker pool 并使用 pause() / resume()。

Streams & ksqlDB

Kafka Streams：转换 topic 的 Java/Scala 库。ksqlDB：构建在 Streams 之上的 SQL。

Streams DSL

Word count（「Hello World」）
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))
   .groupBy((k,w) -> w)
   .count(Materialized.as("counts"))
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Stream-table 连接
用压缩 topic 的最新状态丰富事件流。
KTable<String,User> users = b.table("users");
KStream<String,Order> orders = b.stream("orders");

orders.join(users,
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
窗口聚合
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(
         Duration.ofMinutes(5), Duration.ofMinutes(1)))
      .aggregate(() -> 0L,
                 (k, v, sum) -> sum + v.amount(),
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

ksqlDB

CREATE STREAM / TABLE
CREATE STREAM orders (
  user_id VARCHAR KEY,
  sku VARCHAR,
  qty INT,
  amount DOUBLE
) WITH (kafka_topic='orders', value_format='JSON');

CREATE TABLE users (
  user_id VARCHAR PRIMARY KEY,

CSAS — 派生 stream
CREATE STREAM big_orders AS
SELECT o.user_id, u.tier, o.amount
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id
WHERE o.amount > 100
EMIT CHANGES;
CTAS — 窗口表
CREATE TABLE spend_5min AS
SELECT user_id,
       SUM(amount) AS total
FROM orders
WINDOW TUMBLING (SIZE 5 MINUTES)
GROUP BY user_id
EMIT CHANGES;
Pull 查询
-- 对表的最新状态做时点读取
SELECT total FROM spend_5min

运维与调优

生产中你真正会调的旋钮，以及把你叫醒的告警。

复制与持久性

配置默认含义与场景
replication.factor（每 topic）—生产用 3。配合 min.insync.replicas=2 可承受单 broker 损失。
min.insync.replicas1RF=3 时设为 2。配合 acks=all = 单 broker 故障不丢数据。
unclean.leader.election.enablefalse保持 false。开启会用持久性换可用性。
replica.lag.time.max.ms30000follower 落后超过此值会被踢出 ISR。
auto.create.topics.enabletrue生产环境设 false。topic 创建应作为显式步骤。

保留与存储

配置作用
retention.ms删除超过此时长的消息。默认 7 天。按 topic 设置。
retention.bytes每个 partition 日志大小上限。按 topic 设置。ms / bytes 谁先到谁生效。
segment.bytes / segment.ms每 X 字节或 X 毫秒滚动新段。段越小：删除/压缩越快、文件句柄越多。
cleanup.policy=compact日志压缩 — 永远保留每个 key 的最新值。用于状态 topic、changelog、__consumer_offsets。
cleanup.policy=delete,compact同时启用：压缩 + 按保留期丢弃旧 key。
min.cleanable.dirty.ratio多脏才触发压缩（默认 0.5）。

常见告警

症状排查方向
under-replicated partitions > 0broker 宕机、磁盘慢或网络问题。kafka-topics.sh --describe --under-replicated-partitions。
offline partitions > 0无 leader 可用。集群处于危险状态。
消费者 lag 持续增长消费者慢/卡。检查 max.poll.interval.ms、GC 暂停、下游系统。
生产者 RecordTooLargeException调大生产者 max.request.size、broker 的 message.max.bytes、消费者的 max.partition.fetch.bytes。
消费组 rebalance 风暴慢 handler、不稳定成员、GC 等。使用静态成员 + cooperative-sticky。
ISR 频繁缩水网络或磁盘问题。检查 replica.lag.time.max.ms 与 broker IO wait。

集群规模经验法则

每个 broker 的 partition 数
目标 < 4000。每个 partition 都吃文件句柄、内存和 rebalance 时间。
每个 topic 的 partition 数
≥ 你将来需要的最大并行度。前期取一个慷慨的值；事后增加会破坏按 key 顺序。
磁盘
XFS、大量顺序写。不要与突发型负载共享。预留 50%+ 给日志压缩。
网络
复制也是南北向流量。RF=3 意味着每生产 1 字节，网络上传 ~3×。
JVM
堆 6-8 GB 足够。大部分内存留给 page cache。
KRaft（无 ZooKeeper）
3.3+ 默认 KRaft。4.0 已移除 ZooKeeper。新集群只用 KRaft。

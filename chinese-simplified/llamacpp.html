llama.cpp — 用 C/C++ 在本地运行 LLM

llama.cpp 是一款面向 LLaMA 系列模型(Llama 1/2/3、Mistral、Qwen、Gemma、Phi、DeepSeek 等)的高性能推理引擎, 用纯 C/C++ 编写,运行时不依赖 Python。它支持 CPU、Apple Silicon(Metal)、NVIDIA(CUDA)、AMD(ROCm) 以及任何支持 Vulkan 的 GPU。模型以单个 .gguf 文件保存。 读完本页后,你将亲手敲过构建、量化、运行和提供服务的所有命令。

为什么选 llama.cpp?
零依赖的 C++,不到一秒就能启动。量化模型(Q4_K_M)能把 7B 模型从 ~14 GB(fp16) 缩到 ~4 GB。无论是笔记本、树莓派、M 系列 Mac 还是 24 GB GPU,代码路径完全一致。 是其他人争相模仿的参考实现。
什么是 GGUF?
GGUF(GPT-Generated Unified Format)是一种单文件二进制格式, 把权重 + 分词器 + 聊天模板 + 元数据打包在一起。可内存映射、有版本号, 是本地 LLM 事实上的交换格式,取代了更老的 GGML 格式。

GGUF 与量化

量化通过用更少比特存储权重来压缩模型。每个权重的精度更低, 但更聪明的分块方案(_K_M 系列) 能在很大程度上保住质量。拖动下面的滑块感受一下尺寸/质量的权衡。

量化类型一览

F3232.0~28 GB参考仅训练
F16 / BF1616.0~14 GB无损推理基准 / 转换步骤
Q8_08.5~7.2 GB≈ fp16质量优先
Q6_K6.6~5.5 GB极佳质量 > 大小

Q4_K_M4.85~4.1 GB良好——甜点位多数场景的默认值

Q3_K_M3.9~3.3 GB明显损失内存吃紧

llama-cli — 一次性生成与聊天

直接生成文本的前端工具。给个提示词,获得 token。加上 -cnv 进入对话模式,或通过 stdin 管道输入。 下面所有标志在 llama-server 上同样适用。

最常用的标志

标志速查表
-m .gguf 模型路径
-p 提示词字符串
-f 从文件读取提示词
-n n_predict — 最多新 token 数(-1 = ∞)
-c 上下文大小(如 4096、8192)
-b 提示词的批次大小(默认 2048)
-cnv 对话 / 聊天模式
-sys 系统提示词
-ngl 卸载到 GPU 的层数(99 = 全部)
-t CPU 线程数
--temp 采样温度
--top-k top-k 采样
--top-p 核采样
--repeat-penalty 抑制重复
--mlock 把权重锁在内存里
--seed 随机数种子(-1 = 随机)
标准生成
llama-cli \
    -m models/Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf \
    -p "Explain backprop in 2 sentences." \
    -n 256 \
    -c 4096 \
    -ngl 99 \
    --temp 0.7 --top-p 0.9
交互式聊天(自动检测 Llama-3 聊天模板)
llama-cli -m model.gguf -cnv \
    -sys "You are a terse assistant." \
    -ngl 99 -c 8192

llama-server — OpenAI 兼容的 HTTP

llama-server 通过 HTTP 暴露同一个模型, 使用 OpenAI API 的子集。能直接替代许多 SDK,内置 WebUI 位于 /。

启动服务器
llama-server \
    -m models/Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf \
    --host 0.0.0.0 --port 8080 \
    -c 8192 -ngl 99 \
    --chat-template llama3 \
    --api-key sk-local # 可选鉴权

端点(OpenAI 兼容)

chat — /v1/chat/completions
接受 { messages: [...] }。服务器会应用 GGUF 内嵌的聊天模板。通过 "stream": true 支持流式。
completion — /v1/completions
原始文本补全。{ prompt: "..." }, 完全跳过聊天模板。
embeddings — /v1/embeddings
嵌入模型用——启动时加 --embedding, 返回兼容 OpenAI 客户端的向量。
原生 — /completion、/tokenize、/health
llama.cpp 原生端点,带额外特性:语法、JSON schema、slot 管理。访问 http://localhost:8080 进入 WebUI。
curl 示例 — 聊天补全
curl http://localhost:8080/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "messages": [
            {"role": "system", "content": "You are concise."},
            {"role": "user",   "content": "What is GGUF?"}
        ],
        "temperature": 0.7,
        "max_tokens": 200
    }'
python(openai SDK)
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="sk-local",        # 任何值都行,除非设了 --api-key
)

resp = client.chat.completions.create(
    model="local",             # 会被忽略,只加载了一个模型
    messages=[
        {"role": "system", "content": "You are terse."},
        {"role": "user",   "content": "Why is the sky blue?"},
    ],
    stream=True,
)
for chunk in resp:
    print(chunk.choices[0].delta.content or "", end="", flush=True)

采样参数

模型产生 logits 后,采样器把它们筛成单个 token。默认值已经合理, 下面这些旋钮覆盖 99% 的调参需求。

温度 — --temp
把 logits 除以 T。0 = 贪心 / 确定性; 0.7 ≈ 创造性聊天; 1.0+ = 狂野。代码生成:0.0–0.3。故事:0.7–1.0。
top-k — --top-k
只保留概率最高的 K 个 token。40 是经典默认值;0 表示禁用。
top-p(核) — --top-p
保留累计概率 ≥ p 的最小 token 集合。现代默认: 0.9–0.95。 设为 1.0 来禁用。
repeat-penalty — --repeat-penalty
惩罚已经出现过的 token。1.0 = 关闭。 1.1–1.2 能治大多数循环。太高 → 文风不自然。
min-p — --min-p
丢掉低于 min_p × max_prob 的 token。 比 top-p 更新,有些用户偏好它。
seed — --seed
随机数种子。-1 = 随机。 固定一个种子可以复现运行结果。

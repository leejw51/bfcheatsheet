multi­agent.

让 Claude Code 并行工作的四种方式。
文件： multi-agent-guide.html
运行时： claude-code/cli
~/repo — claude --multi-agent

一个 Claude 窗口已是一支小队。四个窗口就是一间工作室。 本指南介绍这支队伍可以采用的四种形态 —— 以及每种形态各自适合什么场景。

按协作方式选，而非人头数。副线任务 → 子智能体。 独立任务 → agent view。需要协调 → 智能体团队。完全隔离 → 每个终端各用一个 worktree。

实时 · 可交互
i.
四种方法 · 选其一或叠加使用
方法 01 · 会话内委派

子智能体。

一个专精的工作者，运行在独立的上下文窗口里，完成聚焦的任务后， 将摘要返回给调用方会话。用它们可以让主线对话保持干净 —— 调研、验证、探索都很合适。

“重点不在多一个智能体。重点在让你的主对话保持整洁。”

它们以 Markdown 文件的形式定义在 .claude/agents/ 下。 每个都拥有自己的系统提示词、工具白名单，并可（可选地）使用更小的模型， 让工作更便宜。

上下文 · 独立 并行 · 是 文件冲突 · 共享 cwd 适合 · 调研、审计
.claude/agents/researcher.md 
---
name: Researcher
description: 查找先前研究并标注引用
model: haiku
tools:
  - bash
  - read
  - webfetch
---

# You are an expert researcher.
# Return citations as a bullet list.
提示词 · 主会话 
$ /agents
  → 管理定义

$ 让 @Researcher 挖一下限速 RFC，然后总结。
→ 在独立上下文中派生 Researcher…
← "找到 3 篇资料，详见 notes/rfc-rate-limit.md"
方法 02 · 后台仪表盘

Agent view。

一块独立后台会话的仪表盘。下发提示词，走开，回头再看输出。 每一行都是独立进程，关闭终端也能存活。

“一发即走、回头再看。当你有五件小事但只有一颗大脑时很有用。”

用 claude agents 打开。在仪表盘里： 空格偷瞄，回车挂入， ←挂出。

上下文 · 每会话独立 协调 · 由你来做 持久性 · 关闭后存活 适合 · 并行的杂活
shell 
# 派发一个后台会话
$ claude --bg "调查那个不稳定的 e2e 测试"

# 用自定义 agent 派发
$ claude --agent code-reviewer --bg "评审 PR #1234"

# 打开仪表盘
$ claude agents

┌─ 活跃 ─────────────────────────────┐
│ ▸ flaky-e2e        running   2m 14s │
│ ▸ pr-1234-review   running     38s │
│ ▸ docs-cleanup     done      11m 02s │
└────────────────────────────────────┘
方法 03 · 实验性 · 协同

智能体团队。

多个会话围绕同一目标工作，共享任务列表、智能体之间互发消息， 由 lead 统筹工作。比起工作池，它更像一间工作室。

“一个评审者管安全，一个管性能，一个管测试 —— 它们之间会对话。”

目前是实验功能，默认禁用。在 settings.json 中启用， 然后让 Claude 为你的目标拉起一支团队。

上下文 · 每名队员独立 协调 · 内置 状态 · 实验性 适合 · 多维度评审
settings.json 
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
提示词 
$ 拉起一支团队评审 PR #142。
  三位评审者并行工作：
    1. 安全     — 鉴权、输入、密钥
    2. 性能     — 热路径、内存分配、查询
    3. 测试     — 覆盖率、不稳定测试、mock
  Lead 汇总后发布单一评审。

→ 已派生 team:r142，含 3 名队员
→ 共享任务列表已创建（12 项）
← 汇总评审已发布到 #142
方法 04 · 最朴素 · 强烈推荐

多终端。

再开一个终端标签页。运行 claude。完事 —— 你就有了两个 智能体。再加上 git worktree，便能获得真正的隔离： 每个智能体都在自己的分支和工作树上，不会发生编辑冲突。

“最干净的范式，往往也是最笨的那个：多开几个标签页。”

适合任务长时间运行、彼此无关，并且你希望对每个智能体在做什么有完全可见性的场景。 完成后通过 PR 合并。

配置 · 零 隔离 · 完全（worktree） 可见性 · 全部 适合 · 并行特性开发
shell · 每个 agent 一个 worktree 
# 在主仓库中
$ git worktree add ../repo-feat-a -b feat-a
$ git worktree add ../repo-feat-b -b feat-b

# 终端 1 — 特性 A
$ cd ../repo-feat-a && claude

# 终端 2 — 特性 B
$ cd ../repo-feat-b && claude

# 稍后通过 PR 合并 ─ 无冲突
tmux · 分屏变体 
$ tmux new -s squad
  ctrl-b "       水平分屏
  ctrl-b %       垂直分屏
  ctrl-b o       在分屏间循环

PyApp — 把 Python 应用打包成单个二进制

PyApp(ofek/pyapp)是一个 Rust 运行时, 它把 Python 应用打包成一个自包含的可执行文件。你在构建时用环境变量配置它, 运行 cargo build --release, 然后弹出来的就是一个跨平台的二进制。用户机器上不需要 Python —— 至少不需要他们自己去装。

它是什么
一个小巧的 Rust 程序(~2 MB),它知道如何引导一个 Python 发行版, 并把你的项目装进去。每个目标平台编译一次,然后分发生成的二进制。
它不是什么
不是冻结器(PyInstaller)。不是转译器(Nuitka)。不是 zipapp(shiv / pex)。 它把 Python 来源交给真正的 CPython 发行版 —— 默认在首次运行时下载, 也可以选择嵌入。
它好在哪
默认二进制很小。真正的跨平台构建(Linux / macOS / Windows)。一份真实的 Python 安装,所有棱角都还在 —— 原生依赖、动态加载,通通能用。 自更新内置。

一次构建的形态

流程
# 1. 告诉 pyapp 你的项目是什么
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0
export PYAPP_EXEC_MODULE=mytool

# 2. 编译 pyapp 的 Rust 运行时
cargo install pyapp --locked --root dist

# 3. ./dist/bin/pyapp 现在是一个二进制,首次运行时它会
#    下载 CPython,pip 安装 mytool,然后执行 `python -m mytool`。
mv dist/bin/pyapp dist/bin/mytool
本页使用方法:逐节阅读,在 空格 中输入答案, 然后点 Check。Hint 会一次性显示某道练习里所有空格(再点一次隐藏)。 冲连击。一次绿色彩纸喷射就代表你完成了一节。

心智模型

Pyapp 的二进制包含:(a) Rust 启动代码,(b) 你的项目元数据,(c) 可选的内嵌 Python 发行版 和你的 wheel。首次启动时,它会在缓存目录里铺设一个私有 Python,pip 安装你的项目, 然后 exec 进去。后续运行直接跳到 Python 入口点。

工作原理 — 外面是 Rust,里面是 Python

用户双击 mytool。Rust 运行时检查缓存目录, 必要时获取 Python,首次安装项目,然后运行入口点。在下面的图示里切换两种分发模式。

按需 Python 发行版在首次启动时下载。 二进制小,只要联网一次。
内嵌 设置 PYAPP_DISTRIBUTION_EMBED=1, Python 压缩包就会被打进二进制。文件大些,但从第二次起完全离线可用。

首次运行的步骤

  1. 用户运行 ./mytool。
  2. Rust 运行时解析缓存目录 (Linux 上是 ~/.local/share/pyapp/, macOS 上是 ~/Library/Application Support/pyapp/, Windows 上是 %LOCALAPPDATA%\pyapp\)。
  3. 如果当前 (名称、版本、发行版) 三元组对应的 Python 还没装,它要么 从 PYAPP_DISTRIBUTION_SOURCE 下载, 要么解压内嵌的压缩包。
  4. 创建一个 venv(除非启用了完全隔离,这种情况下 Python 安装本身就是 venv)。
  5. pip install 你的项目(以及拉来的依赖)。
  6. 写一个标记文件。后续运行跳过第 3–5 步。
  7. exec 到配置好的入口点,替换掉 Rust 进程。

快速上手

安装 Rust(rustup.rs),设几个环境变量, 跑 cargo install。就这样 —— 不到一分钟你就有一个能用的二进制。

A) 用 cargo 安装
# 已发布到 crates.io 的项目
export PYAPP_PROJECT_NAME=httpie
export PYAPP_PROJECT_VERSION=3.2.2

cargo install pyapp --locked

# 二进制在 $CARGO_HOME/bin/pyapp
mv ~/.cargo/bin/pyapp ~/.cargo/bin/httpie
httpie --version
# → 3.2.2
B) 从源码构建
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0

git clone https://github.com/ofek/pyapp pyapp-src
cd pyapp-src
cargo build --release

# 二进制在 target/release/pyapp
mv target/release/pyapp ~/bin/mytool

交叉编译

用一份源码覆盖所有主要平台
# 一次性添加 rust 目标
rustup target add x86_64-unknown-linux-gnu
rustup target add aarch64-apple-darwin
rustup target add x86_64-pc-windows-msvc

# 然后逐个目标编译
cargo build --release --target x86_64-unknown-linux-gnu
cargo build --release --target aarch64-apple-darwin
cargo build --release --target x86_64-pc-windows-msvc

# pyapp 在运行时挑选匹配的 CPython 发行版
CI 提示:官方仓库带了一个 GitHub Actions 工作流,一次为 七个目标构建 —— Linux x64/aarch64、macOS x64/aarch64、Windows x64/x86 加 musl。直接抄吧。

配置变量

Pyapp 完全通过构建时读取的环境变量来配置。这些值会被烤进生成的二进制里。 下面是最常用的几个。

必填

变量作用示例
PYAPP_PROJECT_NAME 项目的 PyPI / wheel 名。必填。 mytool
PYAPP_PROJECT_VERSION 固定版本。除非设置了 PYAPP_PROJECT_PATH,否则必填。 1.2.3

项目来源

变量作用
PYAPP_PROJECT_PATH 本地 wheel 或 sdist 的路径,作为内嵌项,替代从 PyPI pip 安装。私有应用用这个。
PYAPP_PROJECT_DEPENDENCY_FILE 约束文件路径(pip -c),会被烤进二进制。固定传递依赖以保证可复现。
PYAPP_PIP_EXTRA_ARGS 追加到安装命令后面的额外参数,例如 --index-url https://my.repo/simple

入口点(选一个)

变量作用示例
PYAPP_EXEC_MODULE 运行 python -m <module>。 mytool
PYAPP_EXEC_SCRIPT 运行 wheel 内的脚本文件。 /path/to/run.py
PYAPP_EXEC_SPEC console-script 风格:module:func。 mytool.cli:main
PYAPP_EXEC_NOTEBOOK 通过 Jupyter 启动的 .ipynb 文件路径。 analysis.ipynb

Python 发行版

变量作用默认值
PYAPP_PYTHON_VERSION 使用的 CPython 版本。映射到一个 indygreg/python-build-standalone 发布。 最新稳定版
PYAPP_DISTRIBUTION_SOURCE 覆盖 Pyapp 下载 Python 的 URL。用于内网镜像。 github 发布
PYAPP_DISTRIBUTION_EMBED 设为 1/true 时:把 Python 烤进二进制。无需首次下载。 0
PYAPP_DISTRIBUTION_VARIANT 发行版风味:install_only(默认)或 install_only_stripped(更小)。 install_only

行为微调

变量作用
PYAPP_FULL_ISOLATION 跳过 venv 层;直接装到解压后的 Python 里。启动更快,但去重更少。
PYAPP_UPGRADE_VIRTUALENV 在装项目前,强制升级 venv 内的 pip / setuptools。
PYAPP_EXPOSE_ALL_COMMANDS 允许用户调用项目里任意一个 console script,而不仅仅是配置好的那一个。
PYAPP_ALLOW_UPDATES 设为 0 可以禁用 self update 子命令。
PYAPP_PASS_LOCATION 把二进制路径作为 argv[0] 传给 Python —— 对会自我重新 exec 的工具很有用。
布尔值很宽松:1、true、True、 TRUE、yes 都会启用一个标志。其他任何东西(包括未设置)都算"关"。

入口点 — module、script 还是 spec

Pyapp 可以用四种方式启动你的应用。挑那个最像你开发时怎么跑它的。

PYAPP_EXEC_MODULE
类似 python -m mytool。你的项目必须有 __main__.py。最地道。 
export PYAPP_EXEC_MODULE=mytool
# 包结构:mytool/__main__.py
PYAPP_EXEC_SPEC
Console-script 语法:pkg.module:function。 和你写在 pyproject.toml 里的 [project.scripts] 条目是一样的。 
export PYAPP_EXEC_SPEC=mytool.cli:main
PYAPP_EXEC_SCRIPT
一个 .py 文件的路径(相对于 wheel 根)。 运行 python script.py。老派做法。 
export PYAPP_EXEC_SCRIPT=mytool/runner.py
PYAPP_EXEC_NOTEBOOK
启动一个 Jupyter notebook。Pyapp 会把 jupyter 和你的项目装在一起。 给非程序员分发交互式工具时很有用。 
export PYAPP_EXEC_NOTEBOOK=demo.ipynb
参数转发:你传给二进制的任何参数都会原样转给 Python 解释器。 对 EXEC_MODULE 和 EXEC_SPEC 来说, ./mytool --debug -- --help 都能如你所愿地工作。

分发来源 — 内嵌还是按需

两种获取 CPython 的策略。根据用户是否在线、对二进制大小有多敏感、 以及能不能容忍一次性延迟来选。

按需(默认)
  • 二进制保持小巧(~2 MB)。
  • 首次启动从 indygreg/python-build-standalone 下载 CPython(~30 MB)。
  • 后续启动使用缓存的安装。
  • 多个 pyapp 二进制可以共享同一个缓存的解释器。
  • 首次运行需要联网。
# 不用设置 —— 这是默认
export PYAPP_PYTHON_VERSION=3.12
cargo build --release
内嵌
  • Python 压缩包烤进二进制(总共 ~30 MB)。
  • 整个流程都不需要联网。
  • 可复现:每台机器上的 Python 完全一致。
  • 项目依赖仍然在首次运行时 pip 安装,除非你也把它们嵌入。
  • 用于 CI 工具、内网环境,或者首次启动延迟无法接受的场景。
export PYAPP_PYTHON_VERSION=3.12
export PYAPP_DISTRIBUTION_EMBED=1
cargo build --release

固定到具体压缩包

覆盖 pyapp 下载 Python 的 URL
# 镜像或自带的发布包
export PYAPP_DISTRIBUTION_SOURCE=https://internal.example.com/python-3.12-x64.tar.gz
export PYAPP_DISTRIBUTION_PATH_PREFIX=install/
export PYAPP_DISTRIBUTION_PYTHON_PATH=install/bin/python3

cargo build --release
stripped 变体:设 PYAPP_DISTRIBUTION_VARIANT=install_only_stripped 可以去掉调试符号,内嵌构建能瘦身约 10 MB。

自管理 — mytool self ...

每个 Pyapp 二进制都自带一个 self 命令, 用来处理升级和卸载。无需 CI 也能给出高级用户级体验。

命令
mytool self update从 PyPI 拉取最新版本,就地重建 venv
mytool self restore毁掉 venv 重新安装(从依赖搞坏的状态恢复)
mytool self remove删除缓存目录;二进制不受影响
mytool self metadata打印烤进二进制的 Pyapp 配置
mytool self python进入内嵌的 Python REPL —— 调试时很方便
控制 self-update
# 构建时,完全禁用更新
export PYAPP_ALLOW_UPDATES=0

# 运行时,把用户指向私有索引
export PIP_INDEX_URL=https://my.repo/simple
mytool self update

# 固定到具体版本
mytool self update 1.4.7
这点为什么棒:用户可以 brew install 或 winget install 你的工具一次,然后无限次 mytool self update, 都不必再走系统包管理器。Rust 运行时永不变;变的只是缓存里的 Python 项目。

与替代品对比

Pyapp 处在一个非常拥挤的领域。下面是它和那些显而易见的竞争者的对比。

工具 路线 单一二进制 一台机器跨平台 真 CPython 自更新 取舍
PyApp Rust 加载器 → CPython 发行版 是 是(cargo) 是 是 首次下载或更大的内嵌
PyInstaller 冻结:字节码 + 库压缩到一起 是 否 — 必须按系统构建 嵌入副本 否 容易被杀软误报,二进制大
Nuitka Python → C → 原生二进制 是 否 重新编译 否 构建慢,对依赖挑剔
shiv / pex zipapp 包装 .pyz 需 Python 是 用户需自带 Python 否 其实并不自包含
Briefcase(BeeWare) 平台原生包 .app、.msi、.deb 否 嵌入 否 面向 GUI 应用,配置更多
conda-pack conda 环境的压缩包 否 否 是 否 需要激活 conda

什么时候选什么

选 PyApp 当……
你想要单一二进制,接受首次下载,需要从一个 CI 矩阵做跨平台构建, 并喜欢顺带白嫖自更新的体验。
选 PyInstaller 当……
你必须在没有网络也不能写缓存的环境运行,可以在每个目标系统上构建, 并且不介意偶尔追查找不到 import 的小毛病。
选 shiv / pex 当……
目标机已经装了 Python,你想要真正跨平台的产物,而且只需要一个单文件的 wheel 捆绑包(.pyz)。

练习 — 把命令敲出来

构建 Pyapp

1 从 crates.io 安装 pyapp,锁定 Cargo.lock 以保证可复现:
cargo install pyapp
2 从克隆下来的仓库构建。编译 release 二进制:
cargo build
3 cargo build --release 之后,二进制位于:
target/release/

必填的环境变量

4 把项目名设为 "mytool":
export PYAPP_PROJECT_NAME=
5 把项目版本固定到 1.2.3:
export PYAPP_PROJECT_VERSION=

入口点

6 用 python -m mytool.cli 作为入口。填模块名:
export PYAPP_EXEC_MODULE=
7 用 console-script 风格入口:包 mytool,函数 main:
export PYAPP_EXEC_SPEC=

Python 发行版

8 固定 Python 到 3.12:
export PYAPP_PYTHON_VERSION=
9 把 Python 发行版嵌入到二进制(无需首次下载):
export PYAPP_DISTRIBUTION_EMBED=
10 跳过 venv 层,直接装到解压后的 Python 里:
export PYAPP_FULL_ISOLATION=

自管理

11 用户想把你的工具升级到 PyPI 上的最新版:
mytool self
12 抹掉缓存的 venv 和 Python 安装(二进制保留):
mytool self
13 打印烤进二进制的 Pyapp 元数据:
mytool self

速答题

14 在 Linux 上,pyapp 默认把 Python 装到哪儿?
15 在构建时禁用终端用户的自更新:
export PYAPP_ALLOW_UPDATES=

小项目 — 端到端发布 mytool v1.0

假设你已经在 PyPI 上发布了 mytool 1.0.0。 它的 CLI 在 mytool/__main__.py。我们想要一个 单一、自包含、可离线运行的二进制,使用 Python 3.12 并启用隔离。 填写下面整套可复现配方里的空格,然后点 Build & Run 进行模拟。

补全配方

9 个空格。先按 Check All;全绿后,Build & Run 会动画展示构建过程。 

# --- 1. 告诉 pyapp 你的项目 ---
export PYAPP_PROJECT_NAME=
export PYAPP_PROJECT_VERSION=

# --- 2. 怎么启动它 ---
export PYAPP_EXEC_MODULE=

# --- 3. python 发行版 ---
export PYAPP_PYTHON_VERSION=
export PYAPP_DISTRIBUTION_EMBED=
export PYAPP_FULL_ISOLATION=

# --- 4. 锁定地编译 pyapp 的 rust 运行时 ---
cargo install pyapp 

# --- 5. 重命名并运行 ---
mv ~/.cargo/bin/pyapp ~/bin/mytool
mytool --help

# 之后,用户升级:
mytool

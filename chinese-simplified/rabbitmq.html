1. 什么是消息代理(broker)?

消息代理位于生产者和消费者之间。生产者把消息丢进代理后立刻继续做自己的事;消费者按自己的节奏稍后取走。两者无需同时在线、无需知晓彼此地址、也无需就处理速度达成一致。

为什么队列重要

  • 解耦:更改消费者无需改动生产者。
  • 缓冲:平滑流量尖峰;队列吸收突发。
  • 可靠性:消费者崩溃后,消息仍然存在。
  • 扇出:一个事件触发多个独立的反应。
心智模型:生产者是写明信片的人,代理是邮局,队列是信箱,消费者按自己时间取信。

2. AMQP 核心概念

RabbitMQ 实现了 AMQP 0-9-1。词汇不多,但每个都很精确。

  • 生产者发布消息。
  • 消费者订阅并处理消息。
  • 队列存储消息的命名缓冲区。
  • 交换机路由器;生产者发布到交换机,绝不直接发到队列。
  • 绑定把交换机和队列连起来的规则(常带一个键)。
  • 路由键生产者附在消息上的字符串;交换机据此决定投递目标。
  • 虚拟主机(vhost)分隔用户、交换机、队列的命名空间;逻辑隔离。
  • 连接一个 TCP 连接。信道连接里多路复用的虚拟连接。

整体流程一览

生产者
  └─ publish(routing_key) ─▶ 交换机
                              ├─ 绑定 A ─▶ 队列 1 ─▶ 消费者
                              └─ 绑定 B ─▶ 队列 2 ─▶ 消费者

3. 交换机类型详解

Direct

把消息路由到 路由键与绑定键完全相等 的队列。适用于:定向投递、简单工作队列、命令路由。

binding: queue.email  ⇐ key="email"
publish key="email"   → queue.email
publish key="sms"     → 丢弃(或未路由)

Fanout

忽略路由键。广播给每个绑定的队列。适用于:发布/订阅事件、缓存失效通知、"通知所有人"。

publish 任意 key → 所有绑定的队列都会收到

Topic

按模式路由。路由键以点号分隔,绑定模式可使用:

  • * 匹配 恰好一个 单词。
  • # 匹配 零个或多个 单词。
binding "orders.*.us"   匹配  orders.new.us, orders.cancel.us
                        不匹配 orders.us, orders.new.eu, orders.a.b.us
binding "orders.#"      匹配  orders, orders.new, orders.new.us
binding "*.error.*"     匹配  app.error.db, web.error.api

Headers

基于消息头而非路由键路由。匹配模式 x-match: all 要求所有头都匹配;x-match: any 至少匹配一个。

binding headers: {format: pdf, type: report, x-match: all}
header {format: pdf, type: report}     → 匹配
header {format: pdf}                    → 不匹配

什么时候用哪种

类型选择场景…
direct需要点对点投递,或对单个键的严格相等匹配。
fanout每个订阅者都需要相同的事件。理解成本最低。
topic层级化事件:region.service.severity。订阅者需要部分子集。
headers路由依赖多个不适合放在字符串键里的属性。

默认交换机

未命名的 "" 交换机是一个特殊的 direct 交换机,所有队列都隐式以队列名为绑定键。这就是为什么 basic_publish(exchange='', routing_key='hello') 会直接送到名为 hello 的队列。

死信交换机(DLX)

队列可以配置 x-dead-letter-exchange。被拒绝、过期或溢出的消息会被重新发布到 DLX,由另一条绑定把它们送到死信队列以便检查或重试。

4. 确认(Acknowledgments)

自动 ack vs 手动 ack

  • 自动 ack(auto_ack=True):消息一离开代理就被视为投递成功。快但不安全。消费者中途崩溃,消息就没了。
  • 手动 ack(auto_ack=False):消费者在处理成功后必须调用 basic_ack(delivery_tag)。崩溃则代理在重连后会 重新投递。

nack 与 reject

  • basic_nack(requeue=true) — 把消息放回队列。
  • basic_nack(requeue=false) — 丢弃,或如配置则发到 DLX。
  • basic_reject 是 nack 的单消息版本。

重投递标志

消息被重投时,代理会设置 redelivered=true。消费者应将其视为提示,谨慎处理:幂等性、去重键、消息头中的重试计数器。

5. 持久性 — 重启后留下什么

一条消息要熬过代理崩溃,必须 同时 满足三件相互独立的事:

  1. 持久化队列:queue_declare(durable=True)。否则队列本身会消失。
  2. 持久化消息:delivery_mode=2(或 PERSISTENT_TEXT_PLAIN)。否则消息只在 RAM 中。
  3. 持久化交换机(若依赖它):exchange_declare(durable=True)。
注意:持久化是尽力而为,并非每条消息同步 fsync。如果你需要在向上游确认之前确认消息已落盘,使用 publisher confirms(confirm_delivery)。

publisher confirms vs 事务

AMQP 事务存在但很慢。publisher confirms 是异步的,快几个数量级:代理用 ack 帧告诉你"我收到了"。

6. 预取与 QoS — 公平派发

默认情况下,RabbitMQ 把消息以轮询方式分发给消费者。如果一个消费者慢、另一个快,慢的会越积越多,而快的反而空闲。预取(prefetch)解决了这个问题。

channel.basic_qos(prefetch_count=1)

这告诉代理:"我没有 ack 上一条之前,别给我新的。"现在快的拉得多,慢的拉得少。任务短而均匀时(如 10–50)可设更高。

调优经验法则

  • 长时间且不均匀的任务 → prefetch=1。
  • 短小且均匀的任务 → 更高的 prefetch(减少往返开销)。
  • 用 队列深度 和 消费者利用率 来度量。

7. 集群与 quorum 队列

RabbitMQ 集群在节点间共享用户、vhost、交换机和绑定。队列默认是单主实体;主在某个节点上,可以镜像或被 quorum 替代。

经典镜像队列

已是遗留方案。把状态复制给从节点。在现代部署中已被 quorum 队列取代。

Quorum 队列

  • 构建于 Raft 共识协议之上。
  • 强一致性:多数副本确认前不会向写入方 ack。
  • 能干净地承受节点失败。重要数据的推荐默认值。
  • 权衡:延迟比经典队列高,磁盘 I/O 更多。

Streams(额外)

日志风格的追加队列,适合高吞吐分析级负载。协议不同,访问模式也不同(用偏移量,不是消费即销毁)。

8. 常见模式

工作队列(任务分发)

一个队列,多个工作者,手动 ack,prefetch=1。适用于图片缩放、邮件发送、视频编码。

producer ─▶ "" ─▶ queue:tasks ─▶ worker1
                              └▶ worker2
                              └▶ worker3

发布/订阅

fanout 交换机,每个订阅者通过自己的队列收到每条消息。

producer ─▶ fanout:events ─┬─▶ queue:audit       ─▶ logger
                            ├─▶ queue:cache_invl  ─▶ cache
                            └─▶ queue:notify      ─▶ pusher

更多模式

主题路由

层级化事件。不同队列订阅不同形状。

topic:logs
  binding "*.error.*"  ─▶ queue:alerts
  binding "db.#"       ─▶ queue:dba
  binding "#"          ─▶ queue:audit

RPC(请求/应答)

生产者设置 reply_to(临时、独占的队列)与 correlation_id。消费者把响应发布到 reply_to。生产者按 correlation_id 匹配响应。

client → publish(reply_to=Q1, correlation_id=abc)
server → consume → publish(exchange='', routing_key=Q1, correlation_id=abc)
client → 匹配 abc,完成 promise
提示:给应答队列设置 TTL,避免泄漏的客户端积累孤儿队列。

交互式模拟器

选择交换机类型、声明绑定,然后发布消息观察流向。切换消费者行为可以看到 ack/nack 与重投递的实际效果。每次成功发布都会获得 XP。

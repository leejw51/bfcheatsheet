第1章

所有权与借用

rust 的核心

Rust 中的每个值都有唯一的所有者（owner）。当所有者离开作用域时， 值会被销毁——内存被释放、文件被关闭、锁被释放。这就是 RAII，并且由编译器静态强制执行。

你可以移动所有权（move，转移它），也可以借用（borrow， 借出一个引用）。借用遵循两条规则：&T 共享引用同时可以有多个， 但 &mut T 可变引用必须独占。编译器在编译期就证明这一切。

所有权转移（move）与借用的对比
拥有的值 借用的引用 已失效

经典的 move

main.rs
fn main() {
    let s = String::from("hello");
    let t = s;                 // s 被移动到 t
    // println!("{}", s);      // ❌ 错误：使用了已被移动的值 `s`
    println!("{}", t);         // ✅
}

共享借用与可变借用

borrow.rs
fn len(s: &String) -> usize { s.len() }

fn push_bang(s: &mut String) { s.push('!'); }

fn main() {
    let mut greeting = String::from("hi");
    let n = len(&greeting);          // 共享借用，OK
    push_bang(&mut greeting);        // 独占借用，OK
    println!("{} ({} chars)", greeting, n);
}
两条规则 在任意时刻，你要么拥有一个可变引用，要么拥有任意数量的共享引用。 引用始终必须有效（指向活着的数据）。这就是借用检查器的全部内容。
第2章

生命周期（Lifetimes）

为借用的有效期取个名字

生命周期不过是给某个引用有效的代码区域起的名字。 大多数生命周期是被推断出来的（这叫做 elision，省略）。 你只在编译器无法判断返回的引用来自哪个输入时，才需要手写生命周期。

引用的生命周期不能超过它所借用的值
所有者的生命周期 'a 引用的生命周期 'b

为返回的引用标注生命周期

lifetimes.rs
// 返回的 `&str` 必须活得至少和 x、y 中较短者一样久。
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() >= y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string is long");
    let result;
    {
        let s2 = String::from("short");
        result = longest(&s1, &s2);
        println!("{}", result);   // ✅ 两者都还活着
    }
    // println!("{}", result);     // ❌ s2 已被销毁，会成为悬垂引用
}
生命周期省略规则 如果一个函数取一个引用并返回一个引用，Rust 就假定它们共享同一个生命周期。 如果输入中包含 &self，那么输出会借用 &self 的生命周期。 只有当这些默认规则不正确时，你才需要手写标注。
第3章

Trait 与泛型

把行为抽象出来

trait 是一组方法的集合，类型可以为它实现。 泛型（generic）则是按某个类型参数化的函数或类型。 二者结合，便能在没有继承的前提下提供 ad-hoc 多态： "任何能被打印的 T"、"任何 i32 的迭代器"。

一个 trait、两份 impl、一个泛型函数
traits.rs
trait Greet {
    fn hello(&self) -> String;
    fn shout(&self) -> String {                 // 默认方法
        self.hello().to_uppercase()
    }
}

struct Cat;
struct Robot { id: u32 }

impl Greet for Cat   { fn hello(&self) -> String { "meow".into() } }
impl Greet for Robot { fn hello(&self) -> String { format!("unit-{}", self.id) } }

fn welcome<T: Greet>(t: &T) {
    println!("{} ({})", t.hello(), t.shout());
}

fn main() {
    welcome(&Cat);
    welcome(&Robot { id: 7 });
}
静态分发 vs 动态分发 fn welcome<T: Greet> 会为每个具体类型生成一份代码（零开销）。 fn welcome(t: &dyn Greet) 接收一个胖指针并在运行时分发—— 略慢一些，但允许你存储异构的 Vec<Box<dyn Greet>>。
第4章

Result、Option 与 ? 运算符

把错误当作值

Rust 没有异常机制。失败被编码进类型里：Option<T> 意为 可能存在一个 T；Result<T, E> 意为 一个 T 或一个错误 E。? 运算符把它们串联起来， 在没有值时立即返回。

? 运算符：遇 Err 短路返回，遇 Ok 解包
parse.rs
use std::num::ParseIntError;

fn double_first(input: &str) -> Result<i32, ParseIntError> {
    let first = input.split(',').next().unwrap_or("0");
    let n: i32 = first.trim().parse()?;   // 遇 Err 时提前返回
    Ok(n * 2)
}

fn main() {
    match double_first("21, hello") {
        Ok(v)  => println!("got {v}"),
        Err(e) => eprintln!("bad: {e}"),
    }
}

Option，简明版

option.rs
fn first_word(s: &str) -> Option<&str> {
    s.split_whitespace().next()
}

fn main() {
    let n = first_word("hello world").map(str::len).unwrap_or(0);
    println!("{n}");                     // 5
}
第5章

模式匹配

match、if let、while let

match 是穷尽式（exhaustive）的：编译器不会让你忽略任何变体。 模式可以解构元组、结构体、枚举、切片、引用， 甚至可以用 if 条件作为守卫。

match.rs
enum Shape {
    Circle { r: f64 },
    Rect   { w: f64, h: f64 },
    Triangle(f64, f64, f64),
}

fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle { r }            => std::f64::consts::PI * r * r,
        Shape::Rect   { w, h }         => w * h,
        Shape::Triangle(a, b, c) if a + b > *c => {
            let p = (a + b + c) / 2.0;
            (p * (p - a) * (p - b) * (p - c)).sqrt()
        }
        _ => f64::NAN,                                  // 非法三角形
    }
}
if let、while let 当你只关心其中一个变体时，if let Some(x) = opt { … } 比完整的 match 更清爽。while let 会一直循环到模式不再匹配为止—— 非常适合用来弹栈。
第6章

迭代器与闭包

惰性、流畅、高效

迭代器是惰性的：在你显式索取之前什么都不会发生。 闭包会捕获其环境，并分为三种：Fn（按引用借用）、 FnMut（按可变引用借用）、FnOnce（消耗捕获项）。 编译器会自动选择最宽松的那一种。

iter.rs
fn main() {
    let nums = vec![1, 2, 3, 4, 5, 6];

    let sum_of_even_squares: i32 = nums
        .iter()
        .filter(|&&n| n % 2 == 0)
        .map(|&n| n * n)
        .sum();

    println!("{sum_of_even_squares}");   // 56

    // 闭包与捕获
    let factor = 10;
    let scale = |x: i32| x * factor;     // 借用 factor（Fn）
    println!("{}", scale(3));            // 30
}
零开销 地道的迭代器链编译出来的汇编与手写的循环完全相同。 这种链式风格本质上是免费的——有时反而更快，因为 LLVM 能证明更多的不变量。
第7章

并发

线程、Send/Sync、通道

两个标记 trait 把守着并发的大门。Send 表示一个值可以 安全地移动到其他线程。Sync 表示 &T 可以安全地在多个线程之间共享。多数类型同时实现两者； Rc<T> 和 RefCell<T> 不实现—— 编译器会阻止你写出数据竞争。

通道：生产者 → 消费者，所有权随消息一起转移
channel.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    for i in 0..4 {
        let tx = tx.clone();
        thread::spawn(move || {
            tx.send(format!("hello from worker {i}")).unwrap();
        });
    }
    drop(tx);                     // 关闭原始发送端

    while let Ok(msg) = rx.recv() {
        println!("{msg}");
    }
}

共享可变状态

mutex.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let c = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut n = c.lock().unwrap();
            *n += 1;
        }));
    }
    for h in handles { h.join().unwrap(); }
    println!("count = {}", *counter.lock().unwrap()); // 10
}
第8章

Async / Await

协作式并发

给函数加上 async 标记，会把它的函数体变成实现 Future 的状态机。调用它什么都不会做， 直到你 .await 它，或交给某个运行时（Tokio、async-std、smol）。 每个 .await 都是一个挂起点。

async_main.rs
use tokio::time::{sleep, Duration};

async fn fetch(name: &str) -> String {
    sleep(Duration::from_millis(50)).await;
    format!("hello, {name}")
}

#[tokio::main]
async fn main() {
    let (a, b) = tokio::join!(fetch("alice"), fetch("bob"));
    println!("{a} | {b}");
}
Async 是一种"颜色" 你可以从 async 中调用同步函数，但反过来如果没有运行时就不行。 早早确定一个运行时（Tokio 是默认选择）并坚持下去。
第9章

Cargo 与模块

项目布局

Cargo 集构建系统、包管理器、测试运行器和基准工具于一身。 crate 是一个编译单元，module 是 crate 内的命名空间。 文件系统结构与模块树一一对应。

tree
my_app/
├── Cargo.toml          # 清单：名称、版本、依赖
├── Cargo.lock          # 解析后的精确版本（二进制项目应提交）
└── src/
    ├── main.rs         # 二进制入口          →  crate `my_app`
    ├── lib.rs          # 库的根模块          →  crate `my_app`
    └── parser/
        ├── mod.rs      # 模块 `parser`
        └── ast.rs      # 模块 `parser::ast`
Cargo.toml
[package]
name    = "my_app"
version = "0.1.0"
edition = "2024"

[dependencies]
serde   = { version = "1", features = ["derive"] }
tokio   = { version = "1", features = ["full"] }
anyhow  = "1"

[dev-dependencies]
proptest = "1"
commands
cargo new my_app             # 生成一个二进制 crate 脚手架
cargo run                    # 编译并运行
cargo test                   # 运行单元测试 + 集成测试 + 文档测试
cargo clippy -- -D warnings  # 那个出了名严苛的 linter
cargo fmt                    # rustfmt：固执而最终的格式化器

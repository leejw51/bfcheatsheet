01 工作樹解決的問題

你正深陷某個功能開發。線上系統突然著火。你必須立刻修復 main。沒有工作樹的話,每個選項都很痛苦。

痛點 1 切換脈絡的代價

你 stash、切到 main、修復、推送、切回、pop stash,然後祈禱沒有衝突。stash pop 時遇到 WIP 合併衝突是一種特殊的折磨。

痛點 2 平行作業

你的建置要 6 分鐘。當 CI 在分支 A 上跑時,你想在分支 B 寫程式而不必重新建置。一次檢出 = 一次只能一個分支。

痛點 3 審查 PR

你想 git checkout pr/42 在本地執行 — 但你也不想毀掉目前分支上進行中的修改。

工作樹的解答

新增第二個工作目錄,固定到不同分支。兩者共用同一個 .git 物件資料庫 — fetch、ref、hook、設定 — 但磁碟上的 檔案 是獨立的。透過 cd 切換,而不是 checkout。

# 之前:stash 之舞
git stash push -m wip
git checkout main
# ...修復、提交、推送...
git checkout feature/long-thing
git stash pop   # 🤞

# 之後:平行工作目錄
git worktree add ../hotfix main
cd ../hotfix
# ...修復、提交、推送,然後:
cd ../cro-economy-service
git worktree remove ../hotfix

02 解剖:共用 .git、獨立工作目錄

工作樹是位於主倉庫目錄之外、但仍由主倉庫 .git 追蹤的檢出。

共用的部分

物件資料庫(.git/objects):commits、trees、blobs。

Refs:branches、tags、remotes — 單一可信來源。

設定與 hook:在主倉庫定義一次。

獨立的部分

工作樹:磁碟上實際檢出的檔案。

Index:每個工作樹有自己的暫存區。

HEAD:每個工作樹指向自己的 commit/分支。

中繼資料的存放位置

主倉庫將每個連結的工作樹記錄在 .git/worktrees/<name>/ 之下。每個連結的工作樹有一個 .git 檔案(不是目錄),透過 gitdir: 指回該中繼資料。

04 常見模式

三種立刻能讓打字值回票價的工作流程。

Hotfix 放下一切

線上系統當機。你正在某個功能上深入到第 14 個 commit。

git fetch origin
git worktree add -b hotfix/login \
  ../hotfix origin/main
cd ../hotfix
# 修復、提交、推送
cd -
git worktree remove ../hotfix

審查 在本地執行 PR

審查 PR #42,而不毀掉進行中的工作。

git fetch origin pull/42/head:pr/42
git worktree add ../review pr/42
cd ../review
# 跑測試、四處看看
cd -
git worktree remove ../review

長測試 旁邊跑 CI

在 release/v2 上跑整合測試套件,同時繼續寫程式。

git worktree add ../e2e release/v2
cd ../e2e
make integration-test &
cd -
# 在主倉庫繼續工作

05 值得記住的陷阱

幾乎所有你會遇到的工作樹 bug 都是這五個之一。

! add 時 HEAD 分離

如果你傳入 commit/tag 而非分支,新工作樹會處於分離 HEAD 狀態。要建立新分支,使用 -b。

# 分離:
git worktree add ../tmp v1.4.0
# 分支:
git worktree add -b fix-1.4 ../tmp v1.4.0

! 分支已被檢出

Git 拒絕檢出已在另一個工作樹中的分支(以免兩個 HEAD 同時踐踏一個 ref)。要嘛換分支,要嘛在你真的清楚的情況下使用 --force。

fatal: 'feature-x' is already checked out at '/path/A'
# 強制覆寫(很少是好主意):
git worktree add --force ../B feature-x

! 手動 rm 留下殘留中繼資料

如果你 rm -rf 一個工作樹目錄,.git/worktrees/<name> 會留著。git worktree list 仍會將它顯示為 可清理(prunable)。執行 git worktree prune。

git worktree list
# 顯示:../old  (prunable: gitdir is missing)
git worktree prune -v

! 子模組與每個工作樹的設定

子模組不會自動跨工作樹初始化 — 在每個工作樹中執行 git submodule update --init。某些設定(sparse-checkout、hook)只有當你透過 --worktree 啟用時才屬於每個工作樹獨立。

! 移除錯的那一個

若工作樹有未提交的修改,remove 會拒絕。別反射性地伸手抓 --force — 先 cd 進去檢查。

· 用 lock 來保護

若工作樹位於可移除的磁碟或慢速 NFS 掛載點上,git worktree lock 會告訴 git 不要因為路徑不可達就把它清掉。

07 Bare 倉庫 + 工作樹:單一倉庫的妙招

有些團隊完全跳過「主檢出」,將倉庫視為一個工作樹資料夾。

建置方式

git clone --bare git@github.com:org/repo.git repo.git
cd repo.git
git worktree add ../repo/main main
git worktree add ../repo/feature-x feature-x
git worktree add ../repo/release release/v2

布局:

repo.git/        # 共用物件 DB
repo/
  main/          # 在 main 上的工作樹
  feature-x/     # 在 feature-x 上的工作樹
  release/       # 在 release/v2 上的工作樹

大家為何喜歡

沒有「主要」工作樹會讓人混淆 — 每個分支都是兄弟目錄。工具(IDE、devcontainer、各分支獨立的建置快取)清楚對應到資料夾。

取捨:從 bare 倉庫內部執行某些 git 指令會感覺有點怪。多數人會 alias cd repo/main,然後忘記 bare 的存在。

Kafka 速查表
⌕
v3.7+

概念

多個 broker 組成叢集。主題切分為多個分區。生產者 append;消費者依 offset 讀取。整套機制就是這樣。

叢集解剖

生產者把記錄寫到主題。partitioner 決定要寫到哪個分區。每個分區都會被複製。同一群組裡的消費者會分擔分區。

術語

術語含義
broker一個 Kafka 伺服器。叢集 = 多個 broker。每個 broker 由一個數字 broker.id 標識。
topic有名稱的記錄日誌。建立時指定分區數與副本因子。
partition有序、僅追加(append-only)的 commit log。同一分區內記錄嚴格有序;跨分區則無順序保證。
offset分區內單調遞增的 64-bit 位置。消費者會記錄自己讀到哪個 offset。
leader / follower每個分區由一個 broker 擔任 leader(處理讀寫),其他副本則複製。leader 故障時,某個 follower 會被升級。
ISRIn-Sync Replicas:在 replica.lag.time.max.ms 內跟得上的 follower。只有 ISR 成員才有資格成為 leader。
consumer group共用同一個 group.id 的消費者集合。Kafka 會把每個分區一次只指派給群組內的一位消費者。
rebalance群組成員變動(消費者加入/離開/死亡)時重新指派分區。
__consumer_offsets內部的 compact 主題,用於儲存 committed offset(預設 50 個分區)。

順序規則

只保證每個分區內的順序
同一分區內記錄嚴格有序;跨分區則無順序保證。預設下相同 key → 同一分區 → 該 key 順序固定。
# 順序陷阱
# key=user-42 → 寫入 partition hash(user-42) % N
# 所有 user-42 事件保持有序;user-99 可能交錯。
acks 與耐久性
acks=all + min.insync.replicas≥2 = 單一 broker 故障時不會掉資料。
acks=0    # 開火即忘
acks=1    # 只等 leader
acks=all  # 等所有 ISR
傳遞語意
預設是 at-least-once。Exactly-once 需要冪等生產者 + 交易 + read_committed 消費者。
# exactly-once 配方
enable.idempotence=true
transactional.id=order-app-1
# 消費者:
isolation.level=read_committed
陷阱:事後增加分區並不會搬動已存在的資料。原有的 key 仍留在舊分區;新 key 可能落到別處 — 該 key 的順序會在兩個分區之間切割一段時間。請先把分區數規畫好,或執行能感知 key 的 re-partition 工作。

生產者

生產者把記錄追加到主題。挑 key、ack 政策、partitioner 是設計上 80% 的工作。

快速上手(kafka-python)

基本生產者
from kafka import KafkaProducer
import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",
    enable_idempotence=True,
    linger_ms=5,
    compression_type="zstd",
)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()
冪等 + 重試
重試時不會重複,且 max.in.flight ≤ 5 時不會打亂順序。
KafkaProducer(
    enable_idempotence=True,
    acks="all",
    retries=2**31-1,
    max_in_flight_requests_per_connection=5,
    delivery_timeout_ms=120_000,
)

自訂 partitioner
有 key 時預設用 murmur2(key) % N;無 key 則用 sticky partitioner。
def tenant_partitioner(key, all_parts, available):
    # 把 tenant=premium 路由到專屬分區 0..3
    if key and key.startswith(b"premium:"):
        return hash(key) % 4
    return hash(key) % len(all_parts)

KafkaProducer(partitioner=tenant_partitioner, …)
標頭 + timestamp
p.send(
  "orders", key=b"u1", value=b"...",
  headers=[("trace-id", b"abc123"),
           ("schema", b"OrderV2")],
  timestamp_ms=int(time.time()*1000),
)
非同步 + callback
def on_delivery(err, msg):
    if err:
        log.error("failed: %s", err)
    else:
        log.debug("%s[%d]@%d",
                  msg.topic(), msg.partition(), msg.offset())

p.produce("orders", value=v, on_delivery=on_delivery)
p.poll(0)  # 觸發 callback

acks vs 耐久性 vs 延遲

acks耐久性延遲適用
0無 — 開火即忘最低指標、可被取代的遙測。
1leader 不掛就活低偶爾掉資料可接受的熱路徑。
allmin.insync.replicas 個 broker 都活就活較高所有重要資料的預設值。

壓縮與批次

旋鈕效果
linger.ms即使批次未滿,也最多等這麼久才送出。5–20 ms 通常能大幅提升吞吐。
batch.size每個分區批次的位元組數(預設 16 KB)。高吞吐生產者建議 64 KB+。
compression.typenone · gzip · snappy · lz4 · zstd。zstd 通常在壓縮率和 CPU 之間最平衡。
buffer.memory客戶端緩衝未送出記錄可用的位元組總量(預設 32 MB)。

訊息流

生產者 → leader → followers(複製)→ 消費者。

消費者程式碼

基本 poll 迴圈
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",
    auto_offset_reset="earliest",   # 或 "latest"
    enable_auto_commit=False,
    isolation_level="read_committed",
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # 同步 commit
手動 poll + offset
處理成功後才 commit → at-least-once。
while True:
    batch = c.poll(timeout_ms=1000)
    for tp, msgs in batch.items():
        for m in msgs:
            handle(m)
        c.commit({tp: OffsetAndMetadata(msgs[-1].offset+1, None)})
Rebalance 監聽器
在分區被收回前,把進行中的工作 flush 完。
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()
        consumer.commit()
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)

c.subscribe(["orders"], listener=L())
Seek 到 offset / 時間
# 依 offset
c.seek(TopicPartition("orders", 0), 12345)

# 依 timestamp(自 epoch 起的毫秒)
ts = {TopicPartition("orders", 0): 1714000000000}
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():
    c.seek(tp, off.offset)
靜態成員
穩定的 group.instance.id 可避免滾動重啟引發的 rebalance。
KafkaConsumer(
  group_id="billing",
  group_instance_id="billing-pod-7",
  session_timeout_ms=30_000,
)
協作式 rebalance
增量 rebalance — 只有易主的分區會暫停被處理,其他不會。
partition_assignment_strategy=[
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor"
]

Commit 語意

模式保證風險
enable_auto_commit=true(預設 5 秒)近似 at-most-oncecommit 後、處理前 crash → 該筆被丟掉。
處理完再呼叫 commit()At-least-once處理完、commit 前 crash → 重啟後重跑。Handler 要做成冪等。
交易型 send_offsets_to_transaction()Exactly-once(Kafka 內的 read-process-write)對 Kafka 外的副作用無效(例如非冪等的 HTTP POST)。
陷阱:長時間執行的訊息處理函式若超過 max.poll.interval.ms(預設 5 分鐘)就會被踢出群組 → 引發 rebalance 風暴。請拉高上限、降低 max.poll.records,或把工作交給 worker pool 並用 pause() / resume()。

  --delete --topic orders

Streams & ksqlDB

Kafka Streams:用來轉換主題的 Java/Scala 函式庫。ksqlDB:架在 Streams 上的 SQL。

Streams DSL

字數統計(「hello world」)
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))
   .groupBy((k,w) -> w)
   .count(Materialized.as("counts"))
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Stream-table join
用 compact 主題的最新狀態,enrich 一條事件 stream。
KTable<String,User> users = b.table("users");
KStream<String,Order> orders = b.stream("orders");

orders.join(users,
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
視窗聚合
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(
         Duration.ofMinutes(5), Duration.ofMinutes(1)))
      .aggregate(() -> 0L,
                 (k, v, sum) -> sum + v.amount(),
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

ksqlDB

CREATE STREAM / TABLE
CREATE STREAM orders (
  user_id VARCHAR KEY,
  sku VARCHAR,
  qty INT,
  amount DOUBLE
) WITH (kafka_topic='orders', value_format='JSON');

CREATE TABLE users (
  user_id VARCHAR PRIMARY KEY,
  name VARCHAR, tier VARCHAR
) WITH (kafka_topic='users', value_format='JSON');
CSAS — 衍生 stream
CREATE STREAM big_orders AS
SELECT o.user_id, u.tier, o.amount
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id
WHERE o.amount > 100
EMIT CHANGES;
CTAS — 視窗 table
CREATE TABLE spend_5min AS
SELECT user_id,
       SUM(amount) AS total
FROM orders
WINDOW TUMBLING (SIZE 5 MINUTES)
GROUP BY user_id
EMIT CHANGES;
Pull query
-- 對 table 最新狀態做 point-in-time 讀取
SELECT total FROM spend_5min
WHERE user_id = 'user-42';

維運與調校

在生產環境真正會去動的旋鈕,以及那些會把你叫醒的警報。

複製與耐久性

設定預設用途與時機
replication.factor(每個主題)—生產環境 3。搭配 min.insync.replicas=2,可承受一個 broker 掛掉。
min.insync.replicas1RF=3 時設 2。搭配 acks=all = 單一 broker 故障不掉資料。
unclean.leader.election.enablefalse保持 false。打開等於用耐久性換可用性。
replica.lag.time.max.ms30000follower 慢於此值會被踢出 ISR。
auto.create.topics.enabletrue生產環境設 false。建立主題應該是顯式的步驟。

保留與儲存

設定作用
retention.ms刪除超過此時間的記錄。預設 7 天。每個主題各自設定。
retention.bytes限制每個分區 log 的大小。每個主題各自設定。ms / bytes 哪個先到就生效。
segment.bytes / segment.ms每多少 byte 或多少時間滾動一個新的 log segment。segment 越小,刪除/compaction 越快,但 file handle 越多。
cleanup.policy=compactLog compaction — 永遠保留每個 key 的最新值。用於 state topic、changelog、__consumer_offsets。
cleanup.policy=delete,compact兩者並用:compact 並在保留期過後刪除舊 key。
min.cleanable.dirty.ratio達到多髒才執行 compactor(預設 0.5)。

常見警報

症狀該檢查什麼
Under-replicated partitions > 0broker 掛、磁碟慢、或網路。kafka-topics.sh --describe --under-replicated-partitions。
Offline partitions > 0沒有可用 leader。叢集出大事了。
消費者 lag 變大消費者太慢或卡住。檢查 max.poll.interval.ms、GC 暫停、下游系統。
生產者 RecordTooLargeException調高生產者的 max.request.size + broker 的 message.max.bytes + 消費者的 max.partition.fetch.bytes。
群組 rebalance 風暴handler 太慢、成員不穩、GC。改用靜態成員 + cooperative-sticky。
ISR 經常縮小網路或磁碟問題。檢查 replica.lag.time.max.ms、broker IO wait。

叢集規模經驗法則

每個 broker 的分區數
目標 < 4000 分區/broker。每個分區會吃 file handle、記憶體與 rebalance 時間。
每個主題的分區數
≥ 你會用到的最大平行度。預先給足 — 事後增加會破壞同 key 順序。
磁碟
XFS,大型循序寫入。別跟突發 I/O 程式共用。預留 50% 以上空間給 log compaction。
網路
複製本身也是南北向流量。RF=3 等於每位元組產出大概要走 ~3× 網路。
JVM
heap 6-8 GB 就很夠。剩下的記憶體要留給 page cache。
KRaft(不需要 ZooKeeper)
3.3+ 預設 KRaft。ZooKeeper 在 4.0 移除。新建叢集只用 KRaft。

1. 什麼是訊息代理人?

訊息代理人(message broker)位在生產者與消費者之間。生產者把訊息丟進代理人後就回頭處理自己的事。消費者稍後再以自己的步調取走。雙方不需要同時上線、不必知道彼此的位址,也不必對處理速度有共識。

為什麼需要佇列

  • 解耦:更換消費者不需動到生產者。
  • 緩衝:吸收尖峰流量,佇列把突發攤平。
  • 可靠性:即使消費者當掉,訊息還在。
  • 扇出(fan-out):同一事件觸發多個獨立的後續處理。
心智模型:生產者寫明信片,代理人是郵局,佇列是各自的信箱,消費者有空再來收。

2. AMQP 核心概念

RabbitMQ 實作 AMQP 0-9-1。詞彙不多,但個個都很精確。

  • Producer 生產者 發布訊息。
  • Consumer 消費者 訂閱並處理訊息。
  • Queue 訊息佇列 用來存放訊息的具名緩衝區。
  • Exchange 交換器 路由器;生產者只能發布到交換器,絕不能直接發到佇列。
  • Binding 繫結 把交換器與佇列接起來的規則(通常會配上一個鍵)。
  • Routing Key 路由鍵 生產者附加在訊息上的字串;交換器據此決定要送到哪。
  • Virtual host 虛擬主機(vhost)切割使用者、交換器與佇列的命名空間;邏輯隔離。
  • Connection 連線 一條 TCP 連線。Channel 通道 是其上多工的虛擬連線。

整體流向

producer
  └─ publish(routing_key) ─▶ exchange
                              ├─ binding A ─▶ queue 1 ─▶ consumer
                              └─ binding B ─▶ queue 2 ─▶ consumer

3. 交換器類型深入解析

Direct(直接)

把路由鍵與繫結鍵完全相符的訊息路由出去。適用場合:點對點派發、單純工作佇列、指令路由。

binding: queue.email  ⇐ key="email"
publish key="email"   → queue.email
publish key="sms"     → 被丟棄(unrouted)

Fanout(扇出)

忽略路由鍵,對每個繫結的佇列廣播。適用場合:pub/sub 事件、快取失效、「告知所有人」的通知。

publish 任何 key → 所有繫結的佇列都會收到

Topic(主題)

依模式路由。路由鍵是用點號分隔的單字;繫結模式可使用:

  • * 匹配恰好一個單字。
  • # 匹配零個或多個單字。
binding "orders.*.us"   匹配  orders.new.us, orders.cancel.us
                        不符  orders.us, orders.new.eu, orders.a.b.us
binding "orders.#"      匹配  orders, orders.new, orders.new.us
binding "*.error.*"     匹配  app.error.db, web.error.api

Headers(標頭)

依訊息標頭而非路由鍵來路由。匹配模式 x-match: all 要求所有標頭都符合;x-match: any 只要有一個符合即可。

binding headers: {format: pdf, type: report, x-match: all}
header {format: pdf, type: report}     → 匹配
header {format: pdf}                    → 不符

什麼時候用哪一種

類型適用情境…
direct需要點對點派發、或單一鍵的嚴格相等比對。
fanout每個訂閱者都要收到同樣的事件,推理成本最低。
topic階層式事件如 region.service.severity;訂閱者只想要某個子集。
headers路由依據是多個無法塞進單一字串鍵的屬性。

預設交換器

未具名的 "" 交換器是個特殊的 direct 交換器,每個佇列都隱式地以自己的名字繫結到它。這就是為什麼 basic_publish(exchange='', routing_key='hello') 會直接送到名為 hello 的佇列。

死信交換器(DLX)

佇列可以設定 x-dead-letter-exchange。被拒收、過期、或溢位的訊息會被重新發布到 DLX,再由另一條繫結停在死信佇列裡,以供檢查或重試。

4. 確認(Acknowledgments)

自動確認 vs 手動確認

  • 自動確認(auto_ack=True):訊息一離開代理人就視為已遞送。快但不安全。消費者若在處理途中當掉,訊息就消失了。
  • 手動確認(auto_ack=False):消費者處理成功後必須呼叫 basic_ack(delivery_tag)。若先當掉了,代理人在重連後會重新派發。

nack 與 reject

  • basic_nack(requeue=true) —— 把訊息放回佇列。
  • basic_nack(requeue=false) —— 丟棄,或在有設定 DLX 時送過去。
  • basic_reject 是 nack 的單訊息版本。

重新派發旗標

當訊息被重新派發時,代理人會把 redelivered=true 設起來。消費者應把它當成提醒:小心冪等性、去重鍵、把重試次數記在標頭裡。

5. 持久性 —— 哪些東西能撐過重啟

下列三件事必須同時都有,訊息才能撐過代理人崩潰:

  1. 持久佇列:queue_declare(durable=True)。否則佇列本身就會消失。
  2. 持久訊息:delivery_mode=2(或 PERSISTENT_TEXT_PLAIN)。沒有它,訊息只在記憶體中。
  3. 持久交換器(若你依賴它):exchange_declare(durable=True)。
陷阱:持久化是盡力而為,並不會對每個訊息做同步 fsync。如果你需要在向上游回應之前確認訊息已落盤,請使用 publisher confirms(confirm_delivery)。

publisher confirms vs 交易

AMQP 的 transactions 雖然存在,但很慢。publisher confirms 是非同步的,速度快上好幾個量級:代理人會用一個 ack frame 告訴你「我收下了」。

6. Prefetch 與 QoS —— 公平派發

RabbitMQ 預設以輪詢(round-robin)派送訊息給消費者。如果一個慢、一個快,慢的會堆得滿滿,快的卻在閒置。Prefetch 解決此事。

channel.basic_qos(prefetch_count=1)

它告訴代理人:「在我把上一則 ack 掉之前,別發新的給我。」於是快的拉走比較多工作,慢的則拉得少。如果工作短而均勻,可以調高(例如 10–50)。

調校的拇指規則

  • 長且不均的工作 → prefetch=1。
  • 短且均勻的工作 → 較高的 prefetch(減少 round-trip 開銷)。
  • 用 佇列深度 與 消費者使用率 來量測。

7. 叢集與 quorum 佇列

RabbitMQ 的 cluster 在節點之間共享使用者、vhost、交換器與繫結。佇列預設是單主節點實體;主節點駐在某一節點上,可被鏡像,或改用 quorum 取代。

傳統鏡像佇列

已過時。把狀態複製給跟隨者,在現代部署中已被 quorum 佇列取代。

Quorum 佇列

  • 建立於 Raft 共識協定之上。
  • 強一致性:寫入要在多數副本都收到後才會被確認。
  • 可乾淨地撐過節點故障,重要資料的推薦預設選項。
  • 權衡:延遲比傳統佇列高、磁碟 I/O 較多。

Streams(加碼)

類似 log 的追加式佇列,適合高吞吐分析等級工作負載。協定不同、存取模式不同(用 offset、不會破壞性消費)。

8. 常見模式

工作佇列(任務分派)

一個佇列、多個工人,使用手動 ack 與 prefetch=1。適用情境:圖片縮圖、寄信、影片轉檔。

producer ─▶ "" ─▶ queue:tasks ─▶ worker1
                              └▶ worker2
                              └▶ worker3

發布/訂閱

使用 fanout 交換器,每個訂閱者都透過自己的佇列收到每一則訊息。

producer ─▶ fanout:events ─┬─▶ queue:audit       ─▶ logger
                            ├─▶ queue:cache_invl  ─▶ cache
                            └─▶ queue:notify      ─▶ pusher

更多模式

Topic 路由

階層式事件,不同佇列訂閱不同形狀。

topic:logs
  binding "*.error.*"  ─▶ queue:alerts
  binding "db.#"       ─▶ queue:dba
  binding "#"          ─▶ queue:audit

RPC(請求/回應)

生產者設定 reply_to(一個臨時、獨佔的佇列)和 correlation_id。消費者把回應發布到 reply_to。生產者用 correlation_id 比對回應。

client → publish(reply_to=Q1, correlation_id=abc)
server → consume → publish(exchange='', routing_key=Q1, correlation_id=abc)
client → 比對到 abc,resolve promise
提示:給回應佇列設個 TTL,免得洩漏的 client 累積成孤兒佇列。

互動模擬器

挑一種交換器類型、宣告繫結,然後發布訊息看看流向。切換消費者的行為,觀察 ack/nack 與重新派發如何運作。每次成功發布都會獲得 XP。

生產者 交換器 佇列 消費者 掉落 = 路由失敗(mandatory 關閉時悄悄丟失)

繫結

點選繫結即可移除。

消費者

切換每個消費者的 ack 模式(auto / manual-ack / manual-nack-requeue / manual-nack-DLX)。

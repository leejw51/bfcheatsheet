git worktree umožňuje jednomu repozitáři mít více pracovních adresářů — různé větve, všechny vyzvednuté najednou, všechny sdílí jeden .git. Zvládněte to v dojo řízeném psaním s XP, úspěchy a finálním bossem.

01 Problém, který pracovní stromy řeší

Jste hluboko ve funkci. Produkce hoří. Musíte hned opravit main. Bez pracovních stromů každá možnost bolí.

Bolest 1 Daň za přepínání kontextu

Schováte změny, přepnete na main, opravíte, pošlete, přepnete zpět, vyndáte schování a doufáte, že nic nebude kolidovat. Slučovací konflikty rozpracovaných změn při stash pop jsou zvláštní druh utrpení.

Bolest 2 Paralelní práce

Sestavení trvá 6 minut. Zatímco CI běží na větvi A, chcete psát kód na větvi B bez nového sestavování. Jeden checkout = jedna větev v jednu chvíli.

Bolest 3 Recenzování PR

Chcete git checkout pr/42 a spustit jej lokálně — ale taky nechcete spálit rozpracované změny na aktuální větvi.

Odpověď pracovních stromů

Přidejte druhý pracovní adresář připoutaný k jiné větvi. Oba sdílejí stejnou objektovou databázi .git — fetche, reference, hooky, konfiguraci — ale jejich soubory na disku jsou nezávislé. Přepínáte přes cd, ne přes checkout.

# dříve: tanec se schováním
git stash push -m wip
git checkout main
# ...oprav, commit, push...
git checkout feature/long-thing
git stash pop   # 🤞

# nyní: paralelní pracovní adresář
git worktree add ../hotfix main
cd ../hotfix
# ...oprav, commit, push, pak:
cd ../cro-economy-service
git worktree remove ../hotfix

02 Anatomie: sdílený .git, oddělené pracovní adresáře

Pracovní strom je checkout, který žije mimo adresář hlavního repozitáře, ale stále jej eviduje .git hlavního repozitáře.

Co se sdílí

Objektová databáze (.git/objects): commity, stromy, bloby.

Reference: větve, tagy, vzdálené — jeden zdroj pravdy.

Konfigurace a hooky: definované jednou v hlavním repozitáři.

Co je oddělené

Pracovní strom: skutečné soubory vyzvednuté na disku.

Index: každý pracovní strom má vlastní oblast přípravy (staging).

HEAD: každý pracovní strom ukazuje na svůj vlastní commit/větev.

Kde žijí metadata

Hlavní repozitář eviduje každý propojený pracovní strom v .git/worktrees/<jméno>/. Každý propojený pracovní strom má .git soubor (ne adresář), který odkazuje zpět na tato metadata přes gitdir:.

04 Časté vzory

Tři pracovní postupy, které se vám okamžitě vrátí.

Nouzová oprava nech všeho

Produkce je dole. Jste 14 commitů hluboko ve funkci.

git fetch origin
git worktree add -b hotfix/login \
  ../hotfix origin/main
cd ../hotfix
# oprav, commit, push
cd -
git worktree remove ../hotfix

Recenze spusťte PR lokálně

Recenzování PR #42 bez zničení rozpracované práce.

git fetch origin pull/42/head:pr/42
git worktree add ../review pr/42
cd ../review
# spustit testy, prozkoumat
cd -
git worktree remove ../review

Dlouhý test CI vedle

Spusťte integrační sadu na release/v2, zatímco dál kódujete.

git worktree add ../e2e release/v2
cd ../e2e
make integration-test &
cd -
# pokračujte v práci v hlavním repu

05 Záludnosti, které stojí za zapamatování

Skoro každý problém s pracovním stromem, na který narazíte, je jeden z těchto pěti.

! Odpojený HEAD při add

Pokud předáte commit/tag místo větve, nový pracovní strom je na odpojeném HEADu. Pro vytvoření čerstvé větve použijte -b.

# odpojený:
git worktree add ../tmp v1.4.0
# větev:
git worktree add -b fix-1.4 ../tmp v1.4.0

! Větev je již vyzvednutá

Git odmítá vyzvednout větev, která je již v jiném pracovním stromě (nechtěl by, aby dva HEADy zadupávaly jeden ref). Buď vyberte jinou větev, nebo použijte --force, pokud opravdu víte, co děláte.

fatal: 'feature-x' is already checked out at '/path/A'
# obejít (zřídka dobrý nápad):
git worktree add --force ../B feature-x

! Ruční rm zanechá zastaralá metadata

Pokud rm -rf adresář pracovního stromu, .git/worktrees/<jméno> zůstane. git worktree list jej stále zobrazuje jako prunable (k odklizení). Spusťte git worktree prune.

git worktree list
# ukáže: ../old  (prunable: gitdir is missing)
git worktree prune -v

! Submoduly a konfigurace per-worktree

Submoduly se mezi pracovními stromy automaticky neinicializují — spusťte git submodule update --init v každém. Některá konfigurace (sparse-checkout, hooky) je per-worktree pouze, pokud si to zvolíte přes --worktree.

! Odstranění toho špatného

remove odmítá, pokud má pracovní strom necommitnuté změny. Nesahejte reflexivně po --force — nejdřív cd dovnitř a zkontrolujte.

· Zamknutím chráníte

Pokud pracovní strom žije na vyměnitelném disku nebo pomalém NFS, git worktree lock řekne gitu, ať jej neprořezává jen proto, že je cesta nedostupná.

06 Pracovní strom vs schování vs více klonů

Tři způsoby, jak žonglovat s prací. Nejsou zaměnitelné.

git stashvíce klonůgit worktree
Cena nastavení okamžitá znovu klonovat, znovu fetchovat, znovu konfigurovat jeden příkaz, žádný re-fetch
Disk: objektová databáze sdílená duplikovaná pro každý klon sdílená
Reference / fetche synchronizované n/a každý musí fetchovat jeden fetch, vidí ho všechny stromy
Spustit dvě větve naráz ne ano ano
Riziko ztráty práce stash listy, konflikty při pop nízké soubory zůstanou na disku
Kdy je to správné drobná ad-hoc přepnutí (< 1 min) opravdu odlišná SDK / nástroje pro každou kopii paralelní větve, stejný toolchain

07 Bare repozitář + pracovní stromy: monorepo trik

Některé týmy zcela přeskočí „hlavní checkout“ a zacházejí s repozitářem jako se složkou pracovních stromů.

Nastavení

git clone --bare git@github.com:org/repo.git repo.git
cd repo.git
git worktree add ../repo/main main
git worktree add ../repo/feature-x feature-x
git worktree add ../repo/release release/v2

Rozvržení:

repo.git/        # sdílená objektová DB
repo/
  main/          # pracovní strom na main
  feature-x/     # pracovní strom na feature-x
  release/       # pracovní strom na release/v2

Proč to lidé mají rádi

Žádný „primární“ pracovní strom, který by někoho mátl — každá větev je sourozenecký adresář. Nástroje (IDE, devkontejnery, sestavovací cache pro každou větev) se čistě mapují na složky.

Kompromis: pár git příkazů zevnitř bare repa působí trošku divně. Většina lidí si naaliasuje cd repo/main a na to bare zapomene.

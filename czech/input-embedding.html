Jak se vstupní embedding učí

Tokenizer předá modelu seznam celých čísel jako [464, 3797, 3332]. Vstupní embedding převede každé ID na naučený, hustý vektor o velikosti $d_{model}$. Ale není navržen ručně — je to prostá matice vah, která začíná jako náhodný šum a je trénována zpětným šířením chyby, stejně jako každý jiný parametr. Tato příručka sleduje The Annotated Transformer a přesně ukazuje, jak je tabulka sestavena, naplněna a naučena.

01 Hlavní myšlenka v jedné větě

co to je

Naučitelné vyhledávání

Matice $E \in \mathbb{R}^{V \times D}$ — jeden trénovatelný řádek na každý slovníkový záznam. Embedding tokenu je prostě jeho řádek, načtený podle indexu.

jak se učí

Gradientní sestup

Řádky začínají náhodně (Xavierova inicializace). V každém tréninkovém kroku gradient ztráty posune pouze řádky, které se v batchi vyskytly. Po milionech opakování se geometrie sama uspořádá.

„Používáme naučené embeddingy." Článek je explicitní: vstupní i výstupní token embeddingy nejsou pevně dané — jsou to váhy. Neexistuje žádná samostatná fáze „trénování embeddingů"; tabulka se optimalizuje společně s attention a feed-forward vrstvami pod jednou jazykovou/překladovou ztrátou.

01·b Kde sedí

tvary · embedding přidává osu příznaků D
token ids [B, T] → embed × √D [B, T, D] + PE[:T] [T, D] → encoder [B, T, D]

$B$ = batch, $T$ = délka sekvence, $D = d_{model}$ (512 v základním modelu). Tokenizer vytvoří [B, T] ID; embedding vyhledá D-vektor pro každé ID a škáluje ho; pak poziční kódování přidá pořadí.

Modul Embeddings

Jeden trénovatelný řádek pro každý token

V Annotated Transformeru je embedding malý modul obalující nn.Embedding. Dopředný průchod je vyhledání řádku, pak škálování $\sqrt{d_{model}}$. Matice vah je jediná věc, která se zde učí.

třída Embeddings · doslovně 
class Embeddings(nn.Module):
    def __init__(self, d_model, vocab):
        super(Embeddings, self).__init__()
        self.lut = nn.Embedding(vocab, d_model)   # váhy: [V, D]
        self.d_model = d_model
    def forward(self, x):
        return self.lut(x) * math.sqrt(self.d_model)
co jednotlivé symboly znamenají
  • vocab · V  velikost slovníku — počet řádků. (První argument nn.Embedding.)
  • d_model · D  šířka embeddingu — 512 v základním Transformeru.
  • lut  trénovatelná matice vah [V, D] — doslovná „vyhledávací tabulka".
  • x  celočíselná token ID ve tvaru [B, T]; výstup má tvar [B, T, D].
Proč × √d_model? Článek říká: „ve vrstvách embeddingu násobíme tyto váhy √d_model." Tím se zvýší velikost embeddingu, aby byla srovnatelná s jednotkově škálovaným pozičním kódováním, které se přidává poté — jinak by pozice přehlušila obsah. Při D = 512 je tento faktor √512 ≈ 22,6.
Často je to největší vrstva. Počet parametrů je přesně V × D. Pro slovník o velikosti 32 tisíc tokenů a D = 512 to je ≈ 16,4 M vah — vše v jedné tabulce, vše naučeno gradientním sestupem.
Podívej se na vyhledávání

Vyhledávací laboratoř

Než se tabulky dotkne trénink, uvidíte, čím je. Každý řádek je vektor jednoho tokenu; každý sloupec je jedna dimenze; barva je hodnota. Posuňte token id — jeho řádek se rozsvítí a níže se zobrazí jako D-vektor, který proudí dál. Vyhledávání je čisté indexování: E[id], žádná aritmetika.

Čisté indexování: změna id pouze přesune zvýraznění na jiný řádek — matice se nikdy nepohne. Trénink je opačný pohyb: id zůstává a hodnoty řádku se mění. To je záložka 04.
Srdce · jak se řádky učí

Trénování tabulky

Embedding nemá žádný speciální trénovací postup — jede v normální smyčce modelu. Batch projde dopředně ke ztrátě; loss.backward() pošle gradienty zpět do každé váhy, včetně řádků embeddingu, které byly vyhledány; optimizer provede krok. Klíčové je, že gradient dostávají pouze řádky, které se v batchi vyskytly — aktualizace je řídká.

04·b Proč je aktualizace řídká

Protože vyhledávání se rovná násobení one-hot vektorem, $e_{id} = \mathbf{1}_{id}^{\top} E$, gradient vzhledem k tabulce je všude nulový, kromě řádku id. Každý vyhledaný token dostane příchozí gradient uložený na svůj vlastní řádek, sečtený přes všechny výskyty v batchi.

$$\frac{\partial L}{\partial E_{j}} \;=\; \sum_{t\,:\,ids_t = j} \frac{\partial L}{\partial e_{t}} \qquad\Longrightarrow\qquad \frac{\partial L}{\partial E_{j}} = 0 \;\text{ pro nepoužité řádky } j$$
sandbox řídké aktualizace · pouze řádky batche se pohnou

Tento batch je jedna krátká věta. Její tokeny (s řádkovými ID) jsou jediné řádky, které dostávají gradient. Opakovaně klikejte na trénovací krok: zvýrazněné řádky viditelně driftují, zatímco zmrazené řádky se nikdy nemění — to je řídký gradient embeddingu v akci.

−1+1 orámované = v batchi (dostávají ∇) · ztmavené = zmrazené
krok 0
řádky dotčené v tomto běhu: 0 z 16
Skutečný trénink toto dělá pro miliony kroků na obrovském korpusu. Frekventované tokeny (jako the) se aktualizují neustále; vzácné tokeny se pohybují pomalu, protože se v batchi zřídka vyskytují.

04·c Ztráta & optimizer, které to pohánějí

label-smoothed ztráta · backward · krok 
# cílové rozdělení je vyhlazené: jistota na správném tokenu, ε rozložené na zbytek
criterion = LabelSmoothing(size=V, padding_idx=0, smoothing=0.1)   # uvnitř KLDivLoss

def step(out, targets, norm):
    log_probs = generator(out)                 # Linear → log_softmax  [.., V]
    loss = criterion(log_probs, targets) / norm
    loss.backward()                          # ∇ teče do použitých řádků E
    opt.step()                                # Adam aktualizace s Noam rozvrhem
    opt.optimizer.zero_grad()                 # vymazat gradienty pro další batch
    return loss

Inicializace je důležitá. Před krokem 1 je každá váha s více než jednou dimenzí — včetně embedding tabulky — nastavena pomocí nn.init.xavier_uniform(p). Řádky tedy začínají jako malé náhodné vektory; veškerá struktura, kterou uvidíte v záložce 05, je naučená z tohoto základu.
Co trénink vyprodukuje

Význam se stává geometrií

Po dostatečném počtu kroků byly tokeny se stejnou rolí posunuty k podobným řádkům — protože dobré predikce to odměňují. Kosinová podobnost mezi dvěma řádky nyní měří, jak moc si model myslí, že jsou si podobné, a směry získávají smysl (slavné king − man + woman ≈ queen). Zvolte query token a sledujte, jak se rozsvítí jeho sousedé.

natrénovaný hračkový embedding prostor · cos-sim k query tokenu

Každá tečka je vektor tokenu (promítnutý do 2-D). Zvolené query svítí; čáry sahají k ostatním, čím podobnější, tím jasnější. Seřazený seznam je kosinová podobnost, nejvyšší první.

query tokenking
posuňte pro výběr slova
Slova ve stejném klastru skórují blízko +1; nesouvisející klastry klesají k 0 nebo jdou do záporných hodnot. Nic z toho není naprogramováno ručně — tabulka sem dospěje čistě z predikční ztráty záložky 04.
Netrénované = šum. Při inicializaci xavier_uniform jsou řádky náhodné, takže tato mapa by byla beztvarý oblak. word2vec a GloVe se naučily takové prostory se specializovanými cíli; embedding Transformeru dosahuje stejného druhu struktury jako vedlejší produkt predikce dalšího tokenu.
Jedna matice, tři úlohy

Sdílené váhy

Článek jde dál: vstupní embedding, výstupní (cílový) embedding a pre-softmax lineární vrstva, která produkuje logity, všechny sdílejí stejnou matici vah. Trénování jedné trénuje všechny tři — méně parametrů a vstupní/výstupní geometrie zůstává konzistentní.

Doslova: „Sdílíme stejnou matici vah mezi oběma vrstvami embeddingů a pre-softmax lineární transformací." Logity jsou pak h @ E.T — doslova „jak moc tento skrytý stav ukazuje ve směru řádku každého tokenu?"
výstupní strana · Generator 
class Generator(nn.Module):
    # "Definuj standardní krok generování s linear + softmax."
    def __init__(self, d_model, vocab):
        super(Generator, self).__init__()
        self.proj = nn.Linear(d_model, vocab)   # váhy: [V, D] — sdílené s E
    def forward(self, x):
        return F.log_softmax(self.proj(x), dim=-1)

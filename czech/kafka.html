Koncepty

Brokeři tvoří cluster. Topiky se dělí na partitions. Producenti přidávají; konzumenti čtou podle offsetu. To je celá hra.

Anatomie clusteru

Producent zapisuje záznamy do topiku. Partitioner vybírá, do které partition. Každá partition se replikuje. Konzumenti ve skupině si partitions mezi sebou rozdělí.

Slovník

TermínCo znamená
brokerKafka server. Cluster = mnoho brokerů. Každý je identifikován číselným broker.id.
topicPojmenovaný log záznamů. Vytváří se s počtem partitions a replication faktorem.
partitionUspořádaný, append-only commit log. Záznamy v jedné partition jsou striktně uspořádané. Mezi partitions: žádná záruka pořadí.
offsetMonotónní 64bitová pozice uvnitř partition. Konzumenti sledují, kterou pozici přečetli.
leader / followerPro každou partition je jeden broker leader (řeší čtení i zápis), ostatní replikují. Při výpadku leadera je povýšen follower.
ISRIn-Sync Replicas: followers, kteří jsou dorovnaní v rámci replica.lag.time.max.ms. Pouze členové ISR mohou být povýšeni na leadera.
consumer groupSkupina konzumentů sdílejících group.id. Kafka přiřadí každou partition jednomu konzumentovi ze skupiny.
rebalancePřerozdělení partitions, když se změní členství skupiny (konzument přijde/odejde/spadne).
__consumer_offsetsInterní compacted topic, kde žijí potvrzené offsety (50 partitions defaultně).

Pravidla pořadí

Pořadí pouze v rámci partition
Záznamy v jedné partition jsou striktně uspořádané. Mezi partitions: žádné pořadí. Stejný klíč → stejná partition defaultně → uspořádané pro daný klíč.
# Past pořadí
# key=user-42 → jde do partition hash(user-42) % N
# Všechny user-42 události zůstanou uspořádány. user-99 se může proložit.
acks a trvanlivost
acks=all + min.insync.replicas≥2 = žádná ztráta dat při výpadku jednoho brokera.
acks=0    # pošli a zapomeň
acks=1    # čekej jen na leadera
acks=all  # čekej na všechny ISR
Sémantika doručení
At-least-once je default. Exactly-once vyžaduje idempotentní producenty + transakce + read_committed konzumenty.
# recept exactly-once
enable.idempotence=true
transactional.id=order-app-1
# konzument:
isolation.level=read_committed
Past: pozdější zvýšení počtu partitions nepřesune existující data. Existující klíče zůstanou ve svých starých partitions; nové klíče mohou skončit jinde — pořadí pro daný klíč je chvíli rozdělené přes dvě partitions. Plánujte počet partitions dopředu nebo použijte job pro přepartitionování s ohledem na klíč.

Producenti

Producenti přidávají záznamy do topiku. Volba klíče, ack policy a partitioneru je 80 % designu.

Rychlý start (kafka-python)

Základní producent
from kafka import KafkaProducer
import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",
    enable_idempotence=True,
    linger_ms=5,
    compression_type="zstd",
)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()
Idempotentní + retries
Žádné duplicity při retry, žádné přeházení pořadí, když max.in.flight ≤ 5.
KafkaProducer(
    enable_idempotence=True,
    acks="all",
    retries=2**31-1,
    max_in_flight_requests_per_connection=5,
    delivery_timeout_ms=120_000,
)
Transakční producent
Atomický zápis přes více topiků a partitions. Používá se pro read-process-write aplikace a exactly-once.
# confluent-kafka styl
from confluent_kafka import Producer
p = Producer({
  "bootstrap.servers": "broker1:9092",
  "transactional.id": "order-app-1",
  "enable.idempotence": True,
})
p.init_transactions()
p.begin_transaction()
p.produce("orders", key="u1", value="...")
p.produce("audit", key="u1", value="...")
p.commit_transaction()
Vlastní partitioner
Default používá murmur2(key) % N, když je klíč nastaven; sticky partitioner při prázdném klíči.
def tenant_partitioner(key, all_parts, available):
    # směruj tenant=premium na vyhrazené partitions 0..3
    if key and key.startswith(b"premium:"):
        return hash(key) % 4
    return hash(key) % len(all_parts)

KafkaProducer(partitioner=tenant_partitioner, …)
Hlavičky a timestampy
p.send(
  "orders", key=b"u1", value=b"...",
  headers=[("trace-id", b"abc123"),
           ("schema", b"OrderV2")],
  timestamp_ms=int(time.time()*1000),
)
Async + callback
def on_delivery(err, msg):
    if err:
        log.error("failed: %s", err)
    else:
        log.debug("%s[%d]@%d",
                  msg.topic(), msg.partition(), msg.offset())

p.produce("orders", value=v, on_delivery=on_delivery)
p.poll(0)  # spustí callbacky

Komprese a batching

KnoflíkEfekt
linger.msPočká takto dlouho před odesláním batche, i když není plný. 5–20 ms dává velké zisky propustnosti.
batch.sizeBajtů na partition batch (default 16 KB). 64 KB+ pomáhá pro vysokopropustné producenty.
compression.typenone · gzip · snappy · lz4 · zstd. zstd obvykle nejlepší pro poměr a CPU.
buffer.memoryCelkem bajtů, které klient může použít pro buffer neodeslaných záznamů (default 32 MB).

Tok zpráv

Producent → leader → followers (replikace) → konzument.

Kód konzumenta

Základní poll loop
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",
    auto_offset_reset="earliest",   # nebo "latest"
    enable_auto_commit=False,
    isolation_level="read_committed",
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # synchronní commit
Manuální poll + offsety
Commit jen po úspěšném zpracování → at-least-once.
while True:
    batch = c.poll(timeout_ms=1000)
    for tp, msgs in batch.items():
        for m in msgs:
            handle(m)
        c.commit({tp: OffsetAndMetadata(msgs[-1].offset+1, None)})
Rebalance listener
Vyprázdněte rozpracovanou práci, než budou partitions odebrány.
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()
        consumer.commit()
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)

c.subscribe(["orders"], listener=L())
Seek na offset / čas
# podle offsetu
c.seek(TopicPartition("orders", 0), 12345)

# podle timestampu (ms od epoch)
ts = {TopicPartition("orders", 0): 1714000000000}
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():
    c.seek(tp, off.offset)
Statické členství
Stabilní group.instance.id se vyhne rebalanci při rolling restartu.
KafkaConsumer(
  group_id="billing",
  group_instance_id="billing-pod-7",
  session_timeout_ms=30_000,
)
Cooperative rebalance
Inkrementální rebalance — pozastaví se zpracování jen těch partitions, které mění majitele, ne všech.
partition_assignment_strategy=[
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor"
]

Sémantika commitu

RežimZárukaRiziko
enable_auto_commit=true (default 5 s)At-most-once-ishPád po commitu, před zpracováním → záznam ztracen.
Manuální commit() po zpracováníAt-least-oncePád po zpracování, před commitem → reprocess po startu. Udělejte handlery idempotentní.
Transakční send_offsets_to_transaction()Exactly-once (read-process-write uvnitř Kafky)Nepomůže, když je vedlejší efekt mimo Kafku (např. neidempotentní HTTP POST).
Past: dlouho běžící handler zprávy, který překročí max.poll.interval.ms (default 5 min), je vyhozen ze skupiny → bouře rebalance. Buď zvyšte limit, snižte max.poll.records, nebo přesuňte práci do worker poolu a používejte pause() / resume().

CLI

Shell skripty v $KAFKA_HOME/bin/ jsou váš přítel v produkci. Tahák níže.

Topiky

Vytvořit / vypsat / popsat

kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 12 --replication-factor 3 \
  --config retention.ms=604800000 \
  --config min.insync.replicas=2

# list / describe
kafka-topics.sh --bootstrap-server localhost:9092 --list
kafka-topics.sh --bootstrap-server localhost:9092 --describe --topic orders

kafka-topics.sh --bootstrap-server localhost:9092 \
  --alter --topic orders --partitions 24

# změnit konfiguraci
kafka-configs.sh --bootstrap-server localhost:9092 \
  --entity-type topics --entity-name orders \
  --alter --add-config retention.ms=86400000

# smazat
kafka-topics.sh --bootstrap-server localhost:9092 \
  --delete --topic orders

Streams a ksqlDB

Kafka Streams: Java/Scala knihovna pro transformaci topiků. ksqlDB: SQL nad Streams.

Streams DSL

Word count („hello world“)
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))
   .groupBy((k,w) -> w)
   .count(Materialized.as("counts"))
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Stream-table join
Obohaťte tok událostí o nejnovější stav z compacted topiku.
KTable<String,User> users = b.table("users");
KStream<String,Order> orders = b.stream("orders");

orders.join(users,
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
Okenní agregace
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(
         Duration.ofMinutes(5), Duration.ofMinutes(1)))
      .aggregate(() -> 0L,
                 (k, v, sum) -> sum + v.amount(),
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

ksqlDB

CREATE STREAM / TABLE
CREATE STREAM orders (
  user_id VARCHAR KEY,
  sku VARCHAR,
  qty INT,
  amount DOUBLE
) WITH (kafka_topic='orders', value_format='JSON');

CREATE TABLE users (
  user_id VARCHAR PRIMARY KEY,

CSAS — odvozený stream
CREATE STREAM big_orders AS
SELECT o.user_id, u.tier, o.amount
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id
WHERE o.amount > 100
EMIT CHANGES;
CTAS — okenní tabulka
CREATE TABLE spend_5min AS
SELECT user_id,
       SUM(amount) AS total
FROM orders
WINDOW TUMBLING (SIZE 5 MINUTES)
GROUP BY user_id
EMIT CHANGES;
Pull dotaz
-- bodové čtení nejnovějšího stavu tabulky
SELECT total FROM spend_5min
WHERE user_id = 'user-42';

Provoz a ladění

Knoflíky, které v produkci skutečně pootočíte, plus alarmy, které vás vzbudí.

Replikace a trvanlivost

KonfiguraceDefaultCo a kdy
replication.factor (na topic)—3 v produkci. Přežije ztrátu jednoho brokera s min.insync.replicas=2.
min.insync.replicas1Nastavte 2 při RF=3. V kombinaci s acks=all = žádná ztráta dat při výpadku jednoho brokera.
unclean.leader.election.enablefalseNechte false. Povolení vyměňuje trvanlivost za dostupnost.
replica.lag.time.max.ms30000Followers pomalejší než toto vypadnou z ISR.
auto.create.topics.enabletrueV produkci nastavte false. Vytváření topiků udělejte explicitním krokem.

Retence a úložiště

KonfiguraceCo dělá
retention.msSmaže záznamy starší než toto. Default 7 dní. Per-topic.
retention.bytesStrop velikosti logu na partition. Per-topic. Co (ms / bytes) nastane dříve, vyhrává.
segment.bytes / segment.msOtočit nový log segment každých X bajtů nebo X ms. Menší segmenty = rychlejší mazání/komprimace, více file handles.
cleanup.policy=compactLog compaction — uchovat nejnovější hodnotu na klíč navždy. Použijte pro state topiky, changelogy, __consumer_offsets.
cleanup.policy=delete,compactObojí: compact + zahodit staré klíče po retention.
min.cleanable.dirty.ratioJak špinavý log před spuštěním compactoru (default 0,5).

Časté alarmy

SymptomCo zkontrolovat
Under-replicated partitions > 0Broker padl, pomalý disk nebo síť. kafka-topics.sh --describe --under-replicated-partitions.
Offline partitions > 0Není leader. Cluster v problémech.
Roste consumer lagPomalý/zaseklý konzument. Zkontrolujte max.poll.interval.ms, GC pauzy, downstream systém.
Producent RecordTooLargeExceptionZvyšte max.request.size u producenta + message.max.bytes u brokera + max.partition.fetch.bytes u konzumenta.
Bouře rebalance skupinyPomalí handleři, nestabilní členové, GC. Použijte statické členství + cooperative-sticky.
ISR se často zmenšujeSíť nebo problém s diskem. Zkontrolujte replica.lag.time.max.ms, IO wait brokera.

Pravidla pro dimenzování clusteru

Partitions na brokera
Mířit na < 4000 partitions na brokera. Každá partition stojí file handles, paměť a čas rebalance.
Počet partitions na topic
≥ maximální paralelismus, jaký kdy budete potřebovat. Zvolte štědré číslo dopředu — pozdější zvyšování rozbije pořadí na klíč.
Disky
XFS, velké sekvenční zápisy. Nesdílejte s ničím nárazovým. Plánujte 50 %+ volných pro log compaction.
Síť
Replikace je také north-south provoz. RF=3 znamená, že každý vyprodukovaný bajt projde sítí ~3×.
JVM
6–8 GB heap stačí. Většina paměti by měla jít do page cache.
KRaft (bez ZooKeeperu)
3.3+ defaultně KRaft. ZooKeeper odstraněn ve 4.0. Nové clustery: jen KRaft.

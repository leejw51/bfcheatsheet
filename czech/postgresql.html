PostgreSQL CLI — drilujte psql, dokud nezůstane

psql is the official terminal client for PostgreSQL. It's shipped with every server, scriptable, scriptedly-fast, and packed with backslash meta-commands that beat any GUI. By the end of this page you will have typed the connection flags, the meta-commands, the SQL, the dump/restore, the maintenance one-liners, and a tiny end-to-end project.

What is psql?
A terminal program that speaks the PostgreSQL wire protocol. It runs your SQL but also backslash commands like \dt (list tables) and \d users (describe a table). Tab completion, history, scripting via -c / -f, and bulk loading via \copy.
Why not a GUI?
Because GUIs hide the actual command, can't be scripted, won't run on a remote box over SSH, and break on big result sets. Once \d, \dt+, and \watch live in your fingers you'll be faster than any UI on Earth.
Friends of psql
pg_dump / pg_restore for backups, pg_isready for health checks, createdb / dropdb thin wrappers, vacuumdb, reindexdb. All speak the same connection flags.
Conventions
Default port 5432, default DB and user usually match your OS user. Inside psql: \? lists meta-commands, \h SELECT shows SQL syntax, \q quits.

The psql cheat-sheet you'll memorize

essentials
# connect
psql -h db.example.com -p 5432 -U app -d shop

# inside psql
\l            # list databases
\c shop       # connect to db "shop"
\dt           # list tables
\d users      # describe table "users"
\df           # list functions
\du           # list roles
\timing on    # time every query
\watch 2      # re-run last query every 2s
\copy users TO 'users.csv' CSV HEADER
\q            # quit

Připojení — vstup do psql

Six ways to connect. Pick the one that matches your situation.

all connection styles
# 1. defaults — uses $PGUSER / $PGDATABASE / unix socket
psql

# 2. explicit flags
psql -h db.example.com -p 5432 -U alice -d shop

# 3. URI form (great for env vars / CI)
psql postgresql://alice:secret@db.example.com:5432/shop

# 4. one-shot query, no interactive shell
psql -h host -U alice -d shop -c "SELECT now();"

# 5. run an .sql script
psql -h host -U alice -d shop -f migrate.sql

# 6. with environment variables
PGHOST=db PGUSER=alice PGDATABASE=shop psql

Authentication without typing your password every time

~/.pgpass
# ~/.pgpass — one line per (host, port, db, user, password)
# wildcards allowed for any field
db.example.com:5432:shop:alice:s3cret
*:*:*:postgres:rootpw

# MUST be chmod 600 or psql ignores it
chmod 600 ~/.pgpass
Set your own password from inside psql with \password. It hashes client-side so the plaintext never leaves the terminal — much safer than ALTER USER ... PASSWORD '...' which would land in the server log.

Type the connection flags

1 Connect to host db.example.com, port 5432, as user alice, to database shop. Fill the host flag.
psql  db.example.com -p 5432 -U alice -d shop
2 Same connection, fill the user flag.
psql -h db.example.com -p 5432  alice -d shop
3 URI form — type the default Postgres port number.
psql postgresql://user:pass@host:/dbname
4 Quit psql.
postgres=# \
5 Securely set your own password from inside psql.
postgres=# \
6 What permission must ~/.pgpass have so psql will read it? (octal)
chmod  ~/.pgpass
7 Show built-in help for SQL syntax of a statement.
postgres=# \ SELECT
Keep typing: \? lists every meta-command. \h lists every SQL statement. They're your two best friends.

Cheat karta meta-příkazů

Backslash meta-commands are special — they aren't sent to the server, psql interprets them locally. Add a + for verbose, prefix with \d to describe.

Listing things

CommandDescription

\dnlist schemas
\dvlist views
\dilist indexes
\dflist functions
\dulist roles / users
\dxlist installed extensions

CommandDescription

Inspekce — schéma, typy, indexy, plány

Once connected, the first thing you do is look around. Here's how to know what's in your database.

SQL DDL — types you'll actually use

CREATE TABLE — the rich types
CREATE TABLE users (
    id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    email       TEXT        NOT NULL UNIQUE,
    age         INT         CHECK (age >= 0),
    tags        TEXT[],                                -- ARRAY
    profile     JSONB       NOT NULL DEFAULT '{}',    -- JSONB
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- foreign key
CREATE TABLE orders (
    id       BIGSERIAL PRIMARY KEY,
    user_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    total    NUMERIC(10,2) NOT NULL
);

Indexes — three flavours that cover most jobs

indexing patterns
-- 1. plain B-tree (the default)
CREATE INDEX idx_orders_user ON orders(user_id);

-- 2. unique constraint via index
CREATE UNIQUE INDEX idx_users_email ON users(lower(email));

-- 3. partial — only index "live" rows
CREATE INDEX idx_open_orders ON orders(user_id)
    WHERE status = 'open';

-- 4. GIN for JSONB / arrays / full text
CREATE INDEX idx_users_profile ON users USING GIN (profile);
CREATE INDEX idx_users_tags    ON users USING GIN (tags);

EXPLAIN — read your query plan

explain analyze
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE user_id = '...';
--  Index Scan using idx_orders_user on orders  (cost=0.42..8.44 rows=1 width=...)
--    Index Cond: (user_id = '...'::uuid)
--  Planning Time: 0.12 ms
--  Execution Time: 0.04 ms
Reading EXPLAIN: top-down execution, bottom-up cost. Seq Scan on a big table where you expected an Index Scan means missing/unused index. Use the BUFFERS option to see disk reads vs cache hits.

UPSERT, CTEs, window functions — the good stuff

upsert (INSERT … ON CONFLICT)
INSERT INTO users (id, email)
VALUES ('a-uuid', 'a@x')
ON CONFLICT (id) DO UPDATE SET email = EXCLUDED.email;
CTE + window function
WITH ranked AS (
    SELECT user_id, total,
           ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total DESC) AS rn
    FROM orders
)
SELECT * FROM ranked WHERE rn = 1;   -- top order per user

Užitečné jednorázovky

The queries production DBAs actually paste at 3am. Save them.

Top 5 biggest tables

size detective
SELECT
    schemaname || '.' || relname           AS table,
    pg_size_pretty(pg_total_relation_size(relid))  AS total_size,
    pg_size_pretty(pg_relation_size(relid))        AS data_size,
    pg_size_pretty(pg_total_relation_size(relid) - pg_relation_size(relid)) AS idx_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 5;

Live queries — who is doing what right now

pg_stat_activity
SELECT pid, usename, application_name, state,
       now() - query_start AS runtime,
       left(query, 100) AS q
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY runtime DESC;

Cancel / kill a runaway query

cancel vs terminate
-- polite: ask the query to stop (keeps connection open)
SELECT pg_cancel_backend(12345);

-- nuclear: kill the whole connection
SELECT pg_terminate_backend(12345);

Roles & GRANT

create a read-only user
CREATE ROLE readonly LOGIN PASSWORD 'change-me';
GRANT CONNECT ON DATABASE shop TO readonly;
GRANT USAGE ON SCHEMA public TO readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO readonly;

Extensions worth knowing

pg_stat_statements
Track every query's runtime + plan stats. The first thing a serious DBA installs. 
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
pgcrypto
gen_random_uuid(), crypt(), bcrypt password hashing. 
CREATE EXTENSION IF NOT EXISTS pgcrypto;
uuid-ossp
Older UUID generators (v1, v3, v4, v5). On modern PG prefer pgcrypto's gen_random_uuid(). 
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
pg_trgm
Fuzzy / similarity text search. Combine with a GIN index for blazing LIKE '%foo%' performance. 
CREATE EXTENSION IF NOT EXISTS pg_trgm;
pgvector
Store vector embeddings; <-> = L2, <=> = cosine. Native AI/RAG support. 
CREATE EXTENSION IF NOT EXISTS vector;

Zálohování & obnova — pg_dump / pg_restore

Two binaries, four flags. Master them and your data is safe.

The four formats

FlagFormatWhen to use
-Fpplain SQLtiny DBs, readable, replays with psql -f
-Fccustom (binary)default for ops — compressed, parallel restore, table filters
-Fttaruncompressed dir tarball — rare
-Fddirectoryparallel dump with -j N

Recipes

backup
# single DB, custom format (recommended)
pg_dump -h host -U alice -Fc shop > shop.dump

# compressed plain SQL
pg_dump -h host -U alice shop | gzip > shop.sql.gz

# schema only / data only
pg_dump -Fc --schema-only shop > shop_schema.dump
pg_dump -Fc --data-only   shop > shop_data.dump

# parallel directory dump (4 jobs)
pg_dump -Fd -j 4 -f shop_dir shop

# every database on the cluster (incl. roles)
pg_dumpall -U postgres > cluster.sql
restore
# custom format → use pg_restore
createdb shop_new
pg_restore -h host -U alice -d shop_new -j 4 shop.dump

# plain SQL → use psql -f
psql -h host -U alice -d shop_new -f shop.sql

# list contents of a custom dump (no actual restore)
pg_restore -l shop.dump | head

# restore only one table
pg_restore -d shop_new -t users shop.dump

Maintenance — keep PG happy

vacuum / analyze / reindex
-- reclaim dead rows + refresh planner stats
VACUUM ANALYZE;
VACUUM ANALYZE orders;            -- one table

-- aggressive: locks the table, rewrites it (rarely needed)
VACUUM FULL orders;

-- rebuild bloated indexes
REINDEX TABLE CONCURRENTLY orders;
REINDEX INDEX idx_orders_user;

Drills

pg_restore -d mydb_new - 4 out.dump

Golden rule of restores: always restore into a fresh empty database (createdb shop_new), verify, then rename or repoint your app. Never restore on top of a live DB.

Installation

Installez AWS CLI v2, configurez les identifiants et vérifiez que vous pouvez atteindre votre compte. Cinq minutes.

Installation

macOS
# installeur pkg
curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "AWSCLIV2.pkg"
sudo installer -pkg AWSCLIV2.pkg -target /

# ou via Homebrew
brew install awscli

aws --version
# aws-cli/2.15.x Python/3.x Darwin/24.x
Linux
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install

# Apple Silicon / ARM
curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
Windows
# installeur MSI
msiexec.exe /i https://awscli.amazonaws.com/AWSCLIV2.msi

# ou en Powershell :
winget install -e --id Amazon.AWSCLI

Préférez les rôles IAM (rôle d'instance EC2, rôle d'exécution Lambda, EKS IRSA) — ils distribuent automatiquement des identifiants à courte durée de vie.

Bases d'IAM

Identité (qui) + Politique (quoi) + Ressource (où). Chaque appel d'API est autorisé contre une politique.

Concepts

ÉlémentCe que c'est
UserIdentité à longue durée de vie liée à une personne (ou un service). Peut avoir des clés d'accès, un mot de passe console, du MFA.
RoleUne identité qui peut être assumée. Renvoie des identifiants à courte durée de vie (STS). Utilisé par EC2, Lambda, EKS, l'accès inter-comptes, la fédération.
PolicyDocument JSON. Politique d'identité attachée à un user/role. Politique de ressource attachée à une ressource (par ex. une politique de bucket S3).
GroupSac d'utilisateurs qui partagent des politiques. Pas d'imbrication.
STSSecurity Token Service. Émet des identifiants à courte durée de vie via AssumeRole, AssumeRoleWithWebIdentity, etc.

S3 — l'essentiel

  
Un bucket est un espace de noms plat à l'intérieur d'une région. Les clés ressemblent à des chemins mais ne sont pas des dossiers. Voici les commandes que vous lancerez chaque semaine.


  
Buckets

  

    

      
Créer / lister / supprimer

      
# création — les noms de buckets sont GLOBAUX
aws s3 mb s3://my-bucket --region us-east-1

aws s3 ls
aws s3 ls s3://my-bucket
aws s3 ls s3://my-bucket --recursive --human-readable --summarize

# suppression (doit être vide)
aws s3 rb s3://my-bucket
# ou forcer le vidage + suppression :
aws s3 rb s3://my-bucket --force

    

    

      
cp — un seul fichier

      
aws s3 cp ./report.csv s3://my-bucket/reports/

# téléchargement
aws s3 cp s3://my-bucket/reports/report.csv ./

# inter-buckets
aws s3 cp s3://src/key s3://dst/key

# avec métadonnées + classe de stockage
aws s3 cp ./big.tgz s3://my-bucket/ \
  --storage-class STANDARD_IA \
  --metadata "team=ml,owner=jane" \
  --content-type application/gzip

    

    

      
sync — répertoire

      
Copie uniquement les fichiers qui diffèrent. Par défaut, ne supprime pas les fichiers distants absents en local — passez --delete pour cela.

      
aws s3 sync ./build/ s3://my-bucket/v1/ \
  --exclude "*.tmp" \
  --exclude ".git/*" \
  --include "*.html" \
  --delete \
  --dryrun

    

    

      
rm

      
aws s3 rm s3://my-bucket/old.csv

# préfixe entier (récursif)
aws s3 rm s3://my-bucket/tmp/ --recursive

# avec filtre
aws s3 rm s3://my-bucket/ --recursive \
  --exclude "*" --include "*.log"

    

    

      
URL pré-signée

      
URL signée à durée limitée que n'importe qui peut utiliser sans identifiants AWS. Expiration par défaut : 3600 s.

      
aws s3 presign s3://my-bucket/report.csv \
  --expires-in 900

# pour upload (PUT) — nécessite s3api :
aws s3 presign s3://my-bucket/upload.bin \
  --expires-in 600
# Note : aws s3 presign émet des URLs GET.
# Pour des présignés PUT, utilisez le SDK ou la commande s3api.

S3 Avancé

  
Versioning, chiffrement, cycle de vie, hébergement. Les fonctionnalités qui transforment un bucket de simple décharge en infrastructure.


  
Versioning

  

    

      
Activer / lister / supprimer

      
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled

# lister les versions
aws s3api list-object-versions --bucket my-bucket --prefix reports/

# supprimer une version spécifique
aws s3api delete-object \
  --bucket my-bucket --key reports/q1.csv \
  --version-id 3HL4kqtJlcpXroDTDmJ+rmSpXd3dIbrHY+MTRCxf3vjVBH40Nrjfkd

    

    

      
MFA Delete

      
Force un jeton MFA pour supprimer une version d'objet ou désactiver le versioning. Compte root uniquement.

      
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled,MFADelete=Enabled \
  --mfa "arn:aws:iam::123456789012:mfa/root 123456"

    

  


  
Chiffrement

  

    

      
Chiffrement par défaut du bucket

      
# SSE-S3 (AES256, géré par AWS)
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules":[{"ApplyServerSideEncryptionByDefault":{
      "SSEAlgorithm":"AES256"}}]}'

# SSE-KMS (clé gérée par le client)
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules":[{"ApplyServerSideEncryptionByDefault":{
      "SSEAlgorithm":"aws:kms",
      "KMSMasterKeyID":"alias/my-key"}}]}'

    

    

      
SSE-S3 vs SSE-KMS

      
SSE-S3 : zéro configuration, AWS détient la clé. Gratuit. À utiliser sauf si vous avez besoin d'audit/contrôle IAM sur l'usage de la clé.
SSE-KMS : journaux d'audit par clé, rotation de clé, octrois inter-comptes. Coûte 0,03 $/10k requêtes + frais de clé KMS.

    

    

      
Chiffrement par objet

      
aws s3 cp file.txt s3://my-bucket/ \
  --sse aws:kms \
  --sse-kms-key-id alias/my-key

    

  


  
Politique de bucket et accès public

  

    

      
Blocage d'accès public (à activer)

      
aws s3api put-public-access-block \
  --bucket my-bucket \
  --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

    

    

      
Politique de bucket : autoriser CloudFront uniquement

      
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"Service": "cloudfront.amazonaws.com"},
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-bucket/*",
    "Condition": {"StringEquals":{
      "AWS:SourceArn": "arn:aws:cloudfront::123:distribution/E1ABC..."}}
  }]
}

    

    

      
Appliquer la politique

      
aws s3api put-bucket-policy \
  --bucket my-bucket \
  --policy file://policy.json

aws s3api get-bucket-policy --bucket my-bucket
aws s3api delete-bucket-policy --bucket my-bucket

Hébergement de site statique

      
aws s3 website s3://my-bucket/ \
  --index-document index.html \
  --error-document 404.html

# point de terminaison :
# http://my-bucket.s3-website-us-east-1.amazonaws.com

    

  


  
Aujourd'hui : pour les sites en production, mettez CloudFront devant un bucket S3 dont l'accès public est bloqué et utilisez un Origin Access Control (OAC). Le point de terminaison s3-website nu est en HTTP uniquement et n'a pas de cache edge.







  
Workflows Hello World

  
Trois recettes de bout en bout que vous pouvez exécuter aujourd'hui.


  
① Héberger un site statique sur S3 + CloudFront

  

    

      
1. Créer le bucket

      
BUCKET=my-site-$(uuidgen | tr A-Z a-z)
REGION=us-east-1

aws s3 mb s3://$BUCKET --region $REGION

# gardez l'accès public bloqué — CloudFront servira le contenu
aws s3api put-public-access-block \
  --bucket $BUCKET \
  --public-access-block-configuration \
   "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

    

    

      
2. Synchroniser le site

      
aws s3 sync ./site/ s3://$BUCKET/ \
  --delete \
  --cache-control "public,max-age=3600" \
  --exclude "*.html" \
  --exclude "service-worker.js"

# cache court pour le HTML afin que les déploiements soient visibles vite
aws s3 sync ./site/ s3://$BUCKET/ \
  --exclude "*" --include "*.html" \
  --cache-control "public,max-age=60"

    

    

      
3. CloudFront en façade

      
aws cloudfront create-distribution \
  --origin-domain-name $BUCKET.s3.$REGION.amazonaws.com
# Dans la console : activez OAC, attachez une politique de bucket qui
# autorise le SourceArn CloudFront (voir la carte « Politique de bucket »).

# invalidation après déploiement
aws cloudfront create-invalidation \
  --distribution-id E1ABC... --paths "/*"

Confiance + rôle d'exécution

      
aws iam create-role \
  --role-name lambda-thumb \
  --assume-role-policy-document '{
    "Version":"2012-10-17",
    "Statement":[{"Effect":"Allow",
      "Principal":{"Service":"lambda.amazonaws.com"},
      "Action":"sts:AssumeRole"}]}'

  --role-name lambda-thumb \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

  --policy-name s3rw --policy-document '{
    "Version":"2012-10-17",
    "Statement":[{"Effect":"Allow",
      "Action":["s3:GetObject","s3:PutObject"],
      "Resource":"arn:aws:s3:::my-bucket/*"}]}'

    

    

      
Créer la fonction

      
zip thumb.zip lambda_function.py

aws lambda create-function \
  --function-name thumb \
  --runtime python3.12 \
  --handler lambda_function.handler \
  --role arn:aws:iam::123:role/lambda-thumb \
  --zip-file fileb://thumb.zip

    

    

      
Câbler S3 → Lambda

      
aws lambda add-permission \
  --function-name thumb --statement-id s3 \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --source-arn arn:aws:s3:::my-bucket

aws s3api put-bucket-notification-configuration \
  --bucket my-bucket \
  --notification-configuration '{
    "LambdaFunctionConfigurations":[{
      "LambdaFunctionArn":"arn:aws:lambda:us-east-1:123:function:thumb",
      "Events":["s3:ObjectCreated:*"],
      "Filter":{"Key":{"FilterRules":[
        {"Name":"prefix","Value":"uploads/"},
        {"Name":"suffix","Value":".jpg"}]}}
    }]}'

Erreurs courantes

  

    
SymptômeCause probable et correctif

    
      
An error occurred (AccessDenied)La politique IAM n'accorde pas l'action. Vérifiez à la fois la politique d'identité et la politique de bucket/ressource. Faites d'abord aws sts get-caller-identity pour confirmer quelle identité est utilisée. Servez-vous du IAM Policy Simulator.

      
NoSuchBucketFaute de frappe dans le nom du bucket, ou vous êtes dans la mauvaise région/le mauvais compte. Les noms de buckets sont globaux ; aws s3 ls ne liste que les buckets que votre compte possède.

      
The bucket you are attempting to access must be addressed using the specified endpointMauvaise région. Passez --region us-west-2 ou positionnez AWS_REGION.

      
ExpiredToken / InvalidTokenIdentifiants SSO/STS expirés. Lancez aws sso login --profile foo ou réassumez le rôle.

      
Could not connect to the endpoint URLMauvaise région (S3 n'est pas dans celle-là), politique d'endpoint VPC, ou proxy. Essayez --debug.

      
RequestTimeTooSkewedVotre horloge dérive. Synchronisez l'heure (NTP).

      
An error occurred (SlowDown) when calling …S3 limite le débit par préfixe. Répartissez les clés sur plus de préfixes ; faites du back-off et réessayez. Limite par préfixe : 3500 PUT/COPY/POST/DELETE et 5500 GET req/s.

      
InvalidAccessKeyIdMauvaise clé, clé désactivée, ou la clé appartient à un autre compte. Re-vérifiez ~/.aws/credentials et $AWS_PROFILE.

      
EntityAlreadyExists à la création du bucketLes noms de buckets sont globaux. Choisissez-en un autre.

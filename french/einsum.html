PyTorch · une notation pour gouverner toutes les opérations sur tenseurs

Chaque contraction dans une chaîne

torch.einsum vous permet d'écrire un produit matriciel, un matmul par lots, une transposition, une trace, un produit scalaire, un produit extérieur et des scores d'attention avec une seule chaîne d'indices comme 'ik,kj->ij'. Plus besoin de se rappeler s'il faut @, bmm, matmul, ou un enchevêtrement de transpose + sum — il suffit de nommer les axes et de dire ce que la sortie conserve. Cette page présente les règles, une galerie de recettes et un bac à sable interactif qui exécute réellement l'einsum sur de petits tenseurs.

01 L'idée en une ligne

ce que c'est

Nommer les axes

Donnez une lettre à chaque axe de chaque entrée. 'bik,bkj->bij' signifie « l'opérande A est indexé par b,i,k ; l'opérande B par b,k,j ; le résultat conserve b,i,j ».

la seule règle

Absent = sommé

Toute lettre qui apparaît à gauche mais pas dans la sortie est sommée puis éliminée. Ici k est absent de bij, donc on somme sur k — c'est le produit matriciel.

$$\texttt{'ik,kj->ij'} \quad\Longleftrightarrow\quad C_{ij} = \sum_{k} A_{ik}\, B_{kj}$$
Lisez-la comme une phrase. Les indices sont la formule de sommation d'Einstein : le côté gauche liste les indices de chaque opérande, -> sépare la sortie, et la sommation sur l'indice répété puis abandonné est implicite. einsum ne fait que traduire ces maths en un appel.

01·b Pourquoi on l'adopte

lisible

Auto-documenté

'bhqd,bhkd->bhqk' vous dit exactement ce que font les scores d'attention — les axes sont là, sous vos yeux. Un empilement de permute+matmul le masque.

générale

Une API, de nombreuses opérations

Transposition, somme, trace, produit scalaire, produit extérieur, matmul, par lots et contractions multi-tenseurs : tout se ramène au même appel.

moins de bugs

Pas de diffusion silencieuse

Vous explicitez chaque axe, donc une incompatibilité de forme se voit tout de suite — pas de diffusion (broadcasting) accidentelle qui ferait autre chose que prévu.

Tout le langage en trois règles

Comment lire une chaîne d'indices

La chaîne avant -> décrit les entrées ; celle après décrit la sortie. Trois règles couvrent tout.

①

Même lettre = alignement

Une lettre réutilisée entre opérandes signifie que ces axes s'alignent et sont multipliés élément par élément le long de cette dimension. Dans 'ik,kj', le k de A s'apparie avec le k de B.

②

Lettre abandonnée = somme

Toute lettre d'entrée absente de la sortie est sommée. 'ik,kj->ij' abandonne k, donc on somme sur k — une contraction.

③

Lettre conservée = axe

Une lettre qui apparaît dans la sortie devient un axe du résultat, dans l'ordre où vous l'avez écrite. Réordonnez les lettres de sortie pour transposer : 'ij->ji'.

02·a Indices libres vs. sommés

Deux rôles, deux couleurs utilisés tout au long de cette page :

libre — apparaît dans la sortie, devient un axe du résultat sommé — uniquement à gauche, contracté puis éliminé
$$\underbrace{C_{\textcolor{#fbbf24}{ij}}}_{\text{free}} = \sum_{\textcolor{#818cf8}{k}} A_{\textcolor{#fbbf24}{i}\textcolor{#818cf8}{k}}\, B_{\textcolor{#818cf8}{k}\textcolor{#fbbf24}{j}}$$

02·b La forme implicite (sans flèche)

quand vous omettez ->

Si vous laissez tomber ->sortie, einsum applique une règle implicite : chaque indice qui apparaît exactement une fois à gauche devient un axe de sortie libre, trié par ordre alphabétique. Les indices qui se répètent sont sommés.

  • 'ij'  →  'ij->ij'   identité (chaque lettre une fois).
  • 'ii'  →  'ii->'   la trace (i se répète, donc il est sommé).
  • 'ik,kj'  →  'ik,kj->ij'   matmul (k se répète → sommé ; i,j triés).
Soyez explicite. La forme implicite est pratique, mais la règle du tri alphabétique surprend. Écrire la sortie après -> lève toute ambiguïté d'intention, et c'est ce que font la plupart des bases de code.

02·c Signature & pièges

torch.einsum · l'appel 
import torch

A = torch.randn(2, 3)        # [i=2, k=3]
B = torch.randn(3, 4)        # [k=3, j=4]

C = torch.einsum('ik,kj->ij', A, B)   # [i=2, j=4]  == A @ B

# les opérandes sont passés après la chaîne, dans l'ordre
# le nb de lettres de chaque opérande doit égaler son nb de dims
piège

Lettres ≠ dims

Si un opérande a 3 dimensions, sa chaîne d'indices a besoin d'exactement 3 lettres. 'ij' pour un tenseur 3-D lève une erreur.

piège

Même lettre, même taille

Une lettre répétée doit avoir la même longueur partout où elle apparaît — le k des deux opérandes doit correspondre, sinon l'alignement est impossible.

Points de suspension ... : pour les dimensions « de lot » en tête que vous ne voulez pas nommer, utilisez '...ik,...kj->...ij'. Les ... diffusent sur un nombre quelconque d'axes de tête — parfait pour du code qui doit fonctionner aussi bien avec des entrées par lots que sans.

03·c Le bloc d'attention, en einsum

attention par produit scalaire mis à l'échelle · détaillée 
# Q,K,V: [B, H, T, d]   (lot, têtes, tokens, dim-de-tête)
scores = torch.einsum('bhqd,bhkd->bhqk', Q, K) / math.sqrt(d)
attn   = scores.softmax(dim=-1)                    # [B,H,q,k]
out    = torch.einsum('bhqk,bhkd->bhqd', attn, V)  # [B,H,q,d]
Lisez la ligne des scores : q et k sont les deux axes de tokens ; d (la dimension de tête) est partagé et abandonné, donc sommé — c'est le produit scalaire de chaque requête contre chaque clé. b,h suivent comme axes de lot.

HuggingFace GPT-2 — entraînez-le, tokenisez-le, générez avec lui

GPT-2 est un transformeur décodeur seul entraîné à prédire le prochain token. La bibliothèque transformers de HuggingFace l'enveloppe en trois lignes de Python. Cette page vous fait parcourir toute la pile — pipelines, tokeniseur, modèle, leviers d'échantillonnage et boucles d'entraînement — et vous fait taper les éléments clés pour les ancrer.

les quatre bibliothèques
transformers modèles, tokeniseurs, pipelines, Trainer
datasets diffuse / map / cache des données d'entraînement
tokenizers BPE / WordPiece rapides en Rust
accelerate multi-GPU, précision mixte, sans plomberie
qu'est-ce que GPT-2 ?
Un transformeur décodeur seul. 12 couches (124M), 24 (345M), 36 (774M), 48 (1,5B). Entraîné sur WebText avec un seul objectif : prédire le prochain token. Un masque causal empêche chaque position de regarder vers le futur. Le tokeniseur est un BPE au niveau octet avec un vocabulaire de 50 257 tokens.
deux façons de l'utiliser
facile pipeline("text-generation", model="gpt2")
bas niveau AutoTokenizer + AutoModelForCausalLM + .generate()
Le pipeline est parfait pour les démos. Pour du vrai travail, descendez au couple modèle + tokeniseur afin de contrôler l'échantillonnage, le batching et l'entraînement.

Recette 2 — tokeniseur + modèle

doublet explicite
from transformers import AutoTokenizer, AutoModelForCausalLM

tokenizer = AutoTokenizer.from_pretrained("gpt2")
model     = AutoModelForCausalLM.from_pretrained("gpt2")

inputs  = tokenizer("The future of AI is", return_tensors="pt")
outputs = model.generate(**inputs, max_new_tokens=30, do_sample=True)
text    = tokenizer.decode(outputs[0], skip_special_tokens=True)
print(text)
Choix de la classe : AutoModelForCausalLM est la bonne tête pour la génération de texte autorégressive. Les autres têtes sont AutoModelForSequenceClassification, AutoModelForMaskedLM (famille BERT), AutoModelForQuestionAnswering, etc.

Tokenisation — le pont BPE entre chaînes et tenseurs

GPT-2 ne voit ni caractères ni mots — il voit des identifiants de tokens. Son BPE au niveau octet fusionne les sous-mots fréquents en unités uniques (par ex. "ing"), garde les mots rares en plusieurs tokens et utilise un marqueur d'espace de tête (rendu sous la forme Ġ) pour garantir des aller-retours réversibles.

Génération — ce que fait chaque argument de .generate()

Toute la magie de la « créativité » vit dans model.generate(). Six arguments couvrent 95 % des cas. Bougez-les et regardez la démo pré-enregistrée réagir.

🎛️ terrain de jeu de génération

Choisissez un prompt, tournez les boutons, cliquez sur Générer. La sortie est échantillonnée depuis un petit pool de complétions pré-enregistrées graines par les réglages temperature / top-k / top-p — illustratif uniquement. Les vrais appels .generate() les acceptent de la même façon.

appel canonique de génération
inputs  = tokenizer(prompt, return_tensors="pt")
outputs = model.generate(
    **inputs,
    max_new_tokens=50,
    do_sample=True,            # False = greedy/beam, True = stochastique
    top_k=50,                  # garde les top-50 logits, masque le reste
    top_p=0.9,                 # nucleus : plus petit ensemble dont les probas somment à 0.9
    temperature=0.8,           # <1 plus net, >1 plus fou
    repetition_penalty=1.1,    # décourage les répétitions
    pad_token_id=tokenizer.eos_token_id,
)
text = tokenizer.decode(outputs[0], skip_special_tokens=True)

Stratégies d'échantillonnage — choisir le prochain token

À chaque étape, le modèle émet une probabilité sur l'ensemble des 50 257 tokens du vocabulaire. Une stratégie transforme cette distribution en un token choisi. Les stratégies font un compromis entre cohérence, créativité et coût de calcul.

stratégieflagstyleà utiliser quand
Greedy do_sample=False déterministe, répétitif tâches avec une seule bonne réponse (traduction en mode basse temp)
Beam search num_beams=4 sûr, parfois platement fluide résumé, sortie structurée
Sampling do_sample=True créatif, peut dériver récits, brainstorming
top-k top_k=50 plafonne la pire queue combinez avec sampling pour la stabilité
top-p (nucleus) top_p=0.9 s'adapte à la confiance presque toujours — défaut moderne
temperature temperature=0.8 levier sur la netteté 0.7–1.0 pour le chat, 0.2 pour le code, >1.2 pour le chaos
intuition de temperature
Les logits sont divisés par T avant softmax. T bas affûte (proche du greedy), T haut aplatit (proche de l'uniforme). À T→0 vous retrouvez le greedy. À T=1 vous échantillonnez depuis la vraie distribution du modèle.
intuition de top-k
Jetez chaque token sauf les k plus probables. Renormalisez et échantillonnez. Coupe la longue queue de tokens bizarres et rares. Fragile : même k pour une étape confiante et une étape plate.
top-p (nucleus)
Gardez le plus petit ensemble de tokens dont la probabilité cumulée ≥ p. S'adapte par étape : étape confiante → ensemble minuscule ; étape plate → ensemble large. Bat le top-k brut sur la plupart des tâches. p=0.9 est le défaut sûr.
côte à côte
# 1. greedy
model.generate(**inputs, max_new_tokens=30, do_sample=False)

# 2. beam search
model.generate(**inputs, max_new_tokens=30, num_beams=4, early_stopping=True)

# 3. échantillonnage simple
model.generate(**inputs, max_new_tokens=30, do_sample=True, temperature=1.0)

# 4. top-k
model.generate(**inputs, max_new_tokens=30, do_sample=True, top_k=50)

# 5. top-p (nucleus)
model.generate(**inputs, max_new_tokens=30, do_sample=True, top_p=0.9)

# 6. combo recommandé
model.generate(**inputs, max_new_tokens=50,
               do_sample=True, top_k=50, top_p=0.9, temperature=0.8)
Conseil pratique : commencez par do_sample=True, top_p=0.9, temperature=0.8. Si la sortie est trop aléatoire, baissez T. Si elle boucle, montez repetition_penalty à 1.1–1.3. Si elle sonne fade, poussez T à 1.0 et retirez top_k.

Entraînement — fine-tuner GPT-2 sur vos données

Deux chemins : l'API haut niveau Trainer (moins de 20 lignes), ou une boucle PyTorch brute. Les deux fine-tunent les mêmes poids. Trainer est le bon défaut — il gère le logging, le checkpointing, l'évaluation, l'entraînement distribué et la précision mixte automatiquement.

Voie A — l'API Trainer

fine-tune de bout en bout de gpt2
from transformers import (
    AutoTokenizer, AutoModelForCausalLM,
    Trainer, TrainingArguments, DataCollatorForLanguageModeling,
)
from datasets import load_dataset

tokenizer = AutoTokenizer.from_pretrained("gpt2")
tokenizer.pad_token = tokenizer.eos_token

model = AutoModelForCausalLM.from_pretrained("gpt2")

ds = load_dataset("wikitext", "wikitext-2-raw-v1")

def tok(batch):
    return tokenizer(batch["text"], truncation=True, max_length=512)

ds = ds.map(tok, batched=True, remove_columns=["text"])

collator = DataCollatorForLanguageModeling(
    tokenizer=tokenizer,
    mlm=False,                          # LM causal, pas LM masqué
)

args = TrainingArguments(
    output_dir="./gpt2-ft",
    per_device_train_batch_size=4,
    num_train_epochs=3,
    learning_rate=5e-5,
    warmup_steps=100,
    logging_steps=50,
    save_steps=500,
    fp16=True,
)

trainer = Trainer(
    model=model,
    args=args,
    train_dataset=ds["train"],
    eval_dataset=ds["validation"],
    data_collator=collator,
)

trainer.train()
trainer.save_model("./gpt2-ft")

Voie B — boucle PyTorch brute

sans Trainer, juste torch
import torch
from torch.utils.data import DataLoader
from torch.optim import AdamW

loader    = DataLoader(ds["train"], batch_size=4, collate_fn=collator)
optimizer = AdamW(model.parameters(), lr=5e-5)
device    = "cuda" if torch.cuda.is_available() else "cpu"
model.to(device)
model.train()

for epoch in range(3):
    for batch in loader:
        batch = {k: v.to(device) for k, v in batch.items()}
        outputs = model(**batch)             # labels=input_ids gérés par le collator
        loss    = outputs.loss               # transformers retourne la loss CE
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()
Loss = labels = input_ids : pour les LM causaux, l'entrée et la cible sont la même séquence de tokens — le modèle apprend à prédire chaque token à partir de ceux d'avant. DataCollatorForLanguageModeling(mlm=False) prépare les labels pour vous.

TrainingArguments — chaque levier à connaître

La dataclass TrainingArguments contient chaque hyperparamètre que touche le Trainer. Il y a plus de 100 champs. Voici la douzaine que vous réglerez vraiment.

argumenttypiquece qu'il fait
output_dir "./out" où atterrissent les checkpoints + logs (obligatoire)
per_device_train_batch_size 4–32 batch par GPU ; batch effectif = ceci × num_gpus × grad_accum
gradient_accumulation_steps 1–16 simule de plus gros batches quand la mémoire GPU est juste
num_train_epochs 1–10 passes sur l'ensemble d'entraînement
learning_rate 5e-5 standard pour fine-tuner les transformeurs
warmup_steps 100–1000 rampe linéaire — évite les pics de loss au début
weight_decay 0.01 régularisation AdamW
fp16 / bf16 True précision mixte ; bf16 sur Ampere+, fp16 ailleurs
logging_steps 50 affiche la loss tous les N pas
save_steps 500 checkpoint tous les N pas
evaluation_strategy "steps" "no" / "steps" / "epoch"
load_best_model_at_end True après l'entraînement, restaure le meilleur checkpoint
report_to "wandb" ou "tensorboard" / "none"
config quasi-prod
args = TrainingArguments(
    output_dir="./gpt2-ft",
    per_device_train_batch_size=8,
    gradient_accumulation_steps=4,         # batch effectif = 32
    num_train_epochs=3,
    learning_rate=5e-5,
    warmup_steps=200,
    weight_decay=0.01,
    fp16=True,
    logging_steps=50,
    save_steps=500,
    evaluation_strategy="steps",
    eval_steps=500,
    load_best_model_at_end=True,
    report_to="none",
)
Taille de batch effective : per_device_train_batch_size × num_gpus × gradient_accumulation_steps. Un GPU 24 Go peut tenir en batch 8 ; avec grad-accum 4 vous obtenez la courbe de loss du batch 32.

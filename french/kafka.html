Concepts

Les brokers forment un cluster. Les topics se divisent en partitions. Les producteurs ajoutent ; les consommateurs lisent par offset. C'est tout le jeu.

Anatomie du cluster

Un producteur écrit des enregistrements dans un topic. Le partitionneur choisit quelle partition. Chaque partition est répliquée. Les consommateurs d'un groupe se partagent les partitions.

Vocabulaire

TermeSignification
brokerUn serveur Kafka. Cluster = plusieurs brokers. Chaque broker est identifié par un broker.id numérique.
topicUn journal nommé d'enregistrements. Créé avec un nombre de partitions et un facteur de réplication.
partitionUn journal commit ordonné, en append-only. Les enregistrements à l'intérieur d'une partition sont strictement ordonnés. Entre partitions : aucune garantie d'ordre.
offsetPosition monotone 64 bits dans une partition. Les consommateurs suivent l'offset qu'ils ont lu.
leader / followerPour chaque partition, un broker est leader (gère lectures + écritures), les autres répliquent. Sur défaillance du leader, un follower est promu.
ISRIn-Sync Replicas : followers à jour dans replica.lag.time.max.ms. Seuls les membres ISR peuvent devenir leader.
consumer groupUn ensemble de consommateurs partageant un group.id. Kafka assigne chaque partition à un seul consommateur du groupe à la fois.
rebalanceRéassignation des partitions quand l'appartenance au groupe change (consommateur rejoint/part/meurt).
__consumer_offsetsTopic interne compacté où vivent les offsets validés (50 partitions par défaut).

Règles d'ordre

Ordre par partition uniquement
Les enregistrements dans une partition sont strictement ordonnés. Entre partitions : pas d'ordre. Même clé → même partition par défaut → ordonné pour cette clé.
# Piège d'ordre
# key=user-42 → va vers la partition hash(user-42) % N
# Tous les événements user-42 restent ordonnés. user-99 peut s'intercaler.
acks & durabilité
acks=all + min.insync.replicas≥2 = aucune perte de données sur défaillance d'un seul broker.
acks=0    # envoyer et oublier
acks=1    # attendre seulement le leader
acks=all  # attendre tous les ISR
Sémantique de livraison
Au-moins-une-fois est par défaut. Exactement-une-fois nécessite producteurs idempotents + transactions + consommateurs read_committed.
# recette exactement-une-fois
enable.idempotence=true
transactional.id=order-app-1
# consommateur :
isolation.level=read_committed
Piège : augmenter le nombre de partitions plus tard ne déplace pas les données existantes. Les clés existantes restent dans leurs anciennes partitions ; les nouvelles clés peuvent atterrir ailleurs — l'ordre pour cette clé se divise entre deux partitions un certain temps. Planifiez le nombre de partitions à l'avance ou utilisez un job de re-partitionnement key-aware.

Producteurs

Les producteurs ajoutent des enregistrements à un topic. Choisir une clé, une politique d'ack et un partitionneur représente 80% de la conception.

Démarrage rapide (kafka-python)

Producteur basique
from kafka import KafkaProducer
import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",
    enable_idempotence=True,
    linger_ms=5,
    compression_type="zstd",
)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()
Idempotent + retries
Pas de doublons sur retry, pas de réordonnancement quand max.in.flight ≤ 5.
KafkaProducer(
    enable_idempotence=True,
    acks="all",
    retries=2**31-1,
    max_in_flight_requests_per_connection=5,
    delivery_timeout_ms=120_000,
)
Producteur transactionnel
Écriture atomique sur plusieurs topics + partitions. Utilisé pour les apps lecture-traitement-écriture et exactement-une-fois.
# style confluent-kafka
from confluent_kafka import Producer
p = Producer({
  "bootstrap.servers": "broker1:9092",
  "transactional.id": "order-app-1",
  "enable.idempotence": True,
})
p.init_transactions()
p.begin_transaction()
p.produce("orders", key="u1", value="...")
p.produce("audit", key="u1", value="...")
p.commit_transaction()
Partitionneur personnalisé
Par défaut : murmur2(key) % N quand la clé est définie ; partitionneur sticky quand la clé est null.
def tenant_partitioner(key, all_parts, available):
    # router tenant=premium vers partitions dédiées 0..3
    if key and key.startswith(b"premium:"):
        return hash(key) % 4
    return hash(key) % len(all_parts)

KafkaProducer(partitioner=tenant_partitioner, …)
Headers + timestamps
p.send(
  "orders", key=b"u1", value=b"...",
  headers=[("trace-id", b"abc123"),
           ("schema", b"OrderV2")],
  timestamp_ms=int(time.time()*1000),
)
Asynchrone + callback
def on_delivery(err, msg):
    if err:
        log.error("failed: %s", err)
    else:
        log.debug("%s[%d]@%d",
                  msg.topic(), msg.partition(), msg.offset())

p.produce("orders", value=v, on_delivery=on_delivery)
p.poll(0)  # déclencher les callbacks

Compression & batching

RéglageEffet
linger.msAttendre ce délai avant d'envoyer un batch même incomplet. 5–20 ms apporte de gros gains de débit.
batch.sizeOctets par batch de partition (par défaut 16 Ko). 64 Ko+ aide pour les producteurs à haut débit.
compression.typenone · gzip · snappy · lz4 · zstd. zstd habituellement meilleur pour ratio + CPU.
buffer.memoryOctets totaux que le client peut utiliser pour bufferiser les enregistrements non envoyés (par défaut 32 Mo).

Flux de message

Producteur → leader → followers (réplication) → consommateur.

Code consommateur

Boucle poll basique
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",
    auto_offset_reset="earliest",   # ou "latest"
    enable_auto_commit=False,
    isolation_level="read_committed",
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # commit synchrone
Poll manuel + offsets
Valider seulement après traitement réussi → au-moins-une-fois.
while True:
    batch = c.poll(timeout_ms=1000)
    for tp, msgs in batch.items():
        for m in msgs:
            handle(m)
        c.commit({tp: OffsetAndMetadata(msgs[-1].offset+1, None)})
Listener de rebalance
Vider le travail en cours avant que les partitions soient révoquées.
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()
        consumer.commit()
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)

c.subscribe(["orders"], listener=L())
Seek vers offset / temps
# par offset
c.seek(TopicPartition("orders", 0), 12345)

# par timestamp (ms depuis epoch)
ts = {TopicPartition("orders", 0): 1714000000000}
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():
    c.seek(tp, off.offset)
Appartenance statique
Un group.instance.id stable évite les rebalances sur redémarrages glissants.
KafkaConsumer(
  group_id="billing",
  group_instance_id="billing-pod-7",
  session_timeout_ms=30_000,
)
Rebalance coopératif
Rebalance incrémental — seules les partitions qui changent de mains arrêtent d'être traitées, pas toutes.
partition_assignment_strategy=[
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor"
]

Sémantique de validation

ModeGarantieRisque
enable_auto_commit=true (par défaut 5 s)Au-plus-une-fois-ishCrash après commit, avant traitement → enregistrement perdu.
Manuel commit() après traitementAu-moins-une-foisCrash après traitement, avant commit → retraitement au redémarrage. Rendez les handlers idempotents.
Transactionnel send_offsets_to_transaction()Exactement-une-fois (lecture-traitement-écriture dans Kafka)N'aide pas quand l'effet de bord est hors de Kafka (par ex. un POST HTTP non idempotent).
Piège : un handler de message qui dépasse max.poll.interval.ms (par défaut 5 min) est expulsé du groupe → tempête de rebalance. Soit augmentez la limite, diminuez max.poll.records, ou déplacez le travail vers un pool de workers et utilisez pause() / resume().

CLI

Les scripts shell sous $KAFKA_HOME/bin/ sont vos amis en production. Aide-mémoire ci-dessous.

Topics

Créer / lister / décrire
# créer
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 12 --replication-factor 3 \
  --config retention.ms=604800000 \
  --config min.insync.replicas=2

# lister / décrire
kafka-topics.sh --bootstrap-server localhost:9092 --list
kafka-topics.sh --bootstrap-server localhost:9092 --describe --topic orders
Modifier / supprimer
# ajouter des partitions (impossible de diminuer !)
kafka-topics.sh --bootstrap-server localhost:9092 \
  --alter --topic orders --partitions 24

# changer une config
kafka-configs.sh --bootstrap-server localhost:9092 \
  --entity-type topics --entity-name orders \
  --alter --add-config retention.ms=86400000

# supprimer
kafka-topics.sh --bootstrap-server localhost:9092 \
  --delete --topic orders

Streams & ksqlDB

Kafka Streams : une bibliothèque Java/Scala pour transformer des topics. ksqlDB : SQL au-dessus de Streams.

Streams DSL

Comptage de mots (le « hello world »)
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))
   .groupBy((k,w) -> w)
   .count(Materialized.as("counts"))
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Jointure stream-table
Enrichir un flux d'événements avec le dernier état d'un topic compacté.
KTable<String,User> users = b.table("users");
KStream<String,Order> orders = b.stream("orders");

orders.join(users,
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
Agrégation fenêtrée
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(
         Duration.ofMinutes(5), Duration.ofMinutes(1)))
      .aggregate(() -> 0L,
                 (k, v, sum) -> sum + v.amount(),
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

ksqlDB

CREATE STREAM / TABLE

  user_id VARCHAR KEY,
  sku VARCHAR,
  qty INT,
  amount DOUBLE

CSAS — stream dérivé
CREATE STREAM big_orders AS
SELECT o.user_id, u.tier, o.amount
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id
WHERE o.amount > 100
EMIT CHANGES;
CTAS — table fenêtrée
CREATE TABLE spend_5min AS
SELECT user_id,
       SUM(amount) AS total
FROM orders
WINDOW TUMBLING (SIZE 5 MINUTES)
GROUP BY user_id
EMIT CHANGES;
Pull query
-- lecture point-in-time du dernier état d'une table
SELECT total FROM spend_5min
WHERE user_id = 'user-42';

Ops & Optimisation

Les boutons que vous tournerez réellement en production, plus les alarmes qui vous réveillent.

Réplication & durabilité

ConfigDéfautQuoi & quand
replication.factor (par topic)—3 en prod. Survit à la perte d'un broker avec min.insync.replicas=2.
min.insync.replicas1Mettre 2 avec RF=3. Combiné avec acks=all = pas de perte de données sur défaillance d'un broker unique.
unclean.leader.election.enablefalseGarder false. L'autoriser échange durabilité contre disponibilité.
replica.lag.time.max.ms30000Les followers plus lents que ça sortent de l'ISR.
auto.create.topics.enabletrueMettre false en prod. Faire de la création de topic une étape explicite.

Rétention & stockage

ConfigCe qu'il fait
retention.msSupprime les enregistrements plus anciens que cela. Par défaut 7 jours. Par topic.
retention.bytesPlafonne la taille du log par partition. Par topic. Le premier (ms / bytes) atteint l'emporte.
segment.bytes / segment.msRoule un nouveau segment de log tous les X octets ou X ms. Segments plus petits = suppressions/compactions plus rapides, plus de descripteurs de fichiers.
cleanup.policy=compactCompaction de log — garde la dernière valeur par clé pour toujours. À utiliser pour les topics d'état, changelogs, __consumer_offsets.
cleanup.policy=delete,compactLes deux : compacte + supprime les vieilles clés après rétention.
min.cleanable.dirty.ratioÀ quel point sale avant que le compactor s'exécute (par défaut 0.5).

Alarmes communes

SymptômeQue vérifier
Partitions sous-répliquées > 0Broker arrêté, disque lent, ou réseau. kafka-topics.sh --describe --under-replicated-partitions.
Partitions hors ligne > 0Aucun leader disponible. Le cluster est en difficulté.
Lag de consommateur en croissanceConsommateur lent / bloqué. Vérifier max.poll.interval.ms, pauses GC, système en aval.
Producteur RecordTooLargeExceptionAugmenter max.request.size sur producteur + message.max.bytes sur broker + max.partition.fetch.bytes sur consommateur.
Tempête de rebalance de groupeHandlers lents, membres instables, GC. Utiliser appartenance statique + cooperative-sticky.
L'ISR rétrécit fréquemmentProblème réseau ou disque. Vérifier replica.lag.time.max.ms, IO wait du broker.

Règles empiriques de dimensionnement de cluster

Partitions par broker
Viser < 4000 partitions par broker. Chaque partition coûte des descripteurs de fichiers, mémoire, et temps de rebalance.
Nombre de partitions par topic
≥ parallélisme maximum dont vous aurez besoin. Choisir un nombre généreux à l'avance — l'augmenter plus tard casse l'ordre par clé.
Disques
XFS, grandes écritures séquentielles. Ne pas partager avec quoi que ce soit de bursty. Prévoir 50%+ libre pour la compaction de log.
Réseau
La réplication est aussi du trafic nord-sud. RF=3 signifie que chaque octet produit traverse le réseau ~3×.
JVM
6-8 Go de heap suffisent largement. La majorité de la mémoire devrait aller au cache de page.
KRaft (sans ZooKeeper)
3.3+ par défaut KRaft. ZooKeeper supprimé en 4.0. Nouveaux clusters : KRaft uniquement.

llama.cpp — exécuter des LLM localement, en C/C++

llama.cpp est un moteur d’inférence haute performance pour les modèles de la famille LLaMA (Llama 1/2/3, Mistral, Qwen, Gemma, Phi, DeepSeek, ...) écrit en C/C++ pur, avec zéro Python à l’exécution. Il tourne sur CPU, Apple Silicon (Metal), NVIDIA (CUDA), AMD (ROCm), et tout GPU compatible Vulkan. Les modèles vivent dans un seul fichier .gguf. À la fin de cette page, vous aurez tapé les commandes pour le compiler, quantifier, exécuter et servir.

Pourquoi llama.cpp ?
Du C++ sans dépendances. Démarre en moins d’une seconde. Les modèles quantifiés (Q4_K_M) font passer un modèle 7B de ~14 Go (fp16) à ~4 Go. Tourne sur un laptop, un Raspberry Pi, un Mac M-series, ou un GPU 24 Go — même chemin de code. L’implémentation de référence que tout le monde copie.
C’est quoi GGUF ?
GGUF (GPT-Generated Unified Format) est un binaire mono-fichier qui regroupe poids + tokenizer + chat-template + métadonnées. Memory-mappable, versionné, et le format d’échange de facto pour les LLM locaux. Remplace l’ancien format GGML.
La boîte à outils
cli llama-cli — chat interactif / génération one-shot
server llama-server — API HTTP compatible OpenAI
bench llama-bench — benchmarks tokens/sec
quant llama-quantize — fp16 → Q4_K_M, etc.

La vue d’ensemble (à mémoriser)

# 1. cloner & compiler (Mac avec Metal)
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp && cmake -B build -DGGML_METAL=ON && cmake --build build -j

# 2. récupérer un modèle quantifié (HuggingFace)
huggingface-cli download bartowski/Meta-Llama-3.1-8B-Instruct-GGUF \
    Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf --local-dir .

# 3. l’exécuter
./build/bin/llama-cli -m Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf \
    -p "Pourquoi le ciel est-il bleu ?" -n 256 -ngl 99

# 4. le servir sur le port 8080 (compatible OpenAI)
./build/bin/llama-server -m Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf \
    --host 0.0.0.0 --port 8080 -ngl 99

Depuis la source — le chemin canonique

compilation depuis la source
# cloner
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp

# cpu-only (toute plateforme)
cmake -B build
cmake --build build --config Release -j

# Apple Silicon — Metal
cmake -B build -DGGML_METAL=ON
cmake --build build --config Release -j

# NVIDIA — CUDA (nécessite le toolkit CUDA installé)
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release -j

# AMD — HIP/ROCm
cmake -B build -DGGML_HIP=ON -DAMDGPU_TARGETS=gfx1100

# Vulkan — GPU multi-vendeur
cmake -B build -DGGML_VULKAN=ON
Le chemin classique make fonctionne encore mais est déprécié. Restez avec cmake. Les binaires atterrissent dans build/bin/.

./build/bin/llama-quantize llama3-8b-f16.gguf llama3-8b-q4km.gguf Q4_K_M

./build/bin/llama-perplexity -m llama3-8b-q4km.gguf -f wiki.test.raw

llama-cli — one-shot & chat

Le frontend pour la génération directe. Passez un prompt, obtenez des tokens. Ajoutez -cnv pour le mode conversation, ou pipez via stdin. Tous les flags ci-dessous fonctionnent aussi sur llama-server.

Flags les plus utilisés

antisèche des flags
-m chemin vers le modèle .gguf
-p chaîne de prompt
-f lire le prompt depuis un fichier
-n n_predict — max nouveaux tokens (-1 = ∞)
-c taille du contexte (ex. 4096, 8192)
-b batch size pour le prompt (défaut 2048)
-cnv mode conversation / chat
-sys prompt système
-ngl # couches déchargées sur GPU (99 = toutes)
-t threads CPU
--temp température d’échantillonnage
--top-k échantillonnage top-k
--top-p échantillonnage nucleus
--repeat-penalty contre la répétition
--mlock verrouille les poids en RAM
--seed graine RNG (-1 = aléatoire)
génération canonique
llama-cli \
    -m models/Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf \
    -p "Explique la backprop en 2 phrases." \
    -n 256 \
    -c 4096 \
    -ngl 99 \
    --temp 0.7 --top-p 0.9
chat interactif (chat-template Llama-3 auto-détecté)
llama-cli -m model.gguf -cnv \
    -sys "Tu es un assistant laconique." \
    -ngl 99 -c 8192

llama-server — HTTP compatible OpenAI

llama-server expose le même modèle via HTTP, parlant un sous-ensemble de l’API d’OpenAI. Remplacement direct pour beaucoup de SDK. WebUI intégrée à /.

lancer le serveur
llama-server \
    -m models/Meta-Llama-3.1-8B-Instruct-Q4_K_M.gguf \
    --host 0.0.0.0 --port 8080 \
    -c 8192 -ngl 99 \
    --chat-template llama3 \
    --api-key sk-local # auth optionnelle

Endpoints (compatibles OpenAI)

chat — /v1/chat/completions
Accepte { messages: [...] }. Le serveur applique le chat-template inscrit dans le GGUF. Supporte le streaming via "stream": true.
completion — /v1/completions
Complétion de texte brute. { prompt: "..." }. Saute complètement le chat-template.
embeddings — /v1/embeddings
Pour les modèles d’embeddings — démarrez avec --embedding. Renvoie des vecteurs compatibles avec les clients OpenAI.
natifs — /completion, /tokenize, /health
Endpoints natifs llama.cpp avec extras : grammaire, schéma JSON, gestion des slots. Visitez http://localhost:8080 pour la WebUI.
exemple curl — chat completion
curl http://localhost:8080/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "messages": [
            {"role": "system", "content": "You are concise."},
            {"role": "user",   "content": "What is GGUF?"}
        ],
        "temperature": 0.7,
        "max_tokens": 200
    }'
python (SDK openai)
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="sk-local",        # n’importe quoi marche sauf si --api-key est défini
)

resp = client.chat.completions.create(
    model="local",             # ignoré, un seul modèle chargé
    messages=[
        {"role": "system", "content": "You are terse."},
        {"role": "user",   "content": "Why is the sky blue?"},
    ],
    stream=True,
)
for chunk in resp:
    print(chunk.choices[0].delta.content or "", end="", flush=True)

Paramètres d’échantillonnage

Après que le modèle produit des logits, les samplers les réduisent à un seul token. Les défauts sont sensés ; les boutons ci-dessous couvrent 99% des réglages.

température — --temp
Divise les logits par T. 0 = greedy / déterministe ; 0.7 ≈ chat créatif ; 1.0+ = sauvage. Génération de code : 0,0–0,3. Histoire : 0,7–1,0.
top-k — --top-k
Garde uniquement les K tokens de plus haute probabilité. 40 est le défaut classique ; 0 désactive.
top-p (nucleus) — --top-p
Garde le plus petit ensemble de tokens dont la probabilité cumulée ≥ p. Défaut moderne : 0.9–0.95. Mettez à 1.0 pour désactiver.
repeat-penalty — --repeat-penalty
Punit les tokens déjà vus. 1.0 = off. 1.1–1.2 guérit la plupart des boucles. Trop haut → prose non naturelle.
min-p — --min-p
Élimine les tokens en dessous de min_p × max_prob. Plus récent que top-p ; certains utilisateurs le préfèrent.
seed — --seed
Graine RNG. -1 = aléatoire. Fixez une graine pour des exécutions reproductibles.

Avancé — grammaire & décodage spéculatif

forcer une sortie structurée (grammaire GBNF)
# contraindre la sortie à du JSON valide
llama-cli -m model.gguf -p "return JSON: {name, age}" \
    --grammar-file grammars/json.gbnf

# ou grammaire en ligne
llama-cli -m model.gguf -p "yes or no?" \
    --grammar 'root ::= "yes" | "no"'
décodage spéculatif (petit draft + gros vérificateur)
llama-cli -m big-70b.gguf \
    --draft-model small-3b.gguf \
    --draft 5 \
    -p "explain quicksort" -n 512

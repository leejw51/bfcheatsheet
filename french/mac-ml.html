Le Mac comme station de travail ML

Apple Silicon (M1/M2/M3/M4) est une vraie machine ML. La même puce qui fait tourner votre IDE peut entraîner des transformers, fine-tuner des LoRA et servir des modèles 7B localement — sans location de GPU, sans nvidia-smi. Cette page enseigne les trois outils dont vous avez besoin : Python en arm64 PyTorch avec MPS MLX — en tapant.

Mémoire unifiée
Sur Apple Silicon, CPU et GPU partagent la même RAM. Pas de copies cuda, pas de goulot .to(device). Un modèle qui tient dans 16 Go tourne, simplement, que vous l’appeliez depuis PyTorch (MPS) ou MLX.
PyTorch sur Mac
PyTorch complet avec le backend MPS (Metal Performance Shaders). Même API que partout ailleurs — il suffit de remplacer cuda par mps. La plupart des ops fonctionnent ; les rares qui manquent retombent sur CPU.
MLX
Le framework de tableaux d’Apple, conçu de zéro pour Apple Silicon. À la NumPy, lazy par défaut, différentiation automatique, outillage LLM natif. Plus petit, plus rapide, natif Mac.

Ce que vous obtiendrez

1
Un Python arm64 propre. Vérifié natif, dans un environnement virtuel, prêt pour les paquets ML.
2
PyTorch sur le GPU. L’incantation exacte device = torch.device("mps" if ...).
3
MLX hello-world. Une régression linéaire qui utilise mx.value_and_grad et mx.eval de la bonne manière.
4
LLM locaux. Une commande mlx_lm.generate et vous discutez avec un modèle 7B depuis le terminal.
5
Une règle de décision. Tableau côte à côte : quand choisir MLX vs PyTorch.

PyTorch + MPS — le GPU de votre Mac

Depuis PyTorch 1.12 (mai 2022), le backend MPS utilise les Metal Performance Shaders d’Apple pour exécuter les ops tensorielles sur le GPU intégré. Même API PyTorch, parfum Mac.

installation
# la wheel pytorch standard pour mac est compatible mps
pip install torch torchvision

# les nightlies ont souvent une meilleure couverture mps
pip install --pre torch torchvision \
    --index-url https://download.pytorch.org/whl/nightly/cpu

La danse du device

code portable qui choisit le meilleur device
import torch

# marche sur Mac (mps), Linux (cuda), et machines CPU only
device = torch.device(
    "mps"  if torch.backends.mps.is_available() else
    "cuda" if torch.cuda.is_available()        else
    "cpu"
)

x = torch.randn(1024, 1024).to(device)
W = torch.randn(1024, 1024).to(device)
y = x @ W                         # tourne sur le GPU Apple
print(y.device)                  # mps:0

L’échappatoire CPU-fallback

quand une op n’est pas encore implémentée pour MPS
# exportez ceci AVANT que python démarre (ou en Python avant d’importer torch)
export PYTORCH_ENABLE_MPS_FALLBACK=1

# ou, dans le code :
import os
os.environ["PYTORCH_ENABLE_MPS_FALLBACK"] = "1"
import torch
Ce que ça fait : si PyTorch tombe sur une op pas encore portée vers MPS, il retombe silencieusement sur CPU au lieu de planter. Plus lent, mais votre code finit. Depuis PyTorch 2.x la plupart des ops courantes marchent — le fallback est rare.

Petit exemple bout-en-bout

un MLP à 1 couche, entraîné sur du bruit, sur le GPU
import torch
import torch.nn as nn

device = torch.device("mps" if torch.backends.mps.is_available() else "cpu")

X = torch.randn(256, 8, device=device)
y = torch.randn(256, 1, device=device)

model = nn.Sequential(
    nn.Linear(8, 32), nn.ReLU(),
    nn.Linear(32, 1),
).to(device)

opt = torch.optim.Adam(model.parameters(), lr=1e-3)
loss_fn = nn.MSELoss()

for step in range(200):
    pred = model(X)
    loss = loss_fn(pred, y)
    opt.zero_grad(); loss.backward(); opt.step()

print("final loss", loss.item())

MLX — le framework de tableaux d’Apple

MLX est le framework de recherche d’Apple pour le machine learning sur Apple Silicon. Imaginez NumPy croisé avec JAX et l’ergonomie de PyTorch. C’est lazy, ça supporte la différentiation automatique, et ça exploite la mémoire unifiée nativement. Conçu pour le Mac, par l’équipe qui livre le Mac.

installation
# bibliothèque tableaux core
pip install mlx

# helpers LLM (mlx_lm)
pip install mlx-lm
les imports que vous taperez toujours
import mlx.core as mx
import mlx.nn   as nn
import mlx.optimizers as optim

Trois idées à intérioriser

L
Lazy par défaut. Les ops mx construisent un graphe. Rien ne se calcule tant que vous n’appelez pas mx.eval(x) ou que vous ne lisez pas une valeur (ex. print, numpy()). Cela permet à MLX de fusionner + d’ordonnancer.
U
Mémoire unifiée, pas de « device ». Pas de .to(device). Les tableaux vivent en mémoire partagée ; les ops peuvent tourner sur CPU ou GPU automatiquement.
A
Autograd fonctionnel. Utilisez mx.grad(fn) ou mx.value_and_grad(fn) — elles renvoient une fonction qui renvoie les gradients. Pas de .backward().

Hello-world : régression linéaire en MLX

ajuster y = 2x + 1, avec autograd, en 25 lignes
import mlx.core as mx
import mlx.nn   as nn
import mlx.optimizers as optim

# données synthétiques : y = 2x + 1 + bruit
mx.random.seed(0)
X = mx.random.normal((100, 1))
y = 2.0 * X + 1.0 + 0.1 * mx.random.normal((100, 1))

# modèle = une seule couche Linear
model = nn.Linear(1, 1)

def loss_fn(model, X, y):
    pred = model(X)
    return mx.mean((pred - y) ** 2)

loss_and_grad = nn.value_and_grad(model, loss_fn)
opt = optim.SGD(learning_rate=0.1)

for step in range(200):
    loss, grads = loss_and_grad(model, X, y)
    opt.update(model, grads)
    mx.eval(model.parameters(), opt.state)   # forcer le calcul

print(model.weight.item(), model.bias.item())   # ≈ 2.0, 1.0
L’astuce mx.eval : parce que MLX est lazy, les gradients ne sont pas réellement calculés tant que vous ne le demandez pas. Appelez mx.eval(model.parameters(), opt.state) une fois par étape d’entraînement pour les matérialiser — sinon le graphe ne cesse de grossir.

Essayez

10 Construire une fonction qui renvoie le gradient de loss_fn par rapport à son premier argument :

MLX vs PyTorch — quand utiliser lequel

Les deux sont excellents sur Apple Silicon. Ils se chevauchent beaucoup. Voici comment décider.

MLX PyTorch (MPS)
Conçu pour apple silicon Mac uniquement, natif universel Mac · Linux · Windows · CUDA
Style d’API NumPy + autograd fonctionnel (à la JAX) NumPy + nn.Module + .backward()
Exécution Lazy — mx.eval() pour matérialiser Eager (par défaut) ; torch.compile opt-in
Modèle mémoire Unifié — pas de danse de device Device explicite — .to("mps")
Couverture des ops Bibliothèque plus petite — grandit vite Énorme — presque tout marche (quelques trous MPS)
Service LLM mlx_lm — quantification, LoRA, generate, server Utilisez transformers + accelerate
Écosystème Jeune — mlx-community sur HF, outillage qui grandit Massif — torchvision, HuggingFace, lightning, etc.
Performance sur M-series Souvent le plus rapide (natif + fusion lazy) Excellent, parfois plus rapide sur les ops CNN lourdes
Debug Lazy rend le print difficile — appelez eval Eager — print n’importe où, ça marche
Choisissez quand Travail LLM local, recherche ciblée Mac, perf Apple max Portage de code CUDA, modèle HF existant, multi-plateforme
MLX — choisissez quand…
  • Vous faites tourner des LLM de la classe Llama localement
  • Vous voulez LoRA-fine-tuner sur votre laptop
  • Vous aimez le code fonctionnel à la JAX
  • Votre projet est Mac-only — pas de cible CUDA
  • Vous voulez des dépendances minimales + un démarrage rapide
PyTorch — choisissez quand…
  • Vous portez du code PyTorch existant depuis CUDA
  • Vous avez besoin de torchvision / diffusers / transformers
  • Vous déploierez plus tard sur des GPU Linux
  • Vous voulez la plus grande communauté / docs / Stack Overflow
  • Vous avez besoin de couches de niche (ex. ops spectrales complexes)

Même tâche, deux frameworks

MLX matmul
import mlx.core as mx

A = mx.random.normal((1024, 1024))
B = mx.random.normal((1024, 1024))
C = A @ B
mx.eval(C)
print(C.shape)
PyTorch matmul
import torch

dev = "mps" if torch.backends.mps.is_available() else "cpu"
A = torch.randn(1024, 1024, device=dev)
B = torch.randn(1024, 1024, device=dev)
C = A @ B
print(C.shape)
Réponse honnête : pour la plupart des débutants sur Mac, la bonne réponse est « utilisez les deux ». Commencez par PyTorch (tout est documenté), et passez à MLX quand vous voulez la performance Mac maximale pour les LLM.

mlx-lm — LLM locaux en une commande

mlx-lm est un package compagnon qui livre des scripts LLM prêts à l’emploi : generate pour l’inférence, lora pour le fine-tuning, server pour une API HTTP compatible OpenAI. L’organisation mlx-community sur Hugging Face héberge des centaines de modèles pré-convertis et quantifiés.

installation
pip install mlx-lm

Générer depuis n’importe quel modèle mlx-community

    --model mlx-community/Llama-3.2-3B-Instruct-4bit \

    --max-tokens 200

from mlx_lm import load, generate

Astuces de performance pour le ML sur Mac

Quelques habitudes séparent le ML Mac rapide du ML Mac lent. La plupart consistent à respecter la mémoire unifiée et à éviter le travail superflu.

1
Restez en arm64 de bout en bout. Une seule dépendance Rosetta contamine la chaîne. file $(which python) doit signaler arm64. Pareil pour brew --prefix (ce devrait être /opt/homebrew, pas /usr/local).
2
Épinglez un PyTorch récent. Le support MPS s’améliore à chaque release. Visez PyTorch 2.3+. Les nightlies ont souvent une couverture d’ops que la stable n’a pas livrée.
3
Groupez vos mx.eval. Ne l’appelez pas après chaque op — appelez-le une fois par étape d’entraînement sur les choses à matérialiser : mx.eval(model.parameters(), opt.state).
4
Quantifiez pour l’inférence. Le 4-bit divise la mémoire par ~4 avec une perte de qualité négligeable pour le chat. MLX et PyTorch (via bitsandbytes — limité sur Mac) supportent ça. Le chemin le plus simple : tirer des modèles -4bit depuis mlx-community.
5
Surveillez la mémoire unifiée. Le Mac partage la RAM entre CPU et GPU. Un Mac 64 Go peut faire tourner des modèles 70B-ish en 4-bit ; un Mac 8 Go est plus proche du 3B 4-bit. L’onglet Mémoire du Moniteur d’activité est votre ami.
6
Ne luttez pas contre le graphe lazy. En MLX, accumuler de l’état côté Python (ex. losses.append(loss)) dans une boucle sans eval périodique fait gonfler le graphe. mx.eval(loss) avant append.
7
Évitez les allers-retours CPU↔MPS. Chaque .to("cpu") ou .numpy() en PyTorch vous coûte. Construisez toute l’étape d’entraînement on-device, matérialisez aux frontières (logging, checkpoints).
8
Les types flottants comptent. MLX par défaut en float32 ; mx.float16 et mx.bfloat16 divisent la mémoire par deux et accélèrent souvent l’inférence. PyTorch : torch.set_default_dtype(torch.float16) ou autocast.
9
Utilisez torch.compile sélectivement. Sur MPS ça peut aider et ça peut nuire — mesurez. Wrappez seulement la passe forward intérieure et gardez la boucle d’entraînement extérieure en Python plat.
10
Branchez la machine. Apple Silicon est bridé sur batterie — les perf-cores throttle, les fréquences GPU baissent. Pour les benchmarks, branchez et utilisez un profil haute puissance.

Pièges courants

« mon code MPS affiche inf / nan »
Certaines ops (surtout PyTorch ancien + ops spectrales complexes) débordent sur MPS là où elles ne le font pas sur CUDA. Essayez PYTORCH_ENABLE_MPS_FALLBACK=1 et un learning rate plus petit.
« MLX est plus lent que prévu »
Presque toujours : un mx.eval manquant. Vérifiez que vous matérialisez une fois par étape, pas après chaque op de minibatch ou pire — jamais avant la fin.
« pip install torch échoue »
Vous êtes sur du Python x86_64 sous Rosetta. Passez à un Python arm64 (uv / pyenv / installateur python.org) et vérifiez avec file $(which python).
« mon LLM bloque / OOM »
Quantifiez en 4-bit, baissez max-tokens / longueur de contexte, ou prenez un modèle plus petit (3B au lieu de 7B). La mémoire unifiée est partagée avec tout le reste sur le système.

Mini-projet — script d’entraînement MLX complet

Remplissez les blancs pour compléter un programme MLX entièrement autonome : un petit MLP qui apprend une parabole bruitée. Quand tout est vert, cliquez sur Entraîner le modèle et la page va simuler la courbe de loss en train de descendre — la même courbe que vous verriez si vous lanciez ce code sur votre Mac.

complétez le programme

# mlx_quadratic.py — ajuste y = x^2 + bruit sur Apple Silicon
import mlx.core as 
import mlx.nn as nn
import mlx.optimizers as optim

# --- données ---
mx.random.seed(42)
X = mx.random.uniform(low=-1, high=1, shape=(256, 1))
Y = X ** 2 + 0.05 * mx.random.normal((256, 1))

# --- modèle ---
class MLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.l1 = nn.Linear(, 32)
        self.l2 = nn.Linear(32, 32)
        self.l3 = nn.Linear(32, )

    def __call__(self, x):
        x = nn.relu(self.l1(x))
        x = nn.relu(self.l2(x))
        return self.(x)

model = MLP()

# --- loss + autograd ---
def loss_fn(model, X, Y):
    return mx.mean((model(X) - Y) ** 2)

loss_and_grad = nn.(model, loss_fn)
opt = optim.(learning_rate=3e-3)

# --- boucle d’entraînement ---
for step in range(500):
    loss, grads = loss_and_grad(model, X, Y)
    opt.(model, grads)
    mx.(model.parameters(), opt.state)
    if step % 50 == 0:
        print(f"step {step:4d}  loss {loss.item():.4f}")

# --- sauvegarder les poids ---
mx.("model.safetensors", dict(tree_flatten(model.parameters())))

multi­agent.

quatre façons de faire travailler Claude Code en parallèle.
FICHIER : multi-agent-guide.html
RUNTIME : claude-code/cli
~/repo — claude --multi-agent

Une fenêtre Claude est déjà une petite équipe. Quatre fenêtres sont un studio. Ce guide montre les quatre formes que cette équipe peut prendre — et quand chacune est la bonne forme.

Choisissez par coordination, pas par effectif. Tâche annexe → sous-agent. Indépendant → vue d'agent. Coordonné → équipe. Isolation totale → worktree par terminal.

en direct · interactif
i.

approche 01 · délégation en session

Sous-agents.

Un worker spécialiste qui s'exécute dans sa propre fenêtre de contexte, termine une tâche ciblée, et renvoie un résumé à la session appelante. Utilisez-les pour garder le fil principal propre — recherche, vérification, exploration.

« Le but n'est pas un autre agent. Le but est de garder votre conversation principale dégagée. »

Définissez-les comme des fichiers Markdown sous .claude/agents/. Chacun reçoit son propre prompt système, sa allowlist d'outils, et (optionnellement) un modèle plus petit pour que le travail soit peu coûteux.

contexte · isolé parallèle · oui conflits-fichiers · cwd partagé idéal pour · recherche, audits
.claude/agents/researcher.md 
---
name: Researcher
description: Finds prior art & cites sources
model: haiku
tools:
  - bash
  - read
  - webfetch
---

# You are an expert researcher.
# Return citations as a bullet list.
prompt · session principale 
$ /agents
  → gérer les définitions
$ Have @Researcher dig up the rate-limit RFC, then summarize.
→ spawning Researcher in isolated context…
← "Found 3 sources, see notes/rfc-rate-limit.md"
approche 02 · tableau de bord d'arrière-plan

Vue d'agent.

Un tableau de bord de sessions d'arrière-plan indépendantes. Lancez un prompt, partez, jetez un œil à la sortie plus tard. Chaque ligne est son propre processus et survit à la fermeture du terminal.

« Lancer-et-revenir. Utile quand vous avez cinq petits boulots et un seul cerveau. »

Ouvrez avec claude agents. Dans le tableau de bord : Espace pour jeter un œil, Entrée pour s'attacher, ← pour se détacher.

contexte · par session coordination · vous persistance · survit à la fermeture idéal pour · corvées parallèles
shell 
# lancer une session en arrière-plan
$ claude --bg "investigate the flaky e2e test"

# lancer avec un agent personnalisé
$ claude --agent code-reviewer --bg "review PR #1234"

# ouvrir le tableau de bord
$ claude agents

┌─ actifs ──────────────────────────┐
│ ▸ flaky-e2e        running   2m 14s │
│ ▸ pr-1234-review   running     38s │
│ ▸ docs-cleanup     done      11m 02s │
└────────────────────────────────────┘
approche 03 · expérimental · coordonné

Équipes d'agents.

Plusieurs sessions travaillant sur le même but avec une liste de tâches partagée, des messages inter-agents, et un lead qui gère le travail. Plus proche d'un studio que d'un pool de workers.

« Un reviewer pour la sécurité, un pour la performance, un pour les tests — ils se parlent. »

Expérimental et désactivé par défaut. Activez dans settings.json, puis demandez à Claude de monter une équipe pour votre objectif.

contexte · par coéquipier coordination · intégrée statut · expérimental idéal pour · revue multi-axes
settings.json 
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
prompt 
$ Spin up a team to review PR #142.
  Three reviewers, working in parallel:
    1. security  — auth, input, secrets
    2. perf     — hot paths, allocs, queries
    3. tests    — coverage, flakiness, mocks
  Lead consolidates & posts one review.

→ team:r142 spawned with 3 teammates
→ shared task list created (12 items)
← consolidated review posted to #142
approche 04 · la simple · fortement recommandée

Plusieurs terminaux.

Ouvrez un autre onglet de terminal. Lancez claude. Fini — vous avez deux agents. Combinez ça avec les git worktrees et vous obtenez une vraie isolation : chaque agent sur sa propre branche et arborescence, pas de conflits d'édition.

« Le pattern le plus propre est aussi le plus bête : plus d'onglets. »

Idéal quand les tâches sont longues, sans rapport, et que vous voulez une visibilité totale sur ce que fait chaque agent. Fusionnez via PR quand c'est fait.

setup · zéro isolation · complète (worktree) visibilité · totale idéal pour · fonctionnalités parallèles
shell · worktree-par-agent 
# depuis le repo principal
$ git worktree add ../repo-feat-a -b feat-a
$ git worktree add ../repo-feat-b -b feat-b

# terminal 1 — fonctionnalité A
$ cd ../repo-feat-a && claude

# terminal 2 — fonctionnalité B
$ cd ../repo-feat-b && claude

# fusionner plus tard via PR ─ pas de conflits
tmux · variante split-pane 
$ tmux new -s squad
  ctrl-b "       split horizontal
  ctrl-b %       split vertical
  ctrl-b o       cycle panes

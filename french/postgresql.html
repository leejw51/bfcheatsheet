La CLI PostgreSQL — drillez psql jusqu’à ce que ça tienne

psql est le client terminal officiel pour PostgreSQL. Il est livré avec chaque serveur, scriptable, rapide à scripter, et bourré de méta-commandes backslash qui battent n’importe quelle GUI. À la fin de cette page, vous aurez tapé les flags de connexion, les méta-commandes, le SQL, le dump/restore, les one-liners de maintenance, et un petit projet bout-en-bout.

Qu’est-ce que psql ?
Un programme terminal qui parle le protocole filaire PostgreSQL. Il exécute votre SQL mais aussi des commandes backslash comme \dt (lister les tables) et \d users (décrire une table). Complétion par tabulation, historique, scripting via -c / -f, et chargement en masse via \copy.
Pourquoi pas une GUI ?
Parce que les GUI cachent la commande réelle, ne peuvent être scriptées, ne tournent pas sur une machine distante via SSH, et cassent sur les gros résultats. Une fois que \d, \dt+, et \watch vivent dans vos doigts, vous serez plus rapide que n’importe quelle UI sur Terre.
Les amis de psql
pg_dump / pg_restore pour les sauvegardes, pg_isready pour les health checks, createdb / dropdb en wrappers fins, vacuumdb, reindexdb. Tous parlent les mêmes flags de connexion.
Conventions
Port par défaut 5432, BD et utilisateur par défaut correspondent généralement à votre utilisateur OS. Dans psql : \? liste les méta-commandes, \h SELECT montre la syntaxe SQL, \q quitte.

L’antisèche psql que vous mémoriserez

essentiels
# se connecter
psql -h db.example.com -p 5432 -U app -d shop

# dans psql
\l            # lister les bases de données
\c shop       # se connecter à la BD "shop"
\dt           # lister les tables
\d users      # décrire la table "users"
\df           # lister les fonctions
\du           # lister les rôles
\timing on    # chronométrer chaque requête
\watch 2      # relancer la dernière requête toutes les 2s
\copy users TO 'users.csv' CSV HEADER
\q            # quitter

Connexion — entrer dans psql

Six façons de se connecter. Choisissez celle qui correspond à votre situation.

tous les styles de connexion
# 1. défauts — utilise $PGUSER / $PGDATABASE / unix socket
psql

# 2. flags explicites
psql -h db.example.com -p 5432 -U alice -d shop

# 3. forme URI (idéal pour les variables d’env / CI)
psql postgresql://alice:secret@db.example.com:5432/shop

# 4. requête one-shot, sans shell interactif
psql -h host -U alice -d shop -c "SELECT now();"

# 5. exécuter un script .sql
psql -h host -U alice -d shop -f migrate.sql

# 6. avec des variables d’environnement
PGHOST=db PGUSER=alice PGDATABASE=shop psql

Authentification sans taper son mot de passe à chaque fois

~/.pgpass
# ~/.pgpass — une ligne par (host, port, db, user, password)
# wildcards autorisés pour n’importe quel champ
db.example.com:5432:shop:alice:s3cret
*:*:*:postgres:rootpw

# DOIT être chmod 600 sinon psql l’ignore
chmod 600 ~/.pgpass
Définissez votre propre mot de passe depuis psql avec \password. Il hashe côté client donc le plaintext ne quitte jamais le terminal — bien plus sûr que ALTER USER ... PASSWORD '...' qui atterrirait dans le log serveur.

psql postgresql://user:pass@host:/dbname
4 Quitter psql.

7 Afficher l’aide intégrée pour la syntaxe SQL d’un statement.
postgres=# \ SELECT
Continuez à taper : \? liste toutes les méta-commandes. \h liste toutes les instructions SQL. Ce sont vos deux meilleurs amis.

Aide-mémoire des méta-commandes

Les méta-commandes backslash sont spéciales — elles ne sont pas envoyées au serveur, psql les interprète localement. Ajoutez un + pour le mode verbeux, préfixez avec \d pour décrire.

Lister des choses

CommandeDescription
\llister toutes les bases (tailles, propriétaires, encodage)
\c dbnamese connecter à une autre base
\dtlister les tables du schéma courant
\dt+idem, plus taille et description
\dt *.*lister les tables de tous les schémas
\dnlister les schémas
\dvlister les vues
\dilister les index
\dflister les fonctions
\dulister les rôles / utilisateurs
\dxlister les extensions installées

Décrire des choses

CommandeDescription
\d tablecolonnes, types, index, contraintes, FK
\d+ tabletout, plus stockage et cibles statistiques
\d viewmontre le SELECT sous-jacent
\d idxmontre les colonnes de l’index et sa définition
\sf funcmontre la source complète d’une fonction

Inspection — schéma, types, index, plans

Une fois connecté, la première chose à faire c’est de regarder autour. Voici comment savoir ce qu’il y a dans votre base.

SQL DDL — les types que vous utiliserez vraiment

CREATE TABLE — les types riches
CREATE TABLE users (
    id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    email       TEXT        NOT NULL UNIQUE,
    age         INT         CHECK (age >= 0),
    tags        TEXT[],                                -- ARRAY
    profile     JSONB       NOT NULL DEFAULT '{}',    -- JSONB
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- clé étrangère
CREATE TABLE orders (
    id       BIGSERIAL PRIMARY KEY,
    user_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    total    NUMERIC(10,2) NOT NULL
);

Index — trois saveurs qui couvrent la plupart des besoins

patrons d’indexation
-- 1. B-tree simple (le défaut)
CREATE INDEX idx_orders_user ON orders(user_id);

-- 2. contrainte d’unicité via index
CREATE UNIQUE INDEX idx_users_email ON users(lower(email));

-- 3. partiel — n’indexer que les lignes "live"
CREATE INDEX idx_open_orders ON orders(user_id)
    WHERE status = 'open';

-- 4. GIN pour JSONB / tableaux / full text
CREATE INDEX idx_users_profile ON users USING GIN (profile);
CREATE INDEX idx_users_tags    ON users USING GIN (tags);

EXPLAIN — lire votre plan de requête

explain analyze
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE user_id = '...';
--  Index Scan using idx_orders_user on orders  (cost=0.42..8.44 rows=1 width=...)
--    Index Cond: (user_id = '...'::uuid)
--  Planning Time: 0.12 ms
--  Execution Time: 0.04 ms
Lire EXPLAIN : exécution top-down, coût bottom-up. Seq Scan sur une grosse table où vous attendiez un Index Scan = index manquant/inutilisé. Utilisez l’option BUFFERS pour voir les lectures disque vs cache.

UPSERT, CTE, fonctions de fenêtre — les bonnes choses

upsert (INSERT … ON CONFLICT)
INSERT INTO users (id, email)
VALUES ('a-uuid', 'a@x')
ON CONFLICT (id) DO UPDATE SET email = EXCLUDED.email;
CTE + fonction de fenêtre
WITH ranked AS (
    SELECT user_id, total,
           ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total DESC) AS rn
    FROM orders
)
SELECT * FROM ranked WHERE rn = 1;   -- meilleure commande par utilisateur

One-liners utiles

Les requêtes que les vrais DBA en prod collent à 3h du matin. Sauvegardez-les.

Top 5 plus grosses tables

détective de la taille
SELECT
    schemaname || '.' || relname           AS table,
    pg_size_pretty(pg_total_relation_size(relid))  AS total_size,
    pg_size_pretty(pg_relation_size(relid))        AS data_size,
    pg_size_pretty(pg_total_relation_size(relid) - pg_relation_size(relid)) AS idx_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 5;

Requêtes en cours — qui fait quoi maintenant

pg_stat_activity
SELECT pid, usename, application_name, state,
       now() - query_start AS runtime,
       left(query, 100) AS q
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY runtime DESC;

Annuler / tuer une requête qui s’emballe

cancel vs terminate
-- poli : demander à la requête de s’arrêter (garde la connexion ouverte)
SELECT pg_cancel_backend(12345);

-- nucléaire : tuer toute la connexion
SELECT pg_terminate_backend(12345);

Rôles & GRANT

créer un utilisateur en lecture seule
CREATE ROLE readonly LOGIN PASSWORD 'change-me';
GRANT CONNECT ON DATABASE shop TO readonly;
GRANT USAGE ON SCHEMA public TO readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO readonly;

Extensions à connaître

pg_stat_statements
Suivre le runtime + les stats de plan de chaque requête. La première chose qu’un DBA sérieux installe. 
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
pgcrypto
gen_random_uuid(), crypt(), hashing bcrypt des mots de passe. 
CREATE EXTENSION IF NOT EXISTS pgcrypto;
uuid-ossp
Anciens générateurs d’UUID (v1, v3, v4, v5). Sur PG moderne, préférez le gen_random_uuid() de pgcrypto. 
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
pg_trgm
Recherche texte fuzzy / par similarité. Combinez avec un index GIN pour des perfs LIKE '%foo%' fulgurantes. 
CREATE EXTENSION IF NOT EXISTS pg_trgm;
pgvector
Stocker des vecteurs d’embeddings ; <-> = L2, <=> = cosinus. Support natif AI/RAG. 
CREATE EXTENSION IF NOT EXISTS vector;

Sauvegarde & Restauration — pg_dump / pg_restore

Deux binaires, quatre flags. Maîtrisez-les et vos données sont en sécurité.

Les quatre formats

FlagFormatQuand l’utiliser
-FpSQL platpetites BD, lisible, rejouable avec psql -f
-Fccustom (binaire)défaut pour l’ops — compressé, restore parallèle, filtres de tables
-Fttartarball de répertoire non compressé — rare
-Fdrépertoiredump parallèle avec -j N

Recettes

sauvegarde
# une seule BD, format custom (recommandé)
pg_dump -h host -U alice -Fc shop > shop.dump

# SQL plat compressé
pg_dump -h host -U alice shop | gzip > shop.sql.gz

# schéma seul / données seules
pg_dump -Fc --schema-only shop > shop_schema.dump
pg_dump -Fc --data-only   shop > shop_data.dump

# dump répertoire parallèle (4 jobs)
pg_dump -Fd -j 4 -f shop_dir shop

# toutes les bases du cluster (incl. rôles)
pg_dumpall -U postgres > cluster.sql
restauration
# format custom → utiliser pg_restore
createdb shop_new
pg_restore -h host -U alice -d shop_new -j 4 shop.dump

# SQL plat → utiliser psql -f
psql -h host -U alice -d shop_new -f shop.sql

# lister le contenu d’un dump custom (sans restaurer)
pg_restore -l shop.dump | head

# restaurer une seule table
pg_restore -d shop_new -t users shop.dump

Maintenance — gardez PG content

vacuum / analyze / reindex
-- récupérer les lignes mortes + rafraîchir les stats du planificateur
VACUUM ANALYZE;
VACUUM ANALYZE orders;            -- une seule table

-- agressif : verrouille la table, la réécrit (rarement nécessaire)
VACUUM FULL orders;

-- reconstruire les index gonflés
REINDEX TABLE CONCURRENTLY orders;
REINDEX INDEX idx_orders_user;

Règle d’or des restaurations : restaurez toujours dans une base fraîche et vide (createdb shop_new), vérifiez, puis renommez ou réorientez votre app. Ne restaurez jamais par-dessus une BD live.

9 blancs à remplir. Cliquez Tout vérifier, puis Lancer le pipeline. 

# 1. CRÉER LA BASE (depuis votre shell)
createdb -U alice shop

# 2. CONNEXION
psql -U alice -d shop

# 3. SCHÉMA
shop=> CREATE EXTENSION IF NOT EXISTS ;

shop=> CREATE TABLE users (
    id     PRIMARY KEY DEFAULT gen_random_uuid(),
    email TEXT NOT NULL UNIQUE,
    profile  NOT NULL DEFAULT '{}'
);

shop=> CREATE TABLE orders (
    id      BIGSERIAL PRIMARY KEY,
    user_id UUID NOT NULL  users(id) ON DELETE CASCADE,
    total   NUMERIC(10,2) NOT NULL
);

# 4. INSERT (idempotent via UPSERT)
shop=> INSERT INTO users (email) VALUES ('a@x'), ('b@x')
       ON  (email) DO NOTHING;

# 5. REQUÊTE : meilleure commande par utilisateur
shop=> SELECT user_id, total,
              ROW_NUMBER()  (PARTITION BY user_id ORDER BY total DESC) AS rn
       FROM orders;

# 6. INSPECTION
shop=> \

# 7. SAUVEGARDE (retour au shell)
shop=> \
$ pg_dump - shop > shop.dump

PyApp — livrer une app Python en un seul binaire

PyApp (ofek/pyapp) est un runtime Rust qui empaquette une application Python en exécutable autonome. Vous le configurez avec des variables d’environnement à la compilation, lancez cargo build --release, et ça sort un binaire multi-plateforme. Pas de Python requis sur la machine de l’utilisateur — du moins pas un qu’il doive installer lui-même.

Ce que c’est
Un petit programme Rust (~2 Mo) qui sait amorcer une distribution Python et installer votre projet dedans. Compilez-le une fois par cible, livrez le binaire résultant.
Ce que ce n’est pas
Pas un freezer (PyInstaller). Pas un transpileur (Nuitka). Pas un zipapp (shiv / pex). Il délègue le sourçage de Python à une vraie distribution CPython — par défaut téléchargée au premier lancement, optionnellement embarquée.
Pourquoi c’est sympa
Petits binaires par défaut. Vrais builds multi-plateformes (Linux / macOS / Windows). Une vraie installation Python avec toutes ses arêtes vives qui marchent — deps natives, chargement dynamique, tout. Auto-update intégrée.

La forme d’un build

recette
# 1. dire à pyapp ce qu’est votre projet
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0
export PYAPP_EXEC_MODULE=mytool

# 2. compiler le runtime Rust de pyapp
cargo install pyapp --locked --root dist

# 3. ./dist/bin/pyapp est maintenant un binaire qui, au premier lancement,
#    télécharge CPython, pip-install mytool, et lance `python -m mytool`.
mv dist/bin/pyapp dist/bin/mytool

Modèle mental

Le binaire de pyapp contient : (a) le code de démarrage Rust, (b) les métadonnées de votre projet, (c) optionnellement une distribution Python embarquée et votre wheel. Au premier lancement, il pose un Python privé dans un répertoire de cache, pip-install votre projet, et fait un exec dedans. Les lancements suivants vont direct au point d’entrée Python.

Comment ça marche — Rust dehors, Python dedans

L’utilisateur double-clique sur mytool. Le runtime Rust vérifie un répertoire de cache, sourcer Python si besoin, installe le projet une fois, et lance le point d’entrée. Basculez le diagramme ci-dessous entre les deux modes de distribution.

à la demande distribution Python téléchargée au premier lancement. Petit binaire, en ligne une fois requis.
embarqué mettez PYAPP_DISTRIBUTION_EMBED=1 et le tarball Python est cuit dans le binaire. Plus gros fichier, mais marche hors-ligne dès le second lancement.

La séquence du premier lancement

  1. L’utilisateur lance ./mytool.
  2. Le runtime Rust résout le répertoire de cache (~/.local/share/pyapp/ sur Linux, ~/Library/Application Support/pyapp/ sur macOS, %LOCALAPPDATA%\pyapp\ sur Windows).
  3. Si aucun Python n’est installé pour ce triplet (nom, version, distribution), il télécharge depuis PYAPP_DISTRIBUTION_SOURCE ou extrait le tarball embarqué.
  4. Crée un venv (sauf si l’isolation totale est active, auquel cas l’install python est le venv).
  5. pip install votre projet (et les deps tirées).
  6. Stocke un fichier marqueur. Les lancements suivants sautent les étapes 3–5.
  7. Fait un exec sur le point d’entrée configuré et remplace le processus Rust.

Démarrage rapide

Installez Rust (rustup.rs), définissez quelques variables d’env, et lancez cargo install. C’est tout — vous aurez un binaire fonctionnel en moins d’une minute.

A) installer via cargo
# projet publié sur crates.io
export PYAPP_PROJECT_NAME=httpie
export PYAPP_PROJECT_VERSION=3.2.2

cargo install pyapp --locked

# le binaire est dans $CARGO_HOME/bin/pyapp
mv ~/.cargo/bin/pyapp ~/.cargo/bin/httpie
httpie --version
# → 3.2.2
B) compiler depuis la source
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0

git clone https://github.com/ofek/pyapp pyapp-src
cd pyapp-src
cargo build --release

# binaire dans target/release/pyapp
mv target/release/pyapp ~/bin/mytool

Compilation croisée

toutes les plateformes majeures avec un seul arbre source
# ajouter les cibles rust une fois
rustup target add x86_64-unknown-linux-gnu
rustup target add aarch64-apple-darwin
rustup target add x86_64-pc-windows-msvc

# puis pour chaque cible
cargo build --release --target x86_64-unknown-linux-gnu
cargo build --release --target aarch64-apple-darwin
cargo build --release --target x86_64-pc-windows-msvc

# pyapp choisit la distribution CPython correspondante au runtime
Astuce CI : le repo officiel livre un workflow GitHub Actions qui compile pour sept cibles d’un coup — Linux x64/aarch64, macOS x64/aarch64, Windows x64/x86 et musl. Volez-le.

Variables de configuration

Pyapp se configure exclusivement via des variables d’environnement lues à la compilation. Les valeurs sont cuites dans le binaire résultant. Ci-dessous, les plus utiles.

Requises

VariableEffetExemple
PYAPP_PROJECT_NAME Le nom PyPI / wheel de votre projet. Requis. mytool
PYAPP_PROJECT_VERSION Pin de version. Requis sauf si PYAPP_PROJECT_PATH est défini. 1.2.3

Source du projet

VariableEffet
PYAPP_PROJECT_PATH Chemin vers une wheel ou sdist locale à embarquer au lieu de pip-installer depuis PyPI. Utilisez pour les apps privées.
PYAPP_PROJECT_DEPENDENCY_FILE Chemin vers un fichier de constraints (pip -c) cuit dans le binaire. Pin les deps transitives pour la reproductibilité.
PYAPP_PIP_EXTRA_ARGS Args supplémentaires ajoutés à la commande install. ex. --index-url https://my.repo/simple

Point d’entrée (choisissez-en un)

VariableEffetExemple
PYAPP_EXEC_MODULE Lance python -m <module>. mytool
PYAPP_EXEC_SCRIPT Lance un fichier de script livré dans la wheel. /path/to/run.py
PYAPP_EXEC_SPEC Style console-script : module:func. mytool.cli:main
PYAPP_EXEC_NOTEBOOK Chemin vers un fichier .ipynb lancé via Jupyter. analysis.ipynb

Distribution Python

VariableEffetDéfaut
PYAPP_PYTHON_VERSION Version CPython à utiliser. Mappée à une release indygreg/python-build-standalone. dernière stable
PYAPP_DISTRIBUTION_SOURCE Surcharger l’URL d’où Pyapp télécharge Python. Utilisez pour les miroirs air-gapped. release github
PYAPP_DISTRIBUTION_EMBED Si 1/true : cuit Python dans le binaire. Pas de téléchargement au premier lancement. 0
PYAPP_DISTRIBUTION_VARIANT Saveur de distribution : install_only (défaut) ou install_only_stripped pour des binaires plus petits. install_only

Réglages de comportement

VariableEffet
PYAPP_FULL_ISOLATION Sauter la couche venv ; installer directement dans le Python décompressé. Démarrage plus rapide, moins de déduplication.
PYAPP_UPGRADE_VIRTUALENV Forcer la mise à jour pip / setuptools dans le venv avant d’installer le projet.
PYAPP_EXPOSE_ALL_COMMANDS Permettre aux utilisateurs d’invoquer n’importe quel console script du projet, pas seulement celui configuré.
PYAPP_ALLOW_UPDATES Mettre 0 pour désactiver la sous-commande self update.
PYAPP_PASS_LOCATION Passer le chemin du binaire en argv[0] à Python — utile pour les outils qui se ré-exécutent.
Les valeurs booléennes sont permissives : 1, true, True, TRUE, yes activent toutes un flag. Tout le reste, y compris non défini, c’est « off ».

Points d’entrée — module vs script vs spec

Pyapp peut lancer votre app de quatre façons. Choisissez celle qui correspond à comment vous la lanceriez en développement.

PYAPP_EXEC_MODULE
Comme python -m mytool. Votre projet doit avoir un __main__.py. Le plus idiomatique. 
export PYAPP_EXEC_MODULE=mytool
# layout du package : mytool/__main__.py
PYAPP_EXEC_SPEC
Syntaxe console-script : pkg.module:function. Pareil que les entrées [project.scripts] que vous écririez dans pyproject.toml. 
export PYAPP_EXEC_SPEC=mytool.cli:main
PYAPP_EXEC_SCRIPT
Chemin vers un fichier .py (relatif à la racine de la wheel). Lance python script.py. Old-school. 
export PYAPP_EXEC_SCRIPT=mytool/runner.py
PYAPP_EXEC_NOTEBOOK
Lancer un notebook Jupyter. Pyapp installera jupyter à côté de votre projet. Utile pour livrer un outillage interactif aux non-codeurs. 
export PYAPP_EXEC_NOTEBOOK=demo.ipynb
Forwarding d’arguments : tous les args que vous passez au binaire sont transmis à l’interpréteur Python tels quels. ./mytool --debug -- --help marche comme attendu pour EXEC_MODULE et EXEC_SPEC.

Sources de distribution — embarqué vs à la demande

Deux stratégies pour sourcer CPython. Choisissez selon que vos utilisateurs sont en ligne, à quel point la taille du binaire est sensible, et si vous tolérez un délai unique.

à la demande (défaut)
  • Le binaire reste petit (~2 Mo).
  • Premier lancement télécharge CPython (~30 Mo) depuis indygreg/python-build-standalone.
  • Les lancements suivants utilisent l’install en cache.
  • Plusieurs binaires pyapp peuvent partager le même interpréteur en cache.
  • Réseau requis au premier lancement.
# rien à régler — c’est le défaut
export PYAPP_PYTHON_VERSION=3.12
cargo build --release
embarqué
  • Tarball Python cuit dans le binaire (~30 Mo au total).
  • Aucun accès réseau nécessaire à aucun moment.
  • Reproductible : Python identique sur chaque machine.
  • Les deps du projet font toujours pip-install au premier lancement sauf si vous les embarquez aussi.
  • Utilisez pour l’outillage CI, les envs air-gapped, ou quand la latence du premier lancement est inacceptable.
export PYAPP_PYTHON_VERSION=3.12
export PYAPP_DISTRIBUTION_EMBED=1
cargo build --release

Pin un tarball spécifique

surcharger l’URL d’où pyapp télécharge Python
# mirror ou release vendée
export PYAPP_DISTRIBUTION_SOURCE=https://internal.example.com/python-3.12-x64.tar.gz
export PYAPP_DISTRIBUTION_PATH_PREFIX=install/
export PYAPP_DISTRIBUTION_PYTHON_PATH=install/bin/python3

cargo build --release
Variante stripped : mettez PYAPP_DISTRIBUTION_VARIANT=install_only_stripped pour enlever les symboles de debug et raboter ~10 Mo d’un build embarqué.

Auto-gestion — mytool self ...

Chaque binaire Pyapp livre une commande self qui gère les upgrades et désinstallations. UX power-user sans CI requise.

commandes
mytool self updaterécupère la dernière version depuis PyPI et reconstruit le venv en place
mytool self restorebalaye le venv et réinstalle (récupère d’une dep cassée)
mytool self removesupprime le répertoire de cache ; le binaire est intact
mytool self metadataimprime la config Pyapp cuite dans le binaire
mytool self pythontomber dans le REPL Python embarqué — pratique pour debugger
contrôler self-update
# à la compilation, désactiver les updates entièrement
export PYAPP_ALLOW_UPDATES=0

# au runtime, pointer les utilisateurs vers un index privé
export PIP_INDEX_URL=https://my.repo/simple
mytool self update

# pin sur une version spécifique
mytool self update 1.4.7
Pourquoi c’est génial : les utilisateurs peuvent brew install ou winget install votre outil une fois, puis mytool self update indéfiniment sans relancer le gestionnaire de paquets système. Le runtime Rust ne change jamais ; seul le projet Python dans le cache change.

vs Alternatives

Pyapp se trouve dans un espace très encombré. Voici comment il se compare aux concurrents évidents.

Outil Approche Binaire unique Multi-plateforme depuis un seul host Vrai CPython Auto-update Compromis
PyApp Loader Rust → distro CPython oui oui (cargo) oui oui téléchargement au premier lancement ou embed plus gros
PyInstaller freeze : bytecode + libs en archive oui non — doit compiler par OS copie embarquée non faux positifs antivirus, gros binaires
Nuitka Python → C → binaire natif oui non recompilé non build lent, capricieux sur les deps
shiv / pex wrapper zipapp .pyz a besoin de Python oui l’utilisateur doit avoir Python non pas vraiment autonome
Briefcase (BeeWare) bundles natifs par plateforme .app, .msi, .deb non embarqué non visé GUI, plus de setup
conda-pack tarball d’un env conda non non oui non nécessite l’activation conda

Quand choisir quoi

choisissez PyApp si…
vous voulez un binaire unique, êtes content du téléchargement au premier lancement, avez besoin de builds multi-plateformes depuis une matrice CI unique, et aimez l’idée du self update gratuit.
choisissez PyInstaller si…
vous devez tourner sans réseau ni écriture cache, pouvez compiler sur chaque OS cible, et que ça ne vous dérange pas de chasser les bugs occasionnels d’imports manquants.
choisissez shiv / pex si…
la machine destination a déjà Python, vous voulez une sortie vraiment multi-plateforme, et vous n’avez besoin que d’un wheel-bundle mono-fichier (.pyz).

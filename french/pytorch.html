Tenseurs

Tableaux N-dimensionnels avec support GPU et autograd. L'atome de PyTorch.

🧪 Bac à sable des shapes

Autograd

Différenciation automatique. La raison d'être de PyTorch.

🔄 Flux de gradient

Passe avant → calcule les sorties. Passe arrière ← propage ∂L/∂x via la règle de la chaîne.

activation avant gradient remontant
Piège  Les gradients s'accumulent entre les appels à backward(). Appelez toujours optimizer.zero_grad() (ou passez set_to_none=True) avant chaque step, sinon votre mise à jour mélangera les gradients de différents batches.
requires_grad & backward
Marquez les entrées dont vous voulez les gradients, puis appelez .backward() sur une perte scalaire.

y.backward()
print(x.grad)   # tensor([2., 4.])

Function personnalisée
Définissez votre propre forward/backward pour des opérations spécifiques.
class MyReLU(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x):
        ctx.save_for_backward(x)
        return x.clamp(min=0)

    @staticmethod
    def backward(ctx, grad_out):
        (x,) = ctx.saved_tensors
        return grad_out * (x > 0)

contextes autocast

GradScaler (fp16)
Met la perte à l'échelle pour que les petits gradients survivent en fp16 → annule l'échelle avant step().
scaler = torch.amp.GradScaler("cuda")

for x, y in loader:
    optimizer.zero_grad(set_to_none=True)
    with torch.autocast("cuda", dtype=torch.float16):
        loss = loss_fn(model(x), y)
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)             # pour le clipping de gradient
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()

Distribué : DDP vs FSDP

DDP — tout répliquer
Chaque rank détient le modèle complet + l'état de l'optimiseur. Gradients all-reduced. Optimal quand le modèle tient.
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group("nccl")

torch.cuda.set_device(local_rank)

model = DDP(model.to(local_rank), device_ids=[local_rank])
sampler = torch.utils.data.distributed.DistributedSampler(ds)
# Lancer :  torchrun --nproc_per_node=8 train.py
FSDP — tout sharder
Shards les params, grads et l'état d'optimiseur entre les ranks (ZeRO-3). Pour les modèles trop gros pour un GPU.
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP, MixedPrecision,
)
from torch.distributed.fsdp.wrap import size_based_auto_wrap_policy

mp = MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.float32,
)
model = FSDP(
    model,
    auto_wrap_policy=size_based_auto_wrap_policy,
    mixed_precision=mp,
    device_id=torch.cuda.current_device(),
)
quand choisir lequel
Règles de décision rapides.
# DDP — le modèle tient sur 1 GPU.
#   + le plus simple, le plus rapide par step
#   + chaque rank a le modèle complet (eval/save faciles)
#   - mémoire : O(modèle) par rank

# FSDP — modèle trop gros pour 1 GPU.
#   + mémoire : O(modèle / world_size)
#   + se compose avec bf16 / activation checkpointing
#   - all-gather supplémentaires → un peu plus de comm

# Tensor-parallel + FSDP pour les modèles à plusieurs milliards de params.

Pièges courants

zero_grad oublié : les gradients s'accumulent à chaque appel backward(). Réinitialisez toujours.
view vs reshape : après transpose / permute le tenseur n'est pas contigu — view échouera. Utilisez reshape ou .contiguous() d'abord.
CrossEntropyLoss prend des logits, pas la sortie d'un softmax. Appliquer softmax avant donne des gradients incorrects.
BatchNorm en eval() : les stats running sont utilisées, pas celles du batch. Oublier model.eval() à l'inférence rend les sorties dépendantes de la composition du batch.
tensor.item() ne fonctionne que pour les tenseurs à 1 élément et force une synchronisation CPU — à éviter dans les boucles serrées.

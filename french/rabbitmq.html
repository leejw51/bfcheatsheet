1. Qu'est-ce qu'un courtier de messages ?

Un courtier de messages (message broker) se place entre les producteurs et les consommateurs. Le producteur dépose un message dans le courtier et retourne à son travail. Le consommateur le récupère plus tard, à son propre rythme. Les deux n'ont jamais besoin d'être en ligne en même temps, ni de connaître l'adresse de l'autre, ni de s'accorder sur la vitesse de traitement.

Pourquoi les files d'attente comptent

  • Découplage : changez les consommateurs sans changer les producteurs.
  • Tampon : lissez les pics ; la file absorbe les rafales.
  • Fiabilité : si un consommateur plante, le message est toujours là.
  • Diffusion : un événement déclenche de nombreuses réactions indépendantes.
Modèle mental : les producteurs écrivent des cartes postales, le courtier est la poste, les files sont des boîtes aux lettres, les consommateurs lisent à leur rythme.

2. Concepts fondamentaux d'AMQP

RabbitMQ implémente AMQP 0-9-1. Le vocabulaire est restreint mais précis.

  • Producer publie des messages.
  • Consumer s'abonne et traite les messages.
  • Queue tampon nommé qui stocke les messages.
  • Exchange le routeur ; les producteurs publient vers les échanges, jamais directement vers les files.
  • Binding règle qui relie un échange à une file (souvent avec une clé).
  • Routing Key chaîne que le producteur attache à un message ; l'échange l'utilise pour décider où livrer.
  • Virtual host (vhost) espace de noms partitionnant utilisateurs, échanges et files ; isolation logique.
  • Connection connexion TCP. Channel connexion virtuelle multiplexée à l'intérieur.

Flux en un coup d'œil

producer
  └─ publish(routing_key) ─▶ exchange
                              ├─ binding A ─▶ queue 1 ─▶ consumer
                              └─ binding B ─▶ queue 2 ─▶ consumer

3. Plongée dans les types d'échanges

Direct

Achemine les messages dont la clé de routage correspond exactement à une clé de liaison. À utiliser pour : livraison ciblée, files de travail simples, routage de commandes.

binding: queue.email  ⇐ key="email"
publish key="email"   → queue.email
publish key="sms"     → dropped (or unrouted)

Fanout

Ignore la clé de routage. Diffuse vers chaque file liée. À utiliser pour : événements pub/sub, invalidation de cache, notifications « avertir tout le monde ».

publish ANY key → all bound queues receive it

Topic

Achemine selon un motif. Les clés de routage sont des mots séparés par des points ; les motifs de liaison peuvent utiliser :

  • * correspond à exactement un mot.
  • # correspond à zéro ou plusieurs mots.
binding "orders.*.us"   matches  orders.new.us, orders.cancel.us
                        misses   orders.us, orders.new.eu, orders.a.b.us
binding "orders.#"      matches  orders, orders.new, orders.new.us
binding "*.error.*"     matches  app.error.db, web.error.api

Headers

Achemine selon les en-têtes du message, pas la clé de routage. Le mode x-match: all exige que chaque en-tête corresponde ; x-match: any en exige au moins un.

binding headers: {format: pdf, type: report, x-match: all}
header {format: pdf, type: report}     → match
header {format: pdf}                    → no match

Quand utiliser quoi

TypeChoisissez quand…
directVous voulez du point-à-point ou une égalité stricte sur une clé unique.
fanoutChaque abonné a besoin du même événement. Le plus simple à raisonner.
topicÉvénements hiérarchiques : region.service.severity. Les abonnés veulent des sous-ensembles.
headersLe routage dépend de plusieurs attributs qui ne tiennent pas dans une clé chaîne.

Échange par défaut

L'échange anonyme "" est un échange direct spécial dans lequel chaque file est implicitement liée par son nom. C'est pourquoi basic_publish(exchange='', routing_key='hello') va directement à la file nommée hello.

Échange de lettres mortes (DLX)

Une file peut être configurée avec x-dead-letter-exchange. Les messages rejetés, expirés ou en débordement sont republiés vers le DLX, où une autre liaison les stationne dans une file de lettres mortes pour inspection ou nouvelle tentative.

4. Accusés de réception

Auto-ack vs manual ack

  • Auto-ack (auto_ack=True) : le courtier considère le message comme livré dès qu'il sort. Rapide et risqué. Si le consommateur plante en plein traitement, le message est perdu.
  • Manual ack (auto_ack=False) : le consommateur doit appeler basic_ack(delivery_tag) après succès. S'il meurt avant, le courtier retransmet à la reconnexion.

nack et reject

  • basic_nack(requeue=true) — le remettre dans la file.
  • basic_nack(requeue=false) — le supprimer ou l'envoyer au DLX si configuré.
  • basic_reject est la forme single-message de nack.

Indicateur de retransmission

Quand un message est retransmis, le courtier positionne redelivered=true. Les consommateurs doivent traiter cela comme un signal de prudence : idempotence, clés de déduplication, compteurs de réessais dans les en-têtes.

5. Durabilité — ce qui survit à un redémarrage

Trois choses indépendantes doivent toutes être positionnées pour qu'un message survive à un crash du courtier :

  1. File durable : queue_declare(durable=True). Sinon la file elle-même disparaît.
  2. Message persistant : delivery_mode=2 (ou PERSISTENT_TEXT_PLAIN). Sans cela, le message reste en RAM seulement.
  3. Échange durable (si vous en dépendez) : exchange_declare(durable=True).
Piège : la persistance est en best-effort, pas un fsync synchrone par message. Utilisez les publisher confirms (confirm_delivery) si vous devez savoir qu'un message est sur disque avant d'accuser réception en amont.

Publisher confirms vs transactions

Les transactions AMQP existent mais sont lentes. Les publisher confirms sont asynchrones et plus rapides de plusieurs ordres de grandeur : le courtier vous dit « je l'ai bien reçu » avec une trame ack.

6. Prefetch et QoS — distribution équitable

Par défaut, RabbitMQ distribue les messages aux consommateurs en tourniquet. Si un consommateur est lent et un autre rapide, le lent s'engorge tandis que le rapide reste oisif. Prefetch corrige cela.

channel.basic_qos(prefetch_count=1)

Cela dit au courtier : « ne me donne pas de nouveau message tant que je n'ai pas accusé réception du précédent ». Désormais le consommateur rapide tire plus de travail, le lent en tire moins. Augmentez (par ex. 10–50) pour des tâches courtes et uniformes.

Règle d'optimisation empirique

  • Travaux longs et hétérogènes → prefetch=1.
  • Travaux courts et uniformes → prefetch plus élevé (moins de surcoût aller-retour).
  • Mesurez avec la profondeur de file et l'utilisation des consommateurs.

7. Clustering et files quorum

Un cluster RabbitMQ partage utilisateurs, vhosts, échanges et liaisons à travers les nœuds. Les files sont par défaut une entité à un seul maître ; le maître réside sur un nœud et peut être miroité ou remplacé par un quorum.

Files miroir classiques

Héritage. Réplique l'état vers les suiveurs. Remplacée dans les déploiements modernes par les files quorum.

Files quorum

  • Construites sur le protocole de consensus Raft.
  • Cohérence forte : une écriture n'est acquittée qu'une fois qu'une majorité de réplicas l'a.
  • Survit proprement aux pannes de nœuds. Recommandé par défaut pour les données importantes.
  • Compromis : latence plus élevée que les files classiques, plus d'E/S disque.

Streams (bonus)

Files en mode log-append pour des charges analytiques à haut débit. Protocole différent, modèle d'accès différent (offsets, pas de consommation destructive).

8. Patrons courants

Files de travail (distribution de tâches)

Une file, plusieurs workers, ack manuel, prefetch=1. À utiliser pour le redimensionnement d'images, l'envoi d'e-mails, l'encodage vidéo.

producer ─▶ "" ─▶ queue:tasks ─▶ worker1
                              └▶ worker2
                              └▶ worker3

Publish/Subscribe

Échange fanout, chaque abonné reçoit chaque message via sa propre file.

producer ─▶ fanout:events ─┬─▶ queue:audit       ─▶ logger
                            ├─▶ queue:cache_invl  ─▶ cache
                            └─▶ queue:notify      ─▶ pusher

Plus de patrons

Routage par topic

Événements hiérarchiques. Différentes files s'abonnent à différentes formes.

topic:logs
  binding "*.error.*"  ─▶ queue:alerts
  binding "db.#"       ─▶ queue:dba
  binding "#"          ─▶ queue:audit

RPC (requête/réponse)

Le producteur positionne reply_to (une file temporaire et exclusive) et correlation_id. Le consommateur publie la réponse vers reply_to. Le producteur apparie les réponses par correlation_id.

client → publish(reply_to=Q1, correlation_id=abc)
server → consume → publish(exchange='', routing_key=Q1, correlation_id=abc)
client → matches abc, resolves promise
Astuce : mettez un TTL sur la file de réponse pour éviter que des clients fuités n'accumulent des files orphelines.

Chapitre1

Possession et emprunt

le cœur de rust

Chaque valeur en Rust a un unique propriétaire (owner). Lorsque le propriétaire sort de portée, la valeur est libérée — sa mémoire restituée, ses fichiers fermés, ses verrous relâchés. C'est le RAII, et c'est imposé statiquement.

Vous pouvez déplacer (move) la possession (la transférer), ou bien emprunter (borrow) (prêter une référence). L'emprunt suit deux règles : les références partagées &T peuvent être nombreuses simultanément, mais les références mutables &mut T doivent être exclusives. Le compilateur le prouve à la compilation.

Transfert de possession (move) versus emprunt (borrow)
valeur possédée référence empruntée invalidée

Le déplacement classique

main.rs
fn main() {
    let s = String::from("hello");
    let t = s;                 // s est déplacée dans t
    // println!("{}", s);      // ❌ erreur : emprunt d'une valeur déplacée : `s`
    println!("{}", t);         // ✅
}

Emprunts partagés et mutables

borrow.rs
fn len(s: &String) -> usize { s.len() }

fn push_bang(s: &mut String) { s.push('!'); }

fn main() {
    let mut greeting = String::from("hi");
    let n = len(&greeting);          // emprunt partagé, OK
    push_bang(&mut greeting);        // emprunt exclusif, OK
    println!("{} ({} chars)", greeting, n);
}
Les deux règles À tout instant, vous pouvez avoir soit une référence mutable, soit n'importe quel nombre de références partagées. Les références doivent toujours être valides (pointer vers une donnée vivante). C'est tout le borrow checker.
Chapitre2

Durées de vie (Lifetimes)

nommer la durée d'un emprunt

Une durée de vie (lifetime) est juste un nom pour la région du code où une référence est valide. La plupart des durées de vie sont inférées (on parle d'élision). Vous n'avez besoin de les écrire que lorsque le compilateur ne peut pas déterminer de quelle entrée provient une référence retournée.

Une référence ne peut pas survivre à la valeur qu'elle emprunte
durée de vie 'a du propriétaire durée de vie 'b de la référence

Annoter les références retournées

lifetimes.rs
// Le `&str` retourné doit vivre aussi longtemps que la plus courte de x et y.
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() >= y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string is long");
    let result;
    {
        let s2 = String::from("short");
        result = longest(&s1, &s2);
        println!("{}", result);   // ✅ les deux sont vivantes
    }
    // println!("{}", result);     // ❌ s2 libérée, référence pendante
}
Règles d'élision Si une fonction prend une référence et en retourne une, Rust suppose qu'elles partagent la même durée de vie. Si &self figure parmi les entrées, la sortie emprunte de &self. Vous n'annotez que lorsque ces valeurs par défaut sont incorrectes.
Chapitre3

Traits et génériques

comportement, abstrait

Un trait est un ensemble de méthodes que des types peuvent implémenter. Un générique est une fonction ou un type paramétré par un type. Ensemble, ils offrent un polymorphisme ad-hoc sans héritage : « n'importe quel T qui peut être affiché », « n'importe quel itérateur de i32 ».

Un trait, deux impls, une fonction générique
traits.rs
trait Greet {
    fn hello(&self) -> String;
    fn shout(&self) -> String {                 // méthode par défaut
        self.hello().to_uppercase()
    }
}

struct Cat;
struct Robot { id: u32 }

impl Greet for Cat   { fn hello(&self) -> String { "meow".into() } }
impl Greet for Robot { fn hello(&self) -> String { format!("unit-{}", self.id) } }

fn welcome<T: Greet>(t: &T) {
    println!("{} ({})", t.hello(), t.shout());
}

fn main() {
    welcome(&Cat);
    welcome(&Robot { id: 7 });
}
Dispatch statique vs dynamique fn welcome<T: Greet> génère une copie par type concret (coût nul). fn welcome(t: &dyn Greet) prend un fat pointer et dispatche à l'exécution — légèrement plus lent, mais permet de stocker un Vec<Box<dyn Greet>> hétérogène.
Chapitre4

Result, Option et l'opérateur ?

les erreurs comme valeurs

Rust n'a pas d'exceptions. L'échec est encodé dans le type. Option<T> signifie peut-être un T ; Result<T, E> signifie un T ou une erreur E. L'opérateur ? les enchaîne, en retournant immédiatement s'il n'y a pas de valeur.

L'opérateur ? : court-circuit sur erreur, déballage sur Ok
parse.rs
use std::num::ParseIntError;

fn double_first(input: &str) -> Result<i32, ParseIntError> {
    let first = input.split(',').next().unwrap_or("0");
    let n: i32 = first.trim().parse()?;   // retour anticipé si Err
    Ok(n * 2)
}

fn main() {
    match double_first("21, hello") {
        Ok(v)  => println!("got {v}"),
        Err(e) => eprintln!("bad: {e}"),
    }
}

Option, brièvement

option.rs
fn first_word(s: &str) -> Option<&str> {
    s.split_whitespace().next()
}

fn main() {
    let n = first_word("hello world").map(str::len).unwrap_or(0);
    println!("{n}");                     // 5
}
Chapitre5

Filtrage par motifs

match, if let, while let

match est exhaustif : le compilateur refusera de vous laisser ignorer une variante. Les motifs peuvent déstructurer des tuples, des structs, des enums, des slices, des références, et même filtrer avec des conditions if.

match.rs
enum Shape {
    Circle { r: f64 },
    Rect   { w: f64, h: f64 },
    Triangle(f64, f64, f64),
}

fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle { r }            => std::f64::consts::PI * r * r,
        Shape::Rect   { w, h }         => w * h,
        Shape::Triangle(a, b, c) if a + b > *c => {
            let p = (a + b + c) / 2.0;
            (p * (p - a) * (p - b) * (p - c)).sqrt()
        }
        _ => f64::NAN,                                  // triangle illégal
    }
}
If let, while let Quand vous ne vous intéressez qu'à une seule variante, if let Some(x) = opt { … } est plus net qu'un match complet. while let répète tant qu'un motif filtre — parfait pour dépiler une pile.
Chapitre6

Itérateurs et fermetures

paresseux, fluide, rapide

Les itérateurs sont paresseux : rien ne se passe avant que vous le demandiez. Les fermetures (closures) capturent leur environnement et se déclinent en trois saveurs : Fn (emprunte), FnMut (emprunte mutablement), FnOnce (consomme). Le compilateur choisit automatiquement la plus permissive.

iter.rs
fn main() {
    let nums = vec![1, 2, 3, 4, 5, 6];

    let sum_of_even_squares: i32 = nums
        .iter()
        .filter(|&&n| n % 2 == 0)
        .map(|&n| n * n)
        .sum();

    println!("{sum_of_even_squares}");   // 56

    // fermetures et captures
    let factor = 10;
    let scale = |x: i32| x * factor;     // emprunte factor (Fn)
    println!("{}", scale(3));            // 30
}
Coût nul Les chaînes d'itérateurs idiomatiques compilent vers le même assembleur que des boucles écrites à la main. Le style fluide est essentiellement gratuit — parfois plus rapide, car LLVM démontre davantage d'invariants.
Chapitre7

Concurrence

threads, send/sync, canaux

Deux traits marqueurs gardent la concurrence. Send signifie qu'une valeur peut être déplacée entre threads en toute sûreté. Sync signifie que &T peut être partagé entre threads en toute sûreté. La plupart des types le sont tous deux ; Rc<T> et RefCell<T> non — le compilateur vous empêche d'introduire des courses critiques.

Un canal : producteur → consommateur, avec transfert de possession
channel.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    for i in 0..4 {
        let tx = tx.clone();
        thread::spawn(move || {
            tx.send(format!("hello from worker {i}")).unwrap();
        });
    }
    drop(tx);                     // ferme l'émetteur original

    while let Ok(msg) = rx.recv() {
        println!("{msg}");
    }
}

État mutable partagé

mutex.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let c = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut n = c.lock().unwrap();
            *n += 1;
        }));
    }
    for h in handles { h.join().unwrap(); }
    println!("count = {}", *counter.lock().unwrap()); // 10
}
Chapitre8

Async / Await

concurrence coopérative

Marquer une fonction async transforme son corps en une machine à états implémentant Future. L'appeler ne fait rien jusqu'à ce que vous fassiez .await ou la confiez à un runtime (Tokio, async-std, smol). Chaque .await est un point de suspension.

async_main.rs
use tokio::time::{sleep, Duration};

async fn fetch(name: &str) -> String {
    sleep(Duration::from_millis(50)).await;
    format!("hello, {name}")
}

#[tokio::main]
async fn main() {
    let (a, b) = tokio::join!(fetch("alice"), fetch("bob"));
    println!("{a} | {b}");
}
Async est une couleur Vous pouvez appeler des fonctions sync depuis de l'async, mais pas l'inverse sans un runtime. Choisissez un runtime tôt (Tokio est la valeur par défaut) et tenez-vous-y.
Chapitre9

Cargo et modules

organisation de projet

Cargo est le système de build, le gestionnaire de paquets, le lanceur de tests et le harnais de benchmarks réunis en un. Une crate est une unité de compilation. Un module est un espace de noms à l'intérieur d'une crate. Le système de fichiers reflète l'arborescence des modules.

tree
my_app/
├── Cargo.toml          # manifeste : nom, version, dépendances
├── Cargo.lock          # versions exactes résolues (à committer pour les binaires)
└── src/
    ├── main.rs         # point d'entrée binaire  →  crate `my_app`
    ├── lib.rs          # racine de la bibliothèque  →  crate `my_app`
    └── parser/
        ├── mod.rs      # module `parser`
        └── ast.rs      # module `parser::ast`
Cargo.toml
[package]
name    = "my_app"
version = "0.1.0"
edition = "2024"

[dependencies]
serde   = { version = "1", features = ["derive"] }
tokio   = { version = "1", features = ["full"] }
anyhow  = "1"

[dev-dependencies]
proptest = "1"
commands
cargo new my_app             # générer le squelette d'une crate binaire
cargo run                    # compiler et exécuter
cargo test                   # lancer les tests unitaires + intégration + doc
cargo clippy -- -D warnings  # le linter réputé tranchant
cargo fmt                    # rustfmt, dogmatique et définitif

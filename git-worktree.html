01 The problem worktrees solve

You're deep in a feature. Production catches fire. You need to fix main right now. Without worktrees, every option hurts.

Pain 1 Context switch tax

You stash, switch to main, fix, push, switch back, pop the stash, hope nothing conflicts. WIP merge conflicts on a stash pop are a special kind of misery.

Pain 2 Parallel work

Your build takes 6 minutes. While CI runs on branch A, you want to write code on branch B without rebuilding. One checkout = one branch at a time.

Pain 3 Reviewing PRs

You want to git checkout pr/42 and run it locally — but you also don't want to torch your in-progress changes on the current branch.

The worktree answer

Add a second working directory pinned to a different branch. Both share the same .git object database — fetches, refs, hooks, config — but their files on disk are independent. Switch by cd, not by checkout.

# before: stash dance
git stash push -m wip
git checkout main
# ...fix, commit, push...
git checkout feature/long-thing
git stash pop   # 🤞

# after: a parallel working dir
git worktree add ../hotfix main
cd ../hotfix
# ...fix, commit, push, then:
cd ../cro-economy-service
git worktree remove ../hotfix

02 Anatomy: shared .git, separate working dirs

A worktree is a checkout that lives outside the main repo's directory but is still tracked by the main repo's .git.

What is shared

Object database (.git/objects): commits, trees, blobs.

Refs: branches, tags, remotes — one source of truth.

Config & hooks: defined once in the main repo.

What is separate

Working tree: the actual files checked out on disk.

Index: each worktree has its own staging area.

HEAD: each worktree points at its own commit/branch.

Where the metadata lives

The main repo records each linked worktree under .git/worktrees/<name>/. Each linked worktree has a .git file (not directory) that points back to that metadata via gitdir:.

04 Common patterns

Three workflows that pay for the keystrokes immediately.

Hotfix drop everything

Production is down. You're 14 commits deep on a feature.

git fetch origin
git worktree add -b hotfix/login \
  ../hotfix origin/main
cd ../hotfix
# fix, commit, push
cd -
git worktree remove ../hotfix

Review run the PR locally

Reviewing PR #42 without nuking your in-flight work.

git fetch origin pull/42/head:pr/42
git worktree add ../review pr/42
cd ../review
# run tests, poke around
cd -
git worktree remove ../review

Long test CI on the side

Run an integration suite on release/v2 while you keep coding.

git worktree add ../e2e release/v2
cd ../e2e
make integration-test &
cd -
# keep working in main repo

05 Gotchas worth memorizing

Almost every worktree bug you'll hit is one of these five.

! Detached HEAD on add

If you pass a commit/tag instead of a branch, the new worktree is on a detached HEAD. To create a fresh branch, use -b.

# detached:
git worktree add ../tmp v1.4.0
# branch:
git worktree add -b fix-1.4 ../tmp v1.4.0

! Branch already checked out

Git refuses to check out a branch that's already in another worktree (it would let two HEADs trample one ref). Either pick a different branch or use --force if you really know.

fatal: 'feature-x' is already checked out at '/path/A'
# override (rarely a good idea):
git worktree add --force ../B feature-x

! Manual rm leaves stale metadata

If you rm -rf a worktree directory, .git/worktrees/<name> stays. git worktree list still shows it as prunable. Run git worktree prune.

git worktree list
# shows: ../old  (prunable: gitdir is missing)
git worktree prune -v

! Submodules & per-worktree config

Submodules don't auto-init across worktrees — run git submodule update --init in each. Some config (sparse-checkout, hooks) is per-worktree only when you opt in via --worktree.

! Removing the wrong one

remove refuses if the worktree has uncommitted changes. Don't reach for --force reflexively — first cd in and check.

· Lock to protect

If a worktree lives on a removable disk or a slow NFS mount, git worktree lock tells git not to prune it just because the path is unreachable.

Three ways to juggle work. They're not interchangeable.

07 Bare repo + worktrees: the monorepo trick

Some teams skip the "main checkout" entirely and treat the repo as a folder of worktrees.

The setup

git clone --bare git@github.com:org/repo.git repo.git
cd repo.git
git worktree add ../repo/main main
git worktree add ../repo/feature-x feature-x
git worktree add ../repo/release release/v2

Layout:

repo.git/        # shared object DB
repo/
  main/          # worktree on main
  feature-x/     # worktree on feature-x
  release/       # worktree on release/v2

Why people like it

No "primary" worktree to confuse anyone — every branch is a sibling directory. Tooling (IDE, devcontainers, build caches per branch) maps cleanly onto folders.

Trade-off: a few git commands feel mildly weird from inside a bare repo. Most folks alias cd repo/main and forget the bare exists.

How an input embedding learns

The tokenizer hands the model a list of integers like [464, 3797, 3332]. The input embedding turns each ID into a learned, dense vector of size $d_{model}$. But it isn't designed by hand — it is a plain weight matrix that starts as random noise and is trained by backprop, exactly like every other parameter. This guide follows The Annotated Transformer to show precisely how that table is built, fed, and learned.

01 The one-line idea

what it is

A learnable lookup

A matrix $E \in \mathbb{R}^{V \times D}$ — one trainable row per vocabulary entry. A token's embedding is simply its row, fetched by index.

how it learns

Gradient descent

Rows begin random (Xavier init). Each training step, the loss gradient nudges only the rows that appeared in the batch. Repeat millions of times and the geometry organizes itself.

"We use learned embeddings." The paper is explicit: the input and output token embeddings are not fixed — they are weights. There is no separate "embedding training" phase; the table is optimized jointly with attention and the feed-forward layers under the single language/translation loss.

01·b Where it sits

shapes · the embedding adds a feature axis D
token ids [B, T] → embed × √D [B, T, D] + PE[:T] [T, D] → encoder [B, T, D]

$B$ = batch, $T$ = sequence length, $D = d_{model}$ (512 in the base model). The tokenizer makes the [B, T] of IDs; the embedding looks up a D-vector per ID and scales it; then positional encoding adds order.

The Embeddings module

One trainable row per token

In the Annotated Transformer the embedding is a tiny module wrapping nn.Embedding. The forward pass is a row lookup, then a scale by $\sqrt{d_{model}}$. The weight matrix is the only thing that learns here.

the Embeddings class · verbatim 
class Embeddings(nn.Module):
    def __init__(self, d_model, vocab):
        super(Embeddings, self).__init__()
        self.lut = nn.Embedding(vocab, d_model)   # weight: [V, D]
        self.d_model = d_model
    def forward(self, x):
        return self.lut(x) * math.sqrt(self.d_model)
what each symbol means
  • vocab · V  vocabulary size — the number of rows. (First arg to nn.Embedding.)
  • d_model · D  embedding width — 512 in the base Transformer.
  • lut  the trainable weight matrix [V, D] — a literal "look-up table".
  • x  integer token IDs of shape [B, T]; out comes [B, T, D].
Why × √d_model? The paper says "in the embedding layers, we multiply those weights by √d_model." It boosts the embedding so its magnitude is comparable to the unit-scale positional encoding added next — otherwise position would drown out content. With D = 512 that factor is √512 ≈ 22.6.
It's often the biggest layer. Parameter count is exactly V × D. For a 32k vocab at D = 512 that's ≈ 16.4M weights — all in this one table, all learned by gradient descent.

The heart · how the rows learn

Training the table

The embedding has no special training procedure — it rides the model's normal loop. A batch flows forward to a loss; loss.backward() sends gradients back into every weight, including the embedding rows that were looked up; the optimizer steps. Crucially, only the rows that appeared in the batch get a gradient — the update is sparse.

04·b Why the update is sparse

Because lookup equals a one-hot matmul, $e_{id} = \mathbf{1}_{id}^{\top} E$, the gradient w.r.t. the table is zero everywhere except row id. Each looked-up token gets the incoming gradient deposited onto its own row, summed over its occurrences in the batch.

$$\frac{\partial L}{\partial E_{j}} \;=\; \sum_{t\,:\,ids_t = j} \frac{\partial L}{\partial e_{t}} \qquad\Longrightarrow\qquad \frac{\partial L}{\partial E_{j}} = 0 \;\text{ for unused rows } j$$

04·c The loss & optimizer that drive it

label-smoothed loss · backward · step 
# target distribution is smoothed: confidence on the true token, ε spread over the rest
criterion = LabelSmoothing(size=V, padding_idx=0, smoothing=0.1)   # KLDivLoss inside

def step(out, targets, norm):
    log_probs = generator(out)                 # Linear → log_softmax  [.., V]
    loss = criterion(log_probs, targets) / norm
    loss.backward()                          # ∇ flows into E's used rows
    opt.step()                                # Noam-scheduled Adam update
    opt.optimizer.zero_grad()                 # clear grads for next batch
    return loss

The rate climbs for 4000 steps, then decays as 1/√step — gentle early updates.

Initialization matters. Before step 1, every weight with more than one dimension — the embedding table included — is set with nn.init.xavier_uniform(p). So the rows start as small random vectors; all of the structure you'll see in tab 05 is learned from there.
What training produces

Meaning becomes geometry

After enough steps, tokens that play similar roles have been nudged toward similar rows — because predicting well rewards that. Cosine similarity between two rows now measures how alike the model thinks they are, and directions gain meaning (the famous king − man + woman ≈ queen). Pick a query token and watch its neighbours light up.

Each dot is a token's vector (projected to 2-D). The chosen query glows; lines reach to others, brighter the more similar. The ranked list is cosine similarity, highest first.

Same-cluster words score near +1; unrelated clusters fall toward 0 or go negative. None of this is hand-coded — the table arrives here purely from the prediction loss of tab 04.
Untrained = noise. At xavier_uniform init the rows are random, so this map would be a shapeless cloud. word2vec and GloVe learned such spaces with dedicated objectives; a Transformer's embedding reaches the same kind of structure as a by-product of next-token prediction.
One matrix, three jobs

Shared weights

The paper goes further: the input embedding, the output (target) embedding, and the pre-softmax linear that produces logits all share the same weight matrix. Training one trains all three — fewer parameters, and the in/out geometry stays consistent.

Verbatim: "We share the same weight matrix between the two embedding layers and the pre-softmax linear transformation." Logits are then h @ E.T — literally "how much does this hidden state point along each token's row?"
the output side · Generator 
class Generator(nn.Module):
    # "Define standard linear + softmax generation step."
    def __init__(self, d_model, vocab):
        super(Generator, self).__init__()
        self.proj = nn.Linear(d_model, vocab)   # weight: [V, D] — tie to E
    def forward(self, x):
        return F.log_softmax(self.proj(x), dim=-1)

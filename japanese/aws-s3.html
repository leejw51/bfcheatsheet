AWS S3 CLI — ls から put-bucket-policy まで

2つのCLIファミリー、1つのストレージサービス。aws s3 は日常作業向けの フレンドリーで高レベルなラッパー — copy、sync、presign。aws s3api は REST API の低レベルなミラーで、バージョニング、ライフサイクル、マルチパートアップロード、 バケットポリシーなどの細かな操作向け。両方をタイピングで攻略しよう。

aws s3 — 高レベル
ファイルシステム感覚のコマンド: ls, cp, mv, sync, rm, mb, rb, presign。 マルチスレッドアップロード、大きなファイルのマルチパート、リトライなど自動で賢くこなしてくれる。 90%の場面で最初に手が伸びるのはこちら。
aws s3api — 低レベル
S3 APIの動詞ごとに1つのサブコマンド: list-objects-v2, head-object, put-bucket-versioning, create-multipart-upload, … aws s3 が公開していない機能が必要なとき、または --query / jq に流す生のJSONが欲しいときに使う。

千回はタイプすることになる S3 URI

# scheme   bucket name      key (object path inside the bucket)
s3://my-bucket/photos/2026/cat.jpg

# a "folder" is just a key prefix — it doesn't really exist as an object
s3://my-bucket/photos/2026/
注意: S3のキーは本物のファイルシステムではない。「ディレクトリ」 photos/2026/ はそのプレフィックスで始まるキーを持つオブジェクトが 存在することで初めて見える。空のフォルダは / で終わる0バイトのオブジェクト。

IAM: 最もよく使う4つの S3 アクション

アクション許可される内容リソース例
s3:ListBucket指定したプレフィックスでキーを一覧表示arn:aws:s3:::my-bucket
s3:GetObjectオブジェクトをダウンロードarn:aws:s3:::my-bucket/*
s3:PutObjectオブジェクトをアップロードarn:aws:s3:::my-bucket/*
s3:DeleteObjectオブジェクトを削除（または delete-marker を付与）arn:aws:s3:::my-bucket/*
バケットアクション vs オブジェクトアクション: s3:ListBucket は バケットの ARN（/* なし）が対象。s3:GetObject は中のオブジェクト（/* 付き）が対象。これを混同するのが、 新しいポリシーで「AccessDenied」が出る原因No.1。

一覧と確認 — ls, head-object, list-objects-v2, --query

クラウドストレージの第一の戒め: 触る前にバケットの中身を知ること。 両方のCLIファミリーがオブジェクトを一覧表示できる。s3api は --query による JMESPath で切り出せるJSONを返す。

JMESPath 速習 — --query フラグ

すべての aws コールはJSONを返す。--query 'EXPR' はそれにJMESPathをクライアント側で実行し、--output でフォーマットされる前に処理する。 

# Just the bucket names
$ aws s3api list-buckets --query 'Buckets[].Name'

# Objects bigger than 1 MiB (note the backticks for numbers)
$ aws s3api list-objects-v2 --bucket B \
    --query 'Contents[?Size > `1048576`].Key'

# Newest 5 objects
$ aws s3api list-objects-v2 --bucket B \
    --query 'reverse(sort_by(Contents, &LastModified))[:5].[Key,LastModified]' \
    --output table

# Total size of a prefix (sum)
$ aws s3api list-objects-v2 --bucket B --prefix logs/ \
    --query 'sum(Contents[].Size)'
5 list-buckets からバケット名のみを返す。
$ aws s3api list-buckets --query 'Buckets[].'
6 JMESPath: 1 MiB より厳密に大きいオブジェクトのみ残す。キーワードを入力する。
--query 'Contents[Size > `1048576`].Key'

アップロード、ダウンロード、Sync — cp, mv, sync, exclude/include

この4つの動詞（cp, mv, sync, rm）が、S3 への/からの バイト移動の95%を担う。--delete をタイプする前に --exclude / --include の順序と --dryrun をマスターしよう。

cp — 単一ファイルまたは再帰

# Upload
$ aws s3 cp ./report.pdf s3://my-bucket/reports/

# Download
$ aws s3 cp s3://my-bucket/reports/report.pdf ./

# Recursive — entire directory
$ aws s3 cp ./build/ s3://my-bucket/site/ --recursive

# With storage class, encryption, content-type, and metadata
$ aws s3 cp big.bin s3://my-bucket/data/big.bin \
    --storage-class STANDARD_IA \
    --sse aws:kms --sse-kms-key-id alias/data \
    --content-type application/octet-stream \
    --metadata ingest=2026-05-20,team=ml \
    --acl private
1 ローカルディレクトリ全体を S3 のプレフィックスにアップロードする。
$ aws s3 cp ./build/ s3://my-bucket/site/ --
2 アップロードを Infrequent-Access としてタグ付け。ストレージクラスを入力する。
$ aws s3 cp file.txt s3://b/k --storage-class

mv — コピー + ソース削除

# Promote a build artefact from staging to release prefix
$ aws s3 mv s3://b/staging/v1.tar.gz s3://b/release/v1.tar.gz

# Move local → S3 (deletes the local file on success)
$ aws s3 mv ./done.zip s3://b/archive/

sync — ディレクトリ単位の必殺技

# Mirror local → S3, *adding* and *updating* but never deleting
$ aws s3 sync ./public/ s3://my-site/

# Mirror with deletion: S3 ends up exactly matching local
$ aws s3 sync ./public/ s3://my-site/ --delete

# Skip junk files, only ship JS and CSS
$ aws s3 sync ./dist/ s3://my-site/ \
    --exclude "*" \
    --include "*.js" \
    --include "*.css"

# See what would change without doing it
$ aws s3 sync ./public/ s3://my-site/ --delete --dryrun
順序が重要: --exclude / --include は左から右に評価され、最後にマッチしたものが勝つ。 まず --exclude "*" で除外して、必要なものを再度 include しよう。
3 ローカルディレクトリを S3 にミラーし、削除されたローカルファイルが S3 でも削除され、.tmp ファイルはスキップされるようにする。
$ aws s3  local/ s3://bucket/prefix/ -- --exclude "*.tmp"
4 破壊的な sync が何をするかを実行せずにプレビューする。
$ aws s3 sync ./public/ s3://site/ --delete --

      rm — 慎重に、特に --recursive で

      # A single object
$ aws s3 rm s3://b/old.log

# Whole prefix — confirms nothing!
$ aws s3 rm s3://b/tmp/ --recursive

# Selective: nuke logs/ but keep .json files
$ aws s3 rm s3://b/logs/ --recursive --exclude "*.json"
      バージョニングの警告: バージョニングが有効なバケットで aws s3 rm --recursive を実行すると削除マーカーが 付くだけ — 古いバージョンはまだ残っている。本当に容量を解放するには、 s3api 経由で明示的にバージョンを削除する必要がある。

      s3api パワーツール — S3 API アクションごとに1動詞

      フレンドリーな aws s3 コマンドが力尽きたら、 s3api に降りよう。出力は常にJSON — --query や jq でパイプできる。

      オブジェクトレベルの操作

      # Upload from local file with explicit content-type and metadata
$ aws s3api put-object \
    --bucket my-bucket \
    --key data/blob.bin \
    --body ./blob.bin \
    --content-type application/octet-stream \
    --metadata sha256=abcd...,team=ml

# Download to a file
$ aws s3api get-object --bucket my-bucket --key data/blob.bin out.bin

# Server-side copy (no bytes through your laptop)
$ aws s3api copy-object \
    --bucket dest-bucket \
    --key archive/blob.bin \
    --copy-source my-bucket/data/blob.bin

# Delete one specific version
$ aws s3api delete-object \
    --bucket my-bucket --key data/blob.bin \
    --version-id 3HL4kqtJlcpXroDTDmJ+rmSpXd3dIbrHY
      1 キーをサーバー側で srcB/srcK から dstB/dstK へコピーする。動詞とソースフラグを入力する。
      $ aws s3api  \
    --bucket dstB --key dstK \
    -- srcB/srcK

      オブジェクトのタグ付け

      $ aws s3api put-object-tagging \
    --bucket my-bucket --key data/x.bin \

$ aws s3api get-object-tagging --bucket my-bucket --key data/x.bin

      バケットとポリシー — mb, rb, policy, CORS, lifecycle, public-access block

      バケットレベルの設定はセキュリティとコストの生死を分けることが多い。 これらのコマンドでバージョニング、パブリックアクセスブロック、CORS、ライフサイクルルール、 バケットポリシーを設定する。

      バケットの作成と削除

      # Create. Region matters for the LocationConstraint (us-east-1 is implicit / "none")
$ aws s3 mb s3://my-new-bucket --region us-east-1

# Delete. --force also empties it (high-level only).
$ aws s3 rb s3://my-new-bucket --force

# s3api equivalents (more knobs)
$ aws s3api create-bucket --bucket my-new-bucket \
    --create-bucket-configuration LocationConstraint=eu-west-1

$ aws s3api delete-bucket --bucket my-new-bucket
      1 us-east-1 に cat-photos という新しいバケットを作る。
      $ aws s3  s3://cat-photos --region us-east-1
      2 空でないバケットを一気に削除する。フラグを選ぶ。
      $ aws s3 rb s3://cat-photos --

      すべてのパブリックアクセスをブロック（最初にやろう！）

      $ aws s3api put-public-access-block \
    --bucket my-bucket \
    --public-access-block-configuration \
        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

$ aws s3api get-public-access-block --bucket my-bucket

      Object Lock — 一度書いて何度も読む

      # Enabled at bucket creation only — can't be turned on later
$ aws s3api create-bucket --bucket vault \
    --object-lock-enabled-for-bucket

# Default retention: 7 years in COMPLIANCE mode (immutable for everyone)
$ aws s3api put-object-lock-configuration \
    --bucket vault \
    --object-lock-configuration \
    '{"ObjectLockEnabled":"Enabled","Rule":{"DefaultRetention":{"Mode":"COMPLIANCE","Years":7}}}'
      GOVERNANCE vs COMPLIANCE: GOVERNANCE は権限のある IAM ユーザーが s3:BypassGovernanceRetention で保持期間を上書きできる。COMPLIANCE は 保持期間が終わるまで誰も（ルートユーザーも含む）阻止する。

      暗号化とストレージクラス — SSE, KMS, IA, Glacier

      コスト（および取り出しレイテンシ）のためにストレージクラスを、コンプライアンスのために 暗号化モードを選ぶ。両者は独立しており、すべてのクラスがすべての暗号化モードをサポートする。

      サーバー側暗号化（SSE）

      モード--server-side-encryptionキー管理者監査 / KMS の利用
      SSE-S3AES256S3なし
      SSE-KMSaws:kmsKMS（あなたの CMK）CloudTrail が各 Decrypt を記録
      SSE-KMS DSSEaws:kms:dsseKMS、二重暗号化規制対象ワークロード
      SSE-C顧客提供キーをヘッダーであなたキーも自分で保管
      なし——2023年以降デフォルトで禁止（S3 が AES256 を自動適用）
       
      # SSE-S3 (AES-256, S3 manages the key)
$ aws s3 cp f.bin s3://b/k --sse AES256

# SSE-KMS with a specific key
$ aws s3 cp f.bin s3://b/k \
    --sse aws:kms \
    --sse-kms-key-id alias/data-prod

# Default bucket encryption (everything new gets KMS even if uploader didn't ask)
$ aws s3api put-bucket-encryption --bucket B \
    --server-side-encryption-configuration \
    '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"aws:kms","KMSMasterKeyID":"alias/data-prod"}}]}'

      2 SSE-S3 でアップロード。--sse の値は何？
      $ aws s3 cp f.bin s3://b/k --sse

      マルチパートと署名付き URL — 大きなファイルと共有リンク向け

      高レベルの aws s3 cp は約 8 MB を超えるファイルで自動的に マルチパートを使う。プログラムからアップロードを駆動するときは、s3api のプリミティブを使う。

      マルチパートアップロード — 4ステップのダンス

      ## 1. Start — get an UploadId
$ UID=$(aws s3api create-multipart-upload \
    --bucket B --key big.bin \
    --query 'UploadId' --output text)

## 2. Upload each part (5 MB minimum, except the last)
$ ETAG1=$(aws s3api upload-part \
    --bucket B --key big.bin \
    --part-number 1 --upload-id $UID \
    --body part1.bin \
    --query 'ETag' --output text)
#  …repeat for parts 2..N…

## 3. Complete — give S3 the part list and it stitches them server-side
$ aws s3api complete-multipart-upload \
    --bucket B --key big.bin --upload-id $UID \
    --multipart-upload '{"Parts":[{"ETag":"...","PartNumber":1},...]}'

      練習ドリル — S3 CLI 連射

      最もよくタイプされるコマンドとフラグを20個以上カバー。 aws s3 と aws s3api のミックス。

      1 身元確認 — アカウントIDとARNを返す呼び出しは？
      $ aws sts
      2 1コマンドだけデフォルト以外のプロファイルを使う。
      $ aws s3 ls -- myprofile
      3 1コマンドだけリージョンを上書きする。
      $ aws s3 ls s3://b -- eu-west-1
      4 ローカルツリーを S3 へ再帰コピーする。
      $ aws s3 cp ./site/ s3://my-site/ --
      5 削除付きで sync する。
      $ aws s3 sync ./pub/ s3://my-site/ --
      6 sync ですべて除外し、PNGファイルだけ再度 include する。
      $ aws s3 sync ./img/ s3://b/img/ \
    --exclude "*" \
    -- "*.png"
      7 1日有効な署名付きURL（秒数）を生成する。
      $ aws s3 presign s3://b/k --expires-in
      8 新しいバケットを作る。高レベルの動詞を入力する。
      $ aws s3  s3://photos --region us-east-1
      9 空でないバケットを削除する。
      $ aws s3  s3://photos --force
      10 SSE-S3 暗号化の値。
      $ aws s3 cp f.txt s3://b/k --sse
      11 SSE-KMS 暗号化の値。
      $ aws s3 cp f.txt s3://b/k --server-side-encryption
      12 Intelligent-Tiering のストレージクラス。
      $ aws s3 cp f.txt s3://b/k --storage-class
      13 メタデータのみ取得する s3api 動詞（本体なし）。
      $ aws s3api  --bucket B --key K
      14 プレフィックス配下を最大1000オブジェクトでページ送り。
      $ aws s3api  --bucket B --prefix p/ --max-keys 1000
      15 バージョニングを有効化する。ステータス文字列を入力する。
      $ aws s3api put-bucket-versioning --bucket B --versioning-configuration Status=
      16 ローカルJSONファイルからバケットポリシーを適用する。file:// URI を入力する。
      $ aws s3api put-bucket-policy --bucket B --policy
      17 すべてのパブリックアクセスをブロック — put- の後の動詞を入力する。
      $ aws s3api put- --bucket B \
    --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
      18 JMESPath: 一覧表示された各オブジェクトから Key フィールドだけを射影する。
      --query 'Contents[].'
      19 詰まったマルチパートアップロードをキャンセルする。
      $ aws s3api -multipart-upload --bucket B --key K --upload-id $UID
      20 あるバケットから別のバケットへキーをサーバー側でコピーする。
      $ aws s3api copy-object --bucket dest --key newkey -- srcBucket/srcKey
      21 オブジェクトの特定のバージョンを1つ削除する。
      $ aws s3api delete-object --bucket B --key K -- 3HL4kqtJ
      22 オブジェクトのすべてのバージョン（と delete-marker）を表示する。
      $ aws s3api  --bucket B --prefix data/blob.bin

      $ aws s3api get-bucket- --bucket B

      ボスドリル — エンドツーエンドのS3パイプライン

      1つのシナリオ、10個の空欄。規制対象データセット用の新バケットを構築する: 作成、ロックダウン、バージョニング有効化、暗号化アップロード、一覧、共有リンク発行、 ライフサイクル設定、古いバージョンのクリーンアップ。すべての空欄は実際のコマンドかフラグ。

      パイプライン: コールドストレージ研究データセット

      コンプライアンス要件: KMS暗号化、パブリックアクセス禁止、バージョニング、365日で失効、 90日後にGlacierへ階層化。research プロファイルで eu-west-1 リージョンから操作する。 

      # 1. このシェルセッション全体に正しいプロファイル + リージョンを設定
$ export AWS_=research
$ export AWS_REGION=eu-west-1

# 2. research IAM ユーザーとして実際にログインしているか確認
$ aws sts 

# 3. バケットを作成
$ aws s3  s3://lab-genomics-2026 --region eu-west-1

# 4. パブリックアクセスをすぐにロック
$ aws s3api put- \
    --bucket lab-genomics-2026 \
    --public-access-block-configuration \
        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

# 5. 誤削除を復旧できるようバージョニングを有効化
$ aws s3api put-bucket-versioning --bucket lab-genomics-2026 \
    --versioning-configuration Status=

# 6. KMS でデフォルト暗号化、--sse を「忘れた」アップロードでも暗号化される
$ aws s3api put-bucket-encryption --bucket lab-genomics-2026 \
    --server-side-encryption-configuration \
    '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"","KMSMasterKeyID":"alias/lab-data"}}]}'

# 7. 最初のバッチを明示的な暗号化 + IA ストレージクラスでアップロード
$ aws s3 cp ./batch-001/ s3://lab-genomics-2026/raw/2026-05/ \
    --recursive \
    --sse aws:kms --sse-kms-key-id alias/lab-data \
    --storage-class 

# 8. 確認 — MiB 単位のサイズで一覧表示
$ aws s3 ls s3://lab-genomics-2026/raw/ --recursive -- --summarize

# 9. PI が1ファイルをダウンロードできる24時間有効な共有リンクを生成
$ aws s3  s3://lab-genomics-2026/raw/2026-05/cohort-A.tsv --expires-in 86400

# 10. ライフサイクルをインストール: 90日でGLACIERへ、365日で失効、30日で古いバージョン削除
$ aws s3api put-bucket- \
    --bucket lab-genomics-2026 \
    --lifecycle-configuration file://lc.json

CLI v2

セットアップ

AWS CLI v2 をインストールし、認証情報を設定し、自分のアカウントに接続できるか確認する。所要 5 分。

インストール

macOS
# pkg インストーラー
curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "AWSCLIV2.pkg"
sudo installer -pkg AWSCLIV2.pkg -target /

# または Homebrew 経由
brew install awscli

aws --version
# aws-cli/2.15.x Python/3.x Darwin/24.x
Linux
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install

# Apple Silicon / ARM 用
curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
Windows
# MSI インストーラー
msiexec.exe /i https://awscli.amazonaws.com/AWSCLIV2.msi

# または Powershell で:
winget install -e --id Amazon.AWSCLI

設定

aws configure (デフォルトプロファイル)
aws configure
# AWS Access Key ID: AKIA...
# AWS Secret Access Key: ...
# Default region name: us-east-1
# Default output format: json
名前付きプロファイル
多くのチームは複数アカウント (dev, staging, prod) を持つ。アカウントごとにプロファイルを使う。
aws configure --profile prod
# 各コマンドで:
aws s3 ls --profile prod
# または環境変数で:
export AWS_PROFILE=prod
ディスク上のファイル
# ~/.aws/credentials
[default]
aws_access_key_id = AKIA...
aws_secret_access_key = ...

[prod]
aws_access_key_id = AKIA...
aws_secret_access_key = ...

# ~/.aws/config
[default]
region = us-east-1
output = json

[profile prod]
region = us-west-2
output = json
SSO (人間には推奨)
ノート PC に長期キーを置かない。フェデレーションされた、有効期限付きの認証情報。
aws configure sso
aws sso login --profile prod

# ~/.aws/config 内:
[profile prod]
sso_session = my-sso
sso_account_id = 123456789012
sso_role_name = PowerUser
region = us-west-2
環境変数 (CI / スクリプト)
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...        # STS 用
export AWS_REGION=us-east-1
export AWS_PROFILE=prod              # またはプロファイルを使う
確認
aws sts get-caller-identity
# {
#   "UserId": "AIDA...",
#   "Account": "123456789012",
#   "Arn": "arn:aws:iam::123...:user/jane"
# }
やってはいけない: ~/.aws/credentials をコミットする、Slack にキーを貼る、Docker イメージに焼き込む、Lambda の環境変数に保存する。代わりに IAM ロール (EC2 インスタンスロール、Lambda 実行ロール、EKS IRSA) を使うこと — 短期認証情報を自動で発行してくれる。

IAM の基本

アイデンティティ (誰が) + ポリシー (何を) + リソース (どこで)。すべての API コールはポリシーに対して認可される。

概念

項目意味
User個人 (またはサービス) に紐づく長期アイデンティティ。アクセスキー、コンソールパスワード、MFA を持てる。
Role引き受け可能なアイデンティティ。短期認証情報 (STS) を返す。EC2、Lambda、EKS、クロスアカウントアクセス、フェデレーションで使う。
PolicyJSON ドキュメント。アイデンティティポリシーはユーザー/ロールに付与。リソースポリシーはリソース (例: S3 バケットポリシー) に付与。
Groupポリシーを共有するユーザーの袋。ネスト不可。
STSSecurity Token Service。AssumeRole、AssumeRoleWithWebIdentity 等で短期認証情報を発行する。

最小限の S3 ポリシー

単一バケットの読み取り専用
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::my-bucket",
      "arn:aws:s3:::my-bucket/*"
    ]
  }]
}
特定のプレフィックスへ書き込み
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:PutObject", "s3:DeleteObject"],
    "Resource": "arn:aws:s3:::my-bucket/uploads/*"
  }]
}
信頼ポリシー (EC2 が引き受けるロール)
ロールに付与する。誰が私を引き受けられるかを宣言する。
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"Service": "ec2.amazonaws.com"},
    "Action": "sts:AssumeRole"
  }]
}

S3 の基本

  
バケットはリージョン内のフラットな名前空間。キーはパスのように見えるがフォルダではない。以下は週次で叩くコマンド集。


  
バケット

  

    

      
作成 / 一覧 / 削除

      
# 作成 — バケット名はグローバル
aws s3 mb s3://my-bucket --region us-east-1

aws s3 ls
aws s3 ls s3://my-bucket
aws s3 ls s3://my-bucket --recursive --human-readable --summarize

# 削除 (空である必要がある)
aws s3 rb s3://my-bucket
# または強制削除:
aws s3 rb s3://my-bucket --force

    

    

      
cp — 単一ファイル

      
aws s3 cp ./report.csv s3://my-bucket/reports/

# ダウンロード
aws s3 cp s3://my-bucket/reports/report.csv ./

# バケット間コピー
aws s3 cp s3://src/key s3://dst/key

# メタデータ + ストレージクラス付き
aws s3 cp ./big.tgz s3://my-bucket/ \
  --storage-class STANDARD_IA \
  --metadata "team=ml,owner=jane" \
  --content-type application/gzip

    

    

      
sync — ディレクトリ

      
差分のあるファイルだけコピーする。デフォルトではローカルに無いリモートファイルは削除しない — それが必要なら --delete を渡す。

      
aws s3 sync ./build/ s3://my-bucket/v1/ \
  --exclude "*.tmp" \
  --exclude ".git/*" \
  --include "*.html" \
  --delete \
  --dryrun

    

    

      
rm

      
aws s3 rm s3://my-bucket/old.csv

# プレフィックス全体 (再帰)
aws s3 rm s3://my-bucket/tmp/ --recursive

# フィルタ付き
aws s3 rm s3://my-bucket/ --recursive \
  --exclude "*" --include "*.log"

    

    

      
署名付き URL

      
AWS 認証情報なしで誰でも使える時間制限付きの署名 URL。デフォルトの有効期限は 3600 秒。

      
aws s3 presign s3://my-bucket/report.csv \
  --expires-in 900

# アップロード (PUT) 用 — s3api が必要:
aws s3 presign s3://my-bucket/upload.bin \
  --expires-in 600
# 注意: aws s3 presign は GET URL を発行する。
# PUT 署名には SDK か s3api コマンドを使うこと。

S3 上級

  
バージョニング、暗号化、ライフサイクル、ホスティング。バケットを「ゴミ捨て場」からインフラに変える機能群。


  
バージョニング

  

    

      
有効化 / 一覧 / 削除

      
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled

# バージョン一覧
aws s3api list-object-versions --bucket my-bucket --prefix reports/

# 特定バージョンを削除
aws s3api delete-object \
  --bucket my-bucket --key reports/q1.csv \
  --version-id 3HL4kqtJlcpXroDTDmJ+rmSpXd3dIbrHY+MTRCxf3vjVBH40Nrjfkd

    

    

      
MFA Delete

      
オブジェクトバージョン削除またはバージョニング無効化に MFA トークンを強制する。ルートアカウントのみ。

      
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled,MFADelete=Enabled \
  --mfa "arn:aws:iam::123456789012:mfa/root 123456"

    

  


  
暗号化

  

    

      
バケットのデフォルト暗号化

      
# SSE-S3 (AES256、AWS 管理)
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules":[{"ApplyServerSideEncryptionByDefault":{
      "SSEAlgorithm":"AES256"}}]}'

# SSE-KMS (顧客管理キー)
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules":[{"ApplyServerSideEncryptionByDefault":{
      "SSEAlgorithm":"aws:kms",
      "KMSMasterKeyID":"alias/my-key"}}]}'

    

    

      
SSE-S3 vs SSE-KMS

      
SSE-S3: 設定不要、AWS がキーを所有。無料。キー使用の監査/IAM 制御が不要なら基本これ。
SSE-KMS: キーごとの監査ログ、キーローテーション、クロスアカウント許可。$0.03/10k リクエスト + KMS キー料金。

    

    

      
オブジェクト単位の暗号化

      
aws s3 cp file.txt s3://my-bucket/ \
  --sse aws:kms \
  --sse-kms-key-id alias/my-key

① S3 + CloudFront で静的サイトをホスト

  

    

      
1. バケット作成

      
BUCKET=my-site-$(uuidgen | tr A-Z a-z)
REGION=us-east-1

aws s3 mb s3://$BUCKET --region $REGION

# パブリックアクセスはブロックしたまま — CloudFront がコンテンツを配信する
aws s3api put-public-access-block \
  --bucket $BUCKET \
  --public-access-block-configuration \

2. サイトを sync

      
aws s3 sync ./site/ s3://$BUCKET/ \
  --delete \
  --cache-control "public,max-age=3600" \
  --exclude "*.html" \
  --exclude "service-worker.js"

# HTML はキャッシュ短めにしてデプロイをすぐ反映
aws s3 sync ./site/ s3://$BUCKET/ \
  --exclude "*" --include "*.html" \
  --cache-control "public,max-age=60"

    

    

      
3. CloudFront を前に配置

      
aws cloudfront create-distribution \
  --origin-domain-name $BUCKET.s3.$REGION.amazonaws.com
# コンソールで OAC を有効化し、CloudFront の SourceArn を許可する
# バケットポリシーをアタッチする (上の「バケットポリシー」カード参照)。

# デプロイ後にキャッシュ無効化
aws cloudfront create-invalidation \
  --distribution-id E1ABC... --paths "/*"

S3 → Lambda を結線

      
aws lambda add-permission \
  --function-name thumb --statement-id s3 \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --source-arn arn:aws:s3:::my-bucket

aws s3api put-bucket-notification-configuration \
  --bucket my-bucket \
  --notification-configuration '{
    "LambdaFunctionConfigurations":[{
      "LambdaFunctionArn":"arn:aws:lambda:us-east-1:123:function:thumb",
      "Events":["s3:ObjectCreated:*"],
      "Filter":{"Key":{"FilterRules":[
        {"Name":"prefix","Value":"uploads/"},
        {"Name":"suffix","Value":".jpg"}]}}
    }]}'

よくあるエラー

  

    
症状考えられる原因 & 対処

    
      
An error occurred (AccessDenied)IAM ポリシーがそのアクションを許可していない。アイデンティティポリシーとバケット/リソースポリシーの両方を確認。まず aws sts get-caller-identity でどのアイデンティティかを確認。IAM ポリシーシミュレータを使う。

      
NoSuchBucketバケット名のタイプミス、またはリージョン/アカウント違い。バケット名はグローバル。aws s3 ls は自分のアカウント所有のバケットしか出ない。

      
The bucket you are attempting to access must be addressed using the specified endpointリージョン不一致。--region us-west-2 を渡すか AWS_REGION を設定する。

      
ExpiredToken / InvalidTokenSSO/STS 認証情報の有効期限切れ。aws sso login --profile foo またはロールを再引き受けする。

      
Could not connect to the endpoint URLリージョン違い (そこに S3 が無い)、VPC エンドポイントポリシー、プロキシ。--debug を試す。

      
RequestTimeTooSkewed時計がずれている。NTP で時刻同期する。

      
An error occurred (SlowDown) when calling …S3 がプレフィックス単位でレート制限している。複数プレフィックスにキーを分散し、バックオフして再試行する。プレフィックス当たりの上限: 3500 PUT/COPY/POST/DELETE と 5500 GET req/s。

      
InvalidAccessKeyIdキーが違う、無効化された、別アカウントのキー。~/.aws/credentials と $AWS_PROFILE を確認する。

      
バケット作成時の EntityAlreadyExistsバケット名はグローバル。別の名前を選ぶ。

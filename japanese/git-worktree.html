01 ワークツリーが解決する問題

機能開発に没頭中、本番が炎上。今すぐ main を直さなければならない。ワークツリーがないと、どの選択肢も痛い。

痛み 1 コンテキストスイッチ税

スタッシュして、main に切替えて、修正、push、戻ってスタッシュを pop、コンフリクトしないことを祈る。WIP のスタッシュ pop でのマージコンフリクトは特別な苦しみ。

痛み 2 並列作業

ビルドに 6 分かかる。ブランチ A で CI が走っている間、再ビルドなしでブランチ B のコードを書きたい。1 チェックアウト = 一度に 1 ブランチ。

痛み 3 PR レビュー

git checkout pr/42 してローカルで動かしたい — でも現在のブランチで進行中の変更を吹き飛ばしたくない。

ワークツリーの答え

別のブランチに紐付いた 2 つ目の作業ディレクトリを追加。両者は同じ .git オブジェクトデータベース — fetch、ref、フック、設定 — を共有しますが、ディスク上のファイル は独立。checkout ではなく cd で切替えます。

# 旧: スタッシュダンス
git stash push -m wip
git checkout main
# ...修正、コミット、push...
git checkout feature/long-thing
git stash pop   # 🤞

# 新: 並列の作業ディレクトリ
git worktree add ../hotfix main
cd ../hotfix
# ...修正、コミット、push、その後:
cd ../cro-economy-service
git worktree remove ../hotfix

02 構造: 共有 .git、独立した作業ディレクトリ

ワークツリーは、メインリポジトリのディレクトリの外にあるが、メインリポジトリの .git によって追跡されているチェックアウトです。

共有されるもの

オブジェクトデータベース(.git/objects): コミット、ツリー、ブロブ。

Refs: ブランチ、タグ、リモート — 単一の真実の源。

設定とフック: メインリポジトリで一度だけ定義。

独立しているもの

ワーキングツリー: ディスク上に実際にチェックアウトされたファイル。

インデックス: 各ワークツリーは独自のステージング領域を持つ。

HEAD: 各ワークツリーが独自のコミット/ブランチを指す。

メタデータの保管場所

メインリポジトリは各リンクされたワークツリーを .git/worktrees/<name>/ に記録します。各リンクワークツリーは .git ファイル(ディレクトリではなく)を持ち、gitdir: 経由でそのメタデータを参照します。

04 よくあるパターン

キーストロークの元がすぐ取れる 3 つのワークフロー。

ホットフィックス すべてを投げ出す

本番が落ちている。機能ブランチで 14 コミット深く潜っている。

git fetch origin
git worktree add -b hotfix/login \
  ../hotfix origin/main
cd ../hotfix
# 修正、コミット、push
cd -
git worktree remove ../hotfix

レビュー ローカルで PR を実行

進行中の作業を破壊せずに PR #42 をレビューする。

git fetch origin pull/42/head:pr/42
git worktree add ../review pr/42
cd ../review
# テストを走らせ、つついてみる
cd -
git worktree remove ../review

長時間テスト 並行 CI

コーディングを続けながら release/v2 で統合テストを走らせる。

git worktree add ../e2e release/v2
cd ../e2e
make integration-test &
cd -
# メインリポジトリで作業継続

05 覚えておくべき落とし穴

遭遇するワークツリーのバグはほぼこの 5 つのいずれか。

! add で detached HEAD

ブランチではなくコミット/タグを渡すと、新しいワークツリーは detached HEAD になります。新しいブランチを作るには -b を使いましょう。

# detached:
git worktree add ../tmp v1.4.0
# ブランチ:
git worktree add -b fix-1.4 ../tmp v1.4.0

! ブランチが既にチェックアウト済み

git は別のワークツリーで既にチェックアウトされたブランチのチェックアウトを拒否します(2 つの HEAD が 1 つの ref を踏みにじってしまうため)。別のブランチを選ぶか、本当に分かっているなら --force を使用。

fatal: 'feature-x' is already checked out at '/path/A'
# 上書き(おすすめできない):
git worktree add --force ../B feature-x

! 手動 rm で古いメタデータが残る

ワークツリーディレクトリを rm -rf しても、.git/worktrees/<name> は残ります。git worktree list は依然として prunable として表示。git worktree prune を実行。

git worktree list
# 表示: ../old  (prunable: gitdir is missing)
git worktree prune -v

! サブモジュールとワークツリー単位の設定

サブモジュールはワークツリー間で自動初期化されません — 各ワークツリーで git submodule update --init を実行。一部の設定(sparse-checkout、フック)は --worktree でオプトインしたときのみワークツリー単位になります。

! 間違ったものを削除

ワークツリーに未コミットの変更があると remove は拒否します。反射的に --force に手を伸ばさず、まず cd で入って確認しましょう。

· ロックで保護

取り外し可能ディスクや遅い NFS マウント上のワークツリーには git worktree lock を使い、パスが到達不能になっただけで prune されないようにします。

06 ワークツリー vs スタッシュ vs 複数クローン

作業を並行する 3 つの方法。互換ではありません。

git stash複数クローンgit worktree
セットアップコスト 即時 再クローン、再 fetch、再設定 1 コマンド、再 fetch 不要
ディスク: オブジェクト DB 共有 クローンごとに重複 共有
Refs / fetch の同期 該当なし それぞれで fetch 必要 1 fetch、全ワークツリーが認識
2 ブランチ同時実行 不可 可 可
作業損失リスク スタッシュ一覧、pop でコンフリクト 低 ファイルはディスクに残る
適切な場面 小さなアドホック切替(1 分未満) コピーごとに本当に異なる SDK / ツール 同じツールチェーンで並列ブランチ

07 ベアリポ + ワークツリー: モノレポの技

「メインチェックアウト」を完全にスキップして、リポジトリをワークツリーのフォルダとして扱うチームもあります。

セットアップ

git clone --bare git@github.com:org/repo.git repo.git
cd repo.git
git worktree add ../repo/main main
git worktree add ../repo/feature-x feature-x
git worktree add ../repo/release release/v2

レイアウト:

repo.git/        # 共有オブジェクト DB
repo/
  main/          # main のワークツリー
  feature-x/     # feature-x のワークツリー
  release/       # release/v2 のワークツリー

支持される理由

「主」ワークツリーが存在しないので誰も混乱しない — 全ブランチが兄弟ディレクトリ。ツール(IDE、devcontainer、ブランチごとのビルドキャッシュ)がフォルダにきれいにマップされます。

トレードオフ: ベアリポの中からだと一部の git コマンドが少し奇妙に感じます。多くの人は cd repo/main をエイリアスにしてベアの存在を忘れます。

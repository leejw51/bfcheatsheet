概念

ブローカーがクラスタを形成。トピックはパーティションに分割。プロデューサーが追記、コンシューマーがオフセットで読み取り。それで全部です。

クラスタ構造

プロデューサーがトピックにレコードを書き込み。パーティショナーがどのパーティションかを選択。各パーティションは複製されます。グループ内のコンシューマーがパーティションを分け合います。

用語

用語意味
brokerKafkaサーバー。クラスタ = 複数のブローカー。各ブローカーは数値のbroker.idで識別されます。
topic名前付きのレコードログ。パーティション数とレプリケーション係数で作成します。
partition順序付きの追記専用コミットログ。パーティション内のレコードは厳密に順序付けられます。パーティション間: 順序保証なし。
offsetパーティション内の単調増加64ビット位置。コンシューマーは読み取ったオフセットを追跡します。
leader / follower各パーティションについて、1つのブローカーがリーダー(読み書きを処理)、他がレプリカ。リーダー障害時はフォロワーが昇格。
ISRIn-Sync Replicas: replica.lag.time.max.ms以内に追いついているフォロワー。ISRメンバーのみがリーダーになる資格を持ちます。
consumer groupgroup.idを共有するコンシューマーの集合。Kafkaは各パーティションを同時に1つのコンシューマーに割り当てます。
rebalanceグループメンバーシップが変わった時(コンシューマーが参加/離脱/死亡)のパーティション再割り当て。
__consumer_offsetsコミット済みオフセットが格納される内部compactedトピック(デフォルト50パーティション)。

順序のルール

パーティション内のみ順序保証
1つのパーティション内のレコードは厳密に順序付けられます。パーティション間: 順序なし。同じキー → デフォルトで同じパーティション → そのキーは順序保証。
# 順序の落とし穴
# key=user-42 → partition hash(user-42) % N に行く
# すべてのuser-42イベントは順序を保つ。user-99は割り込むかも。
acks & 耐久性
acks=all + min.insync.replicas≥2 = 単一ブローカー障害でデータ損失なし。
acks=0    # 投げっぱなし
acks=1    # リーダーのみ待機
acks=all  # 全ISRを待機
配信セマンティクス
at-least-onceがデフォルト。Exactly-onceには冪等プロデューサー + トランザクション + read_committedコンシューマーが必要。
# exactly-onceのレシピ
enable.idempotence=true
transactional.id=order-app-1
# コンシューマー:
isolation.level=read_committed
落とし穴: 後でパーティションを増やしても既存データは移動しません。既存のキーは古いパーティションに残り、新しいキーは別の場所に着地するかも — そのキーの順序がしばらく2つのパーティションに分裂します。最初からパーティション数を計画するか、キー対応の再パーティショニングジョブを使ってください。

プロデューサー

プロデューサーはトピックにレコードを追記します。キー、ackポリシー、パーティショナーの選択が設計の80%です。

クイックスタート (kafka-python)

基本のプロデューサー
from kafka import KafkaProducer
import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",
    enable_idempotence=True,
    linger_ms=5,
    compression_type="zstd",
)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()
冪等 + リトライ
リトライ時に重複なし、max.in.flight ≤ 5なら順序の入れ替わりなし。
KafkaProducer(
    enable_idempotence=True,
    acks="all",
    retries=2**31-1,
    max_in_flight_requests_per_connection=5,
    delivery_timeout_ms=120_000,

カスタムパーティショナー
デフォルトはキーありで murmur2(key) % N、キーがnullならsticky partitioner。
def tenant_partitioner(key, all_parts, available):
    # tenant=premium を専用パーティション 0..3 にルーティング
    if key and key.startswith(b"premium:"):
        return hash(key) % 4
    return hash(key) % len(all_parts)

KafkaProducer(partitioner=tenant_partitioner, …)
ヘッダー + タイムスタンプ
p.send(

非同期 + コールバック
def on_delivery(err, msg):
    if err:
        log.error("failed: %s", err)
    else:
        log.debug("%s[%d]@%d",
                  msg.topic(), msg.partition(), msg.offset())

p.produce("orders", value=v, on_delivery=on_delivery)
p.poll(0)  # コールバックを駆動

acks vs 耐久性 vs レイテンシ

acks耐久性レイテンシ用途
0なし — 投げっぱなし最低メトリクス、完全に置換可能なテレメトリ。
1リーダーが残れば生存低偶発的損失が許容できるホットパス。
allmin.insync.replicasブローカーが残れば生存高め重要なものすべてのデフォルト。

圧縮 & バッチング

つまみ効果
linger.msバッチが満杯でなくてもこの時間待ってから送信。5〜20msでスループットが大幅に向上します。
batch.sizeパーティションあたりのバッチバイト数(デフォルト16KB)。高スループットなプロデューサーは64KB以上が有利。
compression.typenone · gzip · snappy · lz4 · zstd。圧縮率とCPUのバランスは通常zstdが最適。
buffer.memoryクライアントが未送信レコードのバッファに使える総バイト数(デフォルト32MB)。

メッセージフロー

プロデューサー → リーダー → フォロワー (レプリケーション) → コンシューマー。

コンシューマーのコード

基本のpollループ
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",
    auto_offset_reset="earliest",   # または "latest"
    enable_auto_commit=False,
    isolation_level="read_committed",
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # 同期コミット
手動poll + オフセット
処理成功後にのみコミット → at-least-once。
while True:
    batch = c.poll(timeout_ms=1000)
    for tp, msgs in batch.items():
        for m in msgs:
            handle(m)
        c.commit({tp: OffsetAndMetadata(msgs[-1].offset+1, None)})
リバランスリスナー
パーティション剥奪前に処理中の作業をフラッシュ。
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()
        consumer.commit()
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)

c.subscribe(["orders"], listener=L())
オフセット / 時刻にseek
# オフセットで
c.seek(TopicPartition("orders", 0), 12345)

# タイムスタンプで (epochからのms)
ts = {TopicPartition("orders", 0): 1714000000000}
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():
    c.seek(tp, off.offset)
静的メンバーシップ
安定したgroup.instance.idでローリング再起動時のリバランスを回避。
KafkaConsumer(
  group_id="billing",
  group_instance_id="billing-pod-7",
  session_timeout_ms=30_000,
)
協調的リバランス
インクリメンタルなリバランス — 移動するパーティションのみ処理が止まり、全部ではありません。
partition_assignment_strategy=[
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor"
]

コミットセマンティクス

モード保証リスク
enable_auto_commit=true (デフォルト 5秒)at-most-once的コミット後・処理前にクラッシュ → レコード喪失。
処理後の手動commit()at-least-once処理後・コミット前にクラッシュ → 再起動時に再処理。ハンドラを冪等に。
トランザクショナルsend_offsets_to_transaction()Exactly-once (Kafka内のread-process-write)副作用がKafka外(冪等でないHTTP POSTなど)では役に立ちません。
落とし穴: max.poll.interval.ms(デフォルト5分)を超える長時間のメッセージハンドラはグループから蹴られる → リバランスストーム。制限を増やすか、max.poll.recordsを減らすか、作業をワーカープールに移してpause()/resume()を使ってください。

Streams & ksqlDB

Kafka Streams: トピックを変換するJava/Scalaライブラリ。ksqlDB: Streamsの上にSQL。

Streams DSL

単語カウント (「ハローワールド」)
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))
   .groupBy((k,w) -> w)
   .count(Materialized.as("counts"))
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Stream-tableジョイン
compactedトピックの最新状態でイベントストリームをエンリッチ。
KTable<String,User> users = b.table("users");
KStream<String,Order> orders = b.stream("orders");

orders.join(users,
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
ウィンドウ集約
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(
         Duration.ofMinutes(5), Duration.ofMinutes(1)))
      .aggregate(() -> 0L,
                 (k, v, sum) -> sum + v.amount(),
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

運用 & チューニング

本番で実際にいじるつまみ、それと夜中に起こされるアラーム。

レプリケーション & 耐久性

設定デフォルト内容と用途
replication.factor (トピック単位)—本番では3。min.insync.replicas=2と組み合わせて1ブローカー喪失に耐える。
min.insync.replicas1RF=3で2にする。acks=allと組み合わせて単一ブローカー障害でデータ損失なし。
unclean.leader.election.enablefalsefalseのまま。許可すると耐久性を可用性と引き換えにすることに。
replica.lag.time.max.ms30000これより遅いフォロワーはISRから外れる。
auto.create.topics.enabletrue本番ではfalseに。トピック作成を明示的なステップにする。

保持 & ストレージ

設定役割
retention.msこれより古いレコードを削除。デフォルト7日。トピック単位。
retention.bytesパーティションあたりのログサイズ上限。トピック単位。msとbytesのどちらか先に到達した方が勝ち。
segment.bytes / segment.msXバイトまたはxミリ秒ごとに新セグメントへ。小さいセグメント = 削除/コンパクションが速い、ファイルハンドルが多い。
cleanup.policy=compactログコンパクション — キーごとに最新値を永続的に保持。状態トピック、changelog、__consumer_offsetsに。
cleanup.policy=delete,compact両方: コンパクト + 保持期間後の古いキー削除。
min.cleanable.dirty.ratioコンパクターが動く前のdirty率(デフォルト0.5)。

よくあるアラーム

症状確認すべきこと
Under-replicated partitions > 0ブローカー停止、低速ディスク、ネットワーク。kafka-topics.sh --describe --under-replicated-partitions。
Offline partitions > 0リーダー不在。クラスタが危機。
コンシューマーラグ増加コンシューマーが遅い/停止。max.poll.interval.ms、GCポーズ、下流システムを確認。
プロデューサーRecordTooLargeExceptionプロデューサーのmax.request.size、ブローカーのmessage.max.bytes、コンシューマーのmax.partition.fetch.bytesを上げる。
グループリバランスストーム遅いハンドラ、不安定なメンバー、GC。静的メンバーシップ + cooperative-stickyを使う。
ISRが頻繁に縮小ネットワークまたはディスク問題。replica.lag.time.max.ms、ブローカーIO待機を確認。

クラスタサイジングの目安

ブローカーあたりのパーティション
ブローカーあたり< 4000パーティションを目指す。各パーティションがファイルハンドル、メモリ、リバランス時間を消費。
トピックあたりのパーティション数
必要となる最大並列度以上。最初に大きめに設定 — 後で増やすとキーごとの順序が壊れる。
ディスク
XFS、大きなシーケンシャル書き込み。バースト的な処理と共有しない。ログコンパクションのために50%以上の空きを計画。
ネットワーク
レプリケーションも南北トラフィック。RF=3はプロデュースされた各バイトがネットワークを~3倍通過することを意味する。
JVM
6〜8GBのヒープで十分。メモリの大半はページキャッシュに。
KRaft (ZooKeeperなし)
3.3+はKRaftがデフォルト。ZooKeeperは4.0で削除。新クラスタはKRaftのみ。

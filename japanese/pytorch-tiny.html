Tiny PyTorch — 2 hidden layers, end to end

The smallest interesting neural network: 4 inputs → 16 → 8 → 3 classes. Two hidden layers, ReLU between them, cross-entropy loss, Adam optimizer. The whole program — model, dataset, loop, inference, save/load — fits in about 50 lines of PyTorch. Type along and you'll have memorized it.

The model, drawn

How shapes flow through

x [B, 4] → Linear(4,16) → [B, 16] → ReLU → [B, 16] → Linear(16,8) → [B, 8] → ReLU → [B, 8] → Linear(8,3) → [B, 3] (logits, one per class)
the goal
Take 4-dim feature vectors and predict one of 3 classes. We'll generate synthetic blobs in 4-D — three Gaussian clusters — and train the MLP to recover them. Same shape as Iris, same shape as half the toy ML problems on the internet.
why two hidden layers?
One linear layer is just a perceptron — it can only draw straight lines. Two layers + ReLU can draw piecewise-linear curves and approximate any continuous function (universal approximation). It's the smallest "real" neural net.

The whole program (≈50 lines)

tiny_mlp.py — copy/paste runnable
import torch
from torch import nn, optim

# 1. model
class TinyMLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(4, 16)
        self.fc2 = nn.Linear(16, 8)
        self.fc3 = nn.Linear(8, 3)
        self.relu = nn.ReLU()
    def forward(self, x):
        x = self.relu(self.fc1(x))
        x = self.relu(self.fc2(x))
        return self.fc3(x)            # raw logits

# 2. data — three 4-D blobs
torch.manual_seed(0)
N = 300
centers = torch.tensor([[2,2,0,0], [-2,2,0,0], [0,-2,2,0]], dtype=torch.float)
y = torch.randint(0, 3, (N,))
x = centers[y] + 0.6 * torch.randn(N, 4)

# 3. loss + optimizer
model     = TinyMLP()
criterion = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=1e-2)

# 4. train
for epoch in range(200):
    optimizer.zero_grad()
    out  = model(x)                          # [N, 3]
    loss = criterion(out, y)
    loss.backward()
    optimizer.step()
    if epoch % 40 == 0:
        print(f"epoch {epoch:3d} · loss {loss.item():.4f}")

# 5. inference
model.eval()
with torch.no_grad():
    preds = model(x).argmax(dim=-1)
print("acc", (preds == y).float().mean().item())

# 6. save / load
torch.save(model.state_dict(), "tiny.pt")
loaded = TinyMLP(); loaded.load_state_dict(torch.load("tiny.pt"))

Define the model — nn.Module

A PyTorch model is a Python class that subclasses nn.Module. Two things to write: __init__ (declare layers as attributes) and forward (run them in order). PyTorch builds the autograd graph for you the moment forward runs.

Skeleton — fill the body

TinyMLP — explicit form
import torch
from torch import nn

class TinyMLP(nn.Module):
    def __init__(self):
        super().__init__()                  # wire up nn.Module bookkeeping
        self.fc1  = nn.Linear(4,  16)       # input → hidden 1
        self.fc2  = nn.Linear(16, 8)        # hidden 1 → hidden 2
        self.fc3  = nn.Linear(8,  3)        # hidden 2 → 3 logits
        self.relu = nn.ReLU()                # reusable activation

    def forward(self, x):
        # x: [B, 4]
        x = self.relu(self.fc1(x))           # [B, 16]
        x = self.relu(self.fc2(x))           # [B, 8]
        return self.fc3(x)                # [B, 3] logits
nn.Linear(in, out)
Stores a weight W of shape [out, in] and a bias b of shape [out]. On forward it computes x @ W.T + b. Argument order: input dim first, output dim second. Easy to flip — many bugs live here.
nn.ReLU()
Element-wise max(0, x). No learnable parameters. You can either store it as an attribute and reuse it, or just call the functional form F.relu(x) inline. Both produce identical computations and gradients.

Same model, nn.Sequential form

If your forward pass is just "apply these in order", nn.Sequential is shorter. No subclass, no forward method.

one-liner equivalent
model = nn.Sequential(
    nn.Linear(4,  16), nn.ReLU(),
    nn.Linear(16, 8),  nn.ReLU(),
    nn.Linear(8,  3),
)
# identical: 4 → 16 → ReLU → 8 → ReLU → 3
Pick one rule: nn.Module subclass when you need branching, skip connections, or anything that isn't a straight pipeline. nn.Sequential when it truly is. For an MLP this small, both are fine.

The training loop — five lines, in order

Every PyTorch training loop boils down to the same five lines. Memorize them in order and the rest of the framework falls into place. We'll go through each one.

the canonical PyTorch loop
for epoch in range(num_epochs):
    optimizer.zero_grad()           # 1. clear last step's gradients
    out  = model(x)                 # 2. forward — build autograd graph
    loss = criterion(out, y)        # 3. score predictions vs targets
    loss.backward()                 # 4. backprop — fill .grad on every weight
    optimizer.step()                # 5. weights ← weights − lr · grad

Each line, in plain English

1zero_grad()

PyTorch accumulates gradients into p.grad by default. Without zeroing, every step would add the new gradient on top of the old one — chaos. Always zero before the next forward.

2out = model(x)

Calls __call__ on the module, which calls forward. Returns logits — raw scores, shape [B, num_classes]. Don't apply softmax; the loss does it for you.

3loss = criterion(out, y)

Compares predictions to targets. For classification with class indices, use nn.CrossEntropyLoss() — it expects logits and integer labels. Returns a scalar tensor.

4loss.backward()

Walks the autograd graph backwards from loss, computing ∂loss/∂param for every parameter. Stores each gradient on param.grad. No weight changes yet.

5optimizer.step()

Reads each param.grad and updates param.data using the optimizer's rule (Adam, SGD, etc.). This is the only line that actually changes weights.

6repeat

Loop over epochs (full passes through the data) or mini-batches. After enough steps, loss drops, accuracy rises, model has learned.

The order matters. Specifically: zero before forward, backward before step. Swap any pair and you either freeze the weights, accumulate stale gradients, or update with last step's numbers. Burn it in: zero → forward → loss → backward → step.

Mini-batch variant (real life)

when the dataset is too big to fit at once
from torch.utils.data import DataLoader, TensorDataset

loader = DataLoader(TensorDataset(x, y), batch_size=32, shuffle=True)

for epoch in range(num_epochs):
    for xb, yb in loader:
        optimizer.zero_grad()
        loss = criterion(model(xb), yb)
        loss.backward()
        optimizer.step()

Data and loss — what goes in, what's measured

Before the loop, we need three things: a tensor x of inputs, a tensor y of class indices, and a loss function that compares the model's output to y. PyTorch is picky about shapes and dtypes — get them right and everything else just works.

Synthetic 3-blob dataset

make 4-D blobs in pure PyTorch
torch.manual_seed(0)
N = 300

# three class centers in 4-D
centers = torch.tensor([
    [ 2,  2,  0,  0],   # class 0
    [-2,  2,  0,  0],   # class 1
    [ 0, -2,  2,  0],   # class 2
], dtype=torch.float)

y = torch.randint(0, 3, (N,))                # labels  [N], dtype=long
x = centers[y] + 0.6 * torch.randn(N, 4)   # features [N, 4], float
x[N=300, 4]float32 · y[N=300]int64 / long, values in {0, 1, 2}
Dtype trap. nn.CrossEntropyLoss requires the target to be integer class indices with dtype long (int64), not one-hot vectors and not floats. torch.randint already gives long; if you load from NumPy, cast with y.long().

Loss — nn.CrossEntropyLoss

logits in, scalar loss out
criterion = nn.CrossEntropyLoss()

# during training:
out  = model(x)              # [B, 3]  raw logits
loss = criterion(out, y)     # scalar; combines log_softmax + nll_loss
expected shapes
logits: [B, C]
target: [B] (long, values 0..C−1)
loss:   scalar
do not pre-softmax
CrossEntropyLoss applies log_softmax internally for numerical stability. Passing softmax probabilities makes it apply softmax twice — gradients get tiny and training stalls. Feed it raw logits.

Optimizer — optim.Adam

a sane default for tiny models
optimizer = optim.Adam(model.parameters(), lr=1e-2)
# model.parameters() returns *every* trainable tensor in the module tree.
# lr=1e-2 is a fine starting point for an MLP this small.
SGD vs Adam: SGD requires careful learning-rate tuning; Adam adapts per-parameter step sizes and works out-of-the-box for most small MLPs. Start with Adam, tune later if you care about the last 1% of accuracy.

Inference & saving — what production looks like

Training is the hard part. Inference is just forward, no gradients. Two flips matter: switch the model to eval mode, and disable autograd so we don't waste memory building a graph we'll never use.

The two-flip dance

predict on new data
model.eval()                          # 1. eval mode — disables dropout, freezes BN stats
with torch.no_grad():                # 2. no autograd — no .grad tracking, no graph
    logits = model(x_new)             # [B, 3]
    preds  = logits.argmax(dim=-1)   # [B]   class indices 0..2
    probs  = logits.softmax(dim=-1)   # [B, 3] probabilities (optional)
model.eval()
Flips a flag on every submodule. For our TinyMLP it's a no-op (no Dropout / BatchNorm), but make it a habit — the moment you add a regularizer this matters. Pair with model.train() before training again.
torch.no_grad()
Context manager that skips autograd bookkeeping. Inference becomes ~2x faster and uses less memory. Without it, every forward pass still builds the full graph for a backward() you'll never call.

Save and load

state_dict — weights only, no Python class
# save
torch.save(model.state_dict(), "tiny.pt")

# load (rebuild the class first, then pour weights in)
loaded = TinyMLP()
loaded.load_state_dict(torch.load("tiny.pt"))
loaded.eval()                          # always eval after loading for inference
Why state_dict? It's a plain dict of tensors — { "fc1.weight": ..., "fc1.bias": ... } — that survives refactors. Saving the whole model object pickles the Python class too, which breaks if you rename files. Always save state_dict, not the module.

one prediction, top to bottom
x_one = torch.tensor([[1.8, 2.1, 0.0, 0.1]])    # shape [1, 4]

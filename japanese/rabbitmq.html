1. メッセージブローカーとは?

メッセージブローカーはプロデューサとコンシューマの間に置かれます。プロデューサはメッセージをブローカーに投げ、すぐ自分の仕事に戻ります。コンシューマは自分のペースで後から取り出します。両者は同時にオンラインである必要も、互いのアドレスを知る必要も、処理速度を合わせる必要もありません。

キューが重要な理由

  • 疎結合: プロデューサを変更せずにコンシューマを変更できる。
  • バッファリング: スパイクを平滑化、キューがバーストを吸収。
  • 信頼性: コンシューマがクラッシュしてもメッセージは残る。
  • ファンアウト: 1 つのイベントが独立した多くの反応を引き起こす。
メンタルモデル: プロデューサがハガキを書き、ブローカーが郵便局、キューがメールボックス、コンシューマは自分の都合で読みます。

2. AMQP のコア概念

RabbitMQ は AMQP 0-9-1 を実装しています。語彙は少ないですが正確です。

  • プロデューサ メッセージを発行する。
  • コンシューマ 購読してメッセージを処理する。
  • キュー メッセージを格納する名前付きバッファ。
  • エクスチェンジ ルーター。プロデューサはエクスチェンジに発行し、キューに直接は発行しない。
  • バインディング エクスチェンジとキューを (多くはキー付きで) 結ぶルール。
  • ルーティングキー プロデューサがメッセージに付ける文字列。エクスチェンジが配送先を決めるのに使う。
  • 仮想ホスト (vhost) ユーザー、エクスチェンジ、キューを区切る名前空間。論理的な分離。
  • コネクション TCP 接続。チャネル その中の多重化された仮想接続。

フローの全体像

producer
  └─ publish(routing_key) ─▶ exchange
                              ├─ binding A ─▶ queue 1 ─▶ consumer
                              └─ binding B ─▶ queue 2 ─▶ consumer

3. エクスチェンジ種別 詳説

Direct

ルーティングキーがバインディングキーと完全一致するメッセージをルーティングします。用途: 狙った配送、シンプルなワークキュー、コマンドのルーティング。

binding: queue.email  ⇐ key="email"
publish key="email"   → queue.email
publish key="sms"     → 破棄 (またはルートなし)

Fanout

ルーティングキーは無視。バインドされた全キューにブロードキャストします。用途: pub/sub イベント、キャッシュ無効化、「全員に通知」系。

publish 任意のキー → バインドされた全キューが受信

Topic

パターンによるルーティング。ルーティングキーはドット区切りの単語列で、バインディングパターンで以下が使えます:

  • * は ちょうど 1 単語 にマッチ。
  • # は 0 個以上の単語 にマッチ。
binding "orders.*.us"   マッチ  orders.new.us, orders.cancel.us
                        ミス    orders.us, orders.new.eu, orders.a.b.us
binding "orders.#"      マッチ  orders, orders.new, orders.new.us
binding "*.error.*"     マッチ  app.error.db, web.error.api

Headers

ルーティングキーではなくメッセージのヘッダーを使ってルーティングします。マッチモード x-match: all は全ヘッダー一致が必要、x-match: any は 1 つでも一致すれば OK。

binding headers: {format: pdf, type: report, x-match: all}
header {format: pdf, type: report}     → マッチ
header {format: pdf}                    → ノーマッチ

どれを使うか

種別こんな時に選ぶ…
direct1 つのキーでポイントツーポイントや厳密な一致がしたい時。
fanoutすべての購読者が同じイベントを必要とする時。最も推論しやすい。
topic階層的なイベント: region.service.severity。購読者は部分集合を望む。
headers文字列キーでは表せない複数の属性に依存するルーティング。

デフォルトエクスチェンジ

名前のない "" エクスチェンジは特別な direct エクスチェンジで、すべてのキューが自身の名前で暗黙的にバインドされています。これが basic_publish(exchange='', routing_key='hello') が hello という名前のキューに直接届く理由です。

デッドレターエクスチェンジ (DLX)

キューには x-dead-letter-exchange を設定できます。reject、TTL 期限切れ、オーバーフローしたメッセージは DLX に再発行され、そこから別のバインディングがデッドレターキューに格納し、検査やリトライに使えます。

4. ACK (確認応答)

auto-ack vs manual ack

  • auto-ack (auto_ack=True): メッセージが出ていった瞬間にブローカーは配送済みとみなします。速いが安全ではありません。 コンシューマが処理中にクラッシュするとメッセージは失われます。
  • manual ack (auto_ack=False): コンシューマは成功後に basic_ack(delivery_tag) を呼ぶ必要があります。先に死んだ場合は再接続時にブローカーが 再配送 します。

nack & reject

  • basic_nack(requeue=true) — キューに戻す。
  • basic_nack(requeue=false) — 破棄、または設定済みなら DLX に送る。
  • basic_reject は単一メッセージ版の nack。

redelivered フラグ

メッセージが再配送されると、ブローカーは redelivered=true を立てます。コンシューマは慎重さを示すヒントとして扱うべきです: 冪等性、重複排除キー、ヘッダーのリトライカウンタなど。

5. 永続性 — 再起動を生き延びるもの

ブローカークラッシュをメッセージが生き延びるには、3 つの独立した条件を すべて 満たす必要があります:

  1. 永続キュー: queue_declare(durable=True)。これがないとキュー自体が消えます。
  2. 永続メッセージ: delivery_mode=2 (または PERSISTENT_TEXT_PLAIN)。これがないとメッセージは RAM 上のみ。
  3. 永続エクスチェンジ (依存している場合): exchange_declare(durable=True)。
注意: 永続化はベストエフォートで、メッセージごとに同期 fsync するわけではありません。ディスクへの書き込みを確実に知ってから上流に ack したい場合は publisher confirms (confirm_delivery) を使いましょう。

publisher confirms vs トランザクション

AMQP トランザクションは存在しますが遅いです。publisher confirms は非同期で、桁違いに高速です: ブローカーが ack フレームで「受け取った」と通知します。

6. Prefetch & QoS — 公平なディスパッチ

デフォルトでは RabbitMQ はメッセージをコンシューマにラウンドロビンで配ります。1 つのコンシューマが遅く、もう 1 つが速いと、遅い方に積み上がり速い方は遊んでしまいます。prefetch がそれを解決します。

channel.basic_qos(prefetch_count=1)

これはブローカーに「直前のメッセージを ack するまで新しいメッセージを渡すな」と伝えます。これで速いコンシューマがより多く、遅い方が少なく取るようになります。短く均一な作業ならもっと高い値 (例 10〜50) に。

チューニングの目安

  • 長く不均一なジョブ → prefetch=1。
  • 短く均一なジョブ → 高めの prefetch (ラウンドトリップ削減)。
  • キュー深さ と コンシューマ稼働率 で計測する。

7. クラスタリング & quorum キュー

RabbitMQ クラスタ はノード間でユーザー、vhost、エクスチェンジ、バインディングを共有します。キューはデフォルトで単一マスタ。マスタは 1 ノード上に存在し、ミラーリングまたは quorum で置き換えできます。

クラシックなミラードキュー

レガシー。状態をフォロワーへ複製します。現代的な構成では quorum キューに置き換えられました。

Quorum キュー

  • Raft 合意プロトコルに基づきます。
  • 強い整合性: 過半数のレプリカが受け取るまで書き込みは ack されません。
  • ノード障害をきれいに乗り越えます。重要なデータには既定の推奨。
  • トレードオフ: クラシックキューよりレイテンシが高く、ディスク I/O も増えます。

Streams (ボーナス)

高スループットの分析ワークロード向けの、ログ形式の追記キュー。プロトコルもアクセスパターンも別 (オフセット指定で、消費しても消えない)。

8. よくあるパターン

ワークキュー (タスク分散)

1 つのキュー、複数のワーカー、manual ack、prefetch=1。画像リサイズ、メール送信、動画エンコードなどに。

producer ─▶ "" ─▶ queue:tasks ─▶ worker1
                              └▶ worker2
                              └▶ worker3

Publish/Subscribe

fanout エクスチェンジ。各購読者は自分のキュー経由で全メッセージを受け取ります。

producer ─▶ fanout:events ─┬─▶ queue:audit       ─▶ logger
                            ├─▶ queue:cache_invl  ─▶ cache
                            └─▶ queue:notify      ─▶ pusher

その他のパターン

トピックルーティング

階層的なイベント。異なる形のキューが異なるパターンを購読します。

topic:logs
  binding "*.error.*"  ─▶ queue:alerts
  binding "db.#"       ─▶ queue:dba
  binding "#"          ─▶ queue:audit

RPC (リクエスト/リプライ)

プロデューサが reply_to (一時的・排他キュー) と correlation_id をセットします。コンシューマは応答を reply_to に発行。プロデューサは correlation_id で応答を突き合わせます。

client → publish(reply_to=Q1, correlation_id=abc)
server → consume → publish(exchange='', routing_key=Q1, correlation_id=abc)
client → abc にマッチ、Promise を解決
ヒント: reply キューに TTL を設定して、リークしたクライアントが孤児キューを溜めないようにしましょう。

インタラクティブシミュレーター

エクスチェンジ種別を選び、バインディングを宣言、メッセージを発行してフローを観察しましょう。コンシューマの挙動を切り替えて、ack/nack と再配送の動きを確認できます。発行成功ごとに XP を獲得します。

プロデューサ エクスチェンジ キュー コンシューマ drop-off = 未ルーティング (mandatory off → 静かに失われる)

バインディング

バインディングをクリックすると削除できます。

コンシューマ

各コンシューマの ack モードを切り替えます (auto / manual-ack / manual-nack-requeue / manual-nack-DLX)。

アクティビティログ

第1章

所有権と借用

rustの心臓部

Rustのすべての値には、たった一人の所有者(オーナー)がいる。所有者がスコープを 抜けると、値はドロップされる ── メモリは解放され、ファイルは閉じられ、 ロックは解放される。これがRAIIであり、静的に強制される。

所有権をムーブ(移転)することもできるし、借用 (参照を貸し出すこと)もできる。借用には2つのルールがある:&Tの共有 参照は同時に複数あってもよいが、&mut Tの可変参照は 排他的でなければならない。コンパイラはこれをコンパイル時に証明する。

所有権の移転(ムーブ)と借用の対比
所有された値 借用された参照 無効化

古典的なムーブ

main.rs
fn main() {
    let s = String::from("hello");
    let t = s;                 // sはtにムーブされる
    // println!("{}", s);      // ❌ エラー: ムーブされた値の借用: `s`
    println!("{}", t);         // ✅
}

共有借用と可変借用

borrow.rs
fn len(s: &String) -> usize { s.len() }

fn push_bang(s: &mut String) { s.push('!'); }

fn main() {
    let mut greeting = String::from("hi");
    let n = len(&greeting);          // 共有借用、OK
    push_bang(&mut greeting);        // 排他借用、OK
    println!("{} ({} chars)", greeting, n);
}
2つのルール いついかなる時も、可変参照を1つ持つか、 あるいは任意の数の共有参照を持つかのどちらかである。参照は常に 有効でなければならない(生きているデータを指す)。これがborrow checkerのすべてだ。
第2章

ライフタイム

借用の期間に名前を付ける

ライフタイムとは、参照が有効であるコード領域の名前にすぎない。 ほとんどのライフタイムは推論される(これは省略(elision)と呼ばれる)。 記述する必要があるのは、返される参照がどの入力に由来するかをコンパイラが 判断できないときだけだ。

参照は、借用先の値より長く生存できない
所有者のライフタイム 'a 参照のライフタイム 'b

返り値の参照に注釈を付ける

lifetimes.rs
// 返される `&str` は、xとyの短い方と同じだけ生存する必要がある。
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() >= y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string is long");
    let result;
    {
        let s2 = String::from("short");
        result = longest(&s1, &s2);
        println!("{}", result);   // ✅ 両方とも生存中
    }
    // println!("{}", result);     // ❌ s2はドロップ済み、宙ぶらりんになる
}
省略のルール 関数が1つの参照を取り、1つを返す場合、Rustは それらが同じライフタイムを共有すると仮定する。入力に&selfがある場合は、 出力は&selfから借用する。これらのデフォルトが 正しくない場合にのみ注釈を書く。
第3章

トレイトとジェネリクス

抽象化された振る舞い

トレイトとは、型が実装できるメソッドの集合である。 ジェネリクスとは、型でパラメータ化された関数や型のことだ。 これらを組み合わせると、継承なしにアドホック多相を得られる。 「Displayできる任意のT」、「i32の任意のイテレータ」といった具合に。

1つのトレイト、2つのimpl、1つのジェネリック関数
traits.rs
trait Greet {
    fn hello(&self) -> String;
    fn shout(&self) -> String {                 // デフォルトメソッド
        self.hello().to_uppercase()
    }
}

struct Cat;
struct Robot { id: u32 }

impl Greet for Cat   { fn hello(&self) -> String { "meow".into() } }
impl Greet for Robot { fn hello(&self) -> String { format!("unit-{}", self.id) } }

fn welcome<T: Greet>(t: &T) {
    println!("{} ({})", t.hello(), t.shout());
}

fn main() {
    welcome(&Cat);
    welcome(&Robot { id: 7 });
}
静的ディスパッチ vs 動的ディスパッチ fn welcome<T: Greet>は具体型ごとにコピーを生成する (ゼロコスト)。fn welcome(t: &dyn Greet)は太いポインタを取り、 実行時にディスパッチする ── 少し遅いが、異種混在の Vec<Box<dyn Greet>>を保持できる。
第4章

Result、Option、? 演算子

値としてのエラー

Rustに例外はない。失敗は型にエンコードされる。Option<T> はもしかするとTを意味し、Result<T, E>はTかエラー Eを意味する。?演算子はこれらを通し抜け、値がなければ 早期リターンする。

? 演算子: エラーで短絡し、Okでアンラップ
parse.rs
use std::num::ParseIntError;

fn double_first(input: &str) -> Result<i32, ParseIntError> {
    let first = input.split(',').next().unwrap_or("0");
    let n: i32 = first.trim().parse()?;   // Errで早期リターン
    Ok(n * 2)
}

fn main() {
    match double_first("21, hello") {
        Ok(v)  => println!("got {v}"),
        Err(e) => eprintln!("bad: {e}"),
    }
}

Optionを簡単に

option.rs
fn first_word(s: &str) -> Option<&str> {
    s.split_whitespace().next()
}

fn main() {
    let n = first_word("hello world").map(str::len).unwrap_or(0);
    println!("{n}");                     // 5
}
第5章

パターンマッチング

match、if let、while let

matchは網羅的だ。コンパイラは、どのバリアントを 無視することも許してくれない。パターンはタプル、構造体、enum、 スライス、参照を分解でき、if条件によるガードも書ける。

match.rs
enum Shape {
    Circle { r: f64 },
    Rect   { w: f64, h: f64 },
    Triangle(f64, f64, f64),
}

fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle { r }            => std::f64::consts::PI * r * r,
        Shape::Rect   { w, h }         => w * h,
        Shape::Triangle(a, b, c) if a + b > *c => {
            let p = (a + b + c) / 2.0;
            (p * (p - a) * (p - b) * (p - c)).sqrt()
        }
        _ => f64::NAN,                                  // 不正な三角形
    }
}
If let、while let 1つのバリアントだけを気にする場合、if let Some(x) = opt { … } は完全なmatchより簡潔だ。while letはパターンが マッチしなくなるまで繰り返す ── スタックをポップするのに最適だ。
第6章

イテレータとクロージャ

遅延的、流暢、高速

イテレータは遅延評価だ。要求するまで何も起こらない。クロージャは 環境をキャプチャし、3種類ある: Fn(借用)、 FnMut(可変借用)、FnOnce(消費)。 コンパイラは最も寛容なものを自動的に選ぶ。

iter.rs
fn main() {
    let nums = vec![1, 2, 3, 4, 5, 6];

    let sum_of_even_squares: i32 = nums
        .iter()
        .filter(|&&n| n % 2 == 0)
        .map(|&n| n * n)
        .sum();

    println!("{sum_of_even_squares}");   // 56

    // クロージャとキャプチャ
    let factor = 10;
    let scale = |x: i32| x * factor;     // factorを借用 (Fn)
    println!("{}", scale(3));            // 30
}
ゼロコスト イディオマティックなイテレータ連鎖は、手書きのループと同じアセンブリに コンパイルされる。流暢なスタイルは本質的に無料だ ── 時には より速いことすらある。LLVMがより多くの不変条件を証明できるからだ。
第7章

並行処理

スレッド、send/sync、チャネル

2つのマーカートレイトが並行処理を制御する。Sendとは、値が スレッド間でムーブするのが安全であることを意味する。Syncとは、 &Tがスレッド間で共有するのが安全であることを意味する。ほとんどの型は 両方を満たすが、Rc<T>とRefCell<T>は満たさない ── コンパイラが データ競合を含むコードの出荷を阻止してくれる。

チャネル: 生産者 → 消費者、所有権の移転を伴う
channel.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    for i in 0..4 {
        let tx = tx.clone();
        thread::spawn(move || {
            tx.send(format!("hello from worker {i}")).unwrap();
        });
    }
    drop(tx);                     // 元の送信者をクローズ

    while let Ok(msg) = rx.recv() {
        println!("{msg}");
    }
}

共有された可変状態

mutex.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let c = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut n = c.lock().unwrap();
            *n += 1;
        }));
    }
    for h in handles { h.join().unwrap(); }
    println!("count = {}", *counter.lock().unwrap()); // 10
}
第8章

Async / Await

協調的並行処理

関数をasyncでマークすると、その本体はFutureを実装する ステートマシンになる。呼び出すだけでは何も起こらない。 .awaitするか、ランタイム(Tokio、async-std、smol)に 渡したときに初めて動く。各.awaitは中断点だ。

async_main.rs
use tokio::time::{sleep, Duration};

async fn fetch(name: &str) -> String {
    sleep(Duration::from_millis(50)).await;
    format!("hello, {name}")
}

#[tokio::main]
async fn main() {
    let (a, b) = tokio::join!(fetch("alice"), fetch("bob"));
    println!("{a} | {b}");
}
Asyncは「色」だ asyncから同期関数を呼べるが、逆は ランタイムなしでは不可能だ。早めにランタイムを選び(Tokioが デフォルト)、それを使い続けよう。

Cargoとモジュール

プロジェクト構成

Cargoは、ビルドシステム、パッケージマネージャ、テストランナー、 ベンチマークハーネスを1つにまとめたものだ。クレートはコンパイルの単位、 モジュールはクレート内の名前空間である。 ファイルシステムがモジュールツリーを反映する。

tree
my_app/
├── Cargo.toml          # マニフェスト: 名前、バージョン、依存
├── Cargo.lock          # 解決された厳密なバージョン (バイナリではコミットする)
└── src/
    ├── main.rs         # バイナリのエントリポイント  →  クレート `my_app`
    ├── lib.rs          # ライブラリのルート         →  クレート `my_app`
    └── parser/
        ├── mod.rs      # モジュール `parser`
        └── ast.rs      # モジュール `parser::ast`
Cargo.toml
[package]
name    = "my_app"

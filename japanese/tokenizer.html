第01章 · あらゆるトランスフォーマーの舌

文字列を 入れて、
整数を 出す。

モデルは文字を読みません。単語も読みません。読むのは 整数 です。 トークナイザーは、生のテキストをトランスフォーマーが実際に消費する整数言語に翻訳し、 生成された整数を再び読めるテキストに翻訳する小さなプログラムです。これがなければ、 GPT、BERT、LLaMA、Claude — どれも沈黙したままです。

一文で言うと
トークナイザーは関数 $T : \text{string} \to \mathbb{Z}^n$ で、 固定の 語彙 から取った トークン にテキストを分け、整数IDを出します。 デトークナイザー $T^{-1}$ がその逆を行います。 
"hello world"
       │
       ▼  encode
[ 31373, 995 ]      # 2トークン、GPT-2 の語彙
       │
       ▼  decode
"hello world"
なぜ単語をそのまま使わない?
  • 語彙の爆発。 英語には100万以上の語形があり、モデルは100万行の埋め込み表が必要になります。
  • 未知語。 タイポや新語は <UNK> となり、モデルは見えなくなります。
  • 形態論。 "running" / "runner" / "runs" は同じ語根を共有しますが、モデルはそれを見ません。
  • 言語の違い。 中国語にはスペースがありません。トルコ語は形態素を貼り合わせます。空白分割では破綻します。
サブワード・トークナイザーはこの4つすべてを解決します — 約3万〜20万ピースの小さな語彙で、組み合わせによりあらゆる文字列をカバーできます。
出会う3つの系統

BPE Byte-Pair Encoding — 最頻の隣接ペアを繰り返しマージします。GPT-2/3/4、LLaMA、Mistral。

WordPiece BPE に似ていますが、コーパス尤度を最大化するマージを選びます。BERT、DistilBERT、ELECTRA。

SentencePiece 空白を通常の文字 (▁) として扱い、言語非依存。Unigram または BPE をラップします。T5、ALBERT、LLaMA。

この先に来るもの
トークンは トークンID となり、埋め込み となり、形状 [batch, seq_len, d_model] のテンソルになります。 トランスフォーマーの他のすべて — アテンション、FFN、出力射影 — は、このテンソルの変換に過ぎません。 トークナイズを正しくやれば、残りは行列の掛け算です。

第02章 · すべてのトークナイザーが踏む5ステップ

トークナイザーが
実際に 行うこと。

現代のあらゆるトークナイザーは、結局どこかの順序で同じ5つのことをしています: 正規化、事前トークナイズ、学習済みのマージ表でサブワードに分割、 各ピースをIDに対応付け、特殊トークンを追加。それぞれを平易に説明します。

1

正規化 (Normalize)

モデルに応じて、小文字化 / NFC ユニコード / アクセント除去などを行います。GPT-2 はここでほとんど何もしません。BERT-uncased は積極的に小文字化します。

"Hèllo!" → "hello!" (BERT-uncased)
2

事前トークナイズ (Pre-tokenize)

マージ・ステップが単語の境界を超えないように、空白と句読点で分割します。GPT-2 は正規表現を使い、SentencePiece は 事前分割しません — 空白を含む生のバイトで動作します。

"hello world!" → ["hello", " world", "!"]
3

マージ / 語彙表を適用

遅い学習時のステップでサブワードの語彙を学びます。推論時には、各事前トークンを既知の最長ピースに分割します。

"tokenization" → ["token", "ization"] (BPE)
4

ピース → 整数IDに対応付け

語彙は str → int の辞書です。各ピースは埋め込み表におけるその行のインデックスになります。

["token", "ization"] → [30001, 1634]
5

特殊トークンの追加 & パディング

BERT は [CLS] を先頭に付け、[SEP] で終わります。GPT 系のモデルは何も付けないことが多いです。バッチ入力は共通の長さに <pad> ID でパディングし、アテンション・マスク がどの位置が実データかを記録します。

[101, 30001, 1634, 102, 0, 0] mask: [1, 1, 1, 1, 0, 0]

02 BPE を60秒で — マージのステップ

Byte-Pair Encoding は 学習中に 語彙を構築します — コーパスで最も頻繁な隣接シンボル ペアを繰り返し見つけ、その連結を新しいシンボルとして追加します。$V$ 回マージを実行すれば、 語彙サイズは ~初期バイト数 + $V$ になります。

小さな実例
コーパス "low low low lower newer" から始め、単語境界マーカー ▁ を付けて文字に分割します: 
# ステップ 0 — バイト
▁ l o w   ▁ l o w   ▁ l o w   ▁ l o w e r   ▁ n e w e r

# 最頻ペア = (l, o)  →  "lo" にマージ
▁ lo w   ▁ lo w   ▁ lo w   ▁ lo w e r   ▁ n e w e r

# 最頻ペア = (lo, w)  →  "low" にマージ
▁ low   ▁ low   ▁ low   ▁ low e r   ▁ n e w e r

# 最頻ペア = (e, r)  →  "er" にマージ
▁ low   ▁ low   ▁ low   ▁ low er   ▁ n e w er

# 最終的な語彙には ▁, l, o, w, e, r, n, lo, low, er, ... が含まれる
推論 では 同じ マージ・リストを順に走査し、依然として適合するマージを適用します。 BPE が決定的かつロスレスなのはそのためで — どのバイト列もユニークなトークナイゼーションを持ちます。

03 頭の中でトークナイゼーションを読む

GPT-2 の出力をしばらく見つめていると、いくつかの目印が分かってきます:

  • 先頭の空白は重要。 文頭の "hello" と文中の " hello" は別のトークンです。GPT-2 はバイトレベル語彙で先頭の空白を Ġ として表示します。
  • よくある単語 → 1トークン。 "the"、"and"、"is" は単一トークンです。
  • 長い・稀な単語 → 多くのトークン。 "tokenization" は分割され、"antidisestablishmentarianism" はもっと細かく分割されます。
  • 数値や絵文字 は細かく刻まれます — "2024" はトークナイザーによって 1、2、または 4 トークンになり得ます。
  • 非ラテン文字 はバイトレベル BPE では バイトトークン になります: 3バイトの UTF-8 コードポイントは 3 トークンを消費します。SentencePiece をその言語で学習すれば回避できます。

第04章 · テンソルはどこから来るのか

文字列から logits へ
8つの形状で。

同じ文 — "Hello, world!" — を取って、文字列から logits までの正確な形状を追跡しましょう。 B = バッチ、T = 系列長、D = 隠れ次元、V = 語彙サイズ。

1

生の文字列

"Hello, world!"

形状: スカラー (まだテンソルではない)
2

tokenize → ID のリスト

[15496, 11, 995, 0] (GPT-2 BPE、4トークン)

shape: [T=4] dtype: int64
3

バッチ化 + パディング

2文をバッチに積み、ID 0 で T=6 までパディング:
[[15496, 11, 995, 0, 0, 0],
 [15496, 11, 5440, 16085, 0, 0]]

input_ids: [B=2, T=6] attention_mask: [B=2, T=6]
4

埋め込みルックアップ

x = embed[input_ids] — ID ごとに [V × D] 表の1行を取り出します。

x: [B=2, T=6, D=768] embed: [V=50257, D=768]
5

+ 位置埋め込み

[T × D] の表を加算 (もしくは RoPE が代わりに Q/K を回転)。これで位置情報が入ります。

x: [B=2, T=6, D=768] pos: [T=6, D=768]
6

L 個のトランスフォーマー・ブロック

各ブロックは形状を保ちます。内部では、アテンションが一時的に scores: [B, H, T, T] を生成します。

x は維持: [B=2, T=6, D=768]
7

出力ヘッド → logits

logits = x @ embed.T — 各位置で語彙トークンごとに1スコア。

logits: [B=2, T=6, V=50257]
8

softmax → 次トークンID

argmax(logits[:, -1, :], dim=-1) で次のIDを選び、トークナイザーがテキストに戻します。

next_id: [B=2] T⁻¹ で "デコードされたテキスト"

04 同じ流れを PyTorch で

HuggingFace で end-to-end
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

tok   = AutoTokenizer.from_pretrained("gpt2")
model = AutoModelForCausalLM.from_pretrained("gpt2")

# 1. encode → int64 テンソルの dict
batch = tok(["Hello, world!", "Hello, friends, again"],
            padding=True, return_tensors="pt")
print(batch.input_ids.shape)        # torch.Size([2, 6])
print(batch.attention_mask.shape)   # torch.Size([2, 6])

# 2. forward — 埋め込み + トランスフォーマー・ブロック
out = model(**batch)
print(out.logits.shape)            # torch.Size([2, 6, 50257])

# 3. greedy 次トークン = 語彙軸での argmax
next_id = out.logits[:, -1, :].argmax(dim=-1)
print(tok.batch_decode(next_id.unsqueeze(-1)))
メモリの裏紙計算。 8k コンテキスト、バッチ 1 の fp16 推論には、 各層の活性に およそ $T \times D \times 2$ バイト、加えて KV キャッシュに $2 \times L \times T \times D \times 2$ バイトが必要です。LLaMA-7B、T=8192 では ~512 MB の KV キャッシュ — トークナイザーの圧縮率が直接効く予算です。
第05章 · サブワードの3系統

BPE · WordPiece · SentencePiece。

この3つはすべてサブワードトークンを生み、すべて頻度統計から始まりますが、マージ規則と 空白の扱い方が異なります。文字列を入力して、それぞれがどう分割するか見てみてください。

BPE — 頻度マージ

最頻の隣接ペアを選び、マージし、繰り返します。推論では同じマージを順に適用します。 貪欲かつロスレス。

採用: GPT-2/3/4、RoBERTa、LLaMA、Mistral、Qwen。

目印: 先頭の空白はトークンの一部になります (Ġ や ▁ として表示)。

WordPiece — 尤度マージ

BPE に似ていますが、マージ規則は単純な頻度ではなくコーパスの対数尤度を最大化します。 継続ピースには ## プレフィックスが付きます。

採用: BERT、DistilBERT、ELECTRA、MobileBERT。

目印: "playing" → ["play", "##ing"]

SentencePiece — バイトは神聖

空白を本物の文字 (▁ = U+2581) として扱い、生テキストから直接学習します — 事前トークナイズなし。Unigram-LM か BPE をラップします。

採用: T5、ALBERT、XLNet、mT5、LLaMA。

目印: ▁hello、▁world のようなトークン。

05 間違ったものを選ぶと何が起きる?

  • トークナイザーの不一致は静かで致命的。 BPE で学習し WordPiece で推論すれば、すべての埋め込みルックアップが間違います。モデルは動きますが、出力は意味不明です。
  • 実行時のトークン追加 は許可されています (チャットテンプレートなど) が、埋め込み表のリサイズ をマッチさせる必要があります。それを忘れると新しいIDでクラッシュします。
  • 言語が違えば請求も違う。 日本語は GPT-2 BPE では英語の約3倍のトークンを消費しますが、日本語で学習した SentencePiece では約1倍です。同じモデル、違うコスト。

Concepts

Brokers form a cluster. Topics split into partitions. Producers append; consumers read by offset. That's the whole game.

Cluster anatomy

A producer writes records to a topic. The partitioner picks which partition. Each partition is replicated. Consumers in a group divide partitions among themselves.

Vocabulary

TermWhat it means
brokerA Kafka server. Cluster = many brokers. Each broker is identified by a numeric broker.id.
topicA named log of records. Created with a partition count and a replication factor.
partitionAn ordered, append-only commit log. Records inside a partition are strictly ordered. Across partitions: no order guarantee.
offsetMonotonic 64-bit position inside a partition. Consumers track which offset they've read.
leader / followerFor each partition, one broker is leader (handles reads + writes), others replicate. On leader failure, a follower is promoted.
ISRIn-Sync Replicas: followers that are caught up within replica.lag.time.max.ms. Only ISR members are eligible to become leader.
consumer groupA set of consumers sharing a group.id. Kafka assigns each partition to one consumer in the group at a time.
rebalanceReassignment of partitions when group membership changes (consumer joins/leaves/dies).
__consumer_offsetsInternal compacted topic where committed offsets live (50 partitions by default).

Ordering rules

Per-partition ordering only
Records in one partition are strictly ordered. Across partitions: no order. Same key → same partition by default → ordered for that key.
# Ordering pitfall
# key=user-42 → goes to partition hash(user-42) % N
# All user-42 events stay ordered. user-99 may interleave.
acks & durability
acks=all + min.insync.replicas≥2 = no data loss on a single broker failure.
acks=0    # fire and forget
acks=1    # wait for leader only
acks=all  # wait for all ISR
Delivery semantics
At-least-once is the default. Exactly-once needs idempotent producers + transactions + read_committed consumers.
# exactly-once recipe
enable.idempotence=true
transactional.id=order-app-1
# consumer:
isolation.level=read_committed
Pitfall: increasing partitions later does not move existing data. Existing keys stay in their old partitions; new keys may land elsewhere — ordering for that key splits across two partitions for a while. Plan partition count up front or use a key-aware re-partitioning job.

Producers

Producers append records to a topic. Picking a key, an ack policy, and a partitioner is 80% of the design.

Quick start (kafka-python)

Basic producer
from kafka import KafkaProducer
import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",
    enable_idempotence=True,
    linger_ms=5,
    compression_type="zstd",
)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()
Idempotent + retries
No duplicates on retry, no re-ordering when max.in.flight ≤ 5.
KafkaProducer(
    enable_idempotence=True,
    acks="all",
    retries=2**31-1,
    max_in_flight_requests_per_connection=5,
    delivery_timeout_ms=120_000,
)
Transactional producer
Atomic write across multiple topics + partitions. Used for read-process-write apps and exactly-once.
# confluent-kafka style
from confluent_kafka import Producer
p = Producer({
  "bootstrap.servers": "broker1:9092",
  "transactional.id": "order-app-1",
  "enable.idempotence": True,
})
p.init_transactions()
p.begin_transaction()
p.produce("orders", key="u1", value="...")
p.produce("audit", key="u1", value="...")
p.commit_transaction()
Custom partitioner
Default uses murmur2(key) % N when key is set; sticky partitioner when key is null.
def tenant_partitioner(key, all_parts, available):
    # route tenant=premium to dedicated partitions 0..3
    if key and key.startswith(b"premium:"):
        return hash(key) % 4
    return hash(key) % len(all_parts)

KafkaProducer(partitioner=tenant_partitioner, …)
Headers + timestamps
p.send(
  "orders", key=b"u1", value=b"...",
  headers=[("trace-id", b"abc123"),
           ("schema", b"OrderV2")],
  timestamp_ms=int(time.time()*1000),
)
Async + callback
def on_delivery(err, msg):
    if err:
        log.error("failed: %s", err)
    else:
        log.debug("%s[%d]@%d",
                  msg.topic(), msg.partition(), msg.offset())

p.produce("orders", value=v, on_delivery=on_delivery)
p.poll(0)  # drive callbacks

acksDurabilityLatencyWhen
0None — fire and forgetLowestMetrics, fully replaceable telemetry.
1Survives if leader survivesLowHot paths where occasional loss is OK.
allSurvives min.insync.replicas brokersHigherDefault for everything important.

Compression & batching

KnobEffect
linger.msWait this long before sending a batch even if it isn't full. 5–20 ms gives big throughput wins.
batch.sizeBytes per partition batch (default 16 KB). 64 KB+ helps for high-throughput producers.
compression.typenone · gzip · snappy · lz4 · zstd. zstd usually best for ratio + CPU.
buffer.memoryTotal bytes the client may use to buffer un-sent records (default 32 MB).

Message flow

Producer → leader → followers (replication) → consumer.

Consumer code

Basic poll loop
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",
    auto_offset_reset="earliest",   # or "latest"
    enable_auto_commit=False,
    isolation_level="read_committed",
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # sync commit
Manual poll + offsets
Commit only after successful processing → at-least-once.
while True:
    batch = c.poll(timeout_ms=1000)
    for tp, msgs in batch.items():
        for m in msgs:
            handle(m)
        c.commit({tp: OffsetAndMetadata(msgs[-1].offset+1, None)})
Rebalance listener
Flush in-flight work before partitions are revoked.
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()
        consumer.commit()
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)

c.subscribe(["orders"], listener=L())
Seek to offset / time
# by offset
c.seek(TopicPartition("orders", 0), 12345)

# by timestamp (ms since epoch)
ts = {TopicPartition("orders", 0): 1714000000000}
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():
    c.seek(tp, off.offset)
Static membership
A stable group.instance.id avoids rebalances on rolling restarts.
KafkaConsumer(
  group_id="billing",
  group_instance_id="billing-pod-7",
  session_timeout_ms=30_000,
)
Cooperative rebalance
Incremental rebalance — only the partitions changing hands stop being processed, not all of them.
partition_assignment_strategy=[
  "org.apache.kafka.clients.consumer.CooperativeStickyAssignor"
]

Commit semantics

ModeGuaranteeRisk
enable_auto_commit=true (default 5s)At-most-once-ishCrash after commit, before processing → record dropped.
Manual commit() after processingAt-least-onceCrash after processing, before commit → reprocess on restart. Make handlers idempotent.
Transactional send_offsets_to_transaction()Exactly-once (read-process-write inside Kafka)Doesn't help when the side-effect is outside Kafka (e.g., a non-idempotent HTTP POST).
Pitfall: a long-running message handler that exceeds max.poll.interval.ms (default 5 min) gets kicked out of the group → rebalance storm. Either increase the limit, decrease max.poll.records, or move the work to a worker pool and use pause() / resume().

CLI

The shell scripts under $KAFKA_HOME/bin/ are your friend in production. Cheat-sheet below.

Topics

Create / list / describe
# create
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 12 --replication-factor 3 \
  --config retention.ms=604800000 \
  --config min.insync.replicas=2

# list / describe
kafka-topics.sh --bootstrap-server localhost:9092 --list
kafka-topics.sh --bootstrap-server localhost:9092 --describe --topic orders
Alter / delete
# add partitions (cannot decrease!)
kafka-topics.sh --bootstrap-server localhost:9092 \
  --alter --topic orders --partitions 24

# change a config
kafka-configs.sh --bootstrap-server localhost:9092 \
  --entity-type topics --entity-name orders \
  --alter --add-config retention.ms=86400000

# delete
kafka-topics.sh --bootstrap-server localhost:9092 \
  --delete --topic orders

Console produce / consume

Produce from stdin
kafka-console-producer.sh \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --property parse.key=true \
  --property key.separator=:
# type:    user-42:{"sku":"X","qty":2}
Consume from start
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --from-beginning \
  --property print.key=true \
  --property print.timestamp=true \
  --max-messages 50
Consume one partition / offset range
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --partition 3 \
  --offset 1000 \
  --max-messages 10

Consumer groups

List, describe lag
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 --list

kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe --group billing
# columns: TOPIC PARTITION CURRENT-OFFSET LOG-END-OFFSET LAG CONSUMER-ID HOST CLIENT-ID
Reset offsets
--dry-run first. Reset only when the group is stopped.
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group billing \
  --reset-offsets --to-earliest \
  --topic orders --execute

# to a timestamp:
  --to-datetime 2026-05-01T00:00:00.000

# shift back 1000 records:
  --shift-by -1000
Delete a group
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --delete --group billing

Misc useful

Get earliest / latest offsets
kafka-get-offsets.sh \
  --bootstrap-server localhost:9092 \
  --topic orders --time -1   # latest

kafka-get-offsets.sh \
  --bootstrap-server localhost:9092 \

kafka-dump-log.sh \

Streams & ksqlDB

Kafka Streams: a Java/Scala library for transforming topics. ksqlDB: SQL on top of Streams.

Streams DSL

Word count (the "hello world")
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))
   .groupBy((k,w) -> w)
   .count(Materialized.as("counts"))
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Stream-table join
Enrich an event stream with the latest state from a compacted topic.
KTable<String,User> users = b.table("users");
KStream<String,Order> orders = b.stream("orders");

orders.join(users,
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
Windowed aggregation
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(
         Duration.ofMinutes(5), Duration.ofMinutes(1)))
      .aggregate(() -> 0L,
                 (k, v, sum) -> sum + v.amount(),
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

ksqlDB

CREATE STREAM / TABLE
CREATE STREAM orders (
  user_id VARCHAR KEY,
  sku VARCHAR,
  qty INT,
  amount DOUBLE
) WITH (kafka_topic='orders', value_format='JSON');

CREATE TABLE users (
  user_id VARCHAR PRIMARY KEY,
  name VARCHAR, tier VARCHAR
) WITH (kafka_topic='users', value_format='JSON');
CSAS — derived stream
CREATE STREAM big_orders AS
SELECT o.user_id, u.tier, o.amount
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id
WHERE o.amount > 100
EMIT CHANGES;
CTAS — windowed table
CREATE TABLE spend_5min AS
SELECT user_id,
       SUM(amount) AS total
FROM orders
WINDOW TUMBLING (SIZE 5 MINUTES)
GROUP BY user_id
EMIT CHANGES;
Pull query
-- point-in-time read of a table's latest state
SELECT total FROM spend_5min
WHERE user_id = 'user-42';

Ops & Tuning

The knobs you'll actually turn in production, plus the alarms that wake you up.

Replication & durability

ConfigDefaultWhat & when
replication.factor (per topic)—3 in prod. Survives one broker loss with min.insync.replicas=2.
min.insync.replicas1Set 2 with RF=3. Combined with acks=all = no data loss on single-broker fault.
unclean.leader.election.enablefalseKeep false. Allowing it trades durability for availability.
replica.lag.time.max.ms30000Followers slower than this drop out of ISR.
auto.create.topics.enabletrueSet false in prod. Make topic creation an explicit step.

Retention & storage

ConfigWhat it does
retention.msDelete records older than this. Default 7 days. Per-topic.
retention.bytesCap per-partition log size. Per-topic. Whichever (ms / bytes) hits first wins.
segment.bytes / segment.msRoll a new log segment every X bytes or X ms. Smaller segments = faster deletes/compaction, more file handles.
cleanup.policy=compactLog compaction — keep latest value per key forever. Use for state topics, changelogs, __consumer_offsets.
cleanup.policy=delete,compactBoth: compact + drop old keys after retention.
min.cleanable.dirty.ratioHow dirty before the compactor runs (default 0.5).

Common alarms

SymptomWhat to check
Under-replicated partitions > 0Broker down, slow disk, or network. kafka-topics.sh --describe --under-replicated-partitions.
Offline partitions > 0No leader available. Cluster is in trouble.
Consumer lag growingConsumer slow / stuck. Check max.poll.interval.ms, GC pauses, downstream system.
Producer RecordTooLargeExceptionBump max.request.size on producer + message.max.bytes on broker + max.partition.fetch.bytes on consumer.
Group rebalance stormSlow handlers, unstable members, GC. Use static membership + cooperative-sticky.
ISR shrinks frequentlyNetwork or disk issue. Check replica.lag.time.max.ms, broker IO wait.

Cluster sizing rules of thumb

Partitions per broker
Aim for < 4000 partitions per broker. Each partition costs file handles, memory, and rebalance time.
Partition count per topic
≥ max parallelism you'll ever need. Pick a generous number up-front — increasing later breaks per-key ordering.
Disks
XFS, large sequential writes. Don't share with anything bursty. Plan on 50%+ free for log compaction.
Network
Replication is north-south traffic too. RF=3 means each produced byte traverses the network ~3×.
JVM
6-8 GB heap is plenty. Most memory should go to the page cache.
KRaft (no ZooKeeper)
3.3+ defaults to KRaft. ZooKeeper removed in 4.0. New clusters: KRaft only.

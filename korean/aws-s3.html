AWS S3 CLI — ls부터 put-bucket-policy까지

두 개의 CLI 패밀리, 하나의 스토리지 서비스. aws s3는 일상 작업용 친절한 고수준 래퍼입니다 — copy, sync, presign. aws s3api는 REST API를 그대로 미러링한 저수준 인터페이스로, 버저닝, 라이프사이클, 멀티파트 업로드, 버킷 정책 같은 세밀한 작업을 다룹니다. 타이핑하면서 둘 다 익혀봅시다.

aws s3 — 고수준
파일 시스템처럼 느껴지는 명령들: ls, cp, mv, sync, rm, mb, rb, presign. 멀티스레드 업로드, 큰 파일 멀티파트, 재시도 같은 똑똑한 동작을 자동으로 처리해주며, 90%의 경우에 이 도구를 사용하게 됩니다.
aws s3api — 저수준
S3 API 동사 하나당 서브커맨드 하나씩: list-objects-v2, head-object, put-bucket-versioning, create-multipart-upload, … aws s3에 없는 기능이 필요하거나, --query / jq에 넘길 raw JSON 출력이 필요할 때 사용합니다.

천 번은 타이핑하게 될 S3 URI

# scheme   버킷 이름        key (버킷 내 객체 경로)
s3://my-bucket/photos/2026/cat.jpg

# "폴더"는 그냥 key prefix일 뿐 — 객체로 실제 존재하지는 않음
s3://my-bucket/photos/2026/
주의: S3 key는 진짜 파일시스템이 아닙니다. "디렉터리" photos/2026/는 그 prefix로 시작하는 key를 가진 객체가 있기 때문에 존재할 뿐입니다. 빈 폴더는 /로 끝나는 0바이트 객체입니다.

IAM: 가장 많이 쓰는 4개의 S3 액션

액션허용 동작리소스 예시
s3:ListBucket주어진 prefix의 key 목록 조회arn:aws:s3:::my-bucket
s3:GetObject객체 다운로드arn:aws:s3:::my-bucket/*
s3:PutObject객체 업로드arn:aws:s3:::my-bucket/*
s3:DeleteObject객체 삭제 (또는 delete-marker 부착)arn:aws:s3:::my-bucket/*
버킷 vs 객체 액션: s3:ListBucket는 버킷 ARN을 대상으로 합니다 (/* 없음). s3:GetObject는 그 안의 객체를 대상으로 합니다 (/* 포함). 이 둘을 혼동하는 것이 새 정책에서 "AccessDenied"가 나는 1순위 원인입니다.

이 페이지 사용법

  • 모든 섹션에는 직접 타이핑하는 드릴이 있습니다. Enter로 확인하고, 힌트로 정답을 보여주거나 다시 숨길 수 있습니다.
  • 상단에 점수, 연속 정답, 진행 바가 업데이트됩니다. 섹션을 완료하면 색종이가 터집니다.
  • 자매 페이지 aws.html는 더 넓은 AWS CLI & IAM을 다룹니다. 이 페이지는 S3 전용입니다.

목록 & 조회 — ls, head-object, list-objects-v2, --query

클라우드 스토리지의 제1계명: 손대기 전에 버킷에 뭐가 있는지부터 알아야 합니다. 두 CLI 패밀리 모두 객체 목록을 보여주지만, s3api는 JSON을 반환해서 --query로 JMESPath를 써서 잘라낼 수 있습니다.

aws s3api list-objects-v2 — 프로그래밍 방식 목록 조회

# prefix p/ 아래 최대 10개 key를 조회하고, key + size만 추출
$ aws s3api list-objects-v2 \
    --bucket my-bucket \
    --prefix p/ \
    --max-items 10 \
    --query 'Contents[].{Key:Key,Size:Size}'

# 연속 토큰을 통한 페이지네이션
$ aws s3api list-objects-v2 --bucket my-bucket --max-keys 1000 \
    --starting-token "$NEXT_TOKEN"
3 나열된 모든 객체에서 Key 필드만 추출하세요.
$ aws s3api list-objects-v2 --bucket my-bucket \
    --query 'Contents[].'

JMESPath 속성 강좌 — --query 플래그

모든 aws 호출은 JSON을 반환합니다. --query 'EXPR'은 --output이 포맷하기 전에 클라이언트 측에서 JMESPath를 실행합니다. 

# 버킷 이름만
$ aws s3api list-buckets --query 'Buckets[].Name'

# 1 MiB보다 큰 객체 (숫자에는 백틱 사용)
$ aws s3api list-objects-v2 --bucket B \
    --query 'Contents[?Size > `1048576`].Key'

# 가장 최근 5개 객체
$ aws s3api list-objects-v2 --bucket B \
    --query 'reverse(sort_by(Contents, &LastModified))[:5].[Key,LastModified]' \
    --output table

# prefix의 총 크기 (합산)
$ aws s3api list-objects-v2 --bucket B --prefix logs/ \
    --query 'sum(Contents[].Size)'
5 list-buckets에서 버킷 이름만 반환하세요.
$ aws s3api list-buckets --query 'Buckets[].'
6 JMESPath: 1 MiB보다 큰 객체만 남기세요. 키워드를 채우세요.
--query 'Contents[Size > `1048576`].Key'

업로드, 다운로드, 동기화 — cp, mv, sync, exclude/include

이 네 가지 동사 (cp, mv, sync, rm)가 S3로 들어가고 나가는 바이트의 95%를 처리합니다. --delete를 입력하기 전에 --exclude / --include 순서와 --dryrun을 먼저 마스터하세요.

cp — 단일 파일 또는 재귀적

# 업로드
$ aws s3 cp ./report.pdf s3://my-bucket/reports/

# 다운로드
$ aws s3 cp s3://my-bucket/reports/report.pdf ./

# 재귀적 — 전체 디렉터리
$ aws s3 cp ./build/ s3://my-bucket/site/ --recursive

# 스토리지 클래스, 암호화, content-type, 메타데이터 포함
$ aws s3 cp big.bin s3://my-bucket/data/big.bin \
    --storage-class STANDARD_IA \
    --sse aws:kms --sse-kms-key-id alias/data \
    --content-type application/octet-stream \
    --metadata ingest=2026-05-20,team=ml \
    --acl private
1 로컬 디렉터리 전체를 S3의 prefix로 업로드하세요.
$ aws s3 cp ./build/ s3://my-bucket/site/ --
2 업로드를 Infrequent-Access로 표시하세요. 스토리지 클래스를 채우세요.
$ aws s3 cp file.txt s3://b/k --storage-class

mv — 복사 + 원본 삭제

# 빌드 아티팩트를 staging에서 release prefix로 승격
$ aws s3 mv s3://b/staging/v1.tar.gz s3://b/release/v1.tar.gz

# 로컬 → S3 이동 (성공하면 로컬 파일 삭제)
$ aws s3 mv ./done.zip s3://b/archive/

sync — 디렉터리 단위의 슈퍼파워

# 로컬 → S3 미러링, *추가*하고 *갱신*하지만 삭제는 안 함
$ aws s3 sync ./public/ s3://my-site/

# 삭제 포함 미러링: S3가 로컬과 정확히 일치하게 됨
$ aws s3 sync ./public/ s3://my-site/ --delete

# 쓸모없는 파일은 건너뛰고, JS와 CSS만 전송
$ aws s3 sync ./dist/ s3://my-site/ \
    --exclude "*" \
    --include "*.js" \
    --include "*.css"

# 실제 실행 없이 변경될 내용 확인
$ aws s3 sync ./public/ s3://my-site/ --delete --dryrun
순서가 중요합니다: --exclude / --include는 왼쪽에서 오른쪽으로 평가되며 마지막 매치가 이깁니다. 먼저 --exclude "*"로 패턴을 잡고, 원하는 부분만 다시 include하세요.
3 로컬 디렉터리를 S3로 미러링하되, 로컬에서 삭제된 파일은 S3에서도 삭제되고 .tmp 파일은 건너뛰도록 하세요.
$ aws s3  local/ s3://bucket/prefix/ -- --exclude "*.tmp"
4 파괴적 sync가 실행했을 결과를 실제 실행 없이 미리 보세요.
$ aws s3 sync ./public/ s3://site/ --delete --

      rm — 특히 --recursive와 함께 쓸 때 조심하세요

      # 단일 객체
$ aws s3 rm s3://b/old.log

# 전체 prefix — 확인 따위 없음!
$ aws s3 rm s3://b/tmp/ --recursive

# 선택적: logs/는 날리되 .json 파일은 유지
$ aws s3 rm s3://b/logs/ --recursive --exclude "*.json"
      버저닝 주의: 버저닝된 버킷에서 aws s3 rm --recursive는 delete marker만 추가합니다 — 이전 버전은 여전히 남아 있습니다. 진짜로 공간을 확보하려면 s3api로 버전을 명시적으로 삭제해야 합니다.

      s3api 파워 도구 — S3 API 액션당 동사 하나

      친절한 aws s3 명령으로 부족할 때 s3api로 내려갑니다. 출력은 항상 JSON이므로 — --query 또는 jq로 파이프하세요.

      객체 단위 작업

      # 명시적 content-type과 메타데이터로 로컬 파일 업로드
$ aws s3api put-object \
    --bucket my-bucket \
    --key data/blob.bin \
    --body ./blob.bin \
    --content-type application/octet-stream \
    --metadata sha256=abcd...,team=ml

# 파일로 다운로드
$ aws s3api get-object --bucket my-bucket --key data/blob.bin out.bin

# 서버 측 복사 (당신의 노트북을 거치는 바이트 없음)
$ aws s3api copy-object \
    --bucket dest-bucket \
    --key archive/blob.bin \
    --copy-source my-bucket/data/blob.bin

# 특정 버전 하나 삭제
$ aws s3api delete-object \
    --bucket my-bucket --key data/blob.bin \
    --version-id 3HL4kqtJlcpXroDTDmJ+rmSpXd3dIbrHY
      1 srcB/srcK에서 dstB/dstK로 서버 측 복사하세요. 동사와 소스 플래그를 채우세요.
      $ aws s3api  \
    --bucket dstB --key dstK \
    -- srcB/srcK

      객체 태그 지정

      $ aws s3api put-object-tagging \
    --bucket my-bucket --key data/x.bin \
    --tagging '{"TagSet":[{"Key":"env","Value":"prod"},{"Key":"team","Value":"ml"}]}'

$ aws s3api get-object-tagging --bucket my-bucket --key data/x.bin
      2 객체에 현재 설정된 태그를 읽어오세요.
      $ aws s3api  --bucket B --key K

      버저닝

      # 버저닝 켜기 (사실상 비가역 — Suspend만 가능, Disable은 불가)
$ aws s3api put-bucket-versioning \
    --bucket my-bucket \
    --versioning-configuration Status=Enabled

# 현재 설정 확인
$ aws s3api get-bucket-versioning --bucket my-bucket
{ "Status": "Enabled" }

# 객체의 모든 버전
$ aws s3api list-object-versions --bucket my-bucket --prefix data/blob.bin
      3 버킷에서 버저닝을 활성화하세요. 상태 값을 채우세요.
      $ aws s3api put-bucket-versioning --bucket B \
    --versioning-configuration Status=

      대형 치트시트 — 가장 자주 보게 될 s3api 동사들

      동사필수 플래그사용 사례
      list-buckets없음계정 전체 버킷 목록
      list-objects-v2--bucketkey 페이징 조회
      head-object--bucket --key메타데이터, ETag, 버전, 암호화
      head-bucket--bucket존재하는지 & 볼 수 있는지?
      put-object--bucket --key --body파일 한 개 업로드
      get-object--bucket --key + outfile디스크로 다운로드
      delete-object--bucket --key삭제 (또는 delete-marker 부착)
      delete-objects--bucket --delete JSON최대 1000개 key 일괄 삭제
      copy-object--bucket --key --copy-source서버 측 이름 변경 / 버킷 간 복사

      버킷 & 정책 — mb, rb, policy, CORS, lifecycle, public-access block

      버킷 단위 설정은 보안과 비용이 살거나 죽는 곳입니다. 이 명령들은 버저닝, 퍼블릭 액세스 차단, CORS, 라이프사이클 규칙, 버킷 정책을 설정합니다.

      버킷 생성 & 삭제

      # 생성. LocationConstraint를 위해 리전이 중요 (us-east-1은 암묵적 / "none")
$ aws s3 mb s3://my-new-bucket --region us-east-1

# 삭제. --force는 비우기까지 함 (고수준 전용).
$ aws s3 rb s3://my-new-bucket --force

# s3api 대응 명령 (더 많은 옵션)
$ aws s3api create-bucket --bucket my-new-bucket \
    --create-bucket-configuration LocationConstraint=eu-west-1

$ aws s3api delete-bucket --bucket my-new-bucket
      1 us-east-1에 cat-photos라는 새 버킷을 생성하세요.
      $ aws s3  s3://cat-photos --region us-east-1
      2 비어 있지 않은 버킷을 한 번에 삭제하세요. 플래그를 고르세요.
      $ aws s3 rb s3://cat-photos --

      모든 퍼블릭 액세스 차단 (가장 먼저 하세요!)

      $ aws s3api put-public-access-block \
    --bucket my-bucket \
    --public-access-block-configuration \
        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

$ aws s3api get-public-access-block --bucket my-bucket
      계정 단위 방어막: 동일한 플래그가 aws s3control put-public-access-block --account-id …로도 존재합니다 — 거기서도 설정해두면 나중에 만든 버킷 정책이 실수로 구멍을 내지 못합니다.

      버킷 정책

      # policy.json — 모두에게 GetObject 허용 (읽기 전용 공개 버킷)
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid":    "PublicRead",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-public-bucket/*"
  }]
}

$ aws s3api put-bucket-policy --bucket my-public-bucket \
    --policy file://policy.json

$ aws s3api get-bucket-policy --bucket my-public-bucket
$ aws s3api delete-bucket-policy --bucket my-public-bucket
      3 디스크에 저장된 정책을 버킷에 적용하세요.
      $ aws s3api put-bucket-policy --bucket B --policy

      멀티파트 & 사전 서명 URL — 큰 파일과 공유 링크용

      고수준 aws s3 cp는 ~8 MB보다 큰 파일에 멀티파트를 자동 적용합니다. 프로그래밍 방식으로 업로드를 직접 다룰 때 쓰는 s3api 원시 명령들은 다음과 같습니다.

      멀티파트 업로드 — 4단계 댄스

      ## 1. 시작 — UploadId 획득
$ UID=$(aws s3api create-multipart-upload \
    --bucket B --key big.bin \
    --query 'UploadId' --output text)

## 2. 각 파트 업로드 (마지막 파트 제외 최소 5 MB)
$ ETAG1=$(aws s3api upload-part \
    --bucket B --key big.bin \
    --part-number 1 --upload-id $UID \
    --body part1.bin \
    --query 'ETag' --output text)
#  …파트 2..N에 대해 반복…

## 3. 완료 — 파트 목록을 S3에 주면 서버 측에서 합쳐줌
$ aws s3api complete-multipart-upload \
    --bucket B --key big.bin --upload-id $UID \
    --multipart-upload '{"Parts":[{"ETag":"...","PartNumber":1},...]}'

## 4 (대체). 취소 — 그렇지 않으면 청구될 스토리지를 회수
$ aws s3api abort-multipart-upload \
    --bucket B --key big.bin --upload-id $UID

## 고아 업로드 보기 (이게 돈 듭니다!)
$ aws s3api list-multipart-uploads --bucket B
      라이프사이클 규칙을 AbortIncompleteMultipartUpload: { DaysAfterInitiation: 7 }로 추가하면 실패한 업로드가 조용히 청구액을 부풀리지 않습니다.
      1 멈춰 있는 멀티파트 업로드를 중지해 스토리지를 회수하세요.

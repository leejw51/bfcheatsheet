Stash 그만.
병렬로 브랜치하기.

git worktree는 하나의 저장소가 여러 작업 디렉토리를 가질 수 있게 합니다 — 서로 다른 브랜치가 모두 동시에 체크아웃되어, 하나의 .git을 공유합니다. XP, 업적, 최종 보스가 있는 타이핑 중심 도장으로 마스터하세요.

~/projects/cro-economy-service
파일시스템1 워크트리
데모 · zsh

01 Worktree가 해결하는 문제

기능 작업에 깊이 빠져있습니다. 프로덕션이 불타오릅니다. 지금 당장 main을 고쳐야 합니다. Worktree 없이는 모든 선택이 고통스럽습니다.

고통 1 컨텍스트 전환 비용

stash하고, main으로 전환하고, 고치고, 푸시하고, 다시 전환하고, stash를 팝하고, 충돌이 없기를 바랍니다. stash pop의 WIP 머지 충돌은 특별한 종류의 고통입니다.

고통 2 병렬 작업

빌드가 6분 걸립니다. CI가 브랜치 A에서 실행되는 동안, 다시 빌드하지 않고 브랜치 B에서 코드를 작성하고 싶습니다. 하나의 체크아웃 = 한 번에 하나의 브랜치.

고통 3 PR 리뷰

git checkout pr/42를 실행해서 로컬에서 돌려보고 싶지만, 현재 브랜치에 진행 중인 변경사항을 날리고 싶지도 않습니다.

Worktree의 해답

다른 브랜치에 고정된 두 번째 작업 디렉토리를 추가합니다. 둘 다 동일한 .git 객체 데이터베이스를 공유합니다 — fetch, ref, hook, config — 하지만 디스크 상의 파일은 독립적입니다. checkout이 아니라 cd로 전환합니다.

# 이전: 스태시 댄스
git stash push -m wip
git checkout main
# ...수정, 커밋, 푸시...
git checkout feature/long-thing
git stash pop   # 🤞

# 이후: 병렬 작업 디렉토리
git worktree add ../hotfix main
cd ../hotfix
# ...수정, 커밋, 푸시, 그리고:
cd ../cro-economy-service
git worktree remove ../hotfix

02 해부: 공유된 .git, 별도의 작업 디렉토리

워크트리는 메인 리포지토리 디렉토리 외부에 존재하지만 여전히 메인 리포지토리의 .git이 추적하는 체크아웃입니다.

공유되는 것

객체 데이터베이스 (.git/objects): 커밋, 트리, 블롭.

Ref: 브랜치, 태그, 리모트 — 단일한 진실의 소스.

Config & hook: 메인 리포지토리에서 한 번만 정의됨.

분리되는 것

작업 트리: 디스크에 실제로 체크아웃된 파일.

Index: 각 워크트리는 자체 스테이징 영역을 가집니다.

HEAD: 각 워크트리는 자체 커밋/브랜치를 가리킵니다.

메타데이터의 위치

메인 리포지토리는 연결된 각 워크트리를 .git/worktrees/<name>/ 아래에 기록합니다. 연결된 각 워크트리는 .git 파일 (디렉토리가 아님)을 가지고, 이는 gitdir:을 통해 해당 메타데이터를 다시 가리킵니다.

04 일반 패턴

키 입력 비용이 즉시 회수되는 세 가지 워크플로우.

핫픽스 모든 걸 던지고

프로덕션이 다운됐습니다. 기능 작업에서 14커밋 깊이 들어가 있습니다.

git fetch origin
git worktree add -b hotfix/login \
  ../hotfix origin/main
cd ../hotfix
# 수정, 커밋, 푸시
cd -
git worktree remove ../hotfix

리뷰 PR을 로컬에서 실행

진행 중인 작업을 망가뜨리지 않고 PR #42를 리뷰합니다.

git fetch origin pull/42/head:pr/42
git worktree add ../review pr/42
cd ../review
# 테스트 실행, 살펴보기
cd -
git worktree remove ../review

긴 테스트 옆에서 CI 실행

코딩을 계속하면서 release/v2에서 통합 테스트 스위트를 실행합니다.

git worktree add ../e2e release/v2
cd ../e2e
make integration-test &
cd -
# 메인 리포지토리에서 작업 계속

05 외울 가치 있는 함정

실제로 부딪힐 워크트리 버그의 거의 전부가 이 다섯 가지 중 하나입니다.

! add 시 Detached HEAD

브랜치 대신 커밋/태그를 전달하면, 새 워크트리는 detached HEAD 상태가 됩니다. 새 브랜치를 만들려면 -b를 사용하세요.

# detached:
git worktree add ../tmp v1.4.0
# 브랜치:
git worktree add -b fix-1.4 ../tmp v1.4.0

! 브랜치가 이미 체크아웃됨

Git은 다른 워크트리에 이미 체크아웃된 브랜치 체크아웃을 거부합니다 (그러면 두 HEAD가 하나의 ref를 짓밟게 됩니다). 다른 브랜치를 고르거나, 정말 확신이 있다면 --force를 사용하세요.

fatal: 'feature-x' is already checked out at '/path/A'
# 강제 (대개 좋은 생각이 아님):
git worktree add --force ../B feature-x

! 수동 rm은 메타데이터를 남깁니다

워크트리 디렉토리를 rm -rf로 지우면 .git/worktrees/<name>이 남습니다. git worktree list는 그것을 여전히 prunable로 표시합니다. git worktree prune을 실행하세요.

git worktree list
# 표시: ../old  (prunable: gitdir is missing)
git worktree prune -v

! 서브모듈 & 워크트리별 config

서브모듈은 워크트리 간에 자동으로 init되지 않습니다 — 각각에서 git submodule update --init을 실행하세요. 일부 config (sparse-checkout, hook)는 --worktree로 옵트인할 때만 워크트리별로 적용됩니다.

! 잘못된 것을 제거하기

remove는 워크트리에 커밋되지 않은 변경사항이 있으면 거부합니다. 반사적으로 --force로 손이 가지 않게 하세요 — 먼저 cd해서 들어가 확인하세요.

· Lock으로 보호하기

워크트리가 분리 가능한 디스크나 느린 NFS 마운트에 있다면, git worktree lock으로 git에게 경로가 도달 불가능하다는 이유만으로 prune하지 말라고 알리세요.

06 Worktree vs stash vs 다중 클론

작업을 저글링하는 세 가지 방법. 서로 교환 가능하지 않습니다.

git stash다중 클론git worktree
설정 비용 즉시 재클론, 재페치, 재구성 한 번의 명령, 재페치 불필요
디스크: 객체 데이터베이스 공유 클론마다 중복 공유
Ref / fetch 동기화 해당 없음 각각 fetch 필요 한 번 fetch, 모든 워크트리가 인지
두 브랜치를 동시에 실행 불가 가능 가능
작업 손실 위험 스태시 목록, pop 시 충돌 낮음 파일은 디스크에 그대로
적합한 상황 아주 짧은 즉석 전환 (< 1분) 복사본마다 진짜로 다른 SDK / 툴링 병렬 브랜치, 동일한 툴체인

07 Bare 저장소 + worktree: 모노레포 트릭

일부 팀은 "메인 체크아웃"을 완전히 건너뛰고, 리포지토리를 워크트리 폴더 모음으로 취급합니다.

설정 방법

git clone --bare git@github.com:org/repo.git repo.git
cd repo.git
git worktree add ../repo/main main
git worktree add ../repo/feature-x feature-x
git worktree add ../repo/release release/v2

레이아웃:

repo.git/        # 공유 객체 DB
repo/
  main/          # main 브랜치의 워크트리
  feature-x/     # feature-x 브랜치의 워크트리
  release/       # release/v2의 워크트리

사람들이 좋아하는 이유

혼란을 줄 "메인" 워크트리가 없습니다 — 모든 브랜치가 동등한 형제 디렉토리입니다. 툴링 (IDE, devcontainer, 브랜치별 빌드 캐시)이 폴더에 깔끔하게 매핑됩니다.

트레이드오프: 베어 리포지토리 안에서는 일부 git 명령이 약간 어색하게 느껴집니다. 대부분은 cd repo/main을 별칭으로 두고 베어가 존재한다는 사실을 잊어버립니다.

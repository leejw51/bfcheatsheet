⌕
v3.7+

개념

브로커는 클러스터를 구성합니다. 토픽은 파티션으로 분할됩니다. 프로듀서는 추가하고, 컨슈머는 오프셋으로 읽습니다. 그게 전부입니다.

클러스터 해부

프로듀서는 토픽에 레코드를 씁니다. 파티셔너가 어떤 파티션인지 선택합니다. 각 파티션은 복제됩니다. 그룹 내 컨슈머들은 자기들끼리 파티션을 나눕니다.

용어

용어의미
brokerKafka 서버. 클러스터 = 여러 브로커. 각 브로커는 숫자형 broker.id로 식별됩니다.
topic이름이 있는 레코드 로그. 파티션 개수와 복제 계수(replication factor)로 생성합니다.
partition순서가 있는 append-only 커밋 로그. 한 파티션 내부의 레코드는 엄격히 순서가 보장됩니다. 파티션 사이에는 순서 보장이 없습니다.
offset파티션 내부의 단조 증가 64비트 위치. 컨슈머는 자신이 어디까지 읽었는지 오프셋으로 추적합니다.
leader / follower각 파티션마다 한 브로커가 리더(읽기 + 쓰기 처리)이고 나머지는 복제합니다. 리더 장애 시 팔로워 중 하나가 승격됩니다.
ISRIn-Sync Replicas: replica.lag.time.max.ms 이내로 따라잡은 팔로워들. ISR 멤버만 리더가 될 자격이 있습니다.
consumer group같은 group.id를 공유하는 컨슈머 집합. Kafka는 각 파티션을 그룹 내 한 컨슈머에게만 동시에 할당합니다.
rebalance그룹 멤버십이 변할 때(컨슈머가 합류/탈퇴/사망) 일어나는 파티션 재할당.
__consumer_offsets커밋된 오프셋이 저장되는 내부 컴팩션 토픽 (기본 50개 파티션).

순서 규칙

파티션 단위 순서만 보장
한 파티션 내 레코드는 엄격히 순서가 보장됩니다. 파티션 간에는 순서 없음. 같은 키 → 기본적으로 같은 파티션 → 그 키에 한해 순서 보장.
# 순서 함정
# key=user-42 → hash(user-42) % N 파티션으로 들어감
# user-42 이벤트는 순서 유지. user-99 와는 인터리빙 가능.
acks & 내구성
acks=all + min.insync.replicas≥2 = 단일 브로커 장애 시 데이터 손실 없음.
acks=0    # 발사 후 망각 (fire and forget)
acks=1    # 리더만 대기
acks=all  # 모든 ISR 대기
전달 시맨틱
기본은 at-least-once. exactly-once는 멱등 프로듀서 + 트랜잭션 + read_committed 컨슈머가 필요.
# exactly-once 레시피
enable.idempotence=true          # 멱등 프로듀서
transactional.id=order-app-1      # 고유 트랜잭션 ID
# 컨슈머 측:
isolation.level=read_committed    # 커밋된 메시지만 읽기
함정: 나중에 파티션을 늘려도 기존 데이터는 이동하지 않습니다. 기존 키는 옛 파티션에 남고, 새 키는 다른 곳에 떨어질 수 있어 — 그 키의 순서가 두 파티션에 걸쳐 잠시 갈라집니다. 파티션 개수는 처음에 넉넉히 잡거나, 키 인지(re-partitioning) 잡을 따로 돌리세요.

프로듀서

프로듀서는 토픽에 레코드를 추가합니다. 키, ack 정책, 파티셔너를 고르는 것이 설계의 80%입니다.

빠른 시작 (kafka-python)

기본 프로듀서

import json

p = KafkaProducer(
    bootstrap_servers="broker1:9092,broker2:9092",
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode() if k else None,
    acks="all",                # 모든 ISR 응답 대기
    enable_idempotence=True,

)
p.send("orders", key="user-42", value={"sku": "X", "qty": 2})
p.flush()

커스텀 파티셔너
기본은 키가 있을 때 murmur2(key) % N; 키가 null이면 sticky 파티셔너.
def tenant_partitioner(key, all_parts, available):
    # tenant=premium 을 전용 파티션 0..3 으로 라우팅
    if key and key.startswith(b"premium:"):
        return hash(key) % 4            # 프리미엄 전용 파티션
    return hash(key) % len(all_parts)   # 일반은 전체 파티션

KafkaProducer(partitioner=tenant_partitioner, …)

    if err:
        log.error("failed: %s", err)    # 실패 로그

acks vs 내구성 vs 지연시간

acks내구성지연시간언제
0없음 — 발사 후 망각최저지표(metrics), 완전히 대체 가능한 텔레메트리.
1리더가 살아남으면 보존낮음가끔 손실이 허용되는 핫 패스.
allmin.insync.replicas 만큼의 브로커가 보존더 높음중요한 모든 것의 기본값.

압축 & 배칭

다이얼효과
linger.ms배치가 차지 않아도 이만큼 기다렸다가 전송. 5–20 ms면 처리량이 크게 늘어납니다.
batch.size파티션당 배치 바이트 (기본 16 KB). 고처리량 프로듀서는 64 KB+ 가 유리.
compression.typenone · gzip · snappy · lz4 · zstd. 압축률 + CPU 균형은 보통 zstd가 최선.
buffer.memory미전송 레코드를 버퍼링하는 데 클라이언트가 사용할 수 있는 총 바이트 (기본 32 MB).

컨슈머 코드

기본 폴 루프
from kafka import KafkaConsumer

c = KafkaConsumer(
    "orders",
    bootstrap_servers="broker1:9092",
    group_id="billing",                # 컨슈머 그룹 식별자
    auto_offset_reset="earliest",   # 또는 "latest"
    enable_auto_commit=False,           # 수동 커밋 권장
    isolation_level="read_committed",  # 트랜잭션 메시지만 읽기
    max_poll_records=500,
    session_timeout_ms=10_000,
)
for msg in c:
    handle(msg.value)
    c.commit()  # 동기 커밋

처리 성공 후에만 커밋 → at-least-once.
while True:
    batch = c.poll(timeout_ms=1000)  # 1초 폴

리밸런스 리스너
파티션이 회수되기 전에 진행 중 작업을 비웁니다(flush).
class L(ConsumerRebalanceListener):
    def on_partitions_revoked(self, revoked):
        flush_buffer()              # 회수 전 진행 중 작업 처리
        consumer.commit()             # 마지막 오프셋 동기 커밋
    def on_partitions_assigned(self, assigned):
        log.info("now own %s", assigned)  # 새로 할당된 파티션 로그

c.subscribe(["orders"], listener=L())
오프셋 / 시간으로 seek
# 오프셋으로
c.seek(TopicPartition("orders", 0), 12345)   # 파티션 0의 오프셋 12345

# 타임스탬프(epoch ms)로
ts = {TopicPartition("orders", 0): 1714000000000}   # 특정 시각
offsets = c.offsets_for_times(ts)
for tp, off in offsets.items():

커밋 시맨틱

모드보장위험
enable_auto_commit=true (기본 5초)대략 at-most-once커밋 후 처리 전 크래시 → 레코드 유실.
처리 후 수동 commit()At-least-once처리 후 커밋 전 크래시 → 재시작 시 재처리. 핸들러를 멱등하게 만드세요.
트랜잭셔널 send_offsets_to_transaction()Exactly-once (Kafka 내부의 read-process-write)부수 효과가 Kafka 외부일 때(예: 비-멱등 HTTP POST)에는 도움이 되지 않습니다.
함정: max.poll.interval.ms(기본 5분)를 초과하는 장시간 메시지 핸들러는 그룹에서 추방됩니다 → 리밸런스 폭풍. 한도를 늘리거나, max.poll.records를 줄이거나, 워커 풀로 작업을 옮기고 pause() / resume()를 사용하세요.

토픽

생성 / 목록 / 상세
# 생성
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 12 --replication-factor 3 \
  --config retention.ms=604800000 \
  --config min.insync.replicas=2

# 목록 / 상세
kafka-topics.sh --bootstrap-server localhost:9092 --list           # 모든 토픽

변경 / 삭제
# 파티션 추가 (감소는 불가!)
kafka-topics.sh --bootstrap-server localhost:9092 \
  --alter --topic orders --partitions 24       # 12 → 24 로 증가

# 설정 변경
kafka-configs.sh --bootstrap-server localhost:9092 \
  --entity-type topics --entity-name orders \
  --alter --add-config retention.ms=86400000   # 보존 1일

# 삭제
kafka-topics.sh --bootstrap-server localhost:9092 \
  --delete --topic orders                      # 비가역적!

Streams & ksqlDB

Kafka Streams: 토픽을 변환하는 Java/Scala 라이브러리. ksqlDB: Streams 위의 SQL.

Streams DSL

단어 세기 ("hello world")
StreamsBuilder b = new StreamsBuilder();
KStream<String,String> src = b.stream("text-input");   // 입력 토픽
src.flatMapValues(v -> Arrays.asList(v.split("\\W+")))   // 단어 분리
   .groupBy((k,w) -> w)                // 단어 키로 그룹
   .count(Materialized.as("counts"))    // 상태 저장소
   .toStream()
   .to("word-counts", Produced.with(Serdes.String(), Serdes.Long()));
Stream-Table 조인
컴팩트 토픽의 최신 상태로 이벤트 스트림을 풍부화(enrich)합니다.
KTable<String,User> users = b.table("users");    // 테이블 = 최신 상태
KStream<String,Order> orders = b.stream("orders");  // 스트림 = 이벤트

orders.join(users,                   // 같은 키로 조인
       (order, user) -> new EnrichedOrder(order, user))
      .to("orders-enriched");
윈도우 집계 (windowed aggregation)
orders.groupByKey()
      .windowedBy(TimeWindows.ofSizeAndGrace(    // 5분 윈도우
         Duration.ofMinutes(5), Duration.ofMinutes(1)))   // 1분 유예
      .aggregate(() -> 0L,                // 초깃값
                 (k, v, sum) -> sum + v.amount(),       // 누산
                 Materialized.with(Serdes.String(), Serdes.Long()))
      .toStream()
      .to("5min-totals");

CREATE STREAM orders (
  user_id VARCHAR KEY,           -- 메시지 키
  sku VARCHAR,
  qty INT,
  amount DOUBLE
) WITH (kafka_topic='orders', value_format='JSON');

CREATE TABLE users (                   -- 컴팩트 토픽 = 테이블
  user_id VARCHAR PRIMARY KEY,
  name VARCHAR, tier VARCHAR
) WITH (kafka_topic='users', value_format='JSON');
CSAS — 파생 스트림
CREATE STREAM big_orders AS      -- 파생 스트림 생성
SELECT o.user_id, u.tier, o.amount
FROM orders o
LEFT JOIN users u ON o.user_id = u.user_id   -- 풍부화
WHERE o.amount > 100                   -- 큰 주문만
EMIT CHANGES;
CTAS — 윈도우 테이블
CREATE TABLE spend_5min AS     -- 윈도우 집계 테이블
SELECT user_id,
       SUM(amount) AS total       -- 사용자별 합계
FROM orders
WINDOW TUMBLING (SIZE 5 MINUTES)  -- 5분 텀블링
GROUP BY user_id
EMIT CHANGES;
Pull 쿼리
-- 테이블의 최신 상태를 시점(point-in-time) 조회
SELECT total FROM spend_5min
WHERE user_id = 'user-42';

복제 & 내구성

설정기본값의미 & 시점
replication.factor (토픽별)—프로덕션은 3. min.insync.replicas=2와 함께면 브로커 1대 손실 시에도 살아남음.
min.insync.replicas1RF=3이면 2로 설정. acks=all과 합치면 단일 브로커 장애 시 데이터 손실 없음.
unclean.leader.election.enablefalsefalse 유지. 허용은 가용성을 위해 내구성을 포기하는 트레이드오프.
replica.lag.time.max.ms30000이보다 느린 팔로워는 ISR에서 떨어짐.
auto.create.topics.enabletrue프로덕션에서는 false. 토픽 생성을 명시적인 단계로 만드세요.

보존 & 저장

설정역할
retention.ms이보다 오래된 레코드 삭제. 기본 7일. 토픽별.
retention.bytes파티션 로그 크기 상한. 토픽별. (ms / bytes 중) 먼저 도달하는 쪽이 우선.
segment.bytes / segment.msX 바이트 또는 X ms 마다 새 로그 세그먼트 생성. 작은 세그먼트 = 빠른 삭제/컴팩션, 더 많은 파일 핸들.
cleanup.policy=compact로그 컴팩션 — 키별 최신 값을 영구 보존. 상태 토픽, 체인지로그, __consumer_offsets에 사용.
cleanup.policy=delete,compact둘 다: 컴팩션 + 보존 기간 후 오래된 키 제거.
min.cleanable.dirty.ratio컴팩터 실행 전 더러움(dirty) 비율 (기본 0.5).

일반적인 알람

증상확인할 것
under-replicated 파티션 > 0브로커 다운, 느린 디스크, 또는 네트워크. kafka-topics.sh --describe --under-replicated-partitions.
offline 파티션 > 0리더가 없음. 클러스터에 큰 문제.
컨슈머 랙 증가컨슈머가 느리거나 막힘. max.poll.interval.ms, GC 일시정지, 다운스트림 시스템 점검.
프로듀서 RecordTooLargeException프로듀서의 max.request.size + 브로커의 message.max.bytes + 컨슈머의 max.partition.fetch.bytes 상향.
그룹 리밸런스 폭풍느린 핸들러, 불안정한 멤버, GC. 정적 멤버십 + cooperative-sticky 사용.
ISR 잦은 축소네트워크/디스크 문제. replica.lag.time.max.ms, 브로커 IO wait 점검.

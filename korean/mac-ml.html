Mac을 ML 워크스테이션으로

Apple Silicon(M1/M2/M3/M4)은 진지한 ML 머신입니다. IDE를 돌리는 같은 칩이 트랜스포머를 학습시키고, LoRA를 파인튜닝하고, 7B 파라미터 모델을 로컬에서 서빙할 수 있습니다 — GPU 대여도, nvidia-smi도 필요 없습니다. 이 페이지는 필요한 도구 셋을 타이핑으로 가르칩니다: arm64 Python MPS가 적용된 PyTorch MLX.

통합 메모리
Apple Silicon에서는 CPU와 GPU가 같은 RAM을 공유합니다. cuda 복사도, .to(device) 병목도 없습니다. 16 GB에 들어가는 모델은 PyTorch (MPS)에서든 MLX에서든 그냥 실행됩니다.
Mac에서의 PyTorch
MPS (Metal Performance Shaders) 백엔드가 적용된 풀 PyTorch. 다른 곳과 같은 API — cuda를 mps로 바꾸면 끝. 대부분의 op가 동작; 드물게 CPU로 폴백됩니다.
MLX
Apple의 어레이 프레임워크로, Apple Silicon을 위해 처음부터 만들어졌습니다. NumPy 같은, 기본적으로 lazy, 자동 미분, 네이티브 LLM 도구들. 더 작고, 더 빠르고, Mac 네이티브.

얻게 될 것들

1
깔끔한 arm64 Python. 네이티브 검증 완료, 가상 환경 안에서 ML 패키지 설치 준비 끝.
2
GPU 위의 PyTorch. 정확한 device = torch.device("mps" if ...) 주문.
3
MLX 헬로월드. mx.value_and_grad와 mx.eval을 올바르게 쓰는 선형 회귀.
4
로컬 LLM. mlx_lm.generate 명령 한 줄이면 터미널에서 7B 모델과 채팅.
5
판단 기준. 나란히 비교 표: MLX vs PyTorch, 언제 무엇을 쓸지.

PyTorch + MPS — Mac의 GPU

PyTorch 1.12(2022년 5월) 이후, MPS 백엔드가 Apple의 Metal Performance Shaders로 텐서 op을 통합 GPU에서 실행합니다. 같은 PyTorch API, Mac 버전 디바이스.

설치
# Mac용 일반 pytorch 휠은 mps가 활성화되어 있습니다
pip install torch torchvision

# nightly가 보통 mps 커버리지가 더 좋습니다
pip install --pre torch torchvision \
    --index-url https://download.pytorch.org/whl/nightly/cpu

디바이스 댄스

최적 디바이스를 고르는 이식 가능한 코드
import torch

# Mac (mps), Linux (cuda), CPU 전용 머신 모두에서 동작
device = torch.device(
    "mps"  if torch.backends.mps.is_available() else
    "cuda" if torch.cuda.is_available()        else
    "cpu"
)

x = torch.randn(1024, 1024).to(device)
W = torch.randn(1024, 1024).to(device)
y = x @ W                         # Apple GPU에서 실행
print(y.device)                  # mps:0

CPU 폴백 비상구

MPS에 아직 구현되지 않은 op이 있을 때
# Python 시작 전에 export (또는 torch import 전에 Python 안에서 설정)
export PYTORCH_ENABLE_MPS_FALLBACK=1

# 또는 코드에서:
import os
os.environ["PYTORCH_ENABLE_MPS_FALLBACK"] = "1"
import torch
이게 하는 일: PyTorch가 아직 MPS로 포팅되지 않은 op을 만나면, 크래시 대신 조용히 CPU로 폴백합니다. 느려지지만, 코드는 끝까지 돌아갑니다. PyTorch 2.x에서는 대부분의 일반 op이 동작합니다 — 폴백은 드뭅니다.

작은 엔드투엔드 예제

노이즈로 학습되는 1-레이어 MLP, GPU에서
import torch
import torch.nn as nn

device = torch.device("mps" if torch.backends.mps.is_available() else "cpu")

X = torch.randn(256, 8, device=device)
y = torch.randn(256, 1, device=device)

model = nn.Sequential(
    nn.Linear(8, 32), nn.ReLU(),
    nn.Linear(32, 1),
).to(device)

opt = torch.optim.Adam(model.parameters(), lr=1e-3)
loss_fn = nn.MSELoss()

for step in range(200):
    pred = model(X)
    loss = loss_fn(pred, y)
    opt.zero_grad(); loss.backward(); opt.step()

print("final loss", loss.item())

MLX — Apple의 어레이 프레임워크

MLX는 Apple Silicon에서 머신러닝을 위한 Apple의 리서치 프레임워크입니다. NumPy + JAX의 만남에 PyTorch 같은 ergonomics를 더한 느낌. lazy하고, 자동 미분을 지원하며, 통합 메모리를 네이티브로 활용합니다. Mac을 만드는 팀이 Mac을 위해 설계.

설치
# 코어 어레이 라이브러리
pip install mlx

# LLM 헬퍼 (mlx_lm)
pip install mlx-lm
항상 쓰게 될 import
import mlx.core as mx
import mlx.nn   as nn
import mlx.optimizers as optim

꼭 내재화해야 할 세 가지 아이디어

L
기본적으로 lazy. mx op은 그래프를 만듭니다. mx.eval(x)를 호출하거나 값을 읽기 전(예: print, numpy())에는 아무것도 계산되지 않습니다. 이로써 MLX가 fuse + 스케줄.
U
"디바이스"가 아닌 통합 메모리. .to(device)가 없습니다. 어레이는 공유 메모리에 살고, op은 자동으로 CPU나 GPU에서 실행됩니다.
A
Autograd는 functional. mx.grad(fn) 또는 mx.value_and_grad(fn)를 쓰세요 — 그래디언트를 반환하는 함수를 돌려줍니다. .backward()는 없습니다.

헬로월드: MLX로 선형 회귀

y = 2x + 1을 autograd로, 25줄에
import mlx.core as mx
import mlx.nn   as nn
import mlx.optimizers as optim

# 합성 데이터: y = 2x + 1 + noise
mx.random.seed(0)
X = mx.random.normal((100, 1))
y = 2.0 * X + 1.0 + 0.1 * mx.random.normal((100, 1))

# 모델 = Linear 한 레이어
model = nn.Linear(1, 1)

def loss_fn(model, X, y):
    pred = model(X)
    return mx.mean((pred - y) ** 2)

loss_and_grad = nn.value_and_grad(model, loss_fn)
opt = optim.SGD(learning_rate=0.1)

for step in range(200):
    loss, grads = loss_and_grad(model, X, y)
    opt.update(model, grads)
    mx.eval(model.parameters(), opt.state)   # 계산 강제

print(model.weight.item(), model.bias.item())   # ≈ 2.0, 1.0
mx.eval 트릭: MLX는 lazy하기 때문에, 그래디언트는 요청하기 전까지 실제로 계산되지 않습니다. 학습 스텝마다 한 번 mx.eval(model.parameters(), opt.state)를 호출해서 materialize하세요 — 그렇지 않으면 그래프가 계속 자랍니다.

실습

8 MLX core 모듈의 관습적 별칭:

9 lazy 그래프를 실제로 계산하도록 강제:

11 한 번의 호출로 손실 과 그래디언트를 모두 받기:

MLX vs PyTorch — 어떤 걸 언제 쓰나

Apple Silicon에서 둘 다 훌륭합니다. 겹치는 부분이 많습니다. 결정 기준을 정리합니다.

MLX PyTorch (MPS)
설계 대상 apple silicon Mac 전용, 네이티브 universal Mac · Linux · Windows · CUDA
API 스타일 NumPy + functional autograd (JAX 같은) NumPy + nn.Module + .backward()
실행 Lazy — mx.eval()로 materialize Eager (기본); torch.compile은 옵트인
메모리 모델 통합 — 디바이스 댄스 없음 디바이스 명시 — .to("mps")
Op 커버리지 작은 라이브러리 — 빠르게 성장 중 거대함 — 거의 모든 게 동작 (일부 MPS 갭)
LLM 서빙 mlx_lm — 양자화, LoRA, generate, 서버 transformers + accelerate 사용
생태계 젊음 — HF의 mlx-community, 도구가 늘어남 거대함 — torchvision, HuggingFace, lightning 등
M 시리즈 성능 대체로 가장 빠름 (네이티브 + lazy fusion) 훌륭함, 무거운 CNN op에서는 더 빠를 때도 있음
디버깅 Lazy라 print가 까다로움 — eval 호출 Eager — 어디서든 print, 그냥 동작
언제 쓸까 로컬 LLM 작업, Mac 타깃 리서치, Apple 최고 성능 CUDA 코드 포팅, 기존 HF 모델 사용, 크로스 플랫폼
MLX — 이럴 때 선택…
  • Llama 급 LLM을 로컬에서 실행
  • 노트북에서 LoRA 파인튜닝
  • JAX 스타일 functional 코드를 좋아함
  • 프로젝트가 Mac 전용 — CUDA 타깃 없음
  • 의존성 최소 + 빠른 시작 시간
PyTorch — 이럴 때 선택…
  • CUDA에서 기존 PyTorch 코드를 포팅
  • torchvision / diffusers / transformers가 필요
  • 나중에 Linux GPU에 배포 예정
  • 가장 큰 커뮤니티 / 문서 / Stack Overflow가 필요
  • 특수 레이어가 필요 (예: 복잡한 spectral op)

같은 작업, 두 프레임워크

MLX matmul
import mlx.core as mx

A = mx.random.normal((1024, 1024))
B = mx.random.normal((1024, 1024))
C = A @ B
mx.eval(C)
print(C.shape)
PyTorch matmul
import torch

dev = "mps" if torch.backends.mps.is_available() else "cpu"
A = torch.randn(1024, 1024, device=dev)
B = torch.randn(1024, 1024, device=dev)
C = A @ B
print(C.shape)
솔직한 답변: Mac 초보자에겐 보통 정답이 "둘 다 쓰세요"입니다. PyTorch로 시작(모든 게 문서화되어 있으니), LLM에서 Mac 최고 성능을 원할 때 MLX를 꺼내세요.

mlx-lm — 명령 한 줄로 로컬 LLM

mlx-lm은 바로 실행 가능한 LLM 스크립트를 제공하는 동반 패키지입니다: 추론용 generate, 파인튜닝용 lora, OpenAI 호환 HTTP API용 server. Hugging Face의 mlx-community 조직에서 미리 변환·양자화된 수백 개의 모델을 호스팅합니다.

Mac ML 성능 팁

빠른 Mac ML과 느린 Mac ML을 가르는 건 몇 가지 습관입니다. 대부분 통합 메모리를 존중하고 불필요한 작업을 피하는 데 관한 것입니다.

1
처음부터 끝까지 arm64. Rosetta 의존성 하나가 체인을 오염시킵니다. file $(which python)은 arm64를 보고해야 합니다. brew --prefix도 마찬가지로 (/usr/local이 아니라 /opt/homebrew여야 함).
2
최신 PyTorch 고정. MPS 지원은 매 릴리스마다 좋아집니다. PyTorch 2.3+를 노리세요. nightly 빌드는 stable에 아직 없는 op 커버리지를 가질 때가 많습니다.
3
mx.eval을 배치하세요. 매 op마다 호출하지 말고 — 학습 스텝마다 한 번씩, materialize가 필요한 것에: mx.eval(model.parameters(), opt.state).
4
추론용으로 양자화. 4비트는 메모리를 약 4배 줄이고 채팅 품질에는 거의 영향이 없습니다. MLX와 PyTorch (Mac에서 제한적인 bitsandbytes를 통해) 모두 지원합니다. 가장 쉬운 길: mlx-community의 -4bit 모델을 받기.
5
통합 메모리를 살피세요. Mac은 CPU와 GPU가 RAM을 공유합니다. 64 GB Mac은 70B 정도의 4비트 모델을 돌릴 수 있고, 8 GB Mac은 3B 4비트에 가깝습니다. 활성 상태 → 메모리 탭을 친구로.
6
lazy 그래프와 싸우지 마세요. MLX에서는 주기적인 eval 없이 루프 안에서 Python 측 상태를 누적(예: losses.append(loss))하면 그래프가 폭증합니다. append 전에 mx.eval(loss).
7
CPU↔MPS 왕복을 피하세요. PyTorch에서 .to("cpu")나 .numpy()를 쓸 때마다 비용이 듭니다. 학습 스텝 전체를 디바이스 위에서 돌리고, 경계(로깅, 체크포인팅)에서만 materialize 하세요.
8
실수 타입이 중요합니다. MLX는 기본 float32; mx.float16과 mx.bfloat16은 메모리를 대략 절반으로 줄이고 추론 속도를 자주 올립니다. PyTorch: torch.set_default_dtype(torch.float16) 또는 autocast.
9
torch.compile은 선택적으로. MPS에서는 도움이 될 수도, 해가 될 수도 있습니다 — 측정하세요. 안쪽 forward pass만 감싸고 바깥 학습 루프는 그냥 Python으로 두세요.
10
전원에 꽂으세요. Apple Silicon은 배터리에서 성능을 낮춥니다 — 성능 코어가 throttle되고, GPU 클록이 떨어집니다. 벤치마크는 전원 연결 + 고성능 프로파일.

흔한 함정

"내 MPS 코드가 inf / nan을 출력해요"
일부 op(특히 오래된 PyTorch + 복잡한 spectral op)은 CUDA에서는 괜찮지만 MPS에서 오버플로우합니다. PYTORCH_ENABLE_MPS_FALLBACK=1과 더 작은 학습률을 시도하세요.
"MLX가 기대보다 느려요"
거의 항상: mx.eval 누락. 미니배치 op마다나 끝까지 절대 안 부르는 것이 아니라, 스텝마다 한 번 materialize하는지 확인하세요.
"pip install torch가 실패해요"
Rosetta 아래의 x86_64 Python을 쓰고 있습니다. arm64 Python(uv / pyenv / python.org 인스톨러)으로 전환하고 file $(which python)으로 검증하세요.
"내 LLM이 멈추거나 OOM이에요"
4비트 양자화, max-tokens / 컨텍스트 길이 줄이기, 또는 더 작은 모델(7B 대신 3B)을 고르세요. 통합 메모리는 시스템의 다른 모든 것과 공유됩니다.

# mlx_quadratic.py — Apple Silicon에서 y = x^2 + noise 학습
import mlx.core as 
import mlx.nn as nn
import mlx.optimizers as optim

# --- 데이터 ---
mx.random.seed(42)
X = mx.random.uniform(low=-1, high=1, shape=(256, 1))
Y = X ** 2 + 0.05 * mx.random.normal((256, 1))

# --- 모델 ---
class MLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.l1 = nn.Linear(, 32)
        self.l2 = nn.Linear(32, 32)
        self.l3 = nn.Linear(32, )

    def __call__(self, x):
        x = nn.relu(self.l1(x))
        x = nn.relu(self.l2(x))
        return self.(x)

model = MLP()

# --- 손실 + autograd ---
def loss_fn(model, X, Y):
    return mx.mean((model(X) - Y) ** 2)

loss_and_grad = nn.(model, loss_fn)
opt = optim.(learning_rate=3e-3)

# --- 학습 루프 ---
for step in range(500):
    loss, grads = loss_and_grad(model, X, Y)
    opt.(model, grads)
    mx.(model.parameters(), opt.state)
    if step % 50 == 0:
        print(f"step {step:4d}  loss {loss.item():.4f}")

# --- 가중치 저장 ---
mx.("model.safetensors", dict(tree_flatten(model.parameters())))

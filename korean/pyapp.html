PyApp — Python 앱을 단일 바이너리로 출시

PyApp (ofek/pyapp)은 Python 애플리케이션을 자기 완결적 실행 파일로 묶어주는 Rust 런타임입니다. 빌드 시점에 환경 변수로 설정하고 cargo build --release를 실행하면 크로스 플랫폼 바이너리가 나옵니다. 사용자 머신에 Python이 필요 없습니다 — 적어도 사용자가 직접 설치할 필요는 없습니다.

정체
Python 배포본을 부팅하고 프로젝트를 설치하는 방법을 아는 작은 Rust 프로그램(약 2 MB). 타깃별로 한 번 컴파일한 뒤 결과 바이너리를 배포하세요.
아닌 것
프리저(PyInstaller)가 아닙니다. 트랜스파일러(Nuitka)가 아닙니다. zipapp(shiv / pex)도 아닙니다. 실제 CPython 배포본에 Python 소싱을 위임합니다 — 기본은 첫 실행 시 다운로드, 선택적으로 임베드.
왜 좋은가
기본적으로 작은 바이너리. 진짜 크로스 플랫폼 빌드(Linux / macOS / Windows). 모든 까다로운 부분이 동작하는 진짜 Python 설치 — 네이티브 의존성, 동적 로딩 전부. 셀프 업데이트 내장.

빌드 모양

레시피
# 1. pyapp에 프로젝트 정보 알려주기
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0
export PYAPP_EXEC_MODULE=mytool

# 2. pyapp의 Rust 런타임 컴파일
cargo install pyapp --locked --root dist

# 3. ./dist/bin/pyapp 가 이제 첫 실행 시
#    CPython을 다운로드하고, mytool을 pip 설치한 뒤 `python -m mytool`을 실행하는 바이너리.
mv dist/bin/pyapp dist/bin/mytool
이 페이지 사용법: 각 섹션을 읽고 빈칸에 직접 타이핑한 뒤 확인을 누르세요. 힌트는 드릴의 모든 빈칸을 공개합니다(다시 토글로 숨김). 연속 정답을 노리세요. 초록 종이 폭죽이 터지면 섹션을 끝낸 겁니다.

멘탈 모델

Pyapp 바이너리에는 (a) Rust 시작 코드, (b) 프로젝트 메타데이터, (c) 선택적으로 임베드된 Python 배포본과 휠이 들어 있습니다. 첫 실행 시 캐시 디렉토리에 비공개 Python을 풀고, 프로젝트를 pip 설치한 뒤, exec로 들어갑니다. 이후 실행은 곧바로 Python 엔트리 포인트로 갑니다.

작동 원리 — Rust 바깥, Python 안쪽

사용자가 mytool을 더블 클릭합니다. Rust 런타임이 캐시 디렉토리를 확인하고, 필요하면 Python을 소싱하고, 프로젝트를 한 번만 설치한 뒤, 엔트리 포인트를 실행합니다. 아래 다이어그램에서 두 배포 모드를 토글해 보세요.

on-demand Python 배포본을 첫 실행 시 다운로드. 바이너리 작음, 처음 한 번 온라인 필요.
embedded PYAPP_DISTRIBUTION_EMBED=1을 설정하면 Python tarball이 바이너리에 구워집니다. 파일은 더 크지만, 두 번째부터 오프라인 작동.

첫 실행 시퀀스

  1. 사용자가 ./mytool을 실행합니다.
  2. Rust 런타임이 캐시 디렉토리를 결정합니다 (Linux: ~/.local/share/pyapp/, macOS: ~/Library/Application Support/pyapp/, Windows: %LOCALAPPDATA%\pyapp\).
  3. 이 (이름, 버전, 배포본) 트리플에 대한 Python이 없으면 PYAPP_DISTRIBUTION_SOURCE에서 다운로드하거나 임베드된 tarball을 추출합니다.
  4. venv를 만듭니다 (전체 격리가 켜진 경우 Python 설치 자체가 venv 역할).
  5. 프로젝트(와 끌려 들어온 의존성)를 pip install합니다.
  6. 마커 파일을 저장합니다. 이후 실행은 3–5단계를 건너뜁니다.
  7. 설정된 엔트리 포인트로 exec하여 Rust 프로세스를 대체합니다.

빠른 시작

Rust(rustup.rs)를 설치하고, 환경 변수 몇 개를 설정한 뒤 cargo install을 실행하세요. 끝 — 1분 이내에 동작하는 바이너리가 생깁니다.

A) cargo로 설치
# crates.io에 게시된 프로젝트
export PYAPP_PROJECT_NAME=httpie
export PYAPP_PROJECT_VERSION=3.2.2

cargo install pyapp --locked

# 바이너리는 $CARGO_HOME/bin/pyapp 에
mv ~/.cargo/bin/pyapp ~/.cargo/bin/httpie
httpie --version
# → 3.2.2
B) 소스에서 빌드
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0

git clone https://github.com/ofek/pyapp pyapp-src
cd pyapp-src
cargo build --release

# 바이너리는 target/release/pyapp 에
mv target/release/pyapp ~/bin/mytool

크로스 컴파일

한 소스 트리로 모든 주요 플랫폼
# rust 타깃을 한 번 추가
rustup target add x86_64-unknown-linux-gnu
rustup target add aarch64-apple-darwin
rustup target add x86_64-pc-windows-msvc

# 그런 다음 각 타깃에 대해
cargo build --release --target x86_64-unknown-linux-gnu
cargo build --release --target aarch64-apple-darwin
cargo build --release --target x86_64-pc-windows-msvc

# pyapp이 런타임에 일치하는 CPython 배포본을 선택합니다
CI 팁: 공식 레포는 한 번에 일곱 타깃을 빌드하는 GitHub Actions 워크플로우를 제공합니다 — Linux x64/aarch64, macOS x64/aarch64, Windows x64/x86, musl. 훔쳐 쓰세요.

설정 변수

Pyapp은 빌드 시점에 읽는 환경 변수로만 설정합니다. 값들은 결과 바이너리에 박힙니다. 아래는 가장 유용한 변수들입니다.

필수

변수효과예시
PYAPP_PROJECT_NAME 프로젝트의 PyPI / 휠 이름. 필수. mytool
PYAPP_PROJECT_VERSION 버전 핀. PYAPP_PROJECT_PATH를 설정하지 않은 경우 필수. 1.2.3

프로젝트 소스

변수효과
PYAPP_PROJECT_PATH PyPI에서 pip 설치하는 대신 임베드할 로컬 휠/sdist 경로. 비공개 앱에 사용.
PYAPP_PROJECT_DEPENDENCY_FILE 바이너리에 박힐 constraints 파일 경로(pip -c). 재현성을 위해 트랜지티브 의존성 핀.
PYAPP_PIP_EXTRA_ARGS 설치 명령에 추가할 인자. 예: --index-url https://my.repo/simple

엔트리 포인트 (하나 선택)

변수효과예시
PYAPP_EXEC_MODULE python -m <module> 실행. mytool
PYAPP_EXEC_SCRIPT 휠 안에 들어 있는 스크립트 파일 실행. /path/to/run.py
PYAPP_EXEC_SPEC console-script 스타일: module:func. mytool.cli:main
PYAPP_EXEC_NOTEBOOK Jupyter로 실행할 .ipynb 파일 경로. analysis.ipynb

Python 배포본

변수효과기본값
PYAPP_PYTHON_VERSION 사용할 CPython 버전. indygreg/python-build-standalone 릴리스에 매핑. 최신 stable
PYAPP_DISTRIBUTION_SOURCE Pyapp이 Python을 다운로드할 URL 오버라이드. 에어갭 미러용. github 릴리스
PYAPP_DISTRIBUTION_EMBED 1/true이면 Python을 바이너리에 굽기. 첫 실행 다운로드 없음. 0
PYAPP_DISTRIBUTION_VARIANT 배포본 종류: install_only(기본) 또는 더 작은 바이너리용 install_only_stripped. install_only

동작 조정

변수효과
PYAPP_FULL_ISOLATION venv 레이어를 건너뛰고 풀린 Python에 직접 설치. 시작 빠름, 중복 제거 적음.
PYAPP_UPGRADE_VIRTUALENV 프로젝트 설치 전에 venv 안에서 pip / setuptools 강제 업그레이드.
PYAPP_EXPOSE_ALL_COMMANDS 설정된 것뿐 아니라 프로젝트의 모든 console script를 호출 가능하게.
PYAPP_ALLOW_UPDATES 0으로 self update 서브커맨드 비활성화.
PYAPP_PASS_LOCATION 바이너리 경로를 argv[0]으로 Python에 전달 — 자기 재실행하는 도구에 유용.
Boolean 값은 관대합니다: 1, true, True, TRUE, yes 모두 플래그 활성화. 그 외는, 미설정 포함, "off"입니다.

엔트리 포인트 — 모듈 vs 스크립트 vs 스펙

Pyapp은 앱을 네 가지 방식으로 시작할 수 있습니다. 개발 중에 실행하는 방식에 맞게 고르세요.

PYAPP_EXEC_MODULE
python -m mytool처럼. 프로젝트에 __main__.py가 있어야 합니다. 가장 관용적. 
export PYAPP_EXEC_MODULE=mytool
# 패키지 구조: mytool/__main__.py
PYAPP_EXEC_SPEC
Console-script 문법: pkg.module:function. pyproject.toml에 쓸 [project.scripts] 항목과 동일. 
export PYAPP_EXEC_SPEC=mytool.cli:main
PYAPP_EXEC_SCRIPT
.py 파일 경로(휠 루트 기준). python script.py를 실행합니다. 옛날 스타일. 
export PYAPP_EXEC_SCRIPT=mytool/runner.py
PYAPP_EXEC_NOTEBOOK
Jupyter 노트북 실행. Pyapp이 jupyter를 프로젝트와 함께 설치합니다. 코더가 아닌 사람들에게 인터랙티브 도구를 배포하기에 좋음. 
export PYAPP_EXEC_NOTEBOOK=demo.ipynb
인자 전달: 바이너리에 전달한 인자는 그대로 Python 인터프리터로 전달됩니다. ./mytool --debug -- --help가 EXEC_MODULE과 EXEC_SPEC 모두에서 예상대로 동작합니다.

배포 소스 — 임베드 vs 온디맨드

CPython 소싱의 두 전략. 사용자가 온라인인지, 바이너리 크기에 민감한지, 일회성 지연을 허용할 수 있는지에 따라 고르세요.

on-demand (기본)
  • 바이너리는 작게 유지(약 2 MB).
  • 첫 실행 시 indygreg/python-build-standalone에서 CPython(약 30 MB) 다운로드.
  • 이후 실행은 캐시된 설치 사용.
  • 여러 pyapp 바이너리가 캐시된 인터프리터를 공유 가능.
  • 첫 실행 시 네트워크 필요.
# 설정할 게 없음 — 기본값
export PYAPP_PYTHON_VERSION=3.12
cargo build --release
embedded
  • Python tarball을 바이너리에 굽기(총 약 30 MB).
  • 어떤 시점에도 네트워크 접근 불필요.
  • 재현성: 모든 머신에서 동일한 Python.
  • 의존성도 임베드하지 않으면 첫 실행 시 pip install.
  • CI 도구, 에어갭 환경, 첫 실행 지연을 못 받을 때 사용.
export PYAPP_PYTHON_VERSION=3.12
export PYAPP_DISTRIBUTION_EMBED=1
cargo build --release

특정 tarball 핀

pyapp이 Python을 다운로드할 URL 오버라이드
# 미러 또는 벤더링된 릴리스
export PYAPP_DISTRIBUTION_SOURCE=https://internal.example.com/python-3.12-x64.tar.gz
export PYAPP_DISTRIBUTION_PATH_PREFIX=install/
export PYAPP_DISTRIBUTION_PYTHON_PATH=install/bin/python3

cargo build --release
Stripped 변종: PYAPP_DISTRIBUTION_VARIANT=install_only_stripped로 디버그 심볼을 빼서 임베드 빌드에서 약 10 MB를 줄이세요.

셀프 관리 — mytool self ...

모든 Pyapp 바이너리는 업그레이드와 제거를 처리하는 self 명령을 함께 출시합니다. CI 없이도 파워 유저 UX.

명령
mytool self updatePyPI에서 최신 버전을 가져와 venv를 인플레이스로 재구축
mytool self restorevenv를 날리고 재설치 (망가진 의존성 복구)
mytool self remove캐시 디렉토리 삭제; 바이너리는 그대로
mytool self metadata바이너리에 박힌 Pyapp 설정 출력
mytool self python임베드된 Python REPL로 진입 — 디버깅에 유용
셀프 업데이트 제어
# 빌드 시점에 업데이트 완전 비활성화
export PYAPP_ALLOW_UPDATES=0

# 런타임에 사용자를 비공개 인덱스로 향하게
export PIP_INDEX_URL=https://my.repo/simple
mytool self update

# 특정 버전 핀
mytool self update 1.4.7
왜 좋은가: 사용자가 한 번 brew install 또는 winget install한 뒤, 시스템 패키지 매니저를 다시 실행하지 않고도 mytool self update로 무한히 업데이트할 수 있습니다. Rust 런타임은 절대 변하지 않고, 캐시 안의 Python 프로젝트만 변합니다.

대안 비교

Pyapp은 매우 붐비는 영역에 있습니다. 명백한 경쟁 도구들과 어떻게 다른지 봅시다.

도구 접근법 단일 바이너리 한 호스트에서 크로스 플랫폼 실제 CPython 셀프 업데이트 트레이드오프
PyApp Rust 로더 → CPython 배포본 예 예 (cargo) 예 예 첫 실행 다운로드 또는 더 큰 임베드
PyInstaller freeze: 바이트코드 + 라이브러리를 아카이브에 예 아니오 — OS별 빌드 필요 임베드된 사본 아니오 안티바이러스 오탐, 큰 바이너리
Nuitka Python → C → 네이티브 바이너리 예 아니오 재컴파일됨 아니오 빌드 느림, 의존성 까다로움
shiv / pex zipapp 래퍼 .pyz는 Python 필요 예 사용자가 Python 필요 아니오 실제로 자기 완결적이지 않음
Briefcase (BeeWare) 플랫폼 네이티브 번들 .app, .msi, .deb 아니오 임베드 아니오 GUI 앱 대상, 설정 더 많음
conda-pack conda 환경의 tarball 아니오 아니오 예 아니오 conda 활성화 필요

언제 무엇을 고를까

PyApp을 고르세요…
단일 바이너리를 원하고, 첫 실행 다운로드를 받아들일 수 있고, 하나의 CI 매트릭스에서 크로스 플랫폼 빌드를 만들고 싶고, 공짜 self update 아이디어가 마음에 든다면.
PyInstaller를 고르세요…
네트워크 없이, 캐시 쓰기 없이 실행해야 하고, 각 타깃 OS에서 빌드할 수 있고, 가끔 import 누락 버그를 추적하는 게 괜찮다면.
shiv / pex를 고르세요…
목적 머신에 이미 Python이 있고, 진짜 크로스 플랫폼 출력이 필요하고, 단일 파일 휠 번들 (.pyz)만 필요하다면.

미니 프로젝트 — mytool v1.0 엔드투엔드 출시

PyPI에 mytool 1.0.0을 게시했다고 가정합시다. CLI는 mytool/__main__.py에 있습니다. 격리된 Python 3.12에서 오프라인 가능한 단일 자기 완결적 바이너리를 원합니다. 아래 완전한 재현 가능한 레시피의 빈칸을 채우고, 빌드 & 실행을 클릭해 시뮬레이션하세요.

레시피 완성하기

9개 빈칸. 먼저 전체 확인; 초록이 되면 빌드 & 실행이 빌드를 애니메이션합니다. 

# --- 1. pyapp에 프로젝트 정보 알려주기 ---
export PYAPP_PROJECT_NAME=
export PYAPP_PROJECT_VERSION=

# --- 2. 어떻게 실행할지 ---
export PYAPP_EXEC_MODULE=

# --- 3. python 배포본 ---
export PYAPP_PYTHON_VERSION=
export PYAPP_DISTRIBUTION_EMBED=
export PYAPP_FULL_ISOLATION=

# --- 4. pyapp의 rust 런타임을 잠긴 상태로 컴파일 ---
cargo install pyapp 

# --- 5. 이름 바꾸고 실행 ---
mv ~/.cargo/bin/pyapp ~/bin/mytool
mytool --help

# 나중에 사용자가 업그레이드:
mytool

    빌드 완료!
    어떤 Linux x64 머신에 떨어뜨려도 되는 단일 32 MB 바이너리 — Python 설치 불필요.
    실제 사용자라면 이제 scp mytool prod:으로 끝낼 겁니다.
    실제 셸에서 같은 환경 변수 시퀀스를 실행하면 똑같은 출력을 실제로 얻을 수 있습니다.

비공개 프로젝트

      

        PyPI에서 가져오는 대신 직접 휠을 임베드하려면
        PYAPP_PROJECT_PATH=./dist/mytool-1.0.0.whl을 추가하세요.
      

    

    

      
크로스 플랫폼

      

        이걸 x86_64-unknown-linux-gnu,
        aarch64-apple-darwin,
        x86_64-pc-windows-msvc CI 매트릭스로 감싸세요 —
        한 소스에서 6개 바이너리.
      

    

    

      
더 작은 바이너리

      

        디버그 심볼을 빼려면 PYAPP_DISTRIBUTION_VARIANT=install_only_stripped을
        추가하세요. C 레벨 디버깅이 어려워지는 대가로 약 10 MB를 줄입니다.

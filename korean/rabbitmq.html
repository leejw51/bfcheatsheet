1. 메시지 브로커란 무엇인가?

메시지 브로커는 프로듀서와 컨슈머 사이에 있습니다. 프로듀서는 메시지를 브로커에 떨어뜨리고 자기 일로 돌아갑니다. 컨슈머는 자신의 페이스에 맞춰 나중에 그것을 가져갑니다. 둘은 동시에 온라인일 필요도 없고, 서로의 주소를 알 필요도 없으며, 처리 속도에 합의할 필요도 없습니다.

왜 큐가 중요한가

  • 디커플링: 프로듀서 변경 없이 컨슈머 변경.
  • 버퍼링: 스파이크를 부드럽게 함; 큐가 버스트를 흡수.
  • 신뢰성: 컨슈머가 크래시되어도 메시지는 그대로 있습니다.
  • 팬아웃: 하나의 이벤트가 여러 독립적인 반응을 트리거.
심상 모델: 프로듀서는 엽서를 쓰고, 브로커는 우체국이며, 큐는 우편함이고, 컨슈머는 여유롭게 읽습니다.

2. AMQP 핵심 개념

RabbitMQ는 AMQP 0-9-1을 구현합니다. 용어는 작지만 정확합니다.

  • 프로듀서 메시지를 발행합니다.
  • 컨슈머 메시지를 구독하고 처리합니다.
  • 큐 메시지를 저장하는 이름 있는 버퍼.
  • 익스체인지 라우터; 프로듀서는 익스체인지에 발행하며, 큐에 직접 발행하지 않습니다.
  • 바인딩 익스체인지를 큐에 연결하는 규칙 (종종 키와 함께).
  • 라우팅 키 프로듀서가 메시지에 붙이는 문자열; 익스체인지는 이를 사용해 어디로 전달할지 결정.
  • 가상 호스트 (vhost) 사용자, 익스체인지, 큐를 분할하는 네임스페이스; 논리적 격리.
  • 커넥션 TCP 연결. 채널 그 안의 멀티플렉스된 가상 연결.

한눈에 보는 흐름

producer
  └─ publish(routing_key) ─▶ exchange
                              ├─ binding A ─▶ queue 1 ─▶ consumer
                              └─ binding B ─▶ queue 2 ─▶ consumer

3. 익스체인지 유형 심층 분석

Direct

메시지의 라우팅 키가 바인딩 키와 정확히 일치하는 경우에 라우팅합니다. 사용처: 타깃 전달, 단순 작업 큐, 명령 라우팅.

binding: queue.email  ⇐ key="email"
publish key="email"   → queue.email
publish key="sms"     → 폐기 (또는 unrouted)

Fanout

라우팅 키를 무시합니다. 바인딩된 모든 큐에 브로드캐스트. 사용처: pub/sub 이벤트, 캐시 무효화, "모두에게 알리기" 알림.

publish 어떤 키든 → 바인딩된 모든 큐가 수신

Topic

패턴 기반 라우팅. 라우팅 키는 점(.)으로 구분된 단어들이며, 바인딩 패턴은 다음을 사용할 수 있습니다:

  • * — 정확히 한 단어와 매치.
  • # — 0개 이상의 단어와 매치.
binding "orders.*.us"   매치     orders.new.us, orders.cancel.us
                        미스     orders.us, orders.new.eu, orders.a.b.us
binding "orders.#"      매치     orders, orders.new, orders.new.us
binding "*.error.*"     매치     app.error.db, web.error.api

Headers

라우팅 키 대신 메시지의 헤더로 라우팅합니다. 매치 모드 x-match: all은 모든 헤더가 일치해야 하고, x-match: any는 하나 이상이면 됩니다.

binding headers: {format: pdf, type: report, x-match: all}
header {format: pdf, type: report}     → 매치
header {format: pdf}                    → 미스

언제 어떤 것을 사용할까

유형이럴 때 선택…
direct점대점(point-to-point) 또는 단일 키에 대한 엄격한 동등성 매칭이 필요할 때.
fanout모든 구독자가 같은 이벤트를 받아야 할 때. 가장 단순.
topic계층적 이벤트(region.service.severity). 구독자가 부분집합을 원할 때.
headers라우팅이 문자열 키 하나에 담기 어려운 여러 속성에 의존할 때.

기본 익스체인지 (default exchange)

이름 없는 "" 익스체인지는 특별한 direct 익스체인지로, 모든 큐가 자기 이름으로 암시적으로 바인딩됩니다. 그래서 basic_publish(exchange='', routing_key='hello')가 곧장 hello라는 큐로 가는 것이죠.

Dead-letter 익스체인지 (DLX)

큐는 x-dead-letter-exchange를 설정할 수 있습니다. 거부되거나, 만료되거나, 오버플로된 메시지는 DLX로 재발행되며, 거기에 바인딩된 큐(데드레터 큐)에서 검사하거나 재시도할 수 있습니다.

4. 확인 응답 (Acknowledgments)

Auto-ack vs manual ack

  • Auto-ack (auto_ack=True): 메시지가 컨슈머에게 떠나는 순간 브로커는 전달이 완료된 것으로 간주합니다. 빠르지만 불안전. 컨슈머가 처리 도중 크래시되면 메시지는 사라집니다.
  • Manual ack (auto_ack=False): 컨슈머가 성공 후 basic_ack(delivery_tag)를 호출해야 합니다. 그 전에 죽으면 브로커는 재연결 시 재전달(redeliver)합니다.

nack & reject

  • basic_nack(requeue=true) — 큐에 다시 넣음.
  • basic_nack(requeue=false) — 폐기 또는 (설정되어 있다면) DLX로 전송.
  • basic_reject는 단일 메시지용 nack.

재전달 플래그 (redelivered)

메시지가 재전달될 때 브로커는 redelivered=true로 표시합니다. 컨슈머는 이를 주의 신호로 받아들여야 합니다: 멱등성 보장, 중복 제거 키, 헤더 기반 재시도 카운터.

5. 내구성 — 재시작 후에도 살아남는 것

메시지가 브로커 크래시 후에도 살아남으려면 다음 세 가지가 모두 설정되어야 합니다:

  1. Durable 큐: queue_declare(durable=True). 그렇지 않으면 큐 자체가 사라집니다.
  2. Persistent 메시지: delivery_mode=2 (또는 PERSISTENT_TEXT_PLAIN). 없으면 메시지는 RAM에만 존재.
  3. Durable 익스체인지 (의존한다면): exchange_declare(durable=True).
주의: 영속화는 메시지마다 동기 fsync가 아닌 best-effort입니다. 디스크 기록을 보장한 뒤 상위로 ack 해야 한다면 publisher confirms (confirm_delivery)를 사용하세요.

Publisher confirms vs 트랜잭션

AMQP 트랜잭션은 존재하지만 느립니다. Publisher confirms는 비동기이며 수십~수백 배 더 빠릅니다: 브로커가 ack 프레임으로 "받았어"라고 알려줍니다.

6. Prefetch & QoS — 공정한 디스패치

RabbitMQ는 기본적으로 컨슈머에게 라운드 로빈으로 메시지를 전달합니다. 한 컨슈머가 느리고 다른 컨슈머가 빠르면, 빠른 쪽이 노는 동안 느린 쪽에 일이 쌓입니다. Prefetch가 이를 해결합니다.

channel.basic_qos(prefetch_count=1)

브로커에게 "이전 메시지를 ack 하기 전에는 새 메시지를 주지 마"라고 말하는 것입니다. 이제 빠른 컨슈머가 더 많은 일을 가져가고, 느린 컨슈머는 덜 가져갑니다. 짧고 균일한 작업이라면 더 높이 (예: 10–50) 설정하세요.

튜닝 경험칙

  • 길고 불균등한 작업 → prefetch=1.
  • 짧고 균일한 작업 → 높은 prefetch (왕복 오버헤드 감소).
  • 큐 깊이와 컨슈머 활용률로 측정하세요.

7. 클러스터링 & quorum 큐

RabbitMQ 클러스터는 노드 간에 사용자, vhost, 익스체인지, 바인딩을 공유합니다. 큐는 기본적으로 단일 마스터 엔티티이며, 마스터는 한 노드에 살고 미러링되거나 quorum으로 대체될 수 있습니다.

클래식 미러링 큐 (classic mirrored)

레거시. 상태를 팔로워에 복제. 현대 배포에서는 quorum 큐가 이를 대체합니다.

Quorum 큐

  • Raft 합의 프로토콜 위에 구축.
  • 강한 일관성: 다수의 복제본이 받기 전에는 쓰기가 ack 되지 않습니다.
  • 노드 장애에서 깨끗이 회복. 중요한 데이터의 권장 기본값.
  • 트레이드오프: 클래식 큐보다 지연시간이 높고 디스크 I/O가 많음.

Streams (보너스)

고처리량 분석 워크로드를 위한 로그 스타일 append 큐. 다른 프로토콜, 다른 접근 패턴(파괴적 소비가 아닌 오프셋 기반).

8. 일반적인 패턴

작업 큐 (work queues, task distribution)

큐 하나, 워커 여럿, manual ack, prefetch=1. 이미지 리사이즈, 이메일 발송, 비디오 인코딩 등에 사용.

producer ─▶ "" ─▶ queue:tasks ─▶ worker1
                              └▶ worker2
                              └▶ worker3

Publish/Subscribe (발행/구독)

Fanout 익스체인지를 사용해, 모든 구독자가 자신의 큐를 통해 모든 메시지를 받습니다.

producer ─▶ fanout:events ─┬─▶ queue:audit       ─▶ logger
                            ├─▶ queue:cache_invl  ─▶ cache
                            └─▶ queue:notify      ─▶ pusher

더 많은 패턴

토픽 라우팅 (topic routing)

계층적 이벤트. 서로 다른 큐가 서로 다른 모양에 구독합니다.

topic:logs
  binding "*.error.*"  ─▶ queue:alerts
  binding "db.#"       ─▶ queue:dba
  binding "#"          ─▶ queue:audit

RPC (요청/응답)

프로듀서는 reply_to(임시 exclusive 큐)와 correlation_id를 설정합니다. 컨슈머는 응답을 reply_to로 발행합니다. 프로듀서는 correlation_id로 응답을 매칭합니다.

client → publish(reply_to=Q1, correlation_id=abc)
server → consume → publish(exchange='', routing_key=Q1, correlation_id=abc)
client → abc 와 매치, promise 해소
팁: reply 큐에 TTL을 설정해, 새는 클라이언트로 인해 고아 큐가 쌓이지 않도록 하세요.

인터랙티브 시뮬레이터

익스체인지 유형을 고르고 바인딩을 선언한 뒤, 메시지를 발행하고 흐름을 관찰하세요. 컨슈머의 동작 모드를 토글해 ack/nack과 재전달을 확인할 수 있습니다. 발행에 성공할 때마다 XP를 얻습니다.

producer exchange queue consumer 탈락 = unrouted (mandatory off → 조용히 소실)

바인딩

바인딩을 클릭하면 제거됩니다.

컨슈머

각 컨슈머의 ack 모드를 토글하세요 (auto / manual-ack / manual-nack-requeue / manual-nack-DLX).

활동 로그

연습 모드 — 답을 입력

객관식이 아닙니다. 용어, 명령, 인자, 또는 yes/no를 입력하세요. 비교는 대소문자 무시이며 vocab/yes-no 문항은 사소한 오타에 관대합니다. 다만 명령과 코드는 정확히 일치해야 합니다.

단계 1 / 18 vocab 0 정답 0 오답

● 보스 모드: 프로덕션 장애

방금 페이저가 울렸습니다. 문항당 30초, 총 10문제. 명령, 용어, 또는 한 단어 답을 입력하세요. 정답 = +50 XP. 오답 또는 시간 초과 = 콤보 끊김. 10문항 모두 살아남으면 장애 진압자(Outage Crusher) 배지를 획득합니다.

문항 1 / 10 시간 30초 0 정답

배지 캐비닛

시뮬레이터를 돌리고, 연습을 끝내고, 장애에서 살아남으면 잠금 해제됩니다.

학습용. 진행 상황은 브라우저에 저장됩니다. 언제든 초기화 가능.

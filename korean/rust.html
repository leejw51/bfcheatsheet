챕터1

소유권 & 빌림

Rust의 심장

Rust의 모든 값에는 단 하나의 소유자(owner)가 있습니다. 소유자가 스코프를 벗어나면 값은 드롭되어 — 메모리는 해제되고, 파일은 닫히고, 락은 풀립니다. 이것이 RAII이며, 정적으로 강제됩니다.

소유권은 이동(move)(전달)할 수도 있고, 빌림(borrow) (참조를 빌려줌)할 수도 있습니다. 빌림은 두 가지 규칙을 따릅니다: &T 공유 참조는 동시에 여러 개 가질 수 있지만, &mut T 가변 참조는 반드시 배타적이어야 합니다. 컴파일러가 이를 컴파일 타임에 증명합니다.

소유권 이전(이동) vs 빌림
소유한 값 빌린 참조 무효화됨

전형적인 이동

main.rs
fn main() {
    let s = String::from("hello");
    let t = s;                 // s가 t로 이동됨
    // println!("{}", s);      // ❌ 에러: 이동된 값의 빌림: `s`
    println!("{}", t);         // ✅
}

공유 빌림과 가변 빌림

borrow.rs
fn len(s: &String) -> usize { s.len() }

fn push_bang(s: &mut String) { s.push('!'); }

fn main() {
    let mut greeting = String::from("hi");
    let n = len(&greeting);          // 공유 빌림, OK
    push_bang(&mut greeting);        // 배타적 빌림, OK
    println!("{} ({} chars)", greeting, n);
}
두 가지 규칙 어느 시점에든, 하나의 가변 참조 또는 임의의 개수의 공유 참조 중 하나만 가질 수 있습니다. 참조는 항상 유효해야 합니다(살아있는 데이터를 가리켜야 합니다). 이것이 빌림 검사기의 전부입니다.
챕터2

라이프타임

빌림의 지속 기간에 이름 붙이기

라이프타임(lifetime)이란 단지 참조가 유효한 코드 영역에 붙은 이름일 뿐입니다. 대부분의 라이프타임은 추론됩니다(이를 elision, 생략이라고 부릅니다). 컴파일러가 반환된 참조가 어느 입력에서 왔는지 구분할 수 없을 때만 직접 적어주면 됩니다.

참조는 자신이 빌린 값보다 오래 살 수 없다
소유자의 라이프타임 'a 참조의 라이프타임 'b

반환되는 참조에 라이프타임 표기하기

lifetimes.rs
// 반환되는 `&str`는 x와 y 중 더 짧은 쪽만큼 살아있어야 함.
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() >= y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string is long");
    let result;
    {
        let s2 = String::from("short");
        result = longest(&s1, &s2);
        println!("{}", result);   // ✅ 둘 다 살아있음
    }
    // println!("{}", result);     // ❌ s2가 드롭됨, 댕글링이 됨
}
생략(elision) 규칙 함수가 하나의 참조를 받아 하나를 반환하면, Rust는 둘이 같은 라이프타임을 공유한다고 가정합니다. 입력에 &self가 있으면, 출력은 &self로부터 빌립니다. 이 기본값이 맞지 않을 때만 직접 표기하면 됩니다.
챕터3

트레이트 & 제네릭

행동을 추상화하다

트레이트(trait)는 타입이 구현할 수 있는 메서드들의 집합입니다. 제네릭(generic)은 타입에 대해 매개변수화된 함수나 타입입니다. 둘을 함께 쓰면 상속 없이 애드혹 다형성을 얻습니다 — "표시될 수 있는 임의의 T", "i32의 임의의 이터레이터".

하나의 트레이트, 두 개의 impl, 하나의 제네릭 함수
traits.rs
trait Greet {
    fn hello(&self) -> String;
    fn shout(&self) -> String {                 // 기본 메서드
        self.hello().to_uppercase()
    }
}

struct Cat;
struct Robot { id: u32 }

impl Greet for Cat   { fn hello(&self) -> String { "meow".into() } }
impl Greet for Robot { fn hello(&self) -> String { format!("unit-{}", self.id) } }

fn welcome<T: Greet>(t: &T) {
    println!("{} ({})", t.hello(), t.shout());
}

fn main() {
    welcome(&Cat);
    welcome(&Robot { id: 7 });
}
정적 디스패치 vs 동적 디스패치 fn welcome<T: Greet>는 구체 타입마다 사본을 생성합니다 (제로 코스트). fn welcome(t: &dyn Greet)는 팻 포인터를 받아 런타임에 디스패치합니다 — 약간 느리지만, 이질적인 Vec<Box<dyn Greet>>를 저장할 수 있게 해줍니다.
챕터4

Result, Option & ? 연산자

에러를 값으로 다루기

Rust에는 예외(exception)가 없습니다. 실패는 타입에 인코딩됩니다. Option<T>는 아마도 T가 있을 수 있음을, Result<T, E>는 T 또는 에러 E를 의미합니다. ? 연산자가 이들을 꿰어, 값이 없을 경우 함수를 일찍 빠져나갑니다.

? 연산자: 에러는 단락(short-circuit), Ok는 언랩
parse.rs
use std::num::ParseIntError;

fn double_first(input: &str) -> Result<i32, ParseIntError> {
    let first = input.split(',').next().unwrap_or("0");
    let n: i32 = first.trim().parse()?;   // Err 시 일찍 반환 (단락 평가)
    Ok(n * 2)
}

fn main() {
    match double_first("21, hello") {
        Ok(v)  => println!("got {v}"),
        Err(e) => eprintln!("bad: {e}"),
    }
}

Option, 간략히

option.rs
fn first_word(s: &str) -> Option<&str> {
    s.split_whitespace().next()
}

fn main() {
    let n = first_word("hello world").map(str::len).unwrap_or(0);
    println!("{n}");                     // 5
}

패턴 매칭

match, if let, while let

match는 망라적(exhaustive)입니다: 컴파일러는 어떤 변형(variant)도 무시하지 못하게 합니다. 패턴은 튜플, 구조체, enum, 슬라이스, 참조를 구조 분해할 수 있고, if 조건으로 가드할 수도 있습니다.

match.rs
enum Shape {
    Circle { r: f64 },
    Rect   { w: f64, h: f64 },
    Triangle(f64, f64, f64),
}

fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle { r }            => std::f64::consts::PI * r * r,
        Shape::Rect   { w, h }         => w * h,
        Shape::Triangle(a, b, c) if a + b > *c => {
            let p = (a + b + c) / 2.0;
            (p * (p - a) * (p - b) * (p - c)).sqrt()
        }
        _ => f64::NAN,                                  // 잘못된 삼각형
    }
}
If let, while let 하나의 변형만 신경 쓰면 될 때, if let Some(x) = opt { … }가 전체 match보다 깔끔합니다. while let은 패턴이 더 이상 매칭되지 않을 때까지 반복합니다 — 스택을 비우는 데 안성맞춤입니다.

이터레이터 & 클로저

지연 평가, 유려함, 빠름

이터레이터는 지연 평가(lazy)됩니다: 요청하기 전까지는 아무 일도 일어나지 않습니다. 클로저는 환경을 캡처하며 세 가지 종류로 나뉩니다: Fn(빌림), FnMut(가변 빌림), FnOnce(소비). 컴파일러가 가장 관대한 것을 자동으로 선택합니다.

iter.rs
fn main() {
    let nums = vec![1, 2, 3, 4, 5, 6];

    let sum_of_even_squares: i32 = nums
        .iter()
        .filter(|&&n| n % 2 == 0)
        .map(|&n| n * n)
        .sum();

    println!("{sum_of_even_squares}");   // 56

    // 클로저와 캡처
    let factor = 10;
    let scale = |x: i32| x * factor;     // factor를 빌림 (Fn)
    println!("{}", scale(3));            // 30
}
제로 코스트 관용적인 이터레이터 체인은 손으로 작성한 루프와 동일한 어셈블리로 컴파일됩니다. 유려한 스타일은 본질적으로 공짜입니다 — 때로는 더 빠릅니다, LLVM이 더 많은 불변식을 증명하기 때문입니다.

동시성

스레드, send/sync, 채널

두 개의 마커 트레이트가 동시성을 통제합니다. Send는 값이 스레드 사이로 이동해도 안전함을 의미합니다. Sync는 &T가 스레드 사이에서 공유되어도 안전함을 의미합니다. 대부분의 타입은 둘 다 가집니다; Rc<T>와 RefCell<T>는 그렇지 않습니다 — 컴파일러가 데이터 레이스 출하를 막아줍니다.

채널: 생산자 → 소비자, 소유권이 함께 이동
channel.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    for i in 0..4 {
        let tx = tx.clone();
        thread::spawn(move || {
            tx.send(format!("hello from worker {i}")).unwrap();
        });
    }
    drop(tx);                     // 원본 송신자 종료 (남은 핸들이 없을 때 채널이 닫힘)

    while let Ok(msg) = rx.recv() {
        println!("{msg}");
    }
}

공유된 가변 상태

mutex.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let c = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut n = c.lock().unwrap();
            *n += 1;
        }));
    }
    for h in handles { h.join().unwrap(); }
    println!("count = {}", *counter.lock().unwrap()); // 10
}

Async / Await

협력적 동시성

함수에 async 표시를 하면 그 본문이 Future를 구현하는 상태 머신으로 변합니다. 호출만으로는 아무 일도 일어나지 않습니다 — .await를 하거나 런타임(Tokio, async-std, smol)에 넘겨야 합니다. 각각의 .await는 일시 정지 지점입니다.

async_main.rs
use tokio::time::{sleep, Duration};

async fn fetch(name: &str) -> String {
    sleep(Duration::from_millis(50)).await;
    format!("hello, {name}")
}

#[tokio::main]
async fn main() {
    let (a, b) = tokio::join!(fetch("alice"), fetch("bob"));
    println!("{a} | {b}");
}
Async는 색깔이다 async에서 동기 함수를 호출할 수는 있지만, 런타임 없이 그 반대는 불가능합니다. 런타임을 일찍 선택하고(Tokio가 기본) 그것으로 일관되게 가세요.

Cargo & 모듈

프로젝트 구성

Cargo는 빌드 시스템, 패키지 매니저, 테스트 러너, 벤치마크 하니스를 하나로 묶은 도구입니다. 크레이트(crate)는 컴파일의 단위입니다. 모듈(module)은 크레이트 내부의 네임스페이스입니다. 파일시스템이 모듈 트리를 그대로 반영합니다.

tree
my_app/
├── Cargo.toml          # 매니페스트: 이름, 버전, 의존성
├── Cargo.lock          # 정확한 해석된 버전 (바이너리는 커밋할 것)
└── src/
    ├── main.rs         # 바이너리 진입점        →  크레이트 `my_app`
    ├── lib.rs          # 라이브러리 루트         →  크레이트 `my_app`
    └── parser/
        ├── mod.rs      # 모듈 `parser`
        └── ast.rs      # 모듈 `parser::ast`
Cargo.toml
[package]
name    = "my_app"
version = "0.1.0"
edition = "2024"

[dependencies]
serde   = { version = "1", features = ["derive"] }
tokio   = { version = "1", features = ["full"] }
anyhow  = "1"

[dev-dependencies]
proptest = "1"
commands
cargo new my_app             # 바이너리 크레이트 스캐폴딩
cargo run                    # 컴파일 후 실행
cargo test                   # 단위 + 통합 + doc 테스트 실행
cargo clippy -- -D warnings  # 유명하게 날카로운 린터
cargo fmt                    # rustfmt, 단호하고 결정적인 포매터

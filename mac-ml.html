Mac as ML workstation

Apple Silicon (M1/M2/M3/M4) is a serious ML machine. The same chip that runs your IDE can train transformers, fine-tune LoRAs, and serve 7B-parameter models locally — no GPU rental, no nvidia-smi. This page teaches the three tools you need: Python on arm64 PyTorch with MPS MLX — by typing.

Unified memory
On Apple Silicon, CPU and GPU share the same RAM. No cuda copies, no .to(device) bottlenecks. A model that fits in 16 GB just runs, whether you call it from PyTorch (MPS) or MLX.
PyTorch on Mac
Full PyTorch with the MPS backend (Metal Performance Shaders). Same API as everywhere else — just swap cuda for mps. Most ops work; rare ones fall back to CPU.
MLX
Apple's array framework, built from scratch for Apple Silicon. NumPy-like, lazy by default, automatic differentiation, native LLM tooling. Smaller, faster, Mac-native.

PyTorch + MPS — the GPU on your Mac

Since PyTorch 1.12 (May 2022), the MPS backend uses Apple's Metal Performance Shaders to run tensor ops on the integrated GPU. Same PyTorch API, Mac-flavored device.

install
# the regular pytorch wheel for mac is mps-enabled
pip install torch torchvision

# nightlies usually have better mps coverage
pip install --pre torch torchvision \
    --index-url https://download.pytorch.org/whl/nightly/cpu

The device dance

portable code that picks the best device
import torch

# works on Mac (mps), Linux (cuda), and CPU-only boxes
device = torch.device(
    "mps"  if torch.backends.mps.is_available() else
    "cuda" if torch.cuda.is_available()        else
    "cpu"
)

x = torch.randn(1024, 1024).to(device)
W = torch.randn(1024, 1024).to(device)
y = x @ W                         # runs on Apple GPU
print(y.device)                  # mps:0

The CPU-fallback escape hatch

when an op isn't yet implemented for MPS
# export this BEFORE python starts (or set in Python before importing torch)
export PYTORCH_ENABLE_MPS_FALLBACK=1

# or, in code:
import os
os.environ["PYTORCH_ENABLE_MPS_FALLBACK"] = "1"
import torch
What this does: if PyTorch hits an op that's not yet ported to MPS, it silently falls back to CPU instead of crashing. Slower, but your code finishes. As of PyTorch 2.x most common ops work — fallback is rare.

Tiny end-to-end example

a 1-layer MLP, trained on noise, on the GPU
import torch
import torch.nn as nn

device = torch.device("mps" if torch.backends.mps.is_available() else "cpu")

X = torch.randn(256, 8, device=device)
y = torch.randn(256, 1, device=device)

model = nn.Sequential(
    nn.Linear(8, 32), nn.ReLU(),
    nn.Linear(32, 1),
).to(device)

opt = torch.optim.Adam(model.parameters(), lr=1e-3)
loss_fn = nn.MSELoss()

for step in range(200):
    pred = model(X)
    loss = loss_fn(pred, y)
    opt.zero_grad(); loss.backward(); opt.step()

print("final loss", loss.item())

MLX — Apple's array framework

MLX is Apple's research framework for machine learning on Apple Silicon. Think NumPy meets JAX with PyTorch ergonomics. It's lazy, supports automatic differentiation, and exploits unified memory natively. Designed for the Mac, by the team that ships the Mac.

install
# core array library
pip install mlx

# LLM helpers (mlx_lm)
pip install mlx-lm
imports you'll always type
import mlx.core as mx
import mlx.nn   as nn
import mlx.optimizers as optim

Three ideas you must internalize

L
Lazy by default. mx ops build a graph. Nothing computes until you call mx.eval(x) or read a value (e.g. print, numpy()). This lets MLX fuse + schedule.
U
Unified memory, not "device". No .to(device). Arrays live in shared memory; ops can run on CPU or GPU automatically.
A
Autograd is functional. Use mx.grad(fn) or mx.value_and_grad(fn) — they return a function that returns gradients. No .backward().

Hello-world: linear regression in MLX

fit y = 2x + 1, with autograd, in 25 lines
import mlx.core as mx
import mlx.nn   as nn
import mlx.optimizers as optim

# synthetic data: y = 2x + 1 + noise
mx.random.seed(0)
X = mx.random.normal((100, 1))
y = 2.0 * X + 1.0 + 0.1 * mx.random.normal((100, 1))

# model = a single Linear layer
model = nn.Linear(1, 1)

def loss_fn(model, X, y):
    pred = model(X)
    return mx.mean((pred - y) ** 2)

loss_and_grad = nn.value_and_grad(model, loss_fn)
opt = optim.SGD(learning_rate=0.1)

for step in range(200):
    loss, grads = loss_and_grad(model, X, y)
    opt.update(model, grads)
    mx.eval(model.parameters(), opt.state)   # force computation

print(model.weight.item(), model.bias.item())   # ≈ 2.0, 1.0
The mx.eval trick: because MLX is lazy, gradients are not actually computed until you ask. Call mx.eval(model.parameters(), opt.state) once per training step to materialize them — otherwise the graph keeps growing.

Try it

MLX vs PyTorch — when to use which

Both are excellent on Apple Silicon. They overlap a lot. Here's how to decide.

MLX PyTorch (MPS)
Designed for apple silicon Mac-only, native universal Mac · Linux · Windows · CUDA
API style NumPy + functional autograd (JAX-like) NumPy + nn.Module + .backward()
Execution Lazy — mx.eval() to materialize Eager (default); torch.compile opt-in
Memory model Unified — no device dance Device-explicit — .to("mps")
Op coverage Smaller library — growing fast Huge — almost everything works (some MPS gaps)
LLM serving mlx_lm — quantization, LoRA, generate, server Use transformers + accelerate
Ecosystem Young — mlx-community on HF, growing tooling Massive — torchvision, HuggingFace, lightning, etc.
Performance on M-series Often fastest (native + lazy fusion) Excellent, sometimes faster on heavy CNN ops
Debugging Lazy makes printing tricky — call eval Eager — print anywhere, just works
Pick it when Local LLM work, Mac-targeted research, peak Apple perf Porting CUDA code, using existing HF model, cross-platform
MLX — pick when…
  • You're running Llama-class LLMs locally
  • You want to LoRA-fine-tune on your laptop
  • You like JAX-style functional code
  • Your project is Mac-only — no CUDA target
  • You want minimal dependencies + fast startup
PyTorch — pick when…
  • You're porting existing PyTorch code from CUDA
  • You need torchvision / diffusers / transformers
  • You'll deploy to Linux GPUs later
  • You want the largest community / docs / Stack Overflow
  • You need niche layers (e.g. complex spectral ops)

Same task, both frameworks

MLX matmul
import mlx.core as mx

A = mx.random.normal((1024, 1024))
B = mx.random.normal((1024, 1024))
C = A @ B
mx.eval(C)
print(C.shape)
PyTorch matmul
import torch

dev = "mps" if torch.backends.mps.is_available() else "cpu"
A = torch.randn(1024, 1024, device=dev)
B = torch.randn(1024, 1024, device=dev)
C = A @ B
print(C.shape)
Honest answer: for most beginners on Mac the right answer is "use both." Start in PyTorch (everything is documented), and reach for MLX when you want maximum Mac performance for LLMs.

Performance tips for Mac ML

A handful of habits separates fast Mac ML from slow Mac ML. Most are about respecting unified memory and avoiding unnecessary work.

1
Stay arm64 end-to-end. One Rosetta dependency contaminates the chain. file $(which python) should report arm64. Same for brew --prefix (it should be /opt/homebrew, not /usr/local).
2
Pin a recent PyTorch. MPS support improves every release. Aim for PyTorch 2.3+. The nightlies often have op coverage the stable hasn't shipped.
3
Batch your mx.eval. Don't call it after every op — call it once per training step on the things that need to materialize: mx.eval(model.parameters(), opt.state).
4
Quantize for inference. 4-bit cuts memory ~4x with negligible quality loss for chat. Both MLX and PyTorch (via bitsandbytes — limited on Mac) support this. Easiest path: pull -4bit models from mlx-community.
5
Watch unified memory. The Mac shares RAM between CPU and GPU. A 64 GB Mac can run 70B-ish 4-bit models; an 8 GB Mac is closer to 3B 4-bit. Activity Monitor → Memory tab is your friend.
6
Don't fight the lazy graph. In MLX, accumulating Python-side state (e.g. losses.append(loss)) inside a loop without periodic eval balloons the graph. mx.eval(loss) before append.
7
Avoid CPU↔MPS round trips. Every .to("cpu") or .numpy() in PyTorch costs you. Build the whole training step on-device, materialize at the boundaries (logging, checkpointing).
8
Float types matter. MLX defaults to float32; mx.float16 and mx.bfloat16 roughly halve memory and often speed inference up. PyTorch: torch.set_default_dtype(torch.float16) or autocast.
9
Use torch.compile selectively. On MPS it can help and it can hurt — measure. Wrap only the inner forward pass and keep the outer training loop as plain Python.
10
Plug in. Apple Silicon is sandbagged on battery — perf-cores throttle, GPU clocks drop. For benchmarks, plug in and use a high-power profile.

Common gotchas

"my MPS code prints inf / nan"
Some ops (especially older PyTorch + complex spectral ops) overflow on MPS where they don't on CUDA. Try PYTORCH_ENABLE_MPS_FALLBACK=1 and a smaller learning rate.
"MLX is slower than expected"
Almost always: missing mx.eval. Check that you're materializing once per step, not every minibatch op or worse — never until the end.
"pip install torch fails"
You're on x86_64 Python under Rosetta. Switch to an arm64 Python (uv / pyenv / python.org installer) and file $(which python) to verify.
"my LLM hangs / OOMs"
4-bit quantize, drop max-tokens / context length, or pick a smaller model (3B instead of 7B). Unified memory is shared with everything else on the system.

# mlx_quadratic.py — fit y = x^2 + noise on Apple Silicon
import mlx.core as 
import mlx.nn as nn
import mlx.optimizers as optim

# --- data ---
mx.random.seed(42)
X = mx.random.uniform(low=-1, high=1, shape=(256, 1))
Y = X ** 2 + 0.05 * mx.random.normal((256, 1))

# --- model ---
class MLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.l1 = nn.Linear(, 32)
        self.l2 = nn.Linear(32, 32)
        self.l3 = nn.Linear(32, )

    def __call__(self, x):
        x = nn.relu(self.l1(x))
        x = nn.relu(self.l2(x))
        return self.(x)

model = MLP()

# --- loss + autograd ---
def loss_fn(model, X, Y):
    return mx.mean((model(X) - Y) ** 2)

loss_and_grad = nn.(model, loss_fn)
opt = optim.(learning_rate=3e-3)

# --- training loop ---
for step in range(500):
    loss, grads = loss_and_grad(model, X, Y)
    opt.(model, grads)
    mx.(model.parameters(), opt.state)
    if step % 50 == 0:
        print(f"step {step:4d}  loss {loss.item():.4f}")

# --- save weights ---
mx.("model.safetensors", dict(tree_flatten(model.parameters())))

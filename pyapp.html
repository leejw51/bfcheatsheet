PyApp — ship a Python app as a single binary

PyApp (ofek/pyapp) is a Rust runtime that bundles a Python application into a self-contained executable. You configure it with environment variables at build time, run cargo build --release, and out pops a cross-platform binary. No Python required on the user's machine — at least, not one they need to install themselves.

What it is
A small Rust program (~2 MB) that knows how to bootstrap a Python distribution and install your project into it. Compile it once per target, ship the resulting binary.
What it isn't
Not a freezer (PyInstaller). Not a transpiler (Nuitka). Not a zipapp (shiv / pex). It defers Python sourcing to a real CPython distribution — by default downloaded on first run, optionally embedded.
Why it's nice
Tiny binaries by default. Genuine cross-platform builds (Linux / macOS / Windows). A real Python install with all its sharp edges working — native deps, dynamic loading, the lot. Self-update built in.

The shape of a build

recipe
# 1. tell pyapp what your project is
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0
export PYAPP_EXEC_MODULE=mytool

# 2. compile pyapp's Rust runtime
cargo install pyapp --locked --root dist

# 3. ./dist/bin/pyapp is now a binary that, on first run,
#    downloads CPython, pip-installs mytool, and runs `python -m mytool`.
mv dist/bin/pyapp dist/bin/mytool

Mental model

Pyapp's binary contains: (a) Rust startup code, (b) your project metadata, (c) optionally an embedded Python distribution and your wheel. On first launch it lays down a private Python in a cache dir, pip-installs your project, and execs into it. Subsequent runs skip straight to the Python entry point.

How it works — Rust outside, Python inside

The user double-clicks mytool. The Rust runtime checks a cache dir, sources Python if needed, installs the project once, and runs the entry point. Toggle the diagram below between the two distribution modes.

on-demand Python distribution downloaded on first launch. Tiny binary, online-once-required.
embedded set PYAPP_DISTRIBUTION_EMBED=1 and the Python tarball is baked into the binary. Bigger file, but works offline from second one.

The first-run sequence

  1. User runs ./mytool.
  2. Rust runtime resolves the cache directory (~/.local/share/pyapp/ on Linux, ~/Library/Application Support/pyapp/ on macOS, %LOCALAPPDATA%\pyapp\ on Windows).
  3. If no Python is installed for this (name, version, distribution) triple, it either downloads from PYAPP_DISTRIBUTION_SOURCE or extracts the embedded tarball.
  4. Creates a venv (unless full isolation is on, in which case the python install is the venv).
  5. pip install your project (and pulled-in deps).
  6. Stores a marker file. Subsequent runs skip steps 3–5.
  7. Execs into the configured entry point and replaces the Rust process.

Quick start

Install Rust (rustup.rs), set a few env vars, and run cargo install. That's it — you'll have a working binary in under a minute.

A) install via cargo
# project published on crates.io
export PYAPP_PROJECT_NAME=httpie
export PYAPP_PROJECT_VERSION=3.2.2

cargo install pyapp --locked

# the binary is in $CARGO_HOME/bin/pyapp
mv ~/.cargo/bin/pyapp ~/.cargo/bin/httpie
httpie --version
# → 3.2.2
B) build from source
export PYAPP_PROJECT_NAME=mytool
export PYAPP_PROJECT_VERSION=1.0.0

git clone https://github.com/ofek/pyapp pyapp-src
cd pyapp-src
cargo build --release

# binary at target/release/pyapp
mv target/release/pyapp ~/bin/mytool

Cross-compile

all major platforms with one source tree
# add rust targets once
rustup target add x86_64-unknown-linux-gnu
rustup target add aarch64-apple-darwin
rustup target add x86_64-pc-windows-msvc

# then for each target
cargo build --release --target x86_64-unknown-linux-gnu
cargo build --release --target aarch64-apple-darwin
cargo build --release --target x86_64-pc-windows-msvc

# pyapp picks the matching CPython distribution at runtime
CI tip: the official repo ships a GitHub Actions workflow that builds for seven targets in one go — Linux x64/aarch64, macOS x64/aarch64, Windows x64/x86 and musl. Steal it.

Configuration variables

Pyapp is configured exclusively via environment variables read at build time. The values get baked into the resulting binary. Below, the most useful ones.

Required

VariableEffectExample
PYAPP_PROJECT_NAME The PyPI / wheel name of your project. Required. mytool
PYAPP_PROJECT_VERSION Version pin. Required unless PYAPP_PROJECT_PATH is set. 1.2.3

Source of the project

VariableEffect
PYAPP_PROJECT_PATH Path to a local wheel or sdist to embed instead of pip-installing from PyPI. Use this for private apps.
PYAPP_PROJECT_DEPENDENCY_FILE Path to a constraints file (pip -c) baked into the binary. Pins transitive deps for reproducibility.
PYAPP_PIP_EXTRA_ARGS Extra args appended to the install command. e.g. --index-url https://my.repo/simple

Entry point (pick one)

VariableEffectExample
PYAPP_EXEC_MODULE Run python -m <module>. mytool
PYAPP_EXEC_SCRIPT Run a script file shipped inside the wheel. /path/to/run.py
PYAPP_EXEC_SPEC Console-script style: module:func. mytool.cli:main
PYAPP_EXEC_NOTEBOOK Path to a .ipynb file launched via Jupyter. analysis.ipynb

Python distribution

VariableEffectDefault
PYAPP_PYTHON_VERSION CPython version to use. Maps to a indygreg/python-build-standalone release. latest stable
PYAPP_DISTRIBUTION_SOURCE Override the URL Pyapp downloads Python from. Use for air-gapped mirrors. github release
PYAPP_DISTRIBUTION_EMBED If 1/true: bake Python into the binary. No first-run download. 0
PYAPP_DISTRIBUTION_VARIANT Distribution flavor: install_only (default) or install_only_stripped for smaller binaries. install_only

Behaviour tweaks

VariableEffect
PYAPP_FULL_ISOLATION Skip the venv layer; install directly into the unpacked Python. Faster startup, less deduplication.
PYAPP_UPGRADE_VIRTUALENV Force pip / setuptools upgrade inside the venv before installing the project.
PYAPP_EXPOSE_ALL_COMMANDS Allow users to invoke any console script in the project, not just the configured one.
PYAPP_ALLOW_UPDATES Set 0 to disable the self update sub-command.
PYAPP_PASS_LOCATION Pass the binary's path as argv[0] to Python — useful for tools that re-exec themselves.
Boolean values are lenient: 1, true, True, TRUE, yes all enable a flag. Anything else, including unset, is "off".

Entry points — module vs script vs spec

Pyapp can launch your app in four ways. Pick the one that matches how you'd run it during development.

PYAPP_EXEC_MODULE
Like python -m mytool. Your project must have a __main__.py. Most idiomatic. 
export PYAPP_EXEC_MODULE=mytool
# package layout: mytool/__main__.py
PYAPP_EXEC_SPEC
Console-script syntax: pkg.module:function. Same as the [project.scripts] entries you'd write in pyproject.toml. 
export PYAPP_EXEC_SPEC=mytool.cli:main
PYAPP_EXEC_SCRIPT
Path to a .py file (relative to the wheel root). Runs python script.py. Old-school. 
export PYAPP_EXEC_SCRIPT=mytool/runner.py
PYAPP_EXEC_NOTEBOOK
Launch a Jupyter notebook. Pyapp will install jupyter alongside your project. Useful for shipping interactive tooling to non-coders. 
export PYAPP_EXEC_NOTEBOOK=demo.ipynb
Argument forwarding: any args you pass to the binary are forwarded to the Python interpreter unchanged. ./mytool --debug -- --help works as you'd expect for both EXEC_MODULE and EXEC_SPEC.

Distribution sources — embedded vs on-demand

Two strategies for sourcing CPython. Pick based on whether your users are online, how sensitive your binary size is, and whether you can tolerate a one-time delay.

on-demand (default)
  • Binary stays small (~2 MB).
  • First launch downloads CPython (~30 MB) from indygreg/python-build-standalone.
  • Subsequent launches use the cached install.
  • Multiple pyapp binaries can share the same cached interpreter.
  • Network required on first run.
# nothing to set — this is the default
export PYAPP_PYTHON_VERSION=3.12
cargo build --release
embedded
  • Python tarball baked into the binary (~30 MB total).
  • No network access needed at any point.
  • Reproducible: identical Python on every machine.
  • Project deps still pip-install on first run unless you also embed them.
  • Use for CI tooling, air-gapped envs, or when first-run latency is unacceptable.
export PYAPP_PYTHON_VERSION=3.12
export PYAPP_DISTRIBUTION_EMBED=1
cargo build --release

Pinning a specific tarball

override the URL pyapp downloads Python from
# mirror or vendored release
export PYAPP_DISTRIBUTION_SOURCE=https://internal.example.com/python-3.12-x64.tar.gz
export PYAPP_DISTRIBUTION_PATH_PREFIX=install/
export PYAPP_DISTRIBUTION_PYTHON_PATH=install/bin/python3

cargo build --release
Stripped variant: set PYAPP_DISTRIBUTION_VARIANT=install_only_stripped to drop debug symbols and shave ~10 MB off an embedded build.

Self-management — mytool self ...

Every Pyapp binary ships with a self command that handles upgrades and uninstalls. Power-user UX with no CI required.

commands
mytool self updatefetches the latest version from PyPI and rebuilds the venv in-place
mytool self restoreblow away the venv and reinstall (recovers from a botched dep)
mytool self removedelete the cache directory; binary is unaffected
mytool self metadataprint Pyapp config baked into the binary
mytool self pythondrop into the embedded Python REPL — handy for debugging
controlling self-update
# at build time, disable updates entirely
export PYAPP_ALLOW_UPDATES=0

# at runtime, point users at a private index
export PIP_INDEX_URL=https://my.repo/simple
mytool self update

# pin to a specific version
mytool self update 1.4.7
Why this is great: users can brew install or winget install your tool once, then mytool self update indefinitely without re-running the system package manager. The Rust runtime never changes; only the Python project inside the cache does.

vs Alternatives

Pyapp sits in a very crowded space. Here's how it compares to the obvious competitors.

Tool Approach Single binary Cross-platform from one host Real CPython Self-update Trade-off
PyApp Rust loader → CPython distro yes yes (cargo) yes yes first-run download or larger embed
PyInstaller freeze: bytecode + libs in archive yes no — must build per-OS embedded copy no antivirus false positives, big binaries
Nuitka Python → C → native binary yes no recompiled no slow build, picky about deps
shiv / pex zipapp wrapper .pyz needs Python yes user must have Python no not actually self-contained
Briefcase (BeeWare) platform-native bundles .app, .msi, .deb no embedded no aimed at GUI apps, more setup
conda-pack tarball of a conda env no no yes no requires conda activation

When to pick what

pick PyApp if…
you want a single binary, are happy with first-run download, need cross-platform builds from one CI matrix, and like the idea of self update for free.
pick PyInstaller if…
you must run with no network and no cache writes, can build on each target OS, and don't mind tracking down occasional missing-import bugs.
pick shiv / pex if…
the destination machine already has Python, you want truly cross-platform output, and you only need a single-file wheel-bundle (.pyz).

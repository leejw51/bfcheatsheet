1. What is a message broker?

A message broker sits between producers and consumers. The producer drops a message into the broker and goes back to its job. The consumer picks it up later, at its own pace. The two never have to be online at the same time, never have to know each other’s address, and never have to agree on processing speed.

Why queues matter

  • Decoupling: change consumers without changing producers.
  • Buffering: smooth out spikes; the queue absorbs bursts.
  • Reliability: if a consumer crashes, the message is still there.
  • Fan-out: one event triggers many independent reactions.
Mental model: producers write postcards, the broker is the post office, queues are mailboxes, consumers read at their leisure.

2. AMQP core concepts

RabbitMQ implements AMQP 0-9-1. The vocabulary is small but precise.

  • Producer publishes messages.
  • Consumer subscribes and processes messages.
  • Queue a named buffer that stores messages.
  • Exchange the router; producers publish to exchanges, never directly to queues.
  • Binding a rule that wires an exchange to a queue (often with a key).
  • Routing Key a string the producer attaches to a message; the exchange uses it to decide where to deliver.
  • Virtual host (vhost) a namespace partitioning users, exchanges, and queues; logical isolation.
  • Connection a TCP connection. Channel a multiplexed virtual connection inside it.

Flow at a glance

producer
  └─ publish(routing_key) ─▶ exchange
                              ├─ binding A ─▶ queue 1 ─▶ consumer
                              └─ binding B ─▶ queue 2 ─▶ consumer

3. Exchange types deep dive

Direct

Routes messages whose routing key exactly matches a binding key. Use for: targeted delivery, simple work queues, command routing.

binding: queue.email  ⇐ key="email"
publish key="email"   → queue.email
publish key="sms"     → dropped (or unrouted)

Fanout

Ignores the routing key. Broadcasts to every bound queue. Use for: pub/sub events, cache invalidation, "tell everyone" notifications.

publish ANY key → all bound queues receive it

Topic

Routes by pattern. Routing keys are dot-separated words; binding patterns may use:

  • * matches exactly one word.
  • # matches zero or more words.
binding "orders.*.us"   matches  orders.new.us, orders.cancel.us
                        misses   orders.us, orders.new.eu, orders.a.b.us
binding "orders.#"      matches  orders, orders.new, orders.new.us
binding "*.error.*"     matches  app.error.db, web.error.api

Headers

Routes based on a message’s headers, not the routing key. Match mode x-match: all requires every header to match; x-match: any requires at least one.

binding headers: {format: pdf, type: report, x-match: all}
header {format: pdf, type: report}     → match
header {format: pdf}                    → no match

When to use which

TypePick when…
directYou want point-to-point or strict equality on a single key.
fanoutEvery subscriber needs the same event. Cheapest to reason about.
topicHierarchical events: region.service.severity. Subscribers want subsets.
headersRouting depends on multiple attributes that don’t fit a string key.

Default exchange

The unnamed "" exchange is a special direct exchange where every queue is implicitly bound by its name. That’s why basic_publish(exchange='', routing_key='hello') goes straight to the queue named hello.

Dead-letter exchange (DLX)

A queue can be configured with x-dead-letter-exchange. Messages that are rejected, expire, or overflow get republished to the DLX, where another binding parks them in a dead-letter queue for inspection or retry.

4. Acknowledgments

Auto-ack vs manual ack

  • Auto-ack (auto_ack=True): the broker considers the message delivered the moment it leaves. Fast and unsafe. If the consumer crashes mid-processing, the message is gone.
  • Manual ack (auto_ack=False): consumer must call basic_ack(delivery_tag) after success. If it dies first, the broker redelivers on reconnect.

nack & reject

  • basic_nack(requeue=true) — put it back in the queue.
  • basic_nack(requeue=false) — drop or send to DLX if configured.
  • basic_reject is the single-message form of nack.

Redelivery flag

When a message is redelivered, the broker sets redelivered=true. Consumers should treat that as a hint to be careful: idempotency, deduplication keys, retry counters in headers.

5. Durability — what survives a restart

Three independent things must all be set for a message to survive a broker crash:

  1. Durable queue: queue_declare(durable=True). Otherwise the queue itself vanishes.
  2. Persistent message: delivery_mode=2 (or PERSISTENT_TEXT_PLAIN). Without this, the message is RAM-only.
  3. Durable exchange (if you rely on it): exchange_declare(durable=True).
Gotcha: persistence is best-effort, not synchronous fsync per message. Use publisher confirms (confirm_delivery) if you need to know a message is on disk before acknowledging upstream.

Publisher confirms vs transactions

AMQP transactions exist but are slow. Publisher confirms are async and orders of magnitude faster: the broker tells you "I’ve got it" with an ack frame.

6. Prefetch & QoS — fair dispatch

By default, RabbitMQ round-robins messages to consumers. If one consumer is slow and another is fast, the slow one piles up while the fast one is idle. Prefetch fixes that.

channel.basic_qos(prefetch_count=1)

This tells the broker: "don’t hand me a new message until I’ve acked the previous one." Now the fast consumer pulls more work, the slow one pulls less. Set higher (e.g., 10–50) for short, uniform work.

Tuning rule of thumb

  • Long, uneven jobs → prefetch=1.
  • Short, uniform jobs → higher prefetch (less round-trip overhead).
  • Measure with queue depth and consumer utilization.

7. Clustering & quorum queues

A RabbitMQ cluster shares users, vhosts, exchanges, and bindings across nodes. Queues are by default a single-master entity; the master lives on one node and can be mirrored or replaced by quorum.

Classic mirrored queues

Legacy. Replicate state to followers. Replaced in modern deployments by quorum queues.

Quorum queues

  • Built on the Raft consensus protocol.
  • Strong consistency: a write isn’t acknowledged until a majority of replicas have it.
  • Survive node failures cleanly. Recommended default for important data.
  • Trade-off: higher latency than classic queues, more disk I/O.

Streams (bonus)

Log-style append queues for high-throughput analytics-grade workloads. Different protocol, different access pattern (offsets, not destructive consumption).

8. Common patterns

Work queues (task distribution)

One queue, many workers, manual ack, prefetch=1. Use for image resizing, email sending, video encoding.

producer ─▶ "" ─▶ queue:tasks ─▶ worker1
                              └▶ worker2
                              └▶ worker3

Publish/Subscribe

Fanout exchange, every subscriber gets every message via its own queue.

producer ─▶ fanout:events ─┬─▶ queue:audit       ─▶ logger
                            ├─▶ queue:cache_invl  ─▶ cache
                            └─▶ queue:notify      ─▶ pusher

More patterns

Topic routing

Hierarchical events. Different queues subscribe to different shapes.

topic:logs
  binding "*.error.*"  ─▶ queue:alerts
  binding "db.#"       ─▶ queue:dba
  binding "#"          ─▶ queue:audit

RPC (request/reply)

Producer sets reply_to (a temporary, exclusive queue) and correlation_id. Consumer publishes the response to reply_to. Producer matches replies by correlation_id.

client → publish(reply_to=Q1, correlation_id=abc)
server → consume → publish(exchange='', routing_key=Q1, correlation_id=abc)
client → matches abc, resolves promise
Tip: set a TTL on the reply queue so leaked clients don’t accumulate orphan queues.
